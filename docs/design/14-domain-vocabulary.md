# คำศัพท์ธุรกิจ + กติกาที่ดีไซน์ต้องเคารพ

> ทุกคำในเอกสารนี้มาจากฐานข้อมูลจริงและกติกาธุรกิจใน `Ready.md`
> **ป้ายไทยในตารางเหล่านี้คือของจริงในระบบ — ห้ามคิดคำใหม่เอง** ถ้าคิดว่าคำไหนไม่ดี ให้เสนอเป็นข้อเสนอแยก อย่าเปลี่ยนเงียบ ๆ

## 1. สถานะออเดอร์ (order_status) — 11 ค่า

**[ห้ามแตะ]** ป้ายไทยชุดนี้เป็นชุดเดียวที่ครบทั้งระบบ ลูกค้าใน LINE กับพนักงานในเว็บเห็นคำเดียวกัน

| ค่า | ป้ายไทย (เว็บ) | ป้ายไทย (LINE) | ความหมายในงานจริง |
|---|---|---|---|
| `pending` | รอชำระ | รอชำระเงิน | AI สร้างออเดอร์ + ส่ง QR ให้แล้ว — **ยังไม่จองสต็อก** · ยกเลิกได้ |
| `paid` | ชำระแล้ว | ชำระแล้ว | สลิปผ่าน → **จองสต็อก** · นาฬิกา cutoff เริ่มนับที่จุดนี้ |
| `confirmed` | เข้ารอบ | เข้ารอบจัดส่ง | ผูกกับรอบชั่วโมงแล้ว (มีเลขจุดจอด) · Grab/Lalamove/Kerry เข้าสถานะนี้โดยไม่มีรอบ |
| `packing` | กำลังแพ็ค | กำลังแพ็ค | Packer รับงานแล้ว — นาฬิกา ~30 นาทีเดินอยู่ · **ยกเลิกไม่ได้แล้ว** |
| `packed` | แพ็คแล้ว | แพ็คเสร็จแล้ว | มีรูปแพ็คแล้ว → **ตัดสต็อกจริง** + บันทึกค่าคอมของ Packer |
| `out_for_delivery` | ออกส่ง | รอไรเดอร์ออกส่ง | ไรเดอร์รับรอบแล้ว ของอยู่บนรถ — **ยังไม่ถึงคิวของลูกค้าคนนี้** |
| `delivering` | กำลังส่ง | ไรเดอร์กำลังไปส่ง | ไรเดอร์กด "เริ่มส่งจุดนี้" — **ถึงคิวคุณแล้ว** |
| `delivered` | ส่งสำเร็จ | ส่งสำเร็จ | ปิดจุดด้วยรูปหลักฐาน (บังคับ) · จบ |
| `returned` | ตีกลับ | ตีกลับ | ลูกค้าไม่มารับ + โทรไม่ติด + รอครบ 5 นาที → ไรเดอร์กดตีกลับ + **บังคับกรอกเหตุผล** · คืนสต็อก |
| `awaiting_redelivery_fee` | รอค่าส่งใหม่ | รอชำระค่าส่งใหม่ | **จบบนออเดอร์เดิม** — จ่ายแล้วระบบสร้างออเดอร์ลูกใหม่ ไม่ย้อนสถานะเดิม |
| `cancelled` | ยกเลิก | ยกเลิก | ยกเลิกได้ก่อนเริ่มแพ็คเท่านั้น · คืนสต็อกที่จองไว้ |

> ⚠️ **กับดักดีไซน์**: `out_for_delivery` กับ `delivering` ฟังดูคล้ายกัน แต่สำหรับลูกค้า**คนละเรื่องกันเลย** — อันแรกคือ "รถออกแล้วแต่ยังไม่ถึงคุณ" อันหลังคือ "ถึงคิวคุณแล้ว เตรียมรับของ" **ต้องแยกด้วยสีและน้ำหนักให้ชัด ห้ามใช้ชิปสีเดียวกัน**

## 2. สถานะการชำระเงิน (payment_status) — 4 ค่า

⚠️ **ค่าเหล่านี้ยังไม่มีป้ายไทยกลางในระบบ** — คำข้างล่างเป็น**ข้อเสนอ** ต้องให้เจ้าของงานเคาะก่อน

| ค่า | ป้ายไทย (เสนอ) | ความหมาย |
|---|---|---|
| `unpaid` | ยังไม่ชำระ | ยังไม่มีสลิปเข้ามา |
| `pending_verify` | รอตรวจสลิป | ระบบตรวจอัตโนมัติไม่ผ่าน → เข้าคิวให้แอดมินตรวจมือ + แจ้งเตือนแอดมิน |
| `paid` | ชำระแล้ว | ผ่านครบทุกเงื่อนไข → เป็น paid อัตโนมัติ ไม่ต้องรอคน |
| `rejected` | ปฏิเสธสลิป | แอดมินตัดสินว่าสลิปใช้ไม่ได้ — ลูกค้าต้องส่งใหม่ |

**เหตุผลที่สลิปเข้าคิวตรวจมือ** (ข้อความจริงในหน้าตรวจสลิป): "บัญชีผู้รับไม่ตรงกับร้าน" / "ตรวจยอดเงินไม่ผ่าน" / "ยังไม่ได้เชื่อม SlipOK" / "โควต้าตรวจสลิปหมด" / "ระบบตรวจสลิปขัดข้อง"

## 3. สถานะรอบส่ง (round_status) — 5 ค่า

| ค่า | ป้ายไทย | ความหมาย |
|---|---|---|
| `open` | เปิดรับออเดอร์ | รอบชั่วโมงถัดไปยังรับออเดอร์เข้าได้ — ยังไม่ถึงนาที :30 |
| `locked` | ปิดรอบ รอไรเดอร์รับ | ระบบปิดรอบที่นาที :30 → Packer เห็นคิว (มีเวลา ~30 นาที) · Rider เห็นปุ่ม "รับรอบนี้" |
| `confirmed` | ไรเดอร์รับแล้ว | ไรเดอร์กดรับ → ลูกค้าทุกคนในรอบได้แจ้งพร้อมกัน |

| `done` | จบรอบ | ทุกจุดปิดแล้ว → **ค่าวิ่ง 25 บาทเข้าระบบเงินเดือน (ต่อรอบ ไม่ใช่ต่อออเดอร์)** |

## 7. หัวใจของระบบ: รอบส่งรายชั่วโมง

**กติกา**: รอบออกทุกต้นชั่วโมง **ทุกวัน 24 ชั่วโมง ไม่มีวันหยุด** · ทุกการคำนวณเป็นเวลาไทย · **เส้นตายคือนาทีที่ :30**

**สูตร**: จ่ายก่อนนาที :30 → ได้รอบต้นชั่วโมงถัดไป (12:29 → รอบ **13:00**) · จ่ายตั้งแต่นาที :30 → ได้รอบ **อีก 2 ชั่วโมงถัดไป** (12:30 → รอบ **14:00**)

### ไทม์ไลน์จริงของรอบ 14:00

| เวลา | เกิดอะไร | ใครรู้สึกอะไร |
|---|---|---|
| 12:30–13:29 | ออเดอร์ที่จ่ายในช่วงนี้ตกเข้ารอบ 14:00 | ลูกค้ารู้ทันทีที่จ่ายว่า "ได้รอบ 14:00" |
| **13:30** | ระบบปิดรอบอัตโนมัติ | **Packer: งานเด้งเข้าพร้อมกันทั้งรอบ มีเวลาแค่ ~30 นาที** · **Rider: ปุ่ม "รับรอบนี้" โผล่** |
| 13:30–14:00 | Packer แพ็คทีละออเดอร์ | ลูกค้าทยอยได้รูปแพ็ค |
| ~13:4x | Rider กดรับรอบ | ลูกค้า**ทุกคนในรอบพร้อมกัน**: "รอบ 14:00 กำลังออก เตรียมรับของ" |
| 14:00 | Rider ออกวิ่ง | — |
| ระหว่างวิ่ง | Rider กด "เริ่มส่งจุดที่ N" **เอง ทีละจุด** | **เฉพาะลูกค้าจุดนั้น**: "ไรเดอร์กำลังไปหาคุณ" |
| ปิดจุด | ถ่ายรูปหลักฐาน → ส่งสำเร็จ · หรือ ตีกลับ + เหตุผล | ลูกค้าได้รูป + คำขอบคุณ / เหตุผลตีกลับ + วิธีจ่ายค่าส่งใหม่ |
| ครบทุกจุด | รอบจบ | **Rider: +25 บาทเข้าเงินเดือน** |

> ### 🎯 ผลต่อดีไซน์ — สำคัญที่สุดในเอกสารนี้
> **ทุก ๆ นาที :30 คือนาทีที่ทุกอย่างเด้งพร้อมกัน — 24 ครั้งต่อวัน ทุกวัน**
> - Packer เจอ burst งานทุกชั่วโมง ตรงกับที่มีเวลาแค่ 30 นาที → **หน้าจอต้องอ่านออกใน 2 วินาที**
> - Rider ใช้งานกลางแดด มือเดียว บนมอเตอร์ไซค์ → **ปุ่มใหญ่ คอนทราสต์สูง ไม่มีอะไรให้คิด**
> - **ระบบทำงานตี 3 เท่ากับบ่าย 2** → ห้ามออกแบบโดยสมมติว่าคนใช้ตอนกลางวันเสมอ
> - Rider ต้องรู้ตลอดเวลาว่า **"เหลืออีกกี่จุด"**

## 8. ใครขยับสถานะไหนได้บ้าง

**[ห้ามแตะ]** ตารางนี้บังคับใช้ในฐานข้อมูล — ถ้าดีไซน์วางปุ่มที่ทำ transition นอกตารางนี้ ระบบจะปฏิเสธและขึ้น error

| จาก | ไป | ใครทำได้ | หมายเหตุ |
|---|---|---|---|
| รอชำระ | ชำระแล้ว | ระบบ, แอดมิน | **ลูกค้าประกาศเองว่าจ่ายแล้วไม่ได้** |
| รอชำระ | ยกเลิก | ลูกค้า, แอดมิน | ลูกค้ายกเลิกเองได้เฉพาะก่อนจ่าย |
| ชำระแล้ว | เข้ารอบ | ระบบ, แอดมิน | ระบบผูกรอบให้เอง |
| ชำระแล้ว | ยกเลิก | แอดมินเท่านั้น | คืนสต็อก |
| เข้ารอบ | กำลังแพ็ค | ระบบ, Packer, แอดมิน | Packer รับงาน |
| เข้ารอบ | ยกเลิก | แอดมินเท่านั้น | **จุดสุดท้ายที่ยกเลิกได้** |
| กำลังแพ็ค | แพ็คแล้ว | Packer, แอดมิน | ตัดสต็อกจริง + บันทึกค่าคอม |
| แพ็คแล้ว | ออกส่ง | ระบบ, Rider, แอดมิน | Rider รับรอบ |
| แพ็คแล้ว | ส่งสำเร็จ | แอดมิน, ระบบ | **ทางลัดของ Kerry/Grab** — ไม่มีรอบไรเดอร์ |
| ออกส่ง | กำลังส่ง | Rider, แอดมิน | Rider กดเริ่มจุดนี้ |
| ออกส่ง / กำลังส่ง | ตีกลับ | Rider, แอดมิน | บังคับกรอกเหตุผล |
| กำลังส่ง | ส่งสำเร็จ | Rider, แอดมิน | บังคับถ่ายรูป |
| ตีกลับ | รอค่าส่งใหม่ | ระบบ, แอดมิน | **Rider ทำไม่ได้** |

**กฎที่ระบบบังคับ:**
- ทำ transition นอกตาราง → ระบบปฏิเสธทันที **ไม่มีทางเลี่ยง**
- ทำซ้ำสถานะเดิม → ไม่ error แต่ไม่เกิดอะไร (กดซ้ำปลอดภัย)
- ทุกการเปลี่ยนสถานะถูกบันทึกเป็น timeline เสมอ — **timeline นี้คือสิ่งที่ AI ใช้ตอบลูกค้าเวลาถามว่า "ของถึงไหนแล้ว"**

## 9. กฎเรื่องเงิน — สิ่งที่ดีไซน์ห้ามฝ่าฝืน

**[ห้ามแตะ ทุกข้อ]**

1. **เงินเป็นจำนวนเต็มบาทเท่านั้น** — ไม่มีสตางค์ ไม่มีทศนิยม ห้ามออกแบบช่องกรอกทศนิยม
2. **ราคาทุกอย่างคำนวณที่เซิร์ฟเวอร์** — ลูกค้า/พนักงานส่งได้แค่ "สินค้าอะไร กี่ชิ้น" ราคาที่ client ส่งมาถูกเมินทั้งหมด
3. **ห้ามพิมพ์ตัวเลขค่าส่งตายตัวลงใน mockup** — เลข 40/55/100/25 เป็นแค่ค่าเริ่มต้น แอดมินแก้ได้ในหน้าตั้งค่า **ทุกที่ที่โชว์ราคาต้องเป็นช่องที่ดึงค่าจริงมาใส่**
4. **ยอดสลิปต้องตรงเป๊ะ ไม่มีการปัดเศษ ไม่มีค่าคลาดเคลื่อนที่ยอมรับได้** — ห้ามออกแบบ UI ที่สื่อว่า "ใกล้เคียงก็ผ่าน"
5. **ปุ่ม "ยืนยันโอนแล้ว" ในหน้าเงินเดือนไม่ได้สั่งจ่ายเงิน** — ระบบไม่โอนเงินเอง มนุษย์โอนเองแล้วมากดบันทึกว่าโอนไปแล้ว + แนบสลิป **ดีไซน์ต้องไม่ทำให้เข้าใจผิดว่ากดแล้วเงินออก**
6. **ไม่มีเก็บเงินปลายทาง (COD)** ในเวอร์ชันนี้

## 10. ประเด็นที่ควรถามเจ้าของงานก่อนเริ่มดีไซน์

1. **`payment_status` ไม่มีป้ายไทยเลยทั้งระบบ** — คำที่เสนอในข้อ 2 ต้องให้เคาะ
2. **`delivery_type` มีชื่อไทย 2 ชุดขัดกัน** — ต้องเลือกชุดเดียว หรือกำหนดให้ชัดว่าที่ไหนใช้ชุดไหน
3. **`drop_option` (วางไว้ / รอรับ)** มีในฐานข้อมูลและในสเปค แต่ยังไม่ถูกใช้ในเว็บเลย — จะออกแบบเผื่อไหม
4. **แท็บล่าง 11 อันของแอดมินบนมือถือ** — ต้องการให้แอดมินใช้บนมือถือจริงไหม หรือ desktop-only ได้
