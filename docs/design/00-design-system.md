# ระบบดีไซน์ที่มีอยู่จริงวันนี้

> เอกสารนี้บันทึก **ของที่อยู่ในโค้ดจริง ณ วันนี้** ไม่ใช่ข้อเสนอ — ใช้เป็นจุดตั้งต้น ไม่ใช่กรงขัง
> อะไรที่ต้องเก็บไว้จะเขียนว่า **[ห้ามแตะ]** อะไรที่รื้อได้จะเขียนว่า **[ปรับได้]**

---

## 1. สี

**สีแบรนด์มีแค่ 2 ค่า** — ทั้งระบบไม่มี token สีอื่นเลย

| ชื่อ | ค่า | ใช้ที่ไหน |
|---|---|---|
| brand | `#0F6E56` (เขียวเข้ม) | ปุ่มหลัก, ตัวเลขเงิน, แท็บที่เลือกอยู่, wordmark, สีแถบบน LINE Flex, `theme-color` ของเบราว์เซอร์ |
| brand-dark | `#0b5544` | สถานะ hover ของปุ่มหลักเท่านั้น |

สีที่เหลือทั้งหมดใช้ Tailwind palette ตรง ๆ (ยังไม่ถูกทำเป็น token) โดยมีธรรมเนียมที่ใช้สม่ำเสมอทั้งระบบ:

| ความหมาย | สีที่ใช้จริง |
|---|---|
| พื้นหลัง / ตัวอักษรกลาง | slate (พื้นแอป = slate-50, ตัวอักษร = slate-900) |
| สำเร็จ / จ่ายแล้ว / โอนแล้ว | green (ป้าย green-100 + ตัวอักษร green-700, toast green-600) |
| รอดำเนินการ / เตือน | amber (ป้าย amber-100 + amber-700, แบนเนอร์ amber-50 + amber-800) |
| ผิดพลาด / ร้องเรียน / ฉุกเฉิน | red (ป้าย red-100 + red-700, toast red-600) |
| รอบที่กำลังวิ่ง | blue (ป้าย blue-100 + blue-700) |
| ข้อมูลแบรนด์ (ประเภทจัดส่ง, บทบาท) | brand จาง (พื้น brand 10% + ตัวอักษร brand) |

> **[ปรับได้]** ระบบสียังไม่มี token กลาง — แต่ละหน้าเขียนสีป้ายเอง เกิด helper ซ้ำกันหลายไฟล์ นี่คือจุดที่ดีไซน์ควรเข้ามาจัดระเบียบ: กำหนดชุด "status color" กลางที่ครอบคลุมทุก enum (ดู `14-domain-vocabulary.md`) แล้วให้ทุกหน้าใช้ชุดเดียวกัน

**light mode เท่านั้น** — โค้ดบังคับ `color-scheme: light` ไว้ ทั้งระบบไม่มี dark mode และไม่มี `dark:` variant ใช้งานเลยแม้แต่จุดเดียว
> **[ปรับได้ แต่ต้องคุยก่อน]** ไรเดอร์ทำงาน 24 ชั่วโมง — dark mode มีเหตุผลทางธุรกิจจริง ถ้าจะเสนอ ต้องเสนอเป็นระบบทั้งชุด ไม่ใช่บางหน้า

## 2. ตัวอักษร

- **ไม่โหลดฟอนต์จากภายนอกเลย** — ใช้ system stack ล้วน: `system-ui, -apple-system, Segoe UI, Roboto, Noto Sans Thai, sans-serif`
- `<html lang="th">` — ไทยเป็นภาษาหลัก ไม่มีชั้น i18n ข้อความไทยทั้งหมด hardcode อยู่ในโค้ด
- น้ำหนักที่ใช้จริงมีแค่ 2 ระดับ: **medium** (ปุ่ม, ป้าย, label) และ **bold** (หัวเรื่อง, ตัวเลข)

**ขนาดที่ใช้จริง** (จากเล็กไปใหญ่): 10–11px (ป้ายนับ, ป้ายเมนูล่าง) → 12px (hint, label) → 14px (เนื้อความ, ปุ่มเล็ก) → 16px (ปุ่ม, ช่องกรอก) → 18px (หัวข้อหน้า, wordmark) → 20px (ตัวเลขใน StatCard, ช่อง Stepper) → 24px (ปุ่ม +/−) → 30–36px (emoji ในสถานะว่าง/ผิดพลาด)

> **[ห้ามแตะ]** ช่องกรอกต้องเป็น **16px เสมอ** — เล็กกว่านี้ Safari บน iPhone จะซูมหน้าจอเองตอนแตะช่อง ทำให้เลย์เอาต์พัง
> **[ปรับได้]** ถ้าจะเสนอ webfont ไทย (เช่น IBM Plex Thai / LINE Seed) ต้องคิดเรื่องน้ำหนักไฟล์ด้วย — ไรเดอร์ใช้ 4G กลางถนน

## 3. ขนาดพื้นที่กด (touch target)

หัวใจของแอปนี้ — คอมเมนต์ในโค้ดเขียนไว้ตรง ๆ ว่า *"Mobile-first: ≥44px tap targets, 16px inputs, safe-area aware"*

| ขนาด | ใช้กับ |
|---|---|
| **44px** | ปุ่มทุกตัว + ช่องกรอกทุกช่อง — ค่าพื้นฐาน (มาตรฐาน Apple HIG) |
| 38px | ปุ่มเล็ก — **ข้อยกเว้นเดียว**ที่ต่ำกว่า 44 |
| 40px | ปุ่มในแท็บ |
| **48px** | ปุ่ม +/− และช่องตัวเลขของ Stepper — ใหญ่กว่าฐานเพราะกดถี่ |
| **56px** | ปุ่มแท็บล่างบนมือถือ (กว้างอย่างน้อย 68px) — คอมเมนต์ระบุเหตุผล: "thumb reach, big targets" |
| 96px | โซนกดถ่ายรูปตอนยังไม่มีรูป |

> **[ห้ามแตะ]** 44px คือพื้น ไม่ใช่เพดาน — Packer/Rider ใส่ถุงมือ ถือของ ยืนกลางแดด ถ้าดีไซน์ใหม่ทำให้ปุ่มเล็กลงกว่านี้ = ถอยหลัง

**safe area**: หน้าเว็บประกาศ `viewport-fit=cover` และมี utility สำหรับหลบ notch บน / home indicator ล่าง — header ใช้ safe-top, แท็บล่างใช้ safe-bottom

## 4. รูปทรง

**บันไดความมน** (ใช้สม่ำเสมอทั้งระบบ):

| ระดับ | ใช้กับ |
|---|---|
| 8px | ปุ่ม, ช่องกรอก, แท็บที่เลือกอยู่ |
| 12px | การ์ด, toast, StatCard, กล่องแท็บ, ตัวถ่ายรูป, ปุ่ม +/− |
| 16px | Modal บนจอใหญ่ / มุมบนของ bottom sheet บนมือถือ |
| วงกลมเต็ม | ป้าย (badge), spinner, ตัวเลขนับในแท็บ |

**การ์ดมาตรฐาน** = พื้นขาว + ขอบ slate อ่อน + มุมมน 12px + padding 16px + เงาบาง ๆ
**กล่องเนื้อหาหลัก** = กว้างสุด 1024px จัดกลาง + padding 16px + เว้นล่าง 112px บนมือถือ (กันแท็บล่างบัง)

## 5. โครงหน้าจอ (Shell)

**มี nav สองแบบ สลับกันที่จอกว้าง 640px — ไม่มี sidebar**

| ส่วน | พฤติกรรม |
|---|---|
| Header | ติดบนตลอด ทุกขนาดจอ · ซ้าย = "PKM Shop" (เขียวแบรนด์ ตัวหนา) + คำว่า "หลังบ้าน" (โผล่เฉพาะจอใหญ่) · ขวา = ชื่อผู้ใช้ (ตัดด้วย … ถ้ายาว) + ปุ่ม "ออก" |
| Top nav | **เฉพาะจอ ≥ 640px** — แถบแท็บแนวนอนใต้ header, ตัวที่เลือกมีเส้นใต้สีแบรนด์ |
| Bottom tab bar | **เฉพาะมือถือ < 640px** — แท็บติดขอบล่างจอ, ตัวที่เลือกเป็นสีแบรนด์, ไอคอนเป็น emoji |

**ลำดับชั้นการซ้อน**: nav/header (20) → Modal (40) → Toast (50)

**แบนเนอร์เตือนยังไม่ผูก LINE** (พื้นเหลือง ใต้ header — แสดงเมื่อพนักงานยังไม่ผูกบัญชี LINE):
> ⚠️ ยังไม่ได้ผูก LINE — เพิ่ม PKM Shop OA เป็นเพื่อน แล้วพิมพ์รหัส **{รหัสของผู้ใช้}** เพื่อรับแจ้งเตือนงาน

> **[ปรับได้]** แอดมินเห็นเมนู **11 รายการ** — บนมือถือแท็บล่าง 11 อันต้องเลื่อนแนวนอน ซึ่งใช้งานจริงแย่มาก นี่คือปัญหาที่ดีไซน์ควรแก้ (จัดกลุ่ม? เมนู "เพิ่มเติม"? แยก admin ออกเป็น desktop-only?) ส่วนบทบาทอื่นเห็นแค่ 1–2 เมนู ไม่มีปัญหา

## 6. คลัง component ที่มีอยู่แล้ว

โค้ดมี component กลาง 12 ตัว — คอมเมนต์ในไฟล์เขียนเหตุผลไว้ว่า *"เพื่อไม่ให้หน้าไหนต้องกลับไปใช้ prompt()/alert()/confirm() ของเบราว์เซอร์"*

| ชื่อ | ทำอะไร | รายละเอียดที่ดีไซน์ต้องรู้ |
|---|---|---|
| **Modal** | กล่องซ้อน | **bottom sheet บนมือถือ / กล่องกลางจอบนเดสก์ท็อป** · สูงไม่เกิน 90% ของจอ · เลื่อนในตัวได้ · แตะพื้นหลังปิด · มีปุ่ม ✕ มุมขวา |
| **Toast** | ข้อความแจ้งผล | ลอยเหนือแท็บล่าง · **หายเองใน 3.6 วินาที** · ซ้อนกันได้ · 3 แบบ: สำเร็จ (เขียว) / ผิดพลาด (แดง) / ข้อมูล (เทาเข้ม) ตัวอักษรขาวทุกแบบ |
| **Confirm** | กล่องยืนยัน | ใช้ Modal ซ้ำ · ปุ่มค่าเริ่มต้น **"ยกเลิก" / "ยืนยัน"** · โหมดอันตรายเปลี่ยนปุ่มยืนยันเป็นแดง |
| **Spinner** | กำลังโหลด | วงกลมหมุน + ข้อความ **"กำลังโหลด…"** (ค่าเริ่มต้น) |
| **EmptyState** | ไม่มีข้อมูล | emoji ใหญ่ + หัวข้อ + คำใบ้ · emoji ค่าเริ่มต้น 📭 |
| **ErrorState** | โหลดไม่สำเร็จ | ⚠️ + **"โหลดข้อมูลไม่สำเร็จ"** + **"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"** + ปุ่ม "ลองใหม่" · **จงใจแยกจาก EmptyState** — คอมเมนต์ระบุว่า fetch พังจริง ห้ามให้ผู้ใช้เข้าใจผิดว่า "ไม่มีข้อมูล" |
| **PageHeader** | หัวข้อหน้า | หัวข้อซ้าย + ช่องใส่ปุ่ม/ข้อความขวา |
| **Field** | label ครอบช่องกรอก | label ด้านบน + hint ด้านล่าง |
| **Tabs** | แท็บแบบ segmented | พื้นเทา ตัวที่เลือกเป็นการ์ดขาวลอยขึ้นมา · **มีตัวเลขนับในแท็บได้** (วงกลมเขียว) · เลื่อนแนวนอนได้ |
| **Stepper** | เลือกจำนวน | ปุ่ม − / ช่องตัวเลข / ปุ่ม + ขนาด 48px + **แถวปุ่มลัด** (เช่น +10 +50 — เป็นการ**บวกเพิ่ม** ไม่ใช่เซ็ตค่า) · คอมเมนต์ในโค้ด: *"no bare browser number input"* — **นี่คือสิ่งที่เจ้าของงานเคยร้องเรียนโดยตรง** |
| **PhotoPicker** | ถ่ายรูป | **เปิดกล้องหลังทันที** · 2 สถานะ: กรอบประ + 📷 (ยังไม่มีรูป) / รูปพรีวิว + ปุ่ม "ถ่ายใหม่" ลอยบนรูป · โหมดบังคับจะต่อท้าย label ว่า " (บังคับ)" |
| **StatCard** | ตัวเลขสรุป | label เล็ก + ตัวเลขใหญ่ + hint · 3 โทน: ปกติ / แบรนด์ (เขียวจาง) / เตือน (เหลือง) |

**ฟังก์ชันจัดรูปแบบเงิน**: `฿` ติดหน้าตัวเลข ไม่เว้นวรรค คั่นหลักพันแบบไทย → **`฿1,250`** — **ไม่มีทศนิยม** (ระบบเก็บเงินเป็นจำนวนเต็มบาทเท่านั้น)

> **[ห้ามแตะ]** ห้ามออกแบบ UI ที่แสดงสตางค์หรือรับ input ทศนิยม — ฐานข้อมูลเป็น integer บาทถ้วนทุกคอลัมน์

## 7. สิ่งที่ยังไม่มี (ช่องว่างจริง ไม่ใช่ข้อเสนอ)

- **ไม่มี icon library** — ไอคอนทั้งระบบเป็น emoji ล้วน
- **ไม่มี Button / Card / Badge เป็น component** — เป็น CSS class ล้วน ทำให้ป้ายไม่มีระบบสีกลาง
- **ไม่มี dark mode** ทั้งระบบ
- **ไม่มีโลโก้เป็นรูป** — wordmark เป็นข้อความล้วน ไม่มีไฟล์ภาพใด ๆ ไม่มี favicon
- **ไม่มี skeleton loading** — มีแค่ spinner กลางจอ
- **ไม่มีชั้น i18n** — ไทย hardcode

> ทั้ง 6 ข้อนี้คือ **[ปรับได้]** ทั้งหมด และคือพื้นที่ที่ดีไซน์สร้างมูลค่าได้มากที่สุด
