# คิวตรวจสลิป (`/admin/slips`)

> แอดมินร้านใช้ทุกครั้งที่มีสลิปโอนเงินที่ระบบตรวจอัตโนมัติไม่ผ่าน เพื่อเปิดดูรูปสลิป เทียบยอดกับเงินเข้าบัญชีจริง แล้วตัดสินใจ "ยืนยันชำระ" หรือ "ปฏิเสธ"

---

## 1. บริบทการใช้งาน

**ใครใช้:** แอดมินร้าน (role `admin`) เท่านั้น — คนเดียวหรือไม่กี่คน role อื่น (`stock` / `packer` / `rider` / `staff`) เข้าหน้านี้ไม่ได้เลย และไม่เห็นเมนูนี้ด้วยซ้ำ

**อุปกรณ์:** **Mobile-first** — เพราะโฟลว์จริงคือแอดมินต้องสลับไปเปิดแอปธนาคาร (mobile banking) เพื่อดูว่าเงินเข้าจริงไหม แล้วสลับกลับมากดยืนยัน คนใช้มีมือถือเครื่องเดียวและสลับแอปไปมาตลอด แต่หน้านี้ก็ต้องใช้ได้ดีบน desktop ด้วย เพราะแอดมินบางคนนั่งโต๊ะ (ปัจจุบันดีไซน์เป็น mobile-first ที่ยืดขึ้น desktop ตรงๆ ไม่มี layout แยก — ดูข้อ 8)

**สถานการณ์จริง:** ไม่ใช่งานภาคสนาม — เป็นงาน "นั่งเทียบเลข" ในร้าน ต้องการสมาธิ ต้องอ่านตัวเลขบนรูปสลิปให้ออก (ยอดเงิน เวลาโอน เลขบัญชีปลายทาง) แล้วเทียบกับยอดในระบบและยอดในแอปธนาคาร

**ความถี่:** เปิดเป็นระยะตลอดวัน ทุกครั้งที่มีสลิปตกคิว หรือเปิดค้างไว้เป็นหน้าเฝ้า (หน้าดึงข้อมูลใหม่เองทุก 15 วินาที)

**ความเร่งรีบ:** เร่งปานกลาง — ลูกค้ารอของ แต่ **ความถูกต้องสำคัญกว่าความเร็ว** เพราะกด "ยืนยันชำระ" = ปล่อยของออกไปจริง ย้อนไม่ได้

---

## 2. โครงหน้า

หน้านี้อยู่ในเปลือกหลังบ้าน (Shell) เสมอ และ **ไม่มีแท็บภายในหน้า** ไม่มีตัวกรอง ไม่มีช่องค้นหา ไม่มีการเรียงลำดับให้เลือก

เรียงจากบนลงล่าง:

1. **เปลือก Shell (ไม่ใช่ส่วนของหน้านี้ แต่ล้อมอยู่เสมอ)**
   - แถบหัวติดบนสุด (sticky): โลโก้ `"PKM Shop"` + คำว่า `"หลังบ้าน"` (ซ่อนบนมือถือ), ชื่อพนักงานที่ล็อกอิน (ตัดด้วย ellipsis ถ้ายาว, ถ้าไม่มีชื่อแสดง `"พนักงาน"`), ปุ่ม `"ออก"`
   - แถบเตือนสีเหลืองเหนือเนื้อหา ถ้าบัญชียังไม่ผูก LINE
   - เนื้อหาหลักกว้างสุดระดับกลาง (ประมาณ 1024px) มีระยะขอบรอบ และเว้นพื้นที่ล่างเยอะบนมือถือเพื่อหลบแท็บล่าง
   - **แท็บนำทาง:** บน desktop/tablet เป็นแท็บแนวนอนใต้แถบหัว; บนมือถือเป็นแท็บล่างแบบตรึง (แต่ละปุ่มสูงอย่างน้อย 56px) — แท็บของหน้านี้คือ ไอคอน 💳 + คำว่า `"ตรวจสลิป"` และจะเป็นสีแบรนด์เมื่ออยู่หน้านี้

2. **หัวข้อหน้า** — ข้อความ `"คิวตรวจสลิป"` ตัวหนาขนาดกลาง **ไม่มีปุ่มใดๆ ด้านขวา** (ไม่มีปุ่มรีเฟรช ไม่มีตัวกรอง ไม่มีช่องค้นหา)

3. **คำอธิบายใต้หัวข้อ** — บรรทัดเดียว ตัวเล็กสีเทาจาง: `"สลิปที่ระบบตรวจอัตโนมัติไม่ผ่าน หรือยังไม่ได้เชื่อม SlipOK จะมารอตรงนี้"` เป็นข้อความคงที่ ไม่เปลี่ยนตามข้อมูล

4. **รายการการ์ดสลิป** — เนื้อหาหลัก ไม่ใช่ตาราง เป็นการ์ดพื้นขาว ขอบมน เรียงต่อกันในแนวตั้งโดยมีระยะห่างระหว่างใบ ทุกใบมีน้ำหนักเท่ากัน **ไม่มีการจัดกลุ่ม ไม่มีหัวข้อย่อย ไม่มี pagination ไม่มี infinite scroll** — โหลดสลิปที่รอตรวจมาแสดงหมดทุกใบในครั้งเดียว **เรียงจากสลิปที่รอนานที่สุดอยู่บนสุด → ใบใหม่สุดอยู่ล่างสุด**

   แต่ละการ์ดมี 4 บล็อกเรียงบนลงล่าง (มีระยะห่างระหว่างบล็อก):

   **บล็อก 1 — แถวหัวการ์ด** (ซ้าย-ขวา ชิดขอบบน)
   - *ฝั่งซ้าย* 3 บรรทัด:
     - เลขออเดอร์ — **ฟอนต์ monospace** ตัวหนาขนาดเล็ก (เช่น `PKM-2607-0143`) — ถ้าข้อมูลออเดอร์หายไป จะแสดง UUID ดิบยาว 36 ตัวอักษรแทน (ต้องเผื่อพื้นที่)
     - ชื่อผู้รับ + ประเภทจัดส่ง คั่นด้วย ` · ` ตัวเล็กสีเทา ตัดด้วย ellipsis ถ้ายาว เช่น `"สมชาย ใจดี · ส่งปกติ (ไรเดอร์)"` — ถ้าไม่มีชื่อผู้รับ (ว่างหรือไม่มีค่า) แสดงคำว่า `"ลูกค้า"` แทน; ถ้าไม่มีข้อมูลออเดอร์เลย ช่องประเภทจัดส่งแสดง `"—"`
     - เวลาส่งสลิป ตัวเล็กสีเทาจางกว่า: รูปแบบ `ส่งสลิปเมื่อ {วันที่+เวลาแบบสั้น}` เช่น `"ส่งสลิปเมื่อ 16/7/2569 14:30"`
   - *ฝั่งขวา* (ชิดขวา ไม่ยุบ) 2 บรรทัด:
     - **ยอดเงินของสลิปใบนี้** ตัวหนาขนาดใหญ่สุดในการ์ด เช่น `"฿1,250"`
     - ประเภทเงิน + ยอดออเดอร์รวม ตัวจิ๋วสีเทาจาง เช่น `"ค่าส่ง · ออเดอร์รวม ฿1,200"` — ส่วน `· ออเดอร์รวม …` จะหายไปถ้าไม่มีข้อมูลออเดอร์

   **บล็อก 2 — แถวป้าย (badge)** เรียงแนวนอน ตัดขึ้นบรรทัดใหม่ได้
   - **ป้ายแดง (มีเสมอทุกใบ ไม่มีวันหาย):** พื้นแดงอ่อน ตัวอักษรแดงเข้ม เนื้อหา `⚠ ` + เหตุผลที่ตกคิวมือ (6 ค่า ดูข้อ 6)
   - **ป้ายเทา (มีเฉพาะเมื่อมีข้อมูลออเดอร์):** พื้นเทาอ่อน ตัวอักษรเทาเข้ม = สถานะออเดอร์ภาษาไทย

   **บล็อก 3 — รูปสลิป**
   - ถ้ามีรูป: รูปสลิปเต็มความกว้างการ์ด **สูงสุดประมาณ 256px** จัดรูปแบบ "แสดงทั้งรูปไม่ crop" บนพื้นเทาอ่อน (สลิปแนวตั้งจะมีแถบเทาข้างซ้ายขวา) ขอบมน — **ทั้งรูปกดได้** ใต้รูปมีคำใบ้จัดกลาง ตัวจิ๋วสีเทา: `"แตะรูปเพื่อขยาย"`
   - ถ้าไม่มีรูป: กล่องพื้นเทาอ่อนมาก ขอบมน ข้อความจัดกลางตัวจิ๋วสีเทา `"ไม่มีรูปสลิปแนบ"` — **ยังกดยืนยัน/ปฏิเสธได้ตามปกติ ไม่มีการบล็อก**

   **บล็อก 4 — แถวปุ่มตัดสินใจ** สองปุ่มเรียงข้างกัน **กว้างเท่ากันครึ่งการ์ดพอดี**
   - ซ้าย: `"ปฏิเสธ"` — ปุ่มโครงร่าง (ขอบแดงอ่อน พื้นขาว ตัวอักษรแดง)
   - ขวา: `"ยืนยันชำระ"` — ปุ่มหลักสีแบรนด์ทึบ

5. **โมดัลต่างๆ** (ไม่ใช่บล็อกในหน้า แต่ซ้อนทับ) — ดูข้อ 3

## 3. ทุก flow ทีละสเต็ป

### Flow 1 — ยืนยันชำระ (เส้นทางหลัก)

1. แอดมินเข้า `/admin/slips` จากแท็บ 💳 `"ตรวจสลิป"` → ระบบตรวจสิทธิ์ role `admin` ผ่าน → เรนเดอร์หน้า
2. โหลดครั้งแรก → แสดงตัวหมุน (spinner) + ข้อความ `"กำลังโหลด…"` จัดกลาง สีเทา (หัวข้อหน้าและคำอธิบายใต้หัวข้อยังแสดงอยู่เหนือ spinner)
3. ระบบดึงสลิปทุกใบที่สถานะ `pending_verify` เรียงจากเก่า→ใหม่ → แสดงเป็นการ์ดทีละใบ (สลิปที่รอนานที่สุดอยู่บนสุด)
4. แอดมินอ่านหัวการ์ด: เลขออเดอร์ / ผู้รับ / ประเภทส่ง / เวลาส่งสลิป / **ยอดสลิป** / ประเภทเงิน / ยอดออเดอร์รวม และป้ายแดงที่บอกว่า **ทำไมสลิปใบนี้ถึงมาที่นี่**
5. (ทางเลือก) แอดมินแตะรูปสลิปเพื่อดูชัดขึ้น → ดู Flow 3
6. แอดมินสลับไปแอปธนาคารเทียบยอดเงินเข้าจริง (นอกระบบ) แล้วกลับมากดปุ่ม `"ยืนยันชำระ"`
7. **เปิดกล่องยืนยัน (confirm dialog)** — บนมือถือเป็น bottom sheet ขอบบนมน, บนจอใหญ่เป็นกล่องกลางจอ พื้นหลังดำโปร่ง
   - หัวข้อ: `"ยืนยันว่าได้รับเงินแล้ว?"`
   - ข้อความ (สร้างจากข้อมูลจริง): รูปแบบ `{เลขออเดอร์} · {ประเภทเงิน} {ยอดสลิป} — ตรวจสลิปกับยอดเข้าบัญชีแล้วใช่ไหม`
     ตัวอย่างจริง: `"PKM-2607-0143 · ค่าสินค้า ฿1,250 — ตรวจสลิปกับยอดเข้าบัญชีแล้วใช่ไหม"`
     (หมายเหตุ: ถ้าไม่มีข้อมูลออเดอร์ เลขออเดอร์จะเป็นค่าว่าง ข้อความจะขึ้นต้นด้วย ` · ` ที่ค้างอยู่)
   - ปุ่ม: `"ยกเลิก"` (ปุ่มโปร่ง) และ `"ยืนยันชำระ"` — **ปุ่มยืนยันเป็นสีแบรนด์ ไม่ใช่สีแดง** (ไม่ได้ตั้งเป็นปุ่มอันตราย)
8. ถ้ากด `"ยกเลิก"` หรือแตะพื้นหลังนอกกล่อง → ปิดไปเฉยๆ ไม่มีอะไรเกิดขึ้น **ไม่มี toast**
9. ถ้ากด `"ยืนยันชำระ"` → **การ์ดใบนั้นใบเดียว** เข้าสถานะกำลังส่ง: ปุ่มทั้งสองของการ์ดนั้นจางลงและกดไม่ได้ ปุ่มขวาเปลี่ยนข้อความเป็น `"กำลังยืนยัน…"` (**การ์ดใบอื่นยังกดได้ตามปกติ** — ดูกับดักในข้อ 8)
10. ระบบส่งคำสั่งไปหลังบ้าน → บันทึกการชำระ → **ส่งแจ้งเตือนไป LINE ลูกค้า** (และถ้าเป็นออเดอร์ด่วน Grab จะยิงแจ้งเตือนเพิ่มอีกชุด)
11. **สำเร็จ** → toast แถบเขียวลอยขึ้นมา: `"ยืนยันชำระแล้ว เข้าสู่รอบจัดส่ง"` → ระบบดึงรายการใหม่ → **การ์ดใบนั้นหายไปจากคิว**
12. ปุ่มกลับมากดได้เสมอทั้งกรณีสำเร็จและพลาด

**กฎการตรวจสอบ:**
- ต้องผ่านกล่องยืนยันก่อนเสมอ — ไม่มีทางยืนยันชำระได้โดยไม่ผ่านขั้นนี้
- **ไม่บังคับว่าต้องมีรูปสลิป** — สลิปที่ไม่มีรูปแนบก็กดยืนยันได้
- ฝั่งเซิร์ฟเวอร์: ต้องเป็น role `admin` และสลิปต้องอยู่ในร้านเดียวกัน
- ฝั่งเซิร์ฟเวอร์กันการชนกัน: ถ้าออเดอร์ถูกเปลี่ยนสถานะไปแล้ว (แอดมินอีกคนทำก่อน) จะปฏิเสธคำสั่ง แทนที่จะขึ้นสำเร็จหลอกๆ

**ข้อความสำเร็จ (verbatim):** `"ยืนยันชำระแล้ว เข้าสู่รอบจัดส่ง"` — toast เขียว

**ข้อความล้มเหลว (verbatim):** toast แดง แสดงข้อความจริงจากหลังบ้าน กรณีที่พบบ่อยที่สุดคือข้อความยาวว่า
`ยืนยันไม่ได้ — ออเดอร์อยู่สถานะ "เข้ารอบ" แล้ว ตรวจสอบหรือปฏิเสธสลิปนี้แทน`
(คำในเครื่องหมายคำพูดเปลี่ยนตามสถานะจริง ใช้ค่าจากรายการสถานะออเดอร์ในข้อ 6)
ถ้าเกิดข้อผิดพลาดที่ไม่ใช่ข้อความจากหลังบ้าน → toast แดง: `"ยืนยันไม่สำเร็จ"`
**สำคัญ: กรณีล้มเหลว การ์ดยังอยู่ในรายการเหมือนเดิม ไม่มีการรีเฟรช** แอดมินอาจกดซ้ำวนได้เรื่อยๆ

### Flow 2 — ปฏิเสธสลิป

1. แอดมินกดปุ่ม `"ปฏิเสธ"` บนการ์ด → **เปิดโมดัลทันที ไม่มีกล่องยืนยันคั่น**
2. โมดัลหัวข้อสร้างจากข้อมูลจริง: รูปแบบ `ปฏิเสธสลิป · {เลขออเดอร์}` เช่น `"ปฏิเสธสลิป · PKM-2607-0143"`
   (ถ้าไม่มีข้อมูลออเดอร์ หัวข้อจะเป็น `"ปฏิเสธสลิป · "` มีตัวคั่นค้างอยู่ท้ายบรรทัด)
3. บรรทัดแรกในโมดัล เป็นคำเตือนตัวขนาดเล็กสีเทาเข้ม: `"ลูกค้าจะได้รับแจ้งเหตุผลนี้ และต้องส่งสลิปที่ถูกต้องใหม่"`
4. **ฟิลด์เดียว** — ป้ายกำกับ: `"เหตุผลที่ปฏิเสธ (บังคับ)"`
   - ช่องพิมพ์ข้อความหลายบรรทัด สูงขั้นต่ำประมาณ 96px
   - ข้อความตัวอย่างในช่อง (placeholder): `"เช่น ยอดเงินไม่ตรงกับออเดอร์ / สลิปไม่ชัด / โอนผิดบัญชี"`
   - **ไม่มีคำอธิบายใต้ฟิลด์ ไม่มีตัวนับตัวอักษร ไม่มีปุ่มเหตุผลสำเร็จรูป** — พิมพ์อิสระล้วน
5. ปุ่มล่าง 2 ปุ่ม กว้างเท่ากันครึ่งละครึ่ง: `"ยกเลิก"` (ปุ่มโปร่ง) และ `"ปฏิเสธสลิป"` (ปุ่มทึบสีแดงเต็มปุ่ม ตัวอักษรขาว)
6. ปุ่ม `"ปฏิเสธสลิป"` **จางและกดไม่ได้** จนกว่าจะพิมพ์เหตุผลที่ไม่ใช่ช่องว่างล้วน (พิมพ์แต่เว้นวรรคยังกดไม่ได้)
7. กด `"ปฏิเสธสลิป"` → ปุ่ม `"ยกเลิก"` ถูกปิดใช้งาน ปุ่มแดงเปลี่ยนข้อความเป็น `"กำลังปฏิเสธ…"` (**ช่องพิมพ์ยังแก้ได้อยู่ ไม่ถูกล็อก**)
8. ระบบส่งคำสั่งไปหลังบ้าน → บันทึกการปฏิเสธ → **ส่งเหตุผลที่พิมพ์นี้ไปหาลูกค้าทาง LINE ตรงๆ**
9. **สำเร็จ** → toast เขียว → ระบบดึงรายการใหม่ → **โมดัลปิดเอง** → การ์ดหายจากคิว

**กฎการตรวจสอบ:**
- เหตุผลเป็น**ฟิลด์บังคับ** (ป้ายกำกับระบุ `(บังคับ)` ชัดเจน) — ปุ่มยืนยันปิดใช้งานจนกว่าจะพิมพ์
- **ความยาวสูงสุด 300 ตัวอักษร** (พิมพ์เกินไม่ได้ ช่องจะหยุดรับ — และฝั่งเซิร์ฟเวอร์จำกัดที่ 300 เท่ากัน)
- **ไม่มีความยาวขั้นต่ำ** — พิมพ์ตัวอักษรเดียวก็ผ่าน
- ไม่มีการบังคับเลือกจากรายการเหตุผลตายตัว — เป็น free text ล้วน

**ข้อความสำเร็จ (verbatim):** `"ปฏิเสธสลิปแล้ว ระบบแจ้งลูกค้าให้เรียบร้อย"` — toast เขียว

**ข้อความล้มเหลว (verbatim):** toast แดงพร้อมข้อความจริงจากหลังบ้าน หรือถ้าไม่ใช่ข้อผิดพลาดที่มีข้อความ → `"ปฏิเสธไม่สำเร็จ"`
กรณีกดทั้งที่ช่องว่าง (มีการป้องกันซ้ำ) → toast แดง: `"กรุณาระบุเหตุผลที่ปฏิเสธ"`
**สำคัญ: กรณีล้มเหลว โมดัลยังเปิดค้างอยู่พร้อมข้อความที่พิมพ์ไว้** ให้กดลองใหม่ได้ทันที

### Flow 3 — ดู/ซูมรูปสลิป

1. การ์ดที่มีรูปสลิป → ระบบขอลิงก์รูปแบบชั่วคราวจากที่เก็บไฟล์ (รูปอยู่ในที่เก็บแบบส่วนตัว เข้าดูตรงๆ ไม่ได้) **แต่ละการ์ดยิงขอลิงก์ของตัวเองแยกกัน 1 คำขอต่อ 1 การ์ด**
2. **ระหว่างรอ** → กล่อง placeholder สูงประมาณ 160px พื้นเทาอ่อน ข้อความจัดกลางตัวจิ๋วสีเทา: `"กำลังโหลดรูปสลิป…"`
3. **ได้ลิงก์แล้ว** → แสดงรูปเต็มความกว้าง สูงสุดประมาณ 256px แบบไม่ crop บนพื้นเทาอ่อน + ใต้รูปมีคำใบ้ `"แตะรูปเพื่อขยาย"`
   (คำอธิบายสำหรับ screen reader ของรูป: `"สลิปโอนเงิน"`; ปุ่มที่ครอบรูปมีคำอธิบาย: `"แตะเพื่อขยายรูปสลิป"`)
4. **แตะรูป** → เปิดโมดัลหัวข้อ `"สลิปโอนเงิน"` แสดงรูปเต็มความกว้างโมดัล (คำอธิบายสำหรับ screen reader: `"สลิปโอนเงิน (ขยาย)"`)
   - **ในโมดัลมีแค่รูปอย่างเดียว ไม่มีปุ่มใดๆ** — **ไม่มี pinch-zoom / ไม่มี pan / หมุนรูปไม่ได้ / ดาวน์โหลดไม่ได้**
   - โมดัลกว้างสุดประมาณ 448px สูงสุด 90% ของจอ เลื่อนในโมดัลได้ถ้ารูปสูง
5. **ปิดโมดัลได้ 2 ทาง:** กดปุ่ม ✕ มุมขวาบน (คำอธิบายสำหรับ screen reader: `"ปิด"`) หรือแตะพื้นหลังดำโปร่ง

**กฎการตรวจสอบ:**
- ลิงก์รูปมีอายุ 1 ชั่วโมง ระบบจะขอลิงก์ใหม่ก่อนหมดอายุอัตโนมัติ (รีเฟรชทุก 30 นาที) → **ในทางปฏิบัติ รูปจะไม่พังกลางทางแม้เปิดหน้าค้างไว้นาน**

**ข้อความสำเร็จ:** รูปแสดงในการ์ด และเปิดซูมได้ (ไม่มี toast)

**ข้อความล้มเหลว (verbatim):** ถ้าขอลิงก์รูปไม่ได้ → **แทนที่เฉพาะช่องรูปในการ์ดใบนั้น** ด้วยกล่องผิดพลาดมาตรฐาน:
ไอคอน ⚠️ ตัวใหญ่ / หัวข้อ `"โหลดข้อมูลไม่สำเร็จ"` / คำอธิบาย `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"` / ปุ่ม `"ลองใหม่"` (ปุ่มโปร่งขนาดเล็ก)
**การ์ดใบอื่นและปุ่มยืนยัน/ปฏิเสธของการ์ดนี้ยังทำงานปกติ** — พังเฉพาะรูป

### Flow 4 — อัปเดตอัตโนมัติระหว่างเปิดหน้าทิ้งไว้

1. หน้าดึงข้อมูลใหม่เอง **ทุก 15 วินาที** ตราบใดที่หน้ายังเปิดอยู่
2. สลิปใหม่ที่ลูกค้าเพิ่งส่งเข้ามาและตรวจอัตโนมัติไม่ผ่าน → **โผล่ต่อท้ายรายการ (ล่างสุด)** เองภายใน ≤15 วินาที
3. สลิปที่แอดมินคนอื่นเพิ่งเคลียร์ไป → **หายจากรายการเอง** ภายใน ≤15 วินาที (รายการอาจขยับใต้นิ้ว)
4. **ไม่มีตัวบ่งชี้ใดๆ บนหน้าจอว่ากำลังดึงข้อมูลอยู่เบื้องหลัง** — ไม่มี spinner ซ้อน ไม่มีป้าย "อัปเดตล่าสุดเมื่อ…" ไม่มี animation ทุกอย่างเปลี่ยนเงียบๆ
5. ไม่มีการเชื่อมต่อแบบ realtime — เป็นการดึงซ้ำเป็นรอบล้วน

**กฎการตรวจสอบ:** ไม่มี

**ข้อความสำเร็จ:** ไม่มี — รายการอัปเดตเงียบๆ ไม่มีสัญญาณภาพใดๆ

**ข้อความล้มเหลว:** ถ้าการดึงข้อมูลเบื้องหลังพัง → **การ์ดทั้งหมดที่เคยเห็นหายไปทันที** ถูกแทนที่ด้วยกล่องผิดพลาดเต็มพื้นที่: ⚠️ + `"โหลดข้อมูลไม่สำเร็จ"` + `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"` + ปุ่ม `"ลองใหม่"`

## 4. สถานะของหน้า

| สถานะ | ผู้ใช้เห็นอะไร | ข้อความจริง (verbatim) |
|---|---|---|
| **กำลังโหลด (ครั้งแรก)** | หัวข้อหน้าและคำอธิบายยังอยู่ ด้านล่างเป็นตัวหมุนเล็ก (ขอบบนสีแบรนด์) + ข้อความจัดกลาง สีเทาจาง มีระยะห่างบน-ล่างพอสมควร **ไม่มี skeleton การ์ด — เป็น spinner ล้วน** | `"กำลังโหลด…"` |
| **กำลังโหลดรูปในการ์ด** | เฉพาะช่องรูปเป็นกล่องเทาอ่อนสูง ~160px ข้อความจัดกลางตัวจิ๋ว — ส่วนอื่นของการ์ดแสดงครบแล้ว | `"กำลังโหลดรูปสลิป…"` |
| **ปกติ (มีข้อมูล)** | รายการการ์ดสลิปเรียงลง เก่าสุดบนสุด ทุกใบมีป้ายแดงเสมอ | — |
| **ว่าง (ไม่มีสลิปรอตรวจ)** | ไอคอน 🎉 ตัวใหญ่มาก จัดกลาง + ข้อความเดียวใต้ไอคอน ตัวเล็กหนา สีเทาเข้ม มีระยะห่างบน-ล่างเยอะ — **ไม่มีบรรทัดคำอธิบายที่สอง ไม่มีปุ่ม** | `"ไม่มีสลิปรอตรวจ"` |
| **ว่างระดับการ์ด (สลิปไม่มีรูปแนบ)** | กล่องพื้นเทาอ่อนมาก ขอบมน แทนที่รูป ข้อความจัดกลางตัวจิ๋วสีเทา — **ปุ่มยืนยัน/ปฏิเสธยังใช้ได้ปกติ** | `"ไม่มีรูปสลิปแนบ"` |
| **ผิดพลาด (ระดับหน้า)** | การ์ดทั้งหมดหายไป แทนที่ด้วยกล่องผิดพลาดจัดกลาง: ไอคอน ⚠️ ตัวใหญ่ / หัวข้อ / คำอธิบายตัวจิ๋ว / ปุ่มโปร่งขนาดเล็ก | `"โหลดข้อมูลไม่สำเร็จ"` / `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"` / ปุ่ม `"ลองใหม่"` |
| **ผิดพลาด (ระดับรูป)** | กล่องผิดพลาดชุดเดียวกันแทนที่**เฉพาะช่องรูป**ในการ์ดใบนั้น การ์ดอื่นไม่กระทบ | ชุดเดียวกับด้านบน |
| **กำลังส่ง — ยืนยันชำระ** | **การ์ดใบนั้นใบเดียว**: ปุ่ม `"ปฏิเสธ"` และปุ่มหลักจางลง (โปร่งประมาณครึ่ง) กดไม่ได้ ปุ่มหลักเปลี่ยนข้อความ **ไม่มี overlay ไม่มี spinner ทับการ์ด** สื่อสารผ่านข้อความปุ่ม + ความจางเท่านั้น การ์ดใบอื่น**ไม่ถูกล็อก** | ปุ่ม: `"ยืนยันชำระ"` → `"กำลังยืนยัน…"` |
| **กำลังส่ง — ปฏิเสธสลิป** | ในโมดัล: ปุ่ม `"ยกเลิก"` จางกดไม่ได้ ปุ่มแดงเปลี่ยนข้อความ ช่องพิมพ์ยังแก้ได้ | ปุ่ม: `"ปฏิเสธสลิป"` → `"กำลังปฏิเสธ…"` |
| **ปุ่มปฏิเสธยังกดไม่ได้ (ในโมดัล)** | ปุ่มแดงจางลงตราบใดที่ช่องเหตุผลยังว่าง/มีแต่เว้นวรรค — **ไม่มีข้อความอธิบายว่าทำไมกดไม่ได้** | — (ไม่มีข้อความ) |
| **ไม่มีสิทธิ์ (permission-denied)** | **ไม่มีหน้าจอ "คุณไม่มีสิทธิ์" ใดๆ ทั้งสิ้น** — ผู้ใช้ที่ไม่ใช่ `admin` จะถูกเด้งออกไปหน้าแรกของ role ตัวเองทันทีแบบเงียบๆ (stock→แค็ตตาล็อก, packer→หน้าแพ็ค, rider→หน้าไรเดอร์, อื่นๆ→หน้าพนักงาน) และเมนู `"ตรวจสลิป"` ก็ไม่แสดงในแท็บนำทางตั้งแต่แรก | — (ไม่มีข้อความ) |
| **สำเร็จ (toast)** | แถบเขียวทึบ ขอบมน มีเงา ลอยด้านล่าง (เหนือแท็บล่างบนมือถือ / ล่างขวาบน desktop) กว้างสุด ~384px **หายเองใน 3.6 วินาที กดปิดเองไม่ได้** ซ้อนกันได้หลายแถบในแนวตั้ง | ดูข้อ 5 |
| **ล้มเหลว (toast)** | แถบแดงทึบ รูปแบบเดียวกับ toast เขียว — ข้อความ 409 ยาวจะขึ้นหลายบรรทัด | ดูข้อ 5 |

## 7. ข้อมูลอัปเดตยังไง

**Polling ล้วน — ไม่มี realtime**

| อะไร | ความถี่ | หมายเหตุ |
|---|---|---|
| รายการสลิปทั้งคิว | **ดึงใหม่อัตโนมัติทุก 15 วินาที** + ดึงทันทีหลังยืนยัน/ปฏิเสธสำเร็จ + ดึงเมื่อกดปุ่ม `"ลองใหม่"` | ไม่มีปุ่มรีเฟรชด้วยตนเองในหน้า |
| ลิงก์รูปสลิปแต่ละใบ | ไม่ดึงซ้ำเป็นรอบ แต่ถือว่าค่าหมดอายุหลัง 30 นาที (ลิงก์จริงอายุ 1 ชม.) | ตั้งใจให้ต่ออายุก่อนหมด รูปจึงไม่พังกลางทาง |

**ผลต่อดีไซน์:**

- **ปัจจุบันไม่มีตัวบอกว่าอัปเดตล่าสุดเมื่อไหร่เลย** และไม่มีสัญญาณใดๆ ตอนดึงข้อมูลเบื้องหลัง — การ์ดหาย/โผล่เงียบๆ ทุก 15 วินาที
- **นี่คือปัญหาจริงที่ควรแก้:** ถ้าแอดมินกำลังเลื่อนอ่านการ์ดใบที่ 5 อยู่ แล้วแอดมินอีกคนเคลียร์ใบที่ 2 ไป → รายการหดขึ้น **เนื้อหาขยับใต้นิ้ว** อาจกดปุ่มผิดการ์ดได้
- 15 วินาทีเป็นรอบที่ถี่พอสำหรับงานนี้ — **ไม่จำเป็นต้องออกแบบให้มีปุ่ม "ดึงใหม่" ที่เด่น** แต่**ควร**มีตัวบอกสถานะ เช่น "อัปเดตล่าสุด HH:mm" หรือสัญญาณเบาๆ ตอนรายการเปลี่ยน
- แอดมินสลับไปแอปธนาคารแล้วกลับมา → เจอข้อมูลสดเสมอ (ข้อดี) แต่ก็แปลว่าตำแหน่งที่เลื่อนค้างไว้อาจไม่ตรงกับที่จำไว้

## 8. สิ่งที่ designer ต้องรู้

### 🔒 ห้ามแตะ (ผูกกับกติกาธุรกิจ / ความปลอดภัย)

- **ต้องมีกล่องยืนยันก่อน "ยืนยันชำระ" เสมอ** — ปุ่มนี้ = ปล่อยของ ปล่อยเงิน ออเดอร์เข้ารอบจัดส่งทันที และ**ยิงแจ้งเตือนไป LINE ลูกค้า** ห้ามทำเป็น one-tap ห้ามทำ swipe-to-confirm ที่ข้ามการยืนยัน **ในหน้านี้ไม่มีปุ่ม undo — กดแล้วย้อนไม่ได้**
- **เหตุผลปฏิเสธเป็นฟิลด์บังคับ และห้ามเกิน 300 ตัวอักษร** — เพราะ**ข้อความนี้ถูกส่งตรงถึงลูกค้าทาง LINE** ไม่ใช่โน้ตภายใน
- **คำเตือน `"ลูกค้าจะได้รับแจ้งเหตุผลนี้ และต้องส่งสลิปที่ถูกต้องใหม่"` ต้องอยู่** — เป็นสิ่งเดียวที่บอกแอดมินว่ากำลังเขียนถึงลูกค้า
- **ป้ายแดงบอกเหตุผลตกคิวมือต้องมีทุกใบ ตัดออกไม่ได้** — เป็นข้อมูลตัดสินใจหลัก
- **รูปสลิปต้องเข้าถึงได้** — เป็นหลักฐานเดียวที่แอดมินใช้ตัดสิน จะย่อ/ซ่อนได้แต่ต้องเปิดดูเต็มได้เสมอ
- **ต้องแสดงทั้งยอดสลิปและยอดออเดอร์รวม** — เพราะสลิป 1 ใบอาจเป็นแค่ค่าส่ง (`฿50`) ของออเดอร์ `฿1,200` ตัดตัวใดตัวหนึ่งออกจะทำให้ยืนยันผิด
- **เลขออเดอร์ใช้ฟอนต์ monospace โดยตั้งใจ** (อ่าน/เทียบตัวเลขง่าย) — ควรคงไว้
- **สิ่งที่ต้องรักษาด้าน accessibility:** คำอธิบายปุ่มรูป `"แตะเพื่อขยายรูปสลิป"`, ปุ่มปิดโมดัล `"ปิด"`, alt ภาษาไทยบนรูปทั้งสองจุด
- **ข้อความไทยทุกคำต้องใช้ตามข้อ 5 verbatim** — ไม่มีระบบหลายภาษา ห้ามคิดคำใหม่

### ⚠️ กับดักที่ต้องกัน

- **ปุ่ม `"ปฏิเสธ"` และ `"ยืนยันชำระ"` อยู่ติดกัน กว้างเท่ากันเป๊ะ (ครึ่งการ์ดต่อปุ่ม) ในระยะนิ้วโป้ง** — จุดเสี่ยงกดผิดที่ชัดที่สุดของหน้านี้ ผลลัพธ์ของสองปุ่มนี้ตรงข้ามกันสุดขั้ว (ปล่อยของ vs. ตีกลับหาลูกค้า) แต่ปัจจุบันได้น้ำหนักภาพเท่ากัน — **ควรแก้**
- **ระบบล็อกทีละการ์ดเท่านั้น** — ระหว่างใบแรกยังส่งไม่เสร็จ แอดมินกดยืนยันใบที่ 2 ซ้อนได้ แล้วสถานะ "กำลังยืนยัน…" จะกระโดดไปการ์ดใบที่ 2 ทำให้ใบแรกดูเหมือนกดได้อีก ทั้งที่ยังทำงานอยู่ **ถ้าจะเพิ่ม bulk action ต้องคิดใหม่ทั้งหมด**
- **ไม่มี optimistic update** — กดยืนยัน → รอ network + บันทึก + ยิง LINE → ค่อยดึงรายการใหม่ → การ์ดถึงหาย ช่วงรออาจนานพอสมควร ระหว่างนั้นการ์ดยังอยู่เต็มจอ **feedback ปัจจุบันมีแค่ข้อความบนปุ่ม** — บาง เกินไป
- **กรณีล้มเหลว การ์ดยังอยู่และไม่มีการรีเฟรช** — โดยเฉพาะข้อความ `ยืนยันไม่ได้ — ออเดอร์อยู่สถานะ "…" แล้ว…` (เกิดเมื่อแอดมินอีกคนทำไปแล้ว) แอดมินอาจกดซ้ำวนอยู่เพราะการ์ดไม่หายไปไหน **toast กว้างแค่ ~384px ข้อความนี้จะขึ้นหลายบรรทัด** และหายเองใน 3.6 วินาที กดปิดไม่ได้ — **ควรออกแบบให้ข้อความชนกันแบบนี้อยู่บนการ์ดเอง ไม่ใช่ใน toast ที่หายไป**
- **ถ้าการดึงข้อมูลเบื้องหลังพังแม้แค่รอบเดียว → การ์ดทั้งหมดที่เห็นอยู่หายเกลี้ยง** ถูกแทนที่ด้วยกล่องผิดพลาด ทั้งที่เมื่อ 15 วินาทีก่อนยังมีข้อมูลดีๆ อยู่ — **ควรแก้ให้ error ระหว่าง refetch ไม่ทำลายข้อมูลที่แสดงอยู่**
- **การ์ดสูงมาก** (หัวการ์ด + ป้าย + รูปสูงถึง ~256px + แถวปุ่ม) → **บนมือถือเห็นได้ไม่ถึง 2 ใบต่อหน้าจอ** ถ้าอยากให้เห็นภาพรวมคิว ต้องคิดเรื่องย่อรูป/พับเก็บ
- **รูปสลิปอัตราส่วนอะไรก็มา** — ลูกค้าแคปจากทุกแอปธนาคาร ต้องยอมรับสลิปแนวตั้งสูงๆ ที่มีแถบเทาข้างซ้ายขวา
- **ป้ายแดง 6 ค่าใช้สีแดงเหมือนกันหมด ทั้งที่น้ำหนักต่างกันมาก** — `บัญชีผู้รับไม่ตรงกับร้าน` = สัญญาณโกง ต้องเด่นที่สุด; แต่ `ยังไม่ได้เชื่อม SlipOK` / `โควต้าตรวจสลิปหมด` = ปัญหาระบบ ไม่ใช่ความผิดลูกค้า และจะขึ้น**เหมือนกันทุกใบ** ทำให้**คิวแดงทั้งหน้าจอ** จนป้ายไม่มีความหมาย — **ควรแยกโทน**
- **ต้องเผื่อป้ายที่มีข้อความอังกฤษยาวผิดคาด** — ถ้าระบบตรวจส่งรหัสใหม่ที่ยังไม่มีคำแปล จะแสดงรหัสดิบตรงๆ

### 💡 จุดที่ดีไซน์ปัจจุบันอ่อน (ปรับได้ / ควรปรับ)

- **การซูมรูปสลิปคือช่องว่าง UX ที่ชัดที่สุด** — แอดมินต้องอ่านยอดเงิน เวลาโอน และเลขบัญชีปลายทางบนรูป แต่ปัจจุบันซูมได้แค่ "เต็มความกว้างโมดัล ~448px" **ไม่มี pinch-zoom / ไม่มี pan / หมุนไม่ได้** และบน desktop โมดัลถูกจำกัดที่ 448px เท่ากัน — **รูปในโมดัลแทบไม่ใหญ่กว่าในการ์ดเลย** สลิปความละเอียดสูงอ่านตัวเลขไม่ออก
- **ตัวเลข 2 ตัวบนการ์ดคนละความหมายแต่วางใกล้กันมาก** — ยอดสลิป (ตัวใหญ่หนา) กับ ออเดอร์รวม (ตัวจิ๋วเทาจาง) ควรทำให้ความสัมพันธ์ "ยอดสลิปนี้ vs ยอดที่ต้องได้" อ่านออกในแวบเดียว เช่น บอกชัดว่าตรงกันหรือไม่ตรงกัน
- **สลิป 1 ออเดอร์มีได้หลายใบ** (ค่าสินค้า / ค่าส่ง / ค่าส่งรอบใหม่) แต่ละใบเป็นการ์ดแยก และ**การ์ดของออเดอร์เดียวกันอาจอยู่คนละที่ในรายการ** เพราะเรียงตามเวลาส่งสลิป ไม่ได้จัดกลุ่มตามออเดอร์ — โอกาสปรับปรุงชัดเจน
- **ไม่มีแท็บ / ตัวกรอง / ค้นหา / เรียงลำดับ เลย** — flat list เรียงเวลาอย่างเดียว ถ้าคิวยาว (30-50 ใบ) แอดมินต้องเลื่อนหาเอง กระโดดหาออเดอร์เฉพาะใบไม่ได้ และ **ไม่มี pagination** โหลดทุกใบพร้อมรูปทุกใบ (1 คำขอลิงก์รูปต่อ 1 การ์ด)
- **โมดัลปฏิเสธ: ไม่มีตัวนับตัวอักษร (ทั้งที่จำกัด 300) และไม่มีปุ่มเหตุผลสำเร็จรูป** ทั้งที่ข้อความตัวอย่างในช่องบอกเหตุผลยอดฮิต 3 อย่างไว้แล้ว (`ยอดเงินไม่ตรงกับออเดอร์ / สลิปไม่ชัด / โอนผิดบัญชี`) — โอกาสชัดเจน เพราะข้อความนี้ส่งตรงถึงลูกค้า **ถ้าเพิ่ม chips ต้องยังพิมพ์เองได้อยู่**
- **ปุ่ม `"ปฏิเสธสลิป"` ที่จางอยู่ไม่มีคำอธิบายว่าทำไมกดไม่ได้**
- **ปฏิเสธไม่มีกล่องยืนยันคั่น** (ต่างจากยืนยันชำระ) — แต่โมดัลบังคับพิมพ์เหตุผลอยู่แล้ว จึงกันการกดพลาดโดยธรรมชาติ ไม่จำเป็นต้องเพิ่มขั้นตอน
- **หน้านี้เข้าถึงรายละเอียดออเดอร์เต็มไม่ได้เลย** — ไม่มีลิงก์ไปหน้าออเดอร์ ไม่มีรายการสินค้า ถ้าแอดมินอยากตรวจลึกต้องออกไปหน้าอื่นเองแล้วหากลับมาใหม่ (แล้วต้องเลื่อนหาการ์ดเดิม)
- **บน desktop การ์ดกินเต็มความกว้าง ~1024px แต่รูปสูงแค่ ~256px** → พื้นที่ว่างเยอะมาก **ยังไม่มี grid หลายคอลัมน์เลย** เป็น mobile-first ที่ยืดขึ้น desktop ตรงๆ — โอกาสใหญ่สำหรับ layout แยกตามขนาดจอ
- **สถานะว่างมีแค่ 🎉 + `"ไม่มีสลิปรอตรวจ"` บรรทัดเดียว ไม่มีคำอธิบายที่สอง ไม่มีปุ่ม** — เพิ่มบรรทัดที่สองได้ **แต่ถ้าเพิ่มต้องขอคำที่เป็นทางการจากเจ้าของ ห้ามคิดเอง**
- **ไม่มีการถ่ายรูปหรือ GPS ในหน้านี้** — รูปสลิปมาจากฝั่งลูกค้าผ่าน LINE ล้วน แอดมินเป็นผู้ดูอย่างเดียว
