# เข้าสู่ระบบพนักงาน (`/login`)

> พนักงานร้านทุกคน (admin / stock / packer / rider / staff) ใช้เปิดแอปหลังบ้าน PKM Shop ด้วยอีเมล + รหัสผ่านที่แอดมินสร้างให้ — เป็นประตูเดียวเข้าระบบ ไม่มีทางเข้าอื่น

---

## 1. บริบทการใช้งาน

**ใครใช้:** ทุกคนก่อน login — ไม่มีการแยกหน้าตามบทบาท ทุก role เห็นฟอร์มเดียวกันหมด ระบบ **ไม่มีลูกค้า** เข้าใช้หน้านี้ (เป็นเว็บพนักงานล้วน) และ **ไม่มีปุ่มสมัครสมาชิกโดยเจตนา** — บัญชีทุกใบถูกสร้างโดยแอดมิน ผู้ใช้ได้อีเมล + รหัสผ่านมาจากแอดมิน ไม่ได้ตั้งเอง

**อุปกรณ์:** `mobile-first` เป็นข้อบังคับของทั้งโปรเจกต์ ไม่ใช่ความชอบส่วนตัว เหตุผลจากสภาพจริงในโค้ด:
- ปุ่มและช่องกรอกทุกตัวถูกบังคับสูงอย่างน้อย 44px (tap target นิ้วโป้ง)
- ช่องกรอกถูกบังคับขนาดตัวอักษร 16px เพื่อไม่ให้ iOS ซูมหน้าจอตอนแตะช่อง — **ห้ามลดขนาดฟอนต์ในช่อง input ต่ำกว่านี้**
- role หลักของระบบคือ rider (ไรเดอร์ส่งของ) และ packer (คนแพ็คของ) ซึ่งทำงานบนมือถือมือเดียว

**สถานการณ์จริง:** ไรเดอร์ยืนอยู่หน้าร้านตอนเริ่มกะ ถือของ/ถือมือถือมือเดียว อาจอยู่กลางแดดจ้า (คอนทราสต์ต้องสูง) และรีบ — ต้องการเข้าให้เร็วที่สุด ส่วนแอดมินอาจนั่งโต๊ะจอใหญ่ แต่หน้านี้ให้ประสบการณ์เหมือนกันทุกจอ

**ความถี่:** ต่ำมาก — ระบบจำ session ให้อัตโนมัติ ผู้ใช้ปกติจะเห็นหน้านี้แค่ครั้งแรกที่ติดตั้ง หรือหลังจากถูก logout เท่านั้น เปิดแอปครั้งถัดๆ ไปจะข้ามหน้านี้ไปเลย **นี่แปลว่าหน้านี้เป็น "first impression" ของแบรนด์ แต่ไม่ใช่หน้าที่ใช้ทุกวัน** — ควรลงทุนกับความประทับใจและความชัดเจน มากกว่าความเร็วเชิงกลไก

---

## 2. โครงหน้า

หน้านี้ **ไม่มี header, ไม่มีแท็บ, ไม่มีแถบเมนูล่าง, ไม่มีปุ่มย้อนกลับ** — ถูกวาดนอกเปลือกแอปหลัก จอมีแค่การ์ดใบเดียวลอยอยู่กึ่งกลางจอทั้งแนวตั้งและแนวนอน บนพื้นหลังสีเทาอ่อนมาก (#F8FAFC) เต็มความสูงหน้าจอ มีระยะขอบรอบนอก 16px

**การ์ด:** พื้นขาว ขอบมนรัศมี 12px เส้นขอบเทาอ่อน เงาบางมาก ระยะขอบใน 16px กว้างเต็มพื้นที่แต่ไม่เกิน 384px ทุกบล็อกในการ์ดเว้นระยะห่างกัน 16px เท่ากันหมด การ์ดทั้งใบเป็นฟอร์มจริง — กด Enter/Go จากคีย์บอร์ดในช่องไหนก็ส่งฟอร์มได้

ลำดับบล็อกจากบนลงล่างในการ์ด:

1. **หัวแบรนด์ (จัดกึ่งกลาง)** — สองบรรทัด
   - บรรทัดบน: `"PKM Shop"` ขนาด 20px ตัวหนา สีเขียวเข้มของแบรนด์ (#0F6E56) — **เป็นข้อความล้วน ไม่ใช่รูปภาพ ระบบไม่มีไฟล์โลโก้เลย**
   - บรรทัดล่าง: `"เข้าสู่ระบบพนักงาน"` ขนาด 14px สีเทาอ่อน (#94A3B8)
2. **ช่องอีเมล** — สูงอย่างน้อย 44px เต็มความกว้าง ขอบมน 8px เส้นขอบเทา ตัวอักษร 16px **ไม่มีป้ายกำกับ (label) แยก** ใช้ placeholder `"อีเมล"` เป็นตัวบอกช่องอย่างเดียว ตอนแตะเข้าไปเส้นขอบเปลี่ยนเป็นสีแบรนด์ (ไม่มีเรืองแสง/เงา) มือถือจะเด้งคีย์บอร์ดแบบอีเมลขึ้นมา และ password manager เติมให้ได้
3. **ช่องรหัสผ่าน** — หน้าตาเหมือนช่องอีเมลทุกประการ placeholder `"รหัสผ่าน"` ปิดบังตัวอักษร **ไม่มีปุ่มลูกตาแสดง/ซ่อนรหัสผ่าน ไม่มีคำเตือน Caps Lock ไม่มีมาตรวัดความแข็งแรง**
4. **แถบข้อความผิดพลาด (โผล่เฉพาะตอนมี error)** — ข้อความสีแดง (#DC2626) ขนาด 14px เปล่าๆ **ไม่มีกล่อง ไม่มีพื้นหลัง ไม่มีไอคอน ไม่มีเส้นขอบ** แทรกอยู่ระหว่างช่องรหัสผ่านกับปุ่ม เมื่อโผล่จะดันปุ่มลงมา (การ์ดสูงขึ้น)
5. **ปุ่มเข้าสู่ระบบ** — ปุ่มเดียวบนหน้า เต็มความกว้างการ์ด สูงอย่างน้อย 44px พื้นเขียวแบรนด์ ตัวอักษรขาว 16px ขอบมน 8px ตอนชี้เมาส์เปลี่ยนเป็นเขียวเข้มขึ้น (#0b5544) ตอนกดย่อลงเล็กน้อย

**ไม่มีสิ่งเหล่านี้บนหน้า** (อย่าออกแบบเพิ่มโดยไม่ถามเจ้าของ): ลิงก์ "ลืมรหัสผ่าน", ปุ่ม "สมัครสมาชิก", ปุ่ม "จดจำฉัน", LINE Login / OAuth / SSO, magic link, OTP, modal ใดๆ, toast ใดๆ

## 3. ทุก flow ทีละสเต็ป

### 3.1 เข้าสู่ระบบสำเร็จ (เส้นทางปกติ — พนักงานที่แอดมินเพิ่มไว้แล้ว)

**ขั้นตอน:**

1. ผู้ใช้เปิดเว็บที่ URL ใดก็ได้ → ถ้าไม่มี session ระบบพากลับมาหน้านี้อัตโนมัติ (**URL ที่พิมพ์มาถูกทิ้ง ไม่มีการจำเพื่อพากลับไปหลัง login**)
2. ระหว่างระบบเช็ค session เดิม ผู้ใช้เห็นจอโหลดเต็มหน้าจอ ข้อความ `"กำลังโหลด…"` สีเทากลางจอ **ยังไม่เห็นฟอร์ม** — จอนี้ไม่มี spinner ไม่มีโลโก้ ไม่มี skeleton
3. ฟอร์มปรากฏ — เห็นการ์ดกลางจอครบทุกบล็อกตามข้อ 2
4. แตะช่อง `"อีเมล"` → คีย์บอร์ดอีเมลเด้ง → password manager เสนอเติมอัตโนมัติ
5. พิมพ์อีเมล → แตะช่อง `"รหัสผ่าน"` → พิมพ์รหัสผ่าน (password manager เติมได้เช่นกัน)
6. กดปุ่ม `"เข้าสู่ระบบ"` **หรือ** กด Enter/Go บนคีย์บอร์ดจากช่องไหนก็ได้
7. ทันทีที่กด: ข้อความ error เก่า (ถ้ามี) หายไป → ปุ่มถูกปิดการใช้งาน จางลงครึ่งหนึ่ง และข้อความในปุ่มเปลี่ยนเป็น `"กำลังเข้าสู่ระบบ…"`
8. ระบบส่งอีเมล + รหัสผ่านไปตรวจ (อีเมลถูกตัดช่องว่างหัวท้ายให้อัตโนมัติ รหัสผ่านส่งดิบไม่ตัด)
9. สำเร็จ → ปุ่มกลับเป็น `"เข้าสู่ระบบ"` ชั่วขณะ **ไม่มี toast ไม่มีข้อความ "สำเร็จ" ใดๆ**
10. ระบบดึงข้อมูลบทบาทของผู้ใช้เบื้องหลัง
11. หน้า Login หายไปเอง ถูกแทนที่ด้วยแอปหลัก + หน้าแรกตามบทบาทอัตโนมัติ (admin → หน้าออเดอร์, stock → หน้าแคตตาล็อก, packer → หน้าคิวแพ็ค, rider → หน้ารอบส่ง, อื่นๆ → หน้าพนักงาน) — **ผู้ใช้เลือกปลายทางเองไม่ได้**

**กฎการตรวจสอบ (validation):**

- ช่องอีเมล: **บังคับกรอก** — ปล่อยว่างแล้วกดส่ง เบราว์เซอร์บล็อกและขึ้น tooltip **ภาษาของระบบเครื่อง ไม่ใช่ภาษาไทยของแอป** (เช่น "Please fill out this field")
- ช่องอีเมล: **ตรวจรูปแบบอีเมลโดยเบราว์เซอร์** — กรอกผิดรูปขึ้น tooltip เจ้าของเบราว์เซอร์ (เช่น "Please include an @ in the email address")
- ช่องรหัสผ่าน: **บังคับกรอก** — ปล่อยว่างแล้วกดส่ง เบราว์เซอร์บล็อก
- **ไม่มีการกำหนดความยาวขั้นต่ำ/สูงสุด ไม่มี pattern** บนช่องใดในหน้านี้ (ฝั่งแอดมินตอนสร้างบัญชีบังคับรหัสผ่าน 6–72 ตัวอักษร แต่หน้า login ไม่ตรวจซ้ำ)
- **ไม่มีการตรวจสอบใดๆ ที่แอปเขียนเอง** — ไม่มี error ใต้ช่องรายตัว ไม่มีการทำช่องเป็นสีแดง ทุกอย่างพึ่งเบราว์เซอร์ล้วน
- **ไม่มีการจำกัดจำนวนครั้ง ไม่มีการล็อกบัญชี ไม่มี CAPTCHA** ที่ฝั่งหน้าจอ

**ข้อความสำเร็จ (verbatim):** ไม่มีเลย — feedback เดียวคือหน้าจอเปลี่ยนไปเป็นแอปหลัก ระหว่างรอผู้ใช้เห็นแค่ปุ่มที่เขียนว่า `"กำลังเข้าสู่ระบบ…"`

**ข้อความล้มเหลว (verbatim):** ดูข้อ 3.2

### 3.2 เข้าสู่ระบบไม่สำเร็จ (รหัสผิด / เน็ตหลุด / เซิร์ฟเวอร์ล่ม)

**ขั้นตอน:**

1. กรอกอีเมล + รหัสผ่าน → กด `"เข้าสู่ระบบ"`
2. ปุ่มเปลี่ยนเป็น `"กำลังเข้าสู่ระบบ…"` + ถูกปิดการใช้งาน + จางลง
3. ระบบตอบกลับว่าผิดพลาด
4. ปุ่มกลับมากดได้ทันที ข้อความกลับเป็น `"เข้าสู่ระบบ"`
5. ข้อความสีแดงปรากฏระหว่างช่องรหัสผ่านกับปุ่ม → **การ์ดสูงขึ้น ปุ่มถูกดันลง (layout shift)**
6. **ค่าในช่องอีเมลและรหัสผ่านยังอยู่ครบ ไม่ถูกล้าง** → ผู้ใช้แก้แล้วกดใหม่ได้เลย
7. กด `"เข้าสู่ระบบ"` อีกครั้ง → ข้อความแดงหายทันทีตอนเริ่มส่งใหม่

**กฎการตรวจสอบ:**

- **ข้อความ error ที่ผู้ใช้เห็นจริงเป็นภาษาอังกฤษ** — มาจากระบบ auth ตรงๆ โดยไม่มีการแปล ทั้งที่หน้าจอเป็นไทย 100% ตัวอย่างที่เจอจริง:
  - `"Invalid login credentials"` (อีเมลหรือรหัสผ่านผิด — เจอบ่อยที่สุด)
  - `"Email not confirmed"`
  - `"Failed to fetch"` (เน็ตหลุด)
- สตริงไทย `"เข้าสู่ระบบไม่สำเร็จ"` เป็นข้อความสำรองที่จะแสดง **เฉพาะกรณีที่ระบบโยนสิ่งที่ไม่ใช่ error object ออกมา** — ในทางปฏิบัติแทบไม่เคยเห็น
- **ไม่มีการแยกแยะประเภท error เลย** — รหัสผิด / เน็ตหลุด / เซิร์ฟเวอร์ล่ม แสดงเป็นข้อความดิบเหมือนกันหมด
- ไม่มีปุ่ม "ลองใหม่" แยก — ผู้ใช้กดปุ่มเดิมซ้ำ

**ข้อความสำเร็จ:** —

**ข้อความล้มเหลว (verbatim):** ข้อความดิบภาษาอังกฤษจากระบบ auth เช่น `"Invalid login credentials"` / `"Failed to fetch"` แสดงเป็นข้อความสีแดงเปล่าๆ เหนือปุ่ม — **ไม่มีไอคอน ไม่มีกล่องสี ไม่มีเสียง ไม่มีการสั่น ไม่มี toast** ข้อความสำรองไทยคือ `"เข้าสู่ระบบไม่สำเร็จ"`

### 3.3 เข้าสู่ระบบสำเร็จ แต่แอดมินยังไม่ได้เพิ่มสิทธิ์ (pending access)

> จอนี้อยู่คนละไฟล์กับ Login แต่ผู้ใช้เจอทันทีต่อจากหน้า Login — **ต้อง redesign พร้อมกันเป็นชุดเดียว**

**ขั้นตอน:**

1. กรอกอีเมล + รหัสผ่านถูกต้อง → กด `"เข้าสู่ระบบ"` → ผ่าน
2. ระบบดึงข้อมูลบทบาท → พบว่าบัญชีนี้ยังไม่ถูกผูกกับหน้าที่ใดในร้าน
3. หน้า Login หายไป ถูกแทนที่ด้วยจอนี้เต็มจอ — **ผู้ใช้เข้าแอปไม่ได้ และย้อนกลับไปหน้า Login เองไม่ได้**
4. จอนี้: การ์ดขาวกลางจอ (กว้างไม่เกิน 384px) จัดข้อความกึ่งกลาง ระยะขอบรอบนอก 24px แต่ละบล็อกเว้น 12px ประกอบด้วย
   - อิโมจิ `🕓` ขนาดใหญ่ (30px) — **เป็นอิโมจิจริง ไม่ใช่ไอคอนที่ออกแบบ**
   - หัวข้อ `"รอผู้ดูแลเพิ่มสิทธิ์"` ขนาด 18px ตัวหนา
   - คำอธิบาย `"บัญชีนี้ยังไม่ได้ถูกกำหนดหน้าที่ในระบบ กรุณาให้แอดมินเพิ่มคุณเข้าทีมก่อน"` ขนาด 14px สีเทา
   - **(มีบ้างไม่มีบ้าง)** บรรทัดรหัส ขนาด 12px สีเทาอ่อน รูปแบบ `"รหัสของคุณ: XXXX"` โดยตัวรหัสเป็นฟอนต์ monospace ตัวหนา — ให้ผู้ใช้อ่านรหัสนี้บอกแอดมินทางไลน์/ปากเปล่า
   - ปุ่ม `"ออกจากระบบ"` เต็มความกว้าง แบบพื้นขาวขอบเทา (ไม่ใช่ปุ่มเขียว) → กดแล้วกลับมาหน้า Login
5. เมื่อแอดมินเพิ่มสิทธิ์ให้แล้ว **จอนี้ไม่อัปเดตเอง** — ผู้ใช้ต้องรีเฟรชหน้า หรือออกจากระบบแล้วเข้าใหม่

**กฎการตรวจสอบ:**

- เข้าจอนี้เมื่อ: login ผ่านแล้ว แต่ไม่มี profile หรือไม่มี role สักตัว
- **บรรทัด "รหัสของคุณ" อาจหายไปเงียบๆ** — ถ้าเซิร์ฟเวอร์ที่แจกรหัสล่ม ระบบจะดึงข้อมูลสำรองซึ่งไม่มีรหัสติดมาด้วย บรรทัดนี้จะไม่แสดง **ดีไซน์ต้องรองรับทั้งกรณีมีรหัสและไม่มีรหัส โดยไม่ดูพัง**

**ข้อความสำเร็จ:** — **ข้อความล้มเหลว:** — (นี่คือสถานะรอ ไม่ใช่ความล้มเหลว น้ำเสียงต้องไม่ตำหนิผู้ใช้)

### 3.4 เข้าสู่ระบบสำเร็จ แต่บัญชีถูกปิดใช้งาน (deactivated)

**ขั้นตอน:**

1. กรอกอีเมล + รหัสผ่านถูกต้อง → กด `"เข้าสู่ระบบ"` → ผ่าน
2. ระบบพบว่าบัญชีถูกปิดใช้งาน → แสดงจอเดียวกับข้อ 3.3 แต่เปลี่ยนเนื้อหา:
   - อิโมจิ `🚫`
   - หัวข้อ `"บัญชีถูกปิดใช้งาน"`
   - คำอธิบาย `"บัญชีนี้ถูกปิดใช้งานโดยผู้ดูแล หากคิดว่าเป็นความผิดพลาด ติดต่อแอดมินของร้าน"`
3. **ไม่แสดงบรรทัด "รหัสของคุณ" ในกรณีนี้เด็ดขาด**
4. ปุ่ม `"ออกจากระบบ"` → กลับหน้า Login

**กฎการตรวจสอบ:**

- เข้าจอนี้เมื่อ: login ผ่าน + มี profile + มี role อย่างน้อย 1 ตัว + แต่สถานะบัญชีถูกปิด
- **ห้ามตัดจอนี้ทิ้ง** — เหตุผลเชิงระบบระบุไว้ชัดในโค้ด: ถ้าไม่ดักไว้ ผู้ใช้ที่ถูกปิดบัญชีจะเห็นแอปทำงานครบทุกหน้าแต่ข้อมูลว่างเปล่าเงียบๆ (เพราะฐานข้อมูลบล็อกอยู่) ซึ่งสับสนกว่ามาก

**ข้อความสำเร็จ:** — **ข้อความล้มเหลว:** —

### 3.5 เปิดแอปทั้งที่มี session ค้างอยู่ (ไม่เห็นหน้า Login เลย)

**ขั้นตอน:**

1. เปิดเว็บ → จอเต็มหน้าจอ ข้อความ `"กำลังโหลด…"` สีเทากลางจอ (ข้อความล้วน ไม่มี spinner ไม่มีโลโก้)
2. ระบบพบ session ที่เก็บไว้ → ดึงข้อมูลบทบาท
3. ข้ามหน้า Login ไปเข้าแอปตามบทบาทเลย — **ผู้ใช้ไม่เคยเห็นฟอร์ม login**

**กฎการตรวจสอบ:**

- **ไม่มีปุ่ม "จดจำฉัน"** — ระบบจำ session ให้เองเสมอ ผู้ใช้ควบคุมจากหน้านี้ไม่ได้ **อย่าออกแบบ checkbox นี้เพิ่ม เพราะไม่มีกลไกรองรับ**
- จอ `"กำลังโหลด…"` นี้คือสิ่งแรกสุดที่ทุกคนเห็นทุกครั้งที่เปิดแอป **บ่อยกว่าหน้า Login มาก** — คุ้มที่จะออกแบบให้ดี

## 4. สถานะของหน้า

| สถานะ | ผู้ใช้เห็นอะไร | ข้อความจริง |
|---|---|---|
| **ก่อนถึงหน้า (กำลังเช็ค session)** | จอเต็มหน้าจอ พื้นเทาอ่อน ข้อความสีเทากลางจอ ไม่มี spinner ไม่มีโลโก้ ไม่มี skeleton | `"กำลังโหลด…"` |
| **ปกติ (ฟอร์มเปล่า)** | การ์ดกลางจอ ช่องทั้งสองว่าง แสดง placeholder สีเทาจางตามค่าเริ่มต้นของเบราว์เซอร์ ปุ่มเขียวกดได้ | `"PKM Shop"` / `"เข้าสู่ระบบพนักงาน"` / `"อีเมล"` / `"รหัสผ่าน"` / `"เข้าสู่ระบบ"` |
| **กำลังโหลด (ตอน mount)** | **ไม่มี** — หน้านี้ไม่ดึงข้อมูลจากเซิร์ฟเวอร์ตอนเปิด เห็นฟอร์มทันที | — |
| **ว่าง (empty)** | **ไม่มี** — ไม่มีลิสต์/ตาราง/ข้อมูลให้ว่าง สถานะที่ใกล้ที่สุดคือฟอร์มเปล่าตอนเปิดหน้า | — |
| **กำลังส่ง (busy)** | ปุ่มถูกปิดการใช้งาน จางลง 50% ข้อความในปุ่มเปลี่ยน **ไม่มี spinner ไม่มี animation ไม่มี overlay ไม่มี progress bar** และ **ช่องอีเมล/รหัสผ่านยังพิมพ์แก้ได้อยู่ (ไม่ถูกปิด)** | `"กำลังเข้าสู่ระบบ…"` (ใช้อักขระจุดไข่ปลาตัวเดียว `…` ไม่ใช่จุดสามจุด) |
| **ผิดพลาด** | ข้อความแดงเปล่าๆ เหนือปุ่ม ไม่มีกล่อง ไม่มีไอคอน ไม่มี border → ดันปุ่มลง | ดิบจากระบบ auth (อังกฤษ) เช่น `"Invalid login credentials"`, `"Failed to fetch"` / สำรองไทย `"เข้าสู่ระบบไม่สำเร็จ"` |
| **ผิดพลาดจากเบราว์เซอร์ (ช่องว่าง/อีเมลผิดรูป)** | tooltip เจ้าของเบราว์เซอร์ชี้ที่ช่อง **ภาษาตามระบบเครื่อง ไม่ใช่ไทยเสมอ** | เช่น `"Please fill out this field"` (ควบคุมไม่ได้) |
| **permission-denied: ยังไม่มีสิทธิ์** | จอเต็มหน้าจอแทนที่ Login — การ์ดกลางจอ อิโมจิ 🕓 + หัวข้อ + คำอธิบาย + (อาจมี) รหัส + ปุ่มออกจากระบบ | `"รอผู้ดูแลเพิ่มสิทธิ์"` / `"บัญชีนี้ยังไม่ได้ถูกกำหนดหน้าที่ในระบบ กรุณาให้แอดมินเพิ่มคุณเข้าทีมก่อน"` / `"รหัสของคุณ:"` / `"ออกจากระบบ"` |
| **permission-denied: บัญชีถูกปิด** | จอเดียวกัน อิโมจิ 🚫 ไม่มีบรรทัดรหัส | `"บัญชีถูกปิดใช้งาน"` / `"บัญชีนี้ถูกปิดใช้งานโดยผู้ดูแล หากคิดว่าเป็นความผิดพลาด ติดต่อแอดมินของร้าน"` / `"ออกจากระบบ"` |

## 6. ข้อมูลที่แสดง

หน้านี้ **แทบไม่มีข้อมูลจากระบบเลย** — เป็นฟอร์มล้วน ไม่มีเงิน ไม่มีเวลา ไม่มีเลขออเดอร์ ไม่มีรูปภาพ

| ฟิลด์ | รูปแบบการแสดงผล |
|---|---|
| อีเมล | ข้อความธรรมดาที่ผู้ใช้พิมพ์ ไม่มีการจัดรูปแบบ ไม่มี mask ช่องว่างหัวท้ายถูกตัดตอนส่ง (ผู้ใช้ไม่เห็นการตัด) |
| รหัสผ่าน | ปิดบังด้วยจุดตามค่าเริ่มต้นของเบราว์เซอร์ **ไม่มีทางดูตัวจริงได้** |
| ข้อความ error | ข้อความดิบจากระบบ ความยาวคาดเดาไม่ได้ **ดีไซน์ต้องรองรับข้อความยาว 2–3 บรรทัด** |
| รหัสผู้ใช้ (link_code — เฉพาะจอรอสิทธิ์) | รูปแบบ `"รหัสของคุณ: XXXX"` ตัวรหัสเป็น **ฟอนต์ monospace ตัวหนา** ตัวอย่างจริง: `รหัสของคุณ: 7K3P` — เป็นรหัสสั้นไว้อ่านให้แอดมินฟัง จึงต้องอ่านง่าย แยก 0/O และ 1/l ได้ **อาจไม่มีค่าและบรรทัดนี้หายไปทั้งบรรทัด** |

**ฟอนต์:** ไม่มีฟอนต์ custom ในระบบ ใช้ฟอนต์ระบบ (system-ui / Segoe UI / Roboto) และตกไปที่ **Noto Sans Thai** สำหรับตัวอักษรไทย

## 7. ข้อมูลอัปเดตยังไง

**โหลดครั้งเดียว — และจริงๆ คือไม่โหลดอะไรเลย** หน้า Login ไม่ดึงข้อมูลจากเซิร์ฟเวอร์ตอนเปิด ไม่มี polling ไม่มีการรีเฟรชตามเวลา ไม่มี realtime subscription

**ผลต่อดีไซน์:**

- **ไม่ต้องมีตัวบอก "อัปเดตล่าสุดเมื่อ…" ไม่ต้องมีปุ่มรีเฟรช ไม่ต้องมี pull-to-refresh** — ไม่มีข้อมูลอะไรให้เก่า
- **แต่มีกลไก event-driven หนึ่งอย่างที่ดีไซน์ต้องรู้:** ระบบเฝ้าฟังสถานะการ login อยู่ตลอด นี่คือเหตุผลที่หน้า Login **หายไปเอง** หลัง login สำเร็จ โดยฟอร์มไม่ได้สั่งเปลี่ยนหน้าเอง → **การเปลี่ยนหน้าเป็นเรื่องที่เกิดขึ้นกับผู้ใช้ ไม่ใช่เรื่องที่ผู้ใช้สั่ง** ช่วงรอยต่อนี้อาจกระตุกและไม่มี transition ใดๆ รองรับ — เป็นจุดที่ดีไซน์ควรใส่ใจ
- **ผลข้างเคียงข้ามแท็บ:** ถ้าเปิดแอปไว้หลายแท็บ การ login/logout ในแท็บหนึ่งจะทำให้แท็บอื่นเปลี่ยนสถานะตามทันทีโดยไม่เตือน (แอดมินที่นั่งจอใหญ่เจอเคสนี้ได้จริง)
- **จอรอสิทธิ์ไม่อัปเดตเอง** — แอดมินเพิ่มสิทธิ์ให้แล้วผู้ใช้ยังค้างจอเดิม ต้องรีเฟรชเอง (ดู §8)

## 8. สิ่งที่ designer ต้องรู้

### ห้ามแตะ (ผูกกับกติกาธุรกิจ/ความปลอดภัย)

- **ห้ามออกแบบปุ่ม "สมัครสมาชิก" หรือทางเข้าแบบ self-service ใดๆ** — ระบบไม่มี public sign-up **โดยเจตนาเชิงความปลอดภัย** ฐานข้อมูลนี้ใช้ร่วมกันหลายร้าน (multi-tenant) โค้ดระบุชัดว่าห้ามให้ฝั่งผู้ใช้สร้างสิทธิ์แอดมินเด็ดขาด แอดมินคนแรกของแต่ละร้านถูกตั้งนอกระบบ ที่เหลือแอดมินสร้างให้ทีละคน
- **ห้ามเพิ่ม LINE Login / OAuth / SSO / magic link / OTP เข้ามาเอง** — ระบบรองรับแค่อีเมล + รหัสผ่าน ถึงแม้แอปทั้งระบบจะผูกกับ LINE ก็ตาม (LINE เป็นฝั่งลูกค้า ไม่ใช่ฝั่งพนักงาน)
- **ห้ามเพิ่มลิงก์ "ลืมรหัสผ่าน"** ถ้ายังไม่ได้คุยกับเจ้าของ — ปัจจุบันไม่มีกลไก reset password เลย พนักงานที่ลืมรหัสต้องให้แอดมินสร้างรหัสใหม่ให้ ลิงก์ที่กดแล้วไม่มีอะไรเกิดขึ้นแย่กว่าไม่มีลิงก์
- **ห้ามเพิ่ม checkbox "จดจำฉัน"** — ระบบจำ session ให้เสมอ ไม่มีกลไกให้ผู้ใช้เลือก
- **ห้ามตัดจอ "บัญชีถูกปิดใช้งาน" ทิ้ง** — เป็นการดักที่จงใจ ถ้าไม่มี ผู้ใช้ที่ถูกปิดบัญชีจะเห็นแอปว่างเปล่าเงียบๆ โดยไม่รู้สาเหตุ
- **ห้ามลดขนาดตัวอักษรในช่อง input ต่ำกว่า 16px** — จะทำให้ iOS ซูมหน้าจอเองตอนแตะช่อง เป็นข้อกำหนดทั้งโปรเจกต์
- **ห้ามทำ tap target เล็กกว่า 44px** — ข้อกำหนดทั้งโปรเจกต์
- **ห้ามออกแบบให้ผู้ใช้เลือกหน้าปลายทางหลัง login** — ระบบพาไปตามบทบาทอัตโนมัติ และไม่จำ URL ที่ผู้ใช้พิมพ์มาก่อน login

### ปรับได้ — และควรปรับ (ทั้งหมดนี้คือ "สภาพจริงที่อ่อน" ไม่ใช่ของที่ทำไว้แล้ว)

- **error เป็นภาษาอังกฤษบน UI ไทย 100%** — ผู้ใช้เห็น `"Invalid login credentials"` ทั้งที่ทั้งจอเป็นไทย นี่คือจุดอ่อนที่ชัดที่สุดของหน้านี้ ควรเสนอชุดข้อความไทยที่แมปกับ error แต่ละแบบ (รหัสผิด / เน็ตหลุด / เซิร์ฟเวอร์ล่ม) **แต่ต้องส่งข้อความไทยที่เสนอให้เจ้าของอนุมัติ อย่าคิดเองแล้วถือว่าเป็น copy จริง**
- **ไม่มี label จริง มีแต่ placeholder** — พอผู้ใช้พิมพ์ placeholder หาย ไม่เหลือตัวบอกว่าช่องไหนคืออะไร และ screen reader อ่านไม่ครบ ควรเสนอ label ลอย หรือ label ถาวรเหนือช่อง
- **ข้อความ error ไม่ถูกประกาศให้ screen reader** — ผู้ใช้ที่ใช้เครื่องอ่านหน้าจอกดปุ่มแล้วไม่รู้ว่าเกิดอะไรขึ้น
- **error โผล่แล้วดันปุ่มลง (layout shift)** — ผู้ใช้ที่กำลังจะกดซ้ำอาจกดพลาด ควรจองพื้นที่ไว้ล่วงหน้าหรือทำเป็นกล่องความสูงคงที่
- **ปุ่มตอนกำลังส่งไม่มี spinner** — เปลี่ยนแค่ข้อความกับความจาง บนเน็ตช้าผู้ใช้อาจไม่แน่ใจว่าระบบทำงานอยู่ไหม (ระบบมีคอมโพเนนต์ spinner กลางอยู่แล้วแต่หน้านี้ไม่ได้เรียกใช้)
- **ช่องกรอกไม่ถูกปิดตอนกำลังส่ง** — ผู้ใช้พิมพ์แก้ระหว่างรอได้ ทำให้สิ่งที่เห็นกับสิ่งที่กำลังส่งไม่ตรงกัน
- **ไม่มีปุ่มลูกตาแสดง/ซ่อนรหัสผ่าน** — พนักงานภาคสนามพิมพ์รหัสบนมือถือมือเดียวกลางแดด ต้องการมากเป็นพิเศษ
- **จอ `"กำลังโหลด…"` เปล่าเกินไป** — ไม่มีโลโก้ ไม่มี spinner ทั้งที่เป็นสิ่งแรกที่ทุกคนเห็นทุกครั้งที่เปิดแอป (บ่อยกว่าหน้า Login มาก) เป็นโอกาสของแบรนด์ที่ถูกทิ้ง
- **จอรอสิทธิ์ใช้อิโมจิ 🕓 / 🚫 เป็นภาพหลัก** — ไม่ใช่ไอคอนที่ออกแบบ หน้าตาต่างกันไปตามระบบปฏิบัติการ ปรับได้
- **จอรอสิทธิ์ไม่มีทางออกอื่นนอกจาก "ออกจากระบบ"** — และไม่รู้ตัวเมื่อแอดมินเพิ่มสิทธิ์ให้แล้ว ผู้ใช้ต้องเดาว่าต้องรีเฟรช อาจเสนอปุ่ม "ตรวจสอบอีกครั้ง" หรือคำแนะนำที่ชัดกว่านี้

### ข้อจำกัดทางเทคนิคที่กระทบดีไซน์โดยตรง

- **ไม่มี dark mode** — ระบบล็อกโหมดสว่างตายตัว อย่าเสนอดีไซน์ dark mode โดยไม่ถามก่อน
- **สีที่มีอยู่จริงในระบบมีน้อยมาก** — มีแค่เขียวแบรนด์ `#0F6E56` + เขียวเข้ม `#0b5544` (ใช้ตอนชี้เมาส์) และสีเทา/แดง/เขียว/เหลืองมาตรฐานของ Tailwind ไม่มี design token อื่น ไม่มี color scale ของแบรนด์ **ถ้าดีไซน์ต้องการสีเพิ่ม ต้องระบุให้ชัดว่าเป็นของใหม่**
- **ไม่มีไฟล์โลโก้เลยในระบบ** — `"PKM Shop"` เป็นข้อความล้วน ถ้าจะใส่โลโก้จริงต้องสร้างใหม่ทั้งหมด
- **หน้านี้แทบไม่มี responsive breakpoint** — การ์ดกว้างสูงสุด 384px กลางจอ หน้าตาเหมือนกันเป๊ะบนมือถือและเดสก์ท็อป ต่างแค่พื้นที่ว่างรอบๆ ถ้าจะทำเดสก์ท็อปให้ต่างออกไป (เช่น split-screen มีภาพครึ่งจอ) ถือเป็นของใหม่ทั้งหมด
- **หน้านี้ไม่ใช้คอมโพเนนต์ร่วมของระบบเลยแม้แต่ตัวเดียว** — เขียนแยกทั้งหมด ระบบมีคอมโพเนนต์กลางอยู่แล้ว (ช่องกรอกพร้อม label, spinner, กล่อง error พร้อมปุ่มลองใหม่, toast) แต่หน้านี้ไม่ได้เรียกใช้ **นี่เป็นโอกาส: การ redesign ให้กลับมาใช้ระบบกลางจะทำให้หน้านี้สอดคล้องกับที่เหลือของแอป**
- **ไม่มี modal ไม่มี toast บนหน้านี้** — ระบบมี toast กลาง (โผล่ล่างจอ หายเองใน ~3.6 วินาที เขียว/แดง/เทา) แต่หน้า Login ไม่ได้ใช้
- **หน้านี้ไม่ใช้ safe-area (notch/home indicator)** — เพราะการ์ดลอยกลางจอจึงไม่ชนอยู่แล้ว แต่ถ้า redesign ให้เนื้อหาชิดขอบบน/ล่าง **ต้องใส่ safe-area เอง** (ระบบมี utility ให้แล้ว)
- **หน้านี้ไม่มีวันเกิด scroll** — ยกเว้นตอนคีย์บอร์ดมือถือเด้งขึ้นมาบังบนจอเตี้ยมาก ซึ่งเป็นเคสที่ควรทดสอบ

### ขอบเขตงานที่แนะนำ

หน้า Login, จอ `"กำลังโหลด…"`, จอรอสิทธิ์ 🕓 และจอบัญชีถูกปิด 🚫 **ทั้งสี่จอใช้ภาษาการออกแบบเดียวกัน (การ์ดขาวกลางจอ กว้างไม่เกิน 384px) และผู้ใช้เจอต่อกันเป็นลำดับ** ควร redesign ไปพร้อมกันเป็นชุด onboarding เดียว ไม่ใช่แยกจอ
