# คิวแพ็คของ (`/packer`)

> พนักงานแพ็ค (role `packer`) และแอดมิน (role `admin`) ใช้ที่สถานีแพ็คทุกวันหลังรอบส่งถูกล็อก เพื่อ "รับงานแพ็ค" แล้วปิดงานด้วยการถ่ายรูปของที่แพ็คเสร็จ

---

## 1. บริบทการใช้งาน

**ใครใช้**
- **แพ็คเกอร์ (`packer`)** — ผู้ใช้หลัก หน้านี้คือหน้าแรกที่เขาเห็นทันทีเมื่อล็อกอิน (เป็น home ของ role นี้)
- **แอดมิน (`admin`)** — เข้าได้ด้วย และเห็นหน้าตาต่างออกไปเล็กน้อย (ดู §8)
- ใครที่ไม่มีทั้งสอง role จะถูกเด้งออกไปหน้าแรกของ role ตัวเอง ไม่มีหน้าจอ "คุณไม่มีสิทธิ์" ให้เห็นเลย — เป็นการ redirect เงียบๆ (ดู §4)

**อุปกรณ์และสถานการณ์จริง**
- **มือถือ ยืนใช้งาน มือเดียว** — แพ็คเกอร์ยืนอยู่ที่โต๊ะแพ็ค มือข้างหนึ่งถือสินค้า/เทปกาว/ถุง อีกข้างถือมือถือ ต้องกดด้วยนิ้วโป้ง
- ต้องใช้ **กล้องหลัง** ของเครื่องถ่ายรูปกล่องที่แพ็คเสร็จ — เป็นอุปกรณ์เดียวที่หน้านี้ต้องใช้ (ไม่มี GPS, ไม่มีแผนที่, ไม่มีสแกนบาร์โค้ด)
- อาจอยู่ในที่แสงจ้าหรือแสงน้อย เน็ตในร้านอาจช้า

**ความถี่และความเร่งรีบ — นี่คือหัวใจของหน้านี้**
- โค้ดอ้าง `Ready.md §3.1` ตรงๆ ว่า: **หลังรอบ lock (นาทีที่ :30) แพ็คเกอร์มีเวลาประมาณ 30 นาที** ในการแพ็คงานทั้งรอบให้เสร็จก่อนไรเดอร์มารับ
- แปลว่างานไม่ได้ไหลเข้ามาเรื่อยๆ แต่ **มาเป็นก้อนพรวดเดียวตอนนาที :30** แล้วต้องรีบเคลียร์
- เจตนาของหน้านี้ตามคอมเมนต์ในโค้ด: ต้องบอกให้ชัดว่า **"แพ็คอะไร กี่ชิ้น รอบไหน" โดยไม่ต้องเปิดหน้าอื่น** — การ์ดหนึ่งใบต้องอ่านจบใน 2–3 วินาที

**Mobile-first — ไม่ใช่แค่ responsive แต่ออกแบบมาเพื่อมือถือก่อน**
เหตุผล: ผู้ใช้ยืนทำงาน ไม่ได้นั่งโต๊ะ. หลักฐานในดีไซน์ปัจจุบัน:
- แถวเช็คลิสต์แต่ละแถวสูงอย่างน้อย 48px และกดได้ทั้งแถวเต็มความกว้าง
- กล่องถ่ายรูปสูงอย่างน้อย 96px
- ปุ่ม action ทุกตัวเป็นปุ่มเต็มความกว้าง — ไม่มีปุ่มเล็กชิดขอบเลย
- Modal เป็น **bottom sheet** บนมือถือ (สไลด์ขึ้นจากขอบล่าง มุมบนมน) และเป็น dialog กลางจอบนจอใหญ่
- มีระยะเผื่อขอบล่างสำหรับ tab bar และรองรับ notch/home indicator ของเครื่อง
- Toast ลอยเหนือ tab bar เสมอ เพื่อไม่ให้ถูกบัง

**จอใหญ่รองรับแบบ "ใช้ได้" เท่านั้น** — เนื้อหาถูกจำกัดความกว้างไว้กลางจอ (คอลัมน์เดียว ไม่แตกเป็นกริด) และแถบเมนูย้ายจากล่างขึ้นบน แค่นั้น ไม่มี layout เฉพาะ desktop

## 2. โครงหน้า

หน้านี้ **ไม่มีแท็บ** ไม่มีฟิลเตอร์ ไม่มีช่องค้นหา ไม่มีปุ่มเรียงลำดับ ไม่มี pagination — เป็นคอลัมน์เดียว เรียงบนลงล่างล้วนๆ

### กรอบนอก (Shell — มีทุกหน้าในระบบหลังบ้าน ไม่ใช่ของหน้านี้โดยเฉพาะ)
1. **แถบหัวบนสุด** — ชื่อระบบ `"PKM Shop"` + คำว่า `"หลังบ้าน"` + ชื่อพนักงานที่ล็อกอิน (ถ้าไม่มีชื่อจะขึ้นว่า `"พนักงาน"`) + ปุ่ม `"ออก"`
2. **แถบเตือน LINE (แสดงเฉพาะเมื่อยังไม่ผูกบัญชี)** — แถบสีเหลือง ข้อความ: `"⚠️ ยังไม่ได้ผูก LINE — เพิ่ม PKM Shop OA เป็นเพื่อน แล้วพิมพ์รหัส {link_code} เพื่อรับแจ้งเตือนงาน"` (`{link_code}` เป็นรหัสของพนักงานคนนั้น)
3. **แถบเมนู** — บนมือถือเป็น **bottom tab bar** (สูงอย่างน้อย 56px), บนจอใหญ่เป็นเมนูแนวนอนใต้หัวเรื่อง. แท็บของหน้านี้คือ `"🎁 แพ็ค"`

### เนื้อหน้า (บนลงล่าง)

1. **หัวเรื่อง (PageHeader)**
   - ซ้าย: หัวข้อตัวหนา `"คิวแพ็คของ"`
   - ขวา: ข้อความเล็กสีเทาอ่อน `"อัปเดตทุก 15 วิ"` — **เป็นแค่ป้ายข้อความ ไม่ใช่ปุ่ม กดไม่ได้**
   - หัวเรื่องนี้ **แสดงตลอดทุกสถานะ** ไม่หายตอนกำลังโหลด/ว่าง/ผิดพลาด

2. **บล็อกเนื้อหาหลัก** — จะเป็นอย่างใดอย่างหนึ่งใน 4 แบบ ตามลำดับความสำคัญนี้ (ดู §4):
   ผิดพลาด → กำลังโหลด → ว่างทั้งหน้า → เนื้อหาจริง (สองเซคชัน)

3. **เซคชัน 1 — "งานของฉัน"** (อยู่ **บนสุด** เสมอ — ตั้งใจ)
   - หัวข้อเซคชัน: `"งานของฉัน"` (ตัวเล็กหนา สีเทาเข้ม) + **badge นับจำนวน สีแบรนด์ ตัวอักษรขาว** (ซ่อนเมื่อเป็น 0)
   - เนื้อ: การ์ดออเดอร์ที่ตัวเองรับงานไว้แล้ว (สถานะ `packing`) แต่ละใบมีปุ่มเต็มความกว้าง `"📷 แพ็คเสร็จ (ถ่ายรูป)"`
   - ถ้าไม่มี: ข้อความบรรทัดเดียวสีเทาอ่อน (ไม่มีไอคอน ไม่มีกรอบ) — ดู §4

4. **เซคชัน 2 — "รอรับงาน"**
   - หัวข้อเซคชัน: `"รอรับงาน"` + **badge นับจำนวน สีเทาอ่อน ตัวอักษรเทาเข้ม** (ซ่อนเมื่อเป็น 0) — สีต่างจากเซคชันบนโดยตั้งใจ เพื่อลดน้ำหนักสายตา
   - เนื้อ: การ์ดออเดอร์ที่ยังไม่มีใครรับ (สถานะ `confirmed`) แต่ละใบมีปุ่มเต็มความกว้าง `"รับงานแพ็ค"`
   - ถ้าไม่มี: ข้อความบรรทัดเดียวสีเทาอ่อน — ดู §4

5. **บรรทัดท้ายสุด — งานของเพื่อนร่วมทีม** (แสดงเฉพาะเมื่อมีจำนวน > 0 และ **ผู้ใช้ไม่ใช่แอดมิน**)
   - ข้อความเล็กสีเทาอ่อนบรรทัดเดียว: `"อีก {N} ออเดอร์กำลังแพ็คโดยเพื่อนร่วมทีม"` เช่น `"อีก 3 ออเดอร์กำลังแพ็คโดยเพื่อนร่วมทีม"`
   - **ไม่ใช่การ์ด ไม่มีปุ่ม กดไม่ได้** — เจตนาคือบอกว่าคิวจริงยาวกว่าที่เห็น แต่ไม่ยั่วให้ไปแย่งงานคนอื่น

### โครงของ "การ์ดออเดอร์" (OrderCard) — ใช้เหมือนกันทั้งสองเซคชัน
การ์ดพื้นขาว ขอบมน มีระยะห่างระหว่างบล็อกภายใน ประกอบด้วย 4 ชั้นจากบนลงล่าง:

1. **แถวหัวการ์ด** (ซ้าย–ขวา ชิดขอบ)
   - ซ้าย: **เลขออเดอร์** ตัวอักษร monospace ขนาดเล็ก ตัวหนา
   - ขวา: กลุ่ม badge เรียงชิดกัน
     - **badge รอบส่ง** (แสดงเฉพาะออเดอร์ที่ผูกกับรอบ): พื้นสีแบรนด์จางมาก ตัวอักษรสีแบรนด์ตัวหนา — `"⏱ รอบ 14:30"`
     - **badge สถานะ**: พื้นเทาอ่อน ตัวอักษรเทา — `"รอแพ็ค"` หรือ `"กำลังแพ็ค"`
2. **เช็คลิสต์รายการสินค้า** (ItemChecklist — ดูรายละเอียดถัดไป)
3. **แถวสรุปท้าย** — ข้อความเล็กสีเทา รูปแบบ: `ชื่อผู้รับ · ยอดเงิน · รวม N ชิ้น` เช่น `"คุณสมชาย · ฿1,250 · รวม 7 ชิ้น"` (ท่อน `· รวม N ชิ้น` จะไม่ขึ้นถ้าไม่มีรายการสินค้า)
4. **ปุ่ม action เต็มความกว้าง** (ต่างกันตามเซคชัน)

> **การ์ดนี้ไม่แสดง**: เบอร์โทร, ที่อยู่, ประเภทการส่ง, ลำดับจุดส่ง, สถานะการชำระเงิน, ค่าส่ง — ถึงแม้ระบบจะดึงข้อมูลเหล่านี้มาแล้วก็ตาม (ดู §8)

### โครงของ "เช็คลิสต์รายการสินค้า" (ItemChecklist)
- **ถ้าไม่มีรายการสินค้า** → แถบเตือนพื้นสีเหลืองอำพันจาง ตัวอักษรสีน้ำตาลอำพัน มุมมน: `"ไม่พบรายการสินค้าของออเดอร์นี้ — เช็คกับแอดมินก่อนแพ็ค"`
- **ถ้ามีรายการ** → รายการแนวตั้ง แต่ละรายการเป็นแถวกดได้เต็มความกว้าง สูงอย่างน้อย 48px มุมมน มีเส้นขอบ ประกอบด้วย 3 ส่วนเรียงซ้าย→ขวา:
  1. **กล่องติ๊ก** สี่เหลี่ยมจัตุรัสเล็ก มุมมน มีเครื่องหมาย ✓ อยู่ข้างใน
  2. **ชื่อสินค้า** ตัวอักษรขนาดปกติ น้ำหนักกลาง (กินพื้นที่ที่เหลือทั้งหมด)
  3. **จำนวน** รูปแบบ `"× 3"` ตัวหนา
- **สองสถานะของแถว:**

| | ยังไม่ติ๊ก | ติ๊กแล้ว |
|---|---|---|
| พื้นแถว | เทาจางมาก (กดค้างจะเข้มขึ้น) | เขียวจาง |
| เส้นขอบ | เทาอ่อน | เขียวอ่อน |
| กล่องติ๊ก | ขาว ขอบเทา ✓ **โปร่งใสมองไม่เห็น** (แต่ยังกินที่อยู่) | เขียวเข้มทึบ ✓ สีขาว |
| ชื่อสินค้า | เทาเข้ม | **เทาอ่อน + ขีดฆ่า** |
| จำนวน | **สีแบรนด์** | เทาอ่อน |

## 3. ทุก flow ทีละสเต็ป

### 3.1 Flow A — รับงานแพ็ค (claim)

**เริ่มจาก:** เซคชัน `"รอรับงาน"`

**ขั้นตอน**
1. แพ็คเกอร์เปิดหน้า `/packer` (ล็อกอินแล้วเด้งมาเอง หรือแตะแท็บ `"🎁 แพ็ค"`)
2. ระบบดึงคิวมาแสดง — ระหว่างรอเห็นวงกลมหมุนพร้อมข้อความ `"กำลังโหลด…"`
3. หน้าแบ่งออเดอร์ออกเป็น 3 กลุ่มอัตโนมัติ (งานของฉัน / รอรับงาน / งานของเพื่อน) เลื่อนลงมาที่เซคชัน `"รอรับงาน"`
4. อ่านการ์ด: เลขออเดอร์ → badge `"⏱ รอบ 14:30"` → badge `"รอแพ็ค"` → รายการสินค้า → `"คุณสมชาย · ฿1,250 · รวม 7 ชิ้น"`
5. *(ทางเลือก)* แตะรายการสินค้าเพื่อติ๊กว่าหยิบแล้ว — **ติ๊กได้ตั้งแต่ยังไม่รับงาน** แต่ค่าติ๊กอยู่แค่ในเครื่อง (ดู Flow C)
6. กดปุ่ม `"รับงานแพ็ค"` (ปุ่มหลักสีแบรนด์ เต็มความกว้าง)
7. **ทันที**: ปุ่ม **ของการ์ดใบนั้นใบเดียว** เปลี่ยนข้อความเป็น `"กำลังรับงาน…"` และกดไม่ได้ — **ปุ่มของการ์ดใบอื่นยังกดได้อยู่** (busy รายใบ ไม่ล็อกทั้งหน้า) ไม่มี spinner ในปุ่ม ไม่มี overlay คลุมจอ
8. ระบบยิงคำสั่งไปหลังบ้าน ตรวจสิทธิ์ + ตรวจว่าออเดอร์ยังว่างอยู่จริง (ตรวจแบบ atomic กันสองคนกดพร้อมกัน)
9. **สำเร็จ** → toast สีเขียวเด้งขึ้นเหนือ tab bar หายเองใน ~3.6 วินาที
10. ระบบดึงคิวใหม่ทันที (ทั้งกรณีสำเร็จและล้มเหลว)
11. การ์ดใบนั้น **ย้ายจาก `"รอรับงาน"` ขึ้นไปโผล่ในเซคชัน `"งานของฉัน"`** — badge สถานะเปลี่ยนจาก `"รอแพ็ค"` เป็น `"กำลังแพ็ค"`, ปุ่มเปลี่ยนเป็น `"📷 แพ็คเสร็จ (ถ่ายรูป)"`, ตัวเลข badge ของทั้งสองเซคชันขยับ
    > **การย้ายนี้เกิดขึ้นแบบตัดฉับ ไม่มี animation** — เป็นจุดอ่อนที่ดีไซน์ควรแก้

**กฎการตรวจสอบ (validation)**
- **ไม่มีฟอร์ม ไม่มีฟิลด์ให้กรอก** — เป็นการกดปุ่มอย่างเดียว
- ปุ่มถูกปิดการใช้งาน **เฉพาะระหว่างที่กำลังยิงคำสั่งของการ์ดใบนั้น** เท่านั้น
- ฝั่งหน้าจอ **ไม่ตรวจอะไรก่อนยิงเลย** — พึ่งหลังบ้านทั้งหมด
- หลังบ้านบังคับ: ผู้ใช้ต้องมี role `packer` (แอดมินผ่านด้วย) / ออเดอร์ต้องอยู่สถานะ `confirmed` / ต้องยังไม่มีใครรับ (หรือเป็นของเราเองอยู่แล้ว — กดซ้ำได้ไม่พัง)
- **ออเดอร์ไรเดอร์ที่รอบยัง "เปิด" อยู่จะไม่โผล่ในคิวเลย** (ไม่ใช่แสดงแบบปุ่มเทา — **แต่ซ่อนทั้งใบ**) จนกว่ารอบจะล็อกที่นาที :30. ส่วนออเดอร์ Grab ด่วน / Lalamove / พัสดุ Kerry แพ็คได้ทันทีไม่ต้องรอรอบ

**ข้อความสำเร็จ (verbatim)**
- toast สีเขียว: `"รับงานแพ็คแล้ว เริ่มแพ็คได้เลย"`

**ข้อความล้มเหลว (verbatim)** — toast สีแดง เลือกตามกรณี:
- ถูกคนอื่นชิงไปก่อน → `"มีคนรับงานนี้แล้ว"`
  *(หลังบ้านส่งข้อความยาวกว่ามาว่า `"ออเดอร์นี้มีคนรับแพ็คแล้ว"` แต่หน้าจอย่อให้สั้นลงเหลือ `"มีคนรับงานนี้แล้ว"` — designer ใช้ข้อความสั้นตัวนี้)*
- ออเดอร์เปลี่ยนสถานะไปแล้ว → `"ออเดอร์นี้ไม่อยู่ในสถานะรอแพ็ค"`
- หาออเดอร์ไม่เจอ → `"Order not found."` *(ข้อความอังกฤษหลุดมาจากหลังบ้าน — เป็นบั๊กภาษาที่ควรแจ้ง)*
- กรณีอื่นๆ ที่ไม่มีข้อความ → `"รับงานไม่สำเร็จ"`
- กรณีระบบเรียกฟังก์ชันไม่ได้เลย → `"ทำรายการไม่สำเร็จ"`

> หลัง error **ระบบดึงคิวใหม่เสมอ** เพื่อให้คิวสะท้อนความจริง — การ์ดที่ถูกชิงไปจะหายจากคิวเองทันที

---

### 3.2 Flow B — ปิดงานแพ็คด้วยรูป (pack)

**เริ่มจาก:** เซคชัน `"งานของฉัน"`

**ขั้นตอน**
1. กดปุ่ม `"📷 แพ็คเสร็จ (ถ่ายรูป)"` บนการ์ดออเดอร์ที่กำลังแพ็คอยู่ (ปุ่มนี้ **ไม่มีสถานะกดไม่ได้** — กดได้เสมอ)
2. Modal เปิดขึ้น — **bottom sheet สไลด์ขึ้นจากขอบล่างบนมือถือ** / dialog กลางจอบนจอใหญ่. ฉากหลังเป็นสีดำโปร่ง. กล่องสูงได้ไม่เกิน 90% ของจอ และเลื่อนภายในได้ถ้าเนื้อยาว
   - หัวข้อ modal: `"แพ็คเสร็จ · {เลขออเดอร์}"` เช่น `"แพ็คเสร็จ · ORD-2601-0042"` (คั่นด้วยจุดกลาง ·)
   - มุมขวาบนมีปุ่มปิด `"✕"` (screen reader อ่านว่า `"ปิด"`)
3. อ่านคำอธิบายใต้หัวข้อ: `"เช็ครายการให้ครบ แล้วถ่ายรูปของที่แพ็คเสร็จเพื่อยืนยัน"` (ตัวเล็กสีเทา)
4. แตะติ๊กรายการสินค้าในเช็คลิสต์ของ modal ทีละชิ้น
   > ⚠️ **กับดัก**: เช็คลิสต์ใน modal เป็น **คนละชุดกับบนการ์ด** — ต่อให้ติ๊กครบบนการ์ดมาแล้ว พอเปิด modal จะ **เริ่มจากยังไม่ติ๊กทั้งหมดใหม่** ดู §8
5. แตะกล่องถ่ายรูป — กล่องสูงอย่างน้อย 96px **เส้นขอบประ (dashed)** พื้นเทาจาง แสดง `"📷"` ตัวใหญ่ + ข้อความ `"ถ่ายรูปของที่แพ็คเสร็จ (บังคับ)"`
   > คำว่า `" (บังคับ)"` ถูกต่อท้ายอัตโนมัติเพราะช่องนี้ถูกกำหนดว่าจำเป็น — ข้อความฐานคือ `"ถ่ายรูปของที่แพ็คเสร็จ"`
6. **บนมือถือกล้องหลังเปิดขึ้นทันที** ไม่ผ่านแกลเลอรี่ (รับเฉพาะไฟล์รูปภาพ)
7. ถ่ายเสร็จ → กล่องสลับเป็น **โหมดพรีวิว**: รูปเต็มความกว้าง สูงไม่เกินประมาณ 224px ครอปแบบ cover (screen reader อ่านว่า `"ตัวอย่างรูป"`) มีแถบไล่สีดำจางคลุมด้านล่างของรูป และปุ่ม `"ถ่ายใหม่"` (ปุ่มโปร่งเล็กพื้นขาวจาง) วางที่มุมขวาล่างของรูป — กดแล้วเปิดกล้องซ้ำ **ทับรูปเดิม** (มีรูปได้ 1 ใบเท่านั้น ไม่ใช่การเพิ่ม)
8. ทันทีที่มีรูป: ข้อความเตือนสีเหลือง `"ต้องแนบรูปก่อนจึงจะยืนยันได้"` **หายไป** และปุ่มยืนยันเปลี่ยนจากกดไม่ได้เป็นกดได้
9. กดปุ่ม `"ยืนยันแพ็คเสร็จ"` (ปุ่มหลักเต็มความกว้าง)
10. **ทันที**: ปุ่มเปลี่ยนข้อความเป็น `"กำลังส่ง…"` และกดไม่ได้ + **modal ถูกล็อก ปิดไม่ได้** — กด `"✕"` หรือแตะฉากหลังจะ **ไม่มีอะไรเกิดขึ้นเลย** (ไม่มีข้อความบอกว่าปิดไม่ได้ — เงียบสนิท)
11. เบื้องหลัง 2 ขั้น: (ก) อัปโหลดรูปเข้าที่เก็บแบบส่วนตัว (ข) ยิงคำสั่งปิดงาน
    > ⚠️ **ช่วงนี้อาจกินเวลาหลายวินาทีบนเน็ตร้านที่ช้า เพราะรูปดิบจากกล้องไม่ถูกย่อขนาดเลย และ ไม่มี progress bar ใดๆ** — ผู้ใช้เห็นแค่คำว่า `"กำลังส่ง…"` ดู §8
12. หลังบ้านบันทึกสถานะเป็น `packed` → บันทึกค่าคอมมิชชันของแพ็คเกอร์ → **ส่งแจ้งเตือนลูกค้าทาง LINE พร้อมรูปที่เพิ่งถ่าย**
13. **สำเร็จ** → toast สีเขียว → modal ปิดเอง → รูปถูกล้าง → ระบบดึงคิวใหม่
14. ออเดอร์เปลี่ยนเป็นสถานะ `packed` ซึ่ง **ไม่อยู่ในขอบเขตของหน้านี้อีกต่อไป** → **การ์ดหายจากหน้าจอทันที** และ badge `"งานของฉัน"` ลดลง 1
15. ถ้าคิวหมดเกลี้ยง → หน้าสลับไปแสดง empty state `"🎁 ไม่มีของต้องแพ็คตอนนี้"`

**กฎการตรวจสอบ (validation)**

| กฎ | รายละเอียด |
|---|---|
| **รูปถ่าย = บังคับ (hard gate)** | ปุ่ม `"ยืนยันแพ็คเสร็จ"` กดไม่ได้จนกว่าจะมีรูป — **ไม่มีทางลัด ไม่มีข้อยกเว้น** |
| ข้อความเตือนแสดงล่วงหน้า | `"ต้องแนบรูปก่อนจึงจะยืนยันได้"` แสดง **ตั้งแต่เปิด modal** ไม่ต้องรอให้ผู้ใช้กดผิดก่อน |
| ชนิดไฟล์ | รูปภาพเท่านั้น + เปิดกล้องหลังเป็นค่าเริ่มต้น |
| ขนาดไฟล์ | **ไม่จำกัด ไม่ย่อ** — อัปโหลดไฟล์ดิบจากกล้อง |
| จำนวนรูป | **1 ใบเท่านั้น** — ถ่ายใหม่ = ทับของเดิม |
| เช็คลิสต์สินค้า | **ไม่ใช่เงื่อนไขบังคับ** — ติ๊ก 0 ชิ้นก็ยืนยันได้ ถ้ามีรูป (เป็นเครื่องมือช่วยจำล้วนๆ) |
| ปิด modal ระหว่างส่ง | **ทำไม่ได้** (กัน double-submit และงานค้างครึ่งทาง) |
| สิทธิ์ | เฉพาะแพ็คเกอร์ที่รับงานใบนั้นไว้เอง หรือแอดมิน เท่านั้น |

**ข้อความสำเร็จ (verbatim)**
- toast สีเขียว: `"แพ็คเสร็จ แจ้งลูกค้าแล้ว"`

**ข้อความล้มเหลว (verbatim)** — toast สีแดง:
- คนอื่นถือครองงานนี้อยู่ → `"ออเดอร์นี้อยู่ในมือแพ็คเกอร์คนอื่น"`
- ที่อยู่ไฟล์รูปไม่ถูกต้อง → `"Invalid photo path."` *(อังกฤษ)*
- เปลี่ยนสถานะไม่สำเร็จ → `"Pack transition failed."` *(อังกฤษ)*
- อัปโหลดรูปล้มเหลว → ข้อความดิบจากระบบเก็บไฟล์ (**อาจเป็นภาษาอังกฤษ ความยาวไม่แน่นอน — designer ต้องเผื่อ toast รองรับข้อความยาวหลายบรรทัด**)
- กรณีอื่น → `"แพ็คไม่สำเร็จ"`

> **สำคัญมากสำหรับดีไซน์**: เมื่อล้มเหลว **modal ไม่ปิด และรูปยังอยู่ในกล่องพรีวิว** — ผู้ใช้กด `"ยืนยันแพ็คเสร็จ"` ซ้ำได้ทันทีโดยไม่ต้องถ่ายใหม่ (พฤติกรรมนี้ดี ควรรักษาไว้). และ **ไม่มีการดึงคิวใหม่หลัง error** — หน้าจอค้างอยู่กับข้อมูลเดิม

### 3.3 Flow C — ติ๊กเช็คลิสต์หยิบของ (เก็บในเครื่องเท่านั้น)

**ขั้นตอน**
1. แตะแถวสินค้าในเช็คลิสต์ (**กดได้ทั้งแถว** พื้นที่กว้างเต็มการ์ด สูงอย่างน้อย 48px)
2. ถ้ายังไม่ติ๊ก → เปลี่ยนเป็นสถานะติ๊กแล้วทันที: กล่องกลายเป็นเขียวทึบมี ✓ ขาว, ชื่อสินค้าเทาอ่อน + ขีดฆ่า, ตัวเลขจำนวนเทาอ่อน, พื้นแถวเขียวจาง
3. แตะซ้ำ → ยกเลิกติ๊ก กลับเป็นสถานะปกติ
4. **ไม่มีการยิงข้อมูลไปเซิร์ฟเวอร์ใดๆ ทั้งสิ้น**

**กฎการตรวจสอบ**
- ไม่มี validation — เป็น UI ช่วยจำล้วนๆ
- **ไม่ถูกส่งไปเซิร์ฟเวอร์ ไม่ถูกเก็บลงเครื่องถาวร**
- **ไม่บล็อกการกดยืนยัน**
- **หายหมด** เมื่อรีเฟรชหน้า / เปิด-ปิด modal / คอมโพเนนต์ถูกสร้างใหม่

**ข้อความสำเร็จ**: ไม่มี toast — feedback เป็นการเปลี่ยนสีของแถวทันที (เขียว + ✓ + ขีดฆ่า)
**ข้อความล้มเหลว**: ไม่มี (ไม่มีทางล้มเหลว)

---

### 3.4 Flow D — ปิด/ยกเลิก modal แพ็คเสร็จ

**ขั้นตอน**
1. กดปุ่ม `"✕"` มุมขวาบนของ modal — **หรือ** แตะฉากหลังสีดำโปร่งนอกกล่อง (แตะในกล่องไม่ปิด)
2. ระบบเช็คว่ากำลังส่งอยู่หรือไม่
3. **ถ้าไม่ได้กำลังส่ง** → modal ปิด และ **รูปที่ถ่ายไว้ถูกทิ้งทันที**
4. **ถ้ากำลังส่งอยู่** → **ไม่มีอะไรเกิดขึ้นเลย** modal ค้างอยู่จนกว่าจะยิงเสร็จ

**กฎการตรวจสอบ**
- ปิดไม่ได้ระหว่างกำลังส่ง
- **ไม่มี confirm dialog ถามยืนยันก่อนทิ้งรูป — ปิดปุ๊บ รูปหายปั๊บ** (จุดอ่อนที่ควรพิจารณา ดู §8)
- **ปิดด้วยปุ่ม Escape ไม่ได้** (ไม่ผูกคีย์บอร์ด)

**ข้อความสำเร็จ**: ไม่มีข้อความ — modal ปิด กลับสู่คิว. **ออเดอร์ยังคงอยู่ใน `"งานของฉัน"` สถานะ `"กำลังแพ็ค"`** — การปิด modal **ไม่ใช่การคืนงาน**
**ข้อความล้มเหลว**: ไม่มี

---

### 3.5 Flow E — โหลดข้อมูลล้มเหลว → ลองใหม่

**ขั้นตอน**
1. การดึงคิวล้มเหลว (เน็ตหลุด / เซสชันหมดอายุ / ถูกปฏิเสธสิทธิ์) 
2. หน้าแทนที่เนื้อหาทั้งหมดด้วยบล็อกผิดพลาด (**หัวเรื่อง `"คิวแพ็คของ"` และ `"อัปเดตทุก 15 วิ"` ยังอยู่ด้านบน**) จัดกึ่งกลาง เรียงบนลงล่าง:
   - `"⚠️"` ตัวใหญ่มาก
   - หัวข้อ: `"โหลดข้อมูลไม่สำเร็จ"` (ตัวเล็ก น้ำหนักกลาง สีเทาเข้ม)
   - คำอธิบาย: `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"` (ตัวจิ๋ว สีเทาอ่อน)
   - ปุ่ม `"ลองใหม่"` (ปุ่มโปร่งขนาดเล็ก)
3. กด `"ลองใหม่"` → ดึงคิวอีกครั้ง
4. สำเร็จ → คิวกลับมาแสดงปกติ / ล้มเหลวซ้ำ → **คงบล็อกผิดพลาดไว้เหมือนเดิม ไม่มี toast เพิ่ม**
5. **หมายเหตุ**: ระบบยังดึงข้อมูลเองทุก 15 วินาทีอยู่เบื้องหลัง — บล็อกผิดพลาดอาจ **หายไปเองโดยที่ผู้ใช้ไม่ได้กดอะไรเลย**

**กฎการตรวจสอบ**
- **บล็อกผิดพลาดมาก่อนบล็อกกำลังโหลด** — ระหว่างที่กด "ลองใหม่" แล้วรอ ผู้ใช้จะ **ยังเห็นหน้าจอผิดพลาดเดิม ไม่เห็น spinner** (ไม่มี feedback ว่าปุ่มทำงานแล้ว — เป็นจุดอ่อน ดู §8)
- บล็อกผิดพลาด **แยกจาก empty state โดยตั้งใจ** — ผู้ใช้ต้องไม่เข้าใจผิดว่า "ไม่มีข้อมูล" ทั้งที่จริงคือ "โหลดไม่ได้"

## 4. สถานะของหน้า

| สถานะ | ผู้ใช้เห็นอะไร | ข้อความจริง (verbatim) |
|---|---|---|
| **ปกติ (มีงาน)** | หัวเรื่อง + สองเซคชันเรียงบนลงล่าง (`งานของฉัน` เหนือ `รอรับงาน`) + การ์ดออเดอร์ + บรรทัดท้าย (ถ้ามี) | `"คิวแพ็คของ"` / `"อัปเดตทุก 15 วิ"` / `"งานของฉัน"` / `"รอรับงาน"` |
| **กำลังโหลด (ครั้งแรกเท่านั้น)** | หัวเรื่องยังอยู่ + ตรงกลางมีวงกลมหมุนเล็ก (ขอบเทา ยอดเป็นสีแบรนด์) วางแนวนอนคู่กับข้อความ มีระยะห่างบน-ล่างกว้าง | `"กำลังโหลด…"` (ใช้ ellipsis `…` ตัวเดียว ไม่ใช่จุดสามจุด) |
| **กำลังโหลด (ระดับแอป — ก่อนรู้ว่าใครล็อกอิน)** | หน้าจอขาวเปล่าเต็มจอ ข้อความสีเทากลางจอ | `"กำลังโหลด…"` |
| **รีเฟรชอัตโนมัติ / หลัง claim / หลัง pack** | **ไม่มี spinner ไม่มีการกระพริบใดๆ** — คิวอัปเดตเงียบๆ การ์ดโผล่/หายแบบตัดฉับ | — |
| **ว่างทั้งหน้า** (ไม่มีทั้ง `confirmed` และ `packing` เลย) | หัวเรื่องยังอยู่ + บล็อกกึ่งกลาง ระยะห่างบน-ล่างกว้าง: ไอคอน `🎁` ตัวใหญ่ → หัวข้อ → คำใบ้ตัวจิ๋ว. **สองเซคชันหายไปหมด ไม่แสดงหัวข้อเซคชันเลย** | `"🎁"` / `"ไม่มีของต้องแพ็คตอนนี้"` / `"งานจะเข้าคิวเมื่อรอบถูกปิด (นาทีที่ :30)"` |
| **ว่างเฉพาะเซคชัน `งานของฉัน`** (แต่มีออเดอร์อื่นในหน้า) | หัวข้อเซคชันยังอยู่ (ไม่มี badge) + ข้อความบรรทัดเดียวสีเทาอ่อน **ไม่มีไอคอน ไม่มีกรอบ ไม่มีปุ่ม** | `ยังไม่มีงานที่รับไว้ — กด "รับงานแพ็ค" ด้านล่าง` (สังเกต: ใช้ฟันหนู `"` ล้อมคำว่า รับงานแพ็ค จริงในโค้ด) |
| **ว่างเฉพาะเซคชัน `รอรับงาน`** | หัวข้อเซคชันยังอยู่ (ไม่มี badge) + ข้อความบรรทัดเดียวสีเทาอ่อน | `"ไม่มีงานรอรับตอนนี้"` |
| **ว่างระดับการ์ด** (ออเดอร์ไม่มีรายการสินค้า) | แทนที่เช็คลิสต์ด้วยแถบเตือนพื้นเหลืองอำพันจาง มุมมน ตัวอักษรน้ำตาลอำพัน. **ปุ่มยังกดได้ปกติ ไม่บล็อก** | `"ไม่พบรายการสินค้าของออเดอร์นี้ — เช็คกับแอดมินก่อนแพ็ค"` |
| **ผิดพลาด (โหลดคิวไม่ได้)** | หัวเรื่องยังอยู่ + บล็อกกึ่งกลาง: `⚠️` ตัวใหญ่ → หัวข้อ → คำอธิบาย → ปุ่ม. **แทนที่เนื้อหาทั้งหมด** | `"⚠️"` / `"โหลดข้อมูลไม่สำเร็จ"` / `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"` / `"ลองใหม่"` |
| **ผิดพลาด (การกระทำ claim/pack)** | **ไม่ใช้บล็อกผิดพลาด** — ใช้ toast สีแดง (พื้นแดงเข้ม ตัวอักษรขาว มีขอบเรืองสีแดงเข้มกว่า) ลอยเหนือ tab bar หายเองใน ~3.6 วินาที | ดู §3.1 / §3.2 |
| **กำลังส่ง — รับงาน** | **ปุ่มของการ์ดใบนั้นใบเดียว** เปลี่ยนข้อความ + กดไม่ได้. **การ์ดใบอื่นยังกดได้** ไม่มี overlay ไม่มี spinner ในปุ่ม | `"กำลังรับงาน…"` |
| **กำลังส่ง — ปิดงานแพ็ค** | ปุ่มใน modal เปลี่ยนข้อความ + กดไม่ได้ **และ modal ล็อกปิดไม่ได้** (กด ✕ / แตะฉากหลัง = เงียบสนิท). **ไม่มี progress bar ตอนอัปโหลดรูป** | `"กำลังส่ง…"` |
| **ยังไม่มีรูป (ใน modal)** | ปุ่มยืนยันเป็นสีจาง กดไม่ได้ + ข้อความเตือนสีเหลืองอำพันตัวจิ๋วเหนือปุ่ม (แสดงตั้งแต่เปิด modal) | `"ต้องแนบรูปก่อนจึงจะยืนยันได้"` |
| **ไม่มีสิทธิ์ (permission-denied)** | **ไม่มีหน้าจอให้เห็นเลย** — ระบบเด้ง (redirect) ไปหน้าแรกของ role ตัวเองทันทีแบบแทนที่ประวัติ. ไม่มีข้อความ ไม่มี toast ไม่มีหน้า 403. ปลายทางตาม role: admin→หน้าออเดอร์ / stock→หน้าแค็ตตาล็อก / rider→หน้าไรเดอร์ / อื่นๆ→หน้าพนักงาน. **แท็บ `"🎁 แพ็ค"` ก็ไม่แสดงในเมนูตั้งแต่แรกสำหรับคนไม่มีสิทธิ์** | — (ไม่มีข้อความใดๆ) |
| **ยังไม่ผูก LINE** | แถบเตือนสีเหลืองเหนือเนื้อหา (ระดับ Shell ไม่ใช่ของหน้านี้) | `"⚠️ ยังไม่ได้ผูก LINE — เพิ่ม PKM Shop OA เป็นเพื่อน แล้วพิมพ์รหัส {link_code} เพื่อรับแจ้งเตือนงาน"` |

**ลำดับความสำคัญของสถานะ (ตายตัว — ห้ามสลับ)**
`ผิดพลาด` → `กำลังโหลด` → `ว่างทั้งหน้า` → `เนื้อหาจริง`

> ผลที่ตามมาที่ designer ต้องรู้: **"ว่าง" ตัดสินจากออเดอร์ทั้งหน้ารวมกัน ไม่ใช่แยกเซคชัน** — ถ้ามีออเดอร์อย่างน้อย 1 ใบ **จะเห็นทั้งสองหัวข้อเซคชันเสมอ** แม้เซคชันใดเซคชันหนึ่งจะว่าง (แล้วจะเห็นข้อความ empty บรรทัดเดียวของเซคชันนั้นแทน)

## 5. ข้อความไทยทั้งหมดบนหน้านี้

> ⚠️ **ทุกข้อความในตารางนี้คือ copy จริงจากระบบ — designer ห้ามคิดข้อความไทยขึ้นเองเด็ดขาด** ถ้าจำเป็นต้องมีข้อความใหม่ ต้องถามเจ้าของก่อน

### หัวเรื่องและเซคชัน
| ที่ใช้ | ข้อความจริง (verbatim) |
|---|---|
| หัวข้อหน้า | `คิวแพ็คของ` |
| ป้ายมุมขวาบนของหัวเรื่อง (ไม่ใช่ปุ่ม) | `อัปเดตทุก 15 วิ` |
| หัวข้อเซคชัน 1 | `งานของฉัน` |
| หัวข้อเซคชัน 2 | `รอรับงาน` |
| บรรทัดท้ายสุด (ไดนามิก) | `อีก {N} ออเดอร์กำลังแพ็คโดยเพื่อนร่วมทีม` — เช่น `อีก 3 ออเดอร์กำลังแพ็คโดยเพื่อนร่วมทีม` |

### ปุ่ม
| ที่ใช้ | ข้อความจริง (verbatim) |
|---|---|
| ปุ่มบนการ์ดในเซคชัน `รอรับงาน` (ปกติ) | `รับงานแพ็ค` |
| ปุ่มเดียวกัน (กำลังส่ง, กดไม่ได้) | `กำลังรับงาน…` |
| ปุ่มบนการ์ดในเซคชัน `งานของฉัน` | `📷 แพ็คเสร็จ (ถ่ายรูป)` |
| ปุ่มยืนยันใน modal (ปกติ) | `ยืนยันแพ็คเสร็จ` |
| ปุ่มเดียวกัน (กำลังส่ง, กดไม่ได้) | `กำลังส่ง…` |
| ปุ่มถ่ายรูปซ้ำ (ทับบนพรีวิว) | `ถ่ายใหม่` |
| ปุ่มลองโหลดใหม่ในบล็อกผิดพลาด | `ลองใหม่` |
| ปุ่มปิด modal (สัญลักษณ์) | `✕` — screen reader อ่านว่า `ปิด` |
| ปุ่มออกจากระบบ (ระดับ Shell) | `ออก` |

> **หมายเหตุตัวอักษร**: ปุ่มสถานะกำลังส่งทั้งหมดใช้อักขระ ellipsis `…` (ตัวเดียว) **ไม่ใช่จุดสามจุด `...`** — ต้องคงไว้

### ป้าย / badge บนการ์ด
| ที่ใช้ | ข้อความจริง (verbatim) |
|---|---|
| badge รอบส่ง (เฉพาะออเดอร์ที่มีรอบ) | `⏱ รอบ {HH:mm}` — เช่น `⏱ รอบ 14:30` |
| badge สถานะ เมื่อยังไม่มีใครรับ | `รอแพ็ค` |
| badge สถานะ เมื่อรับงานแล้ว | `กำลังแพ็ค` |
| badge นับจำนวน (สองเซคชัน) | ตัวเลขล้วน เช่น `3` — **ซ่อนทั้งอันเมื่อเป็น 0** |

### ข้อมูลบนการ์ด
| ที่ใช้ | ข้อความจริง (verbatim) |
|---|---|
| ชื่อผู้รับ เมื่อไม่มีข้อมูล (fallback) | `ลูกค้า` |
| ชื่อสินค้า เมื่อดึงชื่อไม่ได้ (fallback) | `สินค้า` |
| จำนวนต่อรายการ | `× {qty}` — เช่น `× 3` (ใช้อักขระคูณ `×` ไม่ใช่ตัว x) |
| แถวสรุปท้ายการ์ด | `{ชื่อผู้รับ} · {ยอดเงิน} · รวม {N} ชิ้น` — เช่น `คุณสมชาย · ฿1,250 · รวม 7 ชิ้น` |

### Empty state
| ที่ใช้ | ข้อความจริง (verbatim) |
|---|---|
| ไอคอน empty ทั้งหน้า | `🎁` |
| หัวข้อ empty ทั้งหน้า | `ไม่มีของต้องแพ็คตอนนี้` |
| คำใบ้ empty ทั้งหน้า | `งานจะเข้าคิวเมื่อรอบถูกปิด (นาทีที่ :30)` |
| empty เซคชัน `งานของฉัน` | `ยังไม่มีงานที่รับไว้ — กด "รับงานแพ็ค" ด้านล่าง` |
| empty เซคชัน `รอรับงาน` | `ไม่มีงานรอรับตอนนี้` |
| การ์ดไม่มีรายการสินค้า (แถบเหลือง) | `ไม่พบรายการสินค้าของออเดอร์นี้ — เช็คกับแอดมินก่อนแพ็ค` |

### Modal
| ที่ใช้ | ข้อความจริง (verbatim) |
|---|---|
| หัวข้อ modal (ไดนามิก) | `แพ็คเสร็จ · {order_no}` — เช่น `แพ็คเสร็จ · ORD-2601-0042` |
| คำอธิบายใต้หัวข้อ | `เช็ครายการให้ครบ แล้วถ่ายรูปของที่แพ็คเสร็จเพื่อยืนยัน` |
| ป้ายในกล่องถ่ายรูป (ว่าง) | `ถ่ายรูปของที่แพ็คเสร็จ (บังคับ)` — ฐานคือ `ถ่ายรูปของที่แพ็คเสร็จ` + ` (บังคับ)` ต่อท้ายอัตโนมัติ |
| ไอคอนในกล่องถ่ายรูป | `📷` |
| คำอธิบายรูปพรีวิว (screen reader) | `ตัวอย่างรูป` |
| ข้อความเตือนเมื่อยังไม่มีรูป | `ต้องแนบรูปก่อนจึงจะยืนยันได้` |

### Toast — สำเร็จ (สีเขียว)
| ที่ใช้ | ข้อความจริง (verbatim) |
|---|---|
| รับงานสำเร็จ | `รับงานแพ็คแล้ว เริ่มแพ็คได้เลย` |
| ปิดงานแพ็คสำเร็จ | `แพ็คเสร็จ แจ้งลูกค้าแล้ว` |

### Toast — ผิดพลาด (สีแดง)
| ที่ใช้ | ข้อความจริง (verbatim) |
|---|---|
| รับงาน: ถูกชิงไปก่อน (ที่แสดงจริงบนหน้าจอ) | `มีคนรับงานนี้แล้ว` |
| รับงาน: ข้อความดิบจากหลังบ้าน (กรณีชิงงาน) | `ออเดอร์นี้มีคนรับแพ็คแล้ว` *(ถูกย่อเป็นข้อความข้างบนก่อนแสดง)* |
| รับงาน: สถานะเปลี่ยนไปแล้ว | `ออเดอร์นี้ไม่อยู่ในสถานะรอแพ็ค` |
| รับงาน: หาออเดอร์ไม่เจอ | `Order not found.` *(อังกฤษ — บั๊กภาษา)* |
| รับงาน: fallback | `รับงานไม่สำเร็จ` |
| ปิดงาน: คนอื่นถือครองอยู่ | `ออเดอร์นี้อยู่ในมือแพ็คเกอร์คนอื่น` |
| ปิดงาน: path รูปผิด | `Invalid photo path.` *(อังกฤษ)* |
| ปิดงาน: เปลี่ยนสถานะไม่ได้ | `Pack transition failed.` *(อังกฤษ)* |
| ปิดงาน: fallback | `แพ็คไม่สำเร็จ` |
| เรียกฟังก์ชันไม่ได้: fallback ระดับระบบ | `ทำรายการไม่สำเร็จ` |

### บล็อกสถานะระบบ
| ที่ใช้ | ข้อความจริง (verbatim) |
|---|---|
| ข้อความคู่วงกลมหมุน | `กำลังโหลด…` |
| ไอคอนบล็อกผิดพลาด | `⚠️` |
| หัวข้อบล็อกผิดพลาด | `โหลดข้อมูลไม่สำเร็จ` |
| คำอธิบายบล็อกผิดพลาด | `อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ` |

### ระดับ Shell (กรอบนอก — มีทุกหน้า)
| ที่ใช้ | ข้อความจริง (verbatim) |
|---|---|
| ชื่อระบบ | `PKM Shop` |
| คำกำกับใต้ชื่อระบบ | `หลังบ้าน` |
| ชื่อผู้ใช้ fallback | `พนักงาน` |
| แท็บของหน้านี้ | `🎁 แพ็ค` (ไอคอน `🎁` + ป้าย `แพ็ค`) |
| แถบเตือนยังไม่ผูก LINE | `⚠️ ยังไม่ได้ผูก LINE — เพิ่ม PKM Shop OA เป็นเพื่อน แล้วพิมพ์รหัส {link_code} เพื่อรับแจ้งเตือนงาน` |

## 6. ข้อมูลที่แสดง

| ฟิลด์ | รูปแบบการแสดงผล | ตัวอย่างจริง | หมายเหตุ |
|---|---|---|---|
| `order_no` — เลขออเดอร์ | ตัวอักษร **monospace** ขนาดเล็ก ตัวหนา | `ORD-2601-0042` | ใช้ monospace เพื่อให้อ่านตัวเลขทีละตัวได้ง่ายเวลาเทียบกับใบปะหน้า — **อย่าเปลี่ยนเป็นฟอนต์ปกติ** |
| `round.round_at` — เวลารอบส่ง | `⏱ รอบ HH:mm` — **24 ชั่วโมง โซนเวลา Asia/Bangkok เสมอ** (ไม่ว่าเครื่องผู้ใช้ตั้งโซนอะไร) เลขชั่วโมง/นาทีเติม 0 นำหน้าเสมอ | `⏱ รอบ 14:30`, `⏱ รอบ 09:30` | **แสดงเฉพาะออเดอร์ที่ผูกกับรอบ** — ออเดอร์ Grab/Lalamove/Kerry ไม่มี badge นี้ |
| `status` — สถานะ | ข้อความไทยบน badge เทา | `รอแพ็ค` / `กำลังแพ็ค` | แปลงจากค่า `confirmed` → `"รอแพ็ค"`, `packing` → `"กำลังแพ็ค"` **เฉพาะในหน้านี้** (ดู §8 เรื่องคำศัพท์ไม่ตรงกัน) |
| `products.name` — ชื่อสินค้า | ตัวอักษรขนาดปกติ น้ำหนักกลาง — **ขีดฆ่า + เทาอ่อน เมื่อติ๊กแล้ว** | `นมสดพาสเจอไรซ์ 2 ลิตร` | ถ้าดึงชื่อไม่ได้ → `สินค้า` |
| `order_items.qty` — จำนวนต่อรายการ | `× {จำนวน}` ตัวหนา สีแบรนด์ (เทาอ่อนเมื่อติ๊กแล้ว) | `× 3` | ใช้อักขระ `×` |
| `recipient_name` — ชื่อผู้รับ | ข้อความจิ๋วสีเทา | `คุณสมชาย` | ถ้าว่าง → `ลูกค้า` |
| `grand_total` — ยอดรวม | `฿` ติดกับตัวเลข **มีคอมมาคั่นหลักพัน ไม่มีทศนิยม** (จัดรูปแบบไทย) | `฿1,250` / `฿250` / `฿12,500` | สัญลักษณ์บาทนำหน้าเสมอ ไม่มีเว้นวรรค |
| ยอดรวมชิ้น (คำนวณ) | `รวม {ผลรวม qty ทุกรายการ} ชิ้น` | `รวม 7 ชิ้น` | **ไม่แสดงเลยถ้าไม่มีรายการสินค้า** |
| จำนวนงานในเซคชัน (คำนวณ) | ตัวเลขล้วนใน badge | `3` | ซ่อนเมื่อ 0 |
| จำนวนงานเพื่อน (คำนวณ) | ฝังในประโยค | `อีก 3 ออเดอร์กำลังแพ็คโดยเพื่อนร่วมทีม` | แอดมินไม่เห็นบรรทัดนี้ |

**การเรียงลำดับ**: เรียงตามเวลาที่สร้างออเดอร์ **เก่า → ใหม่ (FIFO) ตายตัว** ทั้งสองเซคชัน — ไม่มีตัวเลือกให้ผู้ใช้เปลี่ยน

**ข้อมูลที่ระบบดึงมาแล้วแต่ยังไม่ได้แสดง** (พร้อมใช้ทันทีถ้าดีไซน์ต้องการ ไม่ต้องแก้อะไรฝั่งข้อมูล):
เบอร์โทรผู้รับ, ที่อยู่, **ประเภทการส่ง** (มีคำไทยพร้อมอยู่แล้ว: `ส่งปกติ (ไรเดอร์)` / `ด่วน (Grab)` / `นอกเขต (Lalamove)` / `พัสดุ (Kerry)`), ลำดับจุดส่ง, เลขอ้างอิงภายนอก, สถานะชำระเงิน, ค่าสินค้า, ค่าส่ง

## 7. ข้อมูลอัปเดตยังไง

**กลไก: Polling ทุก 15 วินาที — ไม่มี realtime subscription เลย**

- ระบบดึงคิวใหม่อัตโนมัติ **ทุก 15 วินาที** ตลอดเวลาที่เปิดหน้าอยู่
- มี **ตัวบอกผู้ใช้อยู่แล้ว**: ป้ายข้อความ `"อัปเดตทุก 15 วิ"` มุมขวาบนของหัวเรื่อง
  > **ข้อจำกัดสำคัญ**: ป้ายนี้บอกแค่ *ความถี่* ไม่ได้บอกว่า **อัปเดตล่าสุดเมื่อไหร่** และ **ไม่มีตัวบ่งชี้ว่าตอนนี้กำลังดึงข้อมูลอยู่**. ระบบไม่ได้เก็บ timestamp ของการดึงล่าสุดไว้แสดง — **ถ้าดีไซน์อยากได้ "อัปเดตล่าสุด 14:28" หรือ pull-to-refresh ต้องแจ้งว่าเป็นของใหม่ที่ต้องเพิ่มความสามารถ**
- **การรีเฟรชอัตโนมัติเงียบสนิท** — ไม่มี spinner ไม่มีการกระพริบ ไม่มี skeleton (เพราะมีข้อมูลใน cache แล้ว)
- นอกจาก polling ยังมีการดึงใหม่แบบสั่งเอง 3 จุด:
  1. **หลังกดรับงาน** — ดึงใหม่เสมอ ทั้งกรณีสำเร็จและล้มเหลว
  2. **หลังปิดงานแพ็คสำเร็จ** — **ไม่ดึงใหม่เมื่อล้มเหลว**
  3. **กดปุ่ม `"ลองใหม่"`** ในบล็อกผิดพลาด
- **คิวเดียวเลี้ยงทั้งหน้า** — การแบ่งเป็น 3 กลุ่ม (งานของฉัน / รอรับงาน / งานของเพื่อน) ทำในเครื่องผู้ใช้ทั้งหมด จากรายการเดียว

**ผลต่อดีไซน์ — สำคัญมาก**
1. **การ์ดจะ "โผล่มาเอง" และ "หายไปเอง" ระหว่างที่แพ็คเกอร์กำลังจ้องหน้าจออยู่** โดย **ไม่มี animation ไม่มี transition ไม่มีเสียง ไม่มี vibration ไม่มี toast บอก** — เป็นการตัดฉับล้วนๆ
2. **จังหวะที่รุนแรงที่สุดคือนาที :30** — รอบถูกล็อก แล้ว **ออเดอร์ไรเดอร์ไหลเข้าคิวพรวดเดียวหลายใบ** จากคิวที่อาจดูว่างอยู่เมื่อ 15 วินาทีก่อน
3. **การถูกชิงงานเงียบ**: ถ้าเพื่อนร่วมทีมกดรับก่อน การ์ดจะหายจาก `"รอรับงาน"` ภายใน 15 วินาที **โดยไม่มีคำอธิบายใดๆ** — ผู้ใช้อาจกำลังจะแตะปุ่มพอดี แล้วปุ่มหายไปใต้นิ้ว (**จุดเจ็บจริง ควรออกแบบแก้**)
4. **การแจ้งเตือนงานใหม่ไม่ได้อยู่บนหน้านี้** — ไปทาง LINE OA แทน (จึงมีแถบเตือนให้ผูก LINE ที่ระดับ Shell)
5. **หน่วงสูงสุด 15 วินาที** — ห้ามออกแบบโดยสมมติว่าตัวเลข badge อัปเดตทันที

## 8. สิ่งที่ designer ต้องรู้

### 🔴 ห้ามแตะ — ผูกกับกติกาธุรกิจ (`Ready.md` §3.1, §6) ถ้าจะเปลี่ยนต้องถามเจ้าของก่อน

- **บังคับถ่ายรูปก่อนปิดงาน = hard gate ห้ามผ่อน** — ปุ่ม `"ยืนยันแพ็คเสร็จ"` ต้องกดไม่ได้จนกว่าจะมีรูป และ `"ต้องแนบรูปก่อนจึงจะยืนยันได้"` ต้องโชว์ตั้งแต่เปิด modal ไม่ใช่รอให้กดผิดก่อน. **รูปนี้ไม่ใช่หลักฐานภายใน — ลูกค้าเห็นทาง LINE** (ระบบส่งรูปให้ลูกค้าทันทีที่ปิดงาน)
- **ลำดับเซคชัน `งานของฉัน` เหนือ `รอรับงาน` เสมอ** — งานค้างในมือต้องปิดก่อนไปหยิบงานใหม่ ห้ามสลับ ห้ามเอาไปไว้ในแท็บแยก
- **น้ำหนักสายตาไล่ระดับตามความสำคัญ ห้ามทำให้เท่ากัน**: `งานของฉัน` = badge สีแบรนด์ตัวอักษรขาว (เด่นสุด) → `รอรับงาน` = badge เทาอ่อน (รองลงมา) → `งานเพื่อน` = ข้อความเทาอ่อนบรรทัดเดียว ไม่มีการ์ด (เบาสุด — **ตั้งใจไม่ให้เด่นเพราะไม่อยากยั่วให้ไปแย่งงานคนอื่น**)
- **ออเดอร์ไรเดอร์ที่รอบยังเปิดต้องไม่โผล่ในคิว** — ต้องซ่อนทั้งใบ **ห้ามเปลี่ยนเป็นแสดงแบบปุ่มเทา/greyed out** เพราะกติกาคือแพ็คเกอร์ต้องไม่แตะออเดอร์ที่ลูกค้ายังแก้ได้อยู่
- **ไม่มีทางคืนงาน (un-claim)** — เมื่อกด `"รับงานแพ็ค"` แล้ว ออเดอร์ติดมือถาวรจนกว่าจะปิดด้วยรูป. **การปิด modal ไม่ใช่การคืนงาน**. อย่าออกแบบปุ่ม "ยกเลิกงาน"/"ส่งต่อ" ขึ้นมาเอง — เป็นกติกาที่ต้องถามเจ้าของ
- **ข้อจำกัดเวลา ~30 นาทีหลังรอบล็อก คือเหตุผลของทุกการตัดสินใจในหน้านี้** — การ์ดต้องอ่านจบใน 2–3 วินาที **ห้ามเพิ่มขั้นตอน ห้ามเพิ่ม chrome ที่ต้องแตะเพิ่ม** เจตนาเดิมคือ "ไม่ต้องเปิดหน้าอื่น"
- **เป้ากดขนาดใหญ่ห้ามลด** — แถวเช็คลิสต์ ≥48px, กล่องถ่ายรูป ≥96px, ปุ่ม action เต็มความกว้างทุกตัว. เหตุผล: แพ็คเกอร์ยืนใช้มือเดียวขณะถือของ
- **กล้องต้องเป็นกล้องหลังและเปิดทันที** — ไม่ผ่านแกลเลอรี่

### 🟡 ปรับได้ / ควรปรับปรุง — จุดที่ดีไซน์ปัจจุบันอ่อน

- **ไม่มี progress bar ตอนอัปโหลดรูป** ⭐ **จุดเจ็บที่สุด** — รูปดิบจากกล้อง (อาจหลาย MB) ถูกอัปโหลดโดย **ไม่ถูกย่อขนาดเลย** (ต่างจากหน้าแค็ตตาล็อกที่ย่อเหลือ 1200px). บนเน็ตร้านที่ช้า ช่วง `"กำลังส่ง…"` อาจกินเวลาหลายสิบวินาที **โดยผู้ใช้ไม่มีทางรู้ว่าค้างหรือกำลังไป** และ **modal ก็ปิดไม่ได้ด้วย** → ควรออกแบบ progress indicator หรือ optimistic feedback
- **เช็คลิสต์รีเซ็ตเมื่อเปิด modal** ⭐ **กับดักที่ผู้ใช้เจอจริง** — สถานะติ๊กเก็บในเครื่องล้วน ไม่บันทึก ไม่ส่งเซิร์ฟเวอร์ และ **เช็คลิสต์บนการ์ดกับในการ modal เป็นคนละชุด** → แพ็คเกอร์ติ๊กครบบนการ์ด พอเปิด modal เจอว่างเปล่า แล้วงง. **ถ้าจะแก้พฤติกรรมนี้ต้องยกขึ้นคุยกับเจ้าของก่อน** (อย่าแก้เอง — ดู `AGENTS.md` / `Ready.md`)
- **การ์ดโผล่/หายแบบตัดฉับ ไม่มี animation** — โดยเฉพาะตอนนาที :30 ที่งานไหลเข้าพรวดเดียว และตอนถูกชิงงาน. **ควรมี transition หรือตัวบ่งชี้ "มีงานใหม่เข้า N ใบ"** (แต่กติกาข้อ "ไม่ยั่วให้แย่งงาน" ยังต้องเคารพ)
- **ไม่มีตัวบอกว่าอัปเดตล่าสุดเมื่อไหร่** — มีแค่ป้าย `"อัปเดตทุก 15 วิ"` ที่บอกความถี่ ไม่มี timestamp ไม่มี pull-to-refresh
- **ไม่มีนับถอยหลังถึงรอบถัดไป** — คิวจะ "ดูว่าง" ทั้งที่มีออเดอร์รออยู่จริง จนถึงนาที :30. คำอธิบายเดียวที่ผู้ใช้ได้คือคำใบ้ใน empty state `"งานจะเข้าคิวเมื่อรอบถูกปิด (นาทีที่ :30)"` — **ตัวนับถอยหลังจะช่วยมาก แต่ต้องถามเจ้าของเพราะแตะ business rule**
- **หัวข้อเซคชันไม่ sticky** — เลื่อนหายไปกับเนื้อหา ถ้ารอบหนึ่งมีหลายสิบออเดอร์จะเป็น long scroll ล้วนๆ ไม่มี filter/search/sort/pagination. **sticky section header หรือ jump-to เป็นจุดที่ปรับปรุงได้**
- **กด "ลองใหม่" แล้วไม่มี feedback** — บล็อกผิดพลาดมาก่อนบล็อกกำลังโหลดในลำดับ ผู้ใช้จึงยังเห็นหน้าจอเดิมเป๊ะ ไม่รู้ว่าปุ่มทำงานหรือเปล่า
- **ปิด modal = รูปหายทันที ไม่มีถามยืนยัน** — ถ่ายรูปเสร็จแล้วเผลอแตะฉากหลัง = ต้องถ่ายใหม่. **ระบบมี confirm dialog ให้ใช้อยู่แล้วแต่หน้านี้ไม่ได้เรียกเลย** — เป็นจุดที่เพิ่มได้
- **ดีไซน์ไม่ได้บอกเลยว่ารูปนี้ลูกค้าจะเห็น** — ควรสื่อให้ชัด เพราะมีผลต่อความใส่ใจในการถ่าย
- **ข้อความ error ภาษาอังกฤษหลุดมาบางกรณี** (`Order not found.`, `Invalid photo path.`, `Pack transition failed.` และ error จากระบบเก็บไฟล์) — **toast ต้องออกแบบให้รองรับข้อความยาวหลายบรรทัดและภาษาอังกฤษ**
- **การ์ดโชว์ข้อมูลน้อยกว่าที่มี** — ประเภทการส่ง, ที่อยู่, เบอร์โทร, ลำดับจุดส่ง ดึงมาแล้วแต่ไม่แสดง. **ถ้าอยากโชว์ประเภทการส่ง คำไทยพร้อมอยู่แล้ว ไม่ต้องแก้อะไรฝั่งข้อมูล**

### ⚪ ข้อเท็จจริงอื่นที่ต้องรู้ก่อนออกแบบ

- **แอดมินเห็นหน้านี้ต่างจากแพ็คเกอร์**: ออเดอร์ที่ `packing` ของ **ทุกคน** จะถูกรวมเข้าเซคชัน `"งานของฉัน"` (ซึ่ง**ชื่อกลายเป็นคำที่ผิดสำหรับแอดมิน**) และแอดมิน **ไม่เห็นบรรทัด `"อีก N ออเดอร์กำลังแพ็คโดยเพื่อนร่วมทีม"` เลย**. ถ้าจะแยกมุมมองแอดมิน นี่คือจุดที่ต้องคิด (และต้องถามเจ้าของ)
- **คำศัพท์ไม่ตรงกันข้ามหน้า**: หน้านี้เรียกสถานะ `confirmed` ว่า `"รอแพ็ค"` แต่หน้าอื่นในระบบเรียกว่า `"เข้ารอบ"`. คำว่า `"รอแพ็ค"` เหมาะกับบริบทแพ็คเกอร์มากกว่า — **ถ้าจะรวมภาษาให้ตรงกันทั้งระบบต้องถามเจ้าของก่อน**
- **สีมีความหมายชัดเจน ควรรักษาระบบนี้**:
  - **สีแบรนด์** = ปุ่มหลัก, badge รอบส่ง (พื้นจาง+ตัวอักษรเข้ม), badge นับงานของฉัน, ตัวเลขจำนวนที่ยังไม่ติ๊ก
  - **เขียว** = ติ๊กแล้ว + toast สำเร็จ
  - **เหลืองอำพัน** = คำเตือนที่**ไม่ใช่ error** (ไม่พบรายการสินค้า, ต้องแนบรูป, ยังไม่ผูก LINE)
  - **แดง** = toast error เท่านั้น
  - **เทา** = ทุกอย่างที่เหลือ + สถานะรอง
- **บล็อกผิดพลาดต้องแยกจาก empty state ให้ชัดโดยตั้งใจ** — ผู้ใช้ต้องไม่เข้าใจผิดว่า "ไม่มีข้อมูล" ทั้งที่จริงคือ "โหลดไม่ได้"
- **Busy เป็นรายใบ ไม่ล็อกทั้งหน้า** — ระหว่างกำลังรับงานใบหนึ่ง ปุ่มของใบอื่นยังกดได้ (ตั้งใจ — คิวไม่ควรหยุดทั้งคิวเพราะรอ 1 คำสั่ง)
- **การ์ดไม่มีรายการสินค้ายังกดปุ่มได้ปกติ** — แถบเตือนไม่บล็อกอะไร และ **ไม่มีลิงก์ไปไหนเลย** ทางออกเดียวคือข้อความ `"เช็คกับแอดมินก่อนแพ็ค"` (ไม่มีปุ่มแจ้งปัญหา/ติดต่อแอดมินบนหน้านี้)
- **ไม่มี GPS ไม่มีแผนที่ ไม่มีสแกนบาร์โค้ด ไม่มีเสียง/vibration** (ต่างจากหน้าไรเดอร์) — อุปกรณ์เดียวที่ใช้คือกล้อง
- **Accessibility ปัจจุบันมีช่องโหว่ที่ควรแก้ในการ redesign**:
  - แถวเช็คลิสต์เป็นปุ่มธรรมดา **ไม่สื่อสถานะติ๊ก/ไม่ติ๊กให้ screen reader เลย** (ควรเป็น checkbox หรือมี pressed state)
  - เครื่องหมาย ✓ ตอนยังไม่ติ๊กใช้วิธี**ทำให้โปร่งใส** — มีอยู่ใน DOM แต่มองไม่เห็น
  - Modal **ไม่ล็อก focus, ปิดด้วย Escape ไม่ได้, ไม่ประกาศตัวเป็น dialog** — ปุ่มปิดมีแค่ป้ายเสียง `"ปิด"`
  - **Toast ไม่ถูกอ่านออกเสียง** — ข้อความสำเร็จ/ผิดพลาดหายไปสำหรับผู้ใช้ screen reader
- **สิ่งที่ระบบมีให้ใช้แต่หน้านี้ไม่ได้ใช้เลย** (พร้อมใช้ถ้าดีไซน์ต้องการ): แท็บ (Tabs), ฟิลด์ฟอร์ม (Field), สเต็ปเปอร์ (Stepper), การ์ดตัวเลขสถิติ (StatCard), กล่องยืนยัน (confirm dialog)
