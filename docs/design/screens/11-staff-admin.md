# ทีมงาน (`/admin/staff`)

> แอดมินใช้หน้านี้เพื่อจัดการ "คน" และ "กะการทำงาน" ของร้าน — ดูรายชื่อพนักงาน สร้างบัญชีเข้าระบบใหม่ แจกรหัสผูก LINE แก้หน้าที่ เปิด/ปิดใช้งาน และเพิ่ม/แก้กะ

---

## 1. บริบทการใช้งาน

**ใครใช้**
- **แอดมินเท่านั้น** เส้นทาง `/admin/staff` ถูกครอบด้วยการ์ดตรวจสิทธิ์ (Guard) ที่ต้องการบทบาท `admin` เท่านั้น ผู้ใช้ที่ไม่ใช่แอดมินจะถูกดีดออกไปหน้าแรกตามบทบาทของตัวเองทันที (บทบาท `stock` → หน้าแคตตาล็อก, `packer` → หน้าแพ็ค, `rider` → หน้าไรเดอร์, บทบาทอื่น → หน้าพนักงานทั่วไป) — **หมายความว่าไม่มี "หน้าจอ permission-denied" ในหน้านี้เลย ผู้ใช้ที่ไม่มีสิทธิ์จะไม่มีวันเห็นหน้านี้ ไม่ต้องออกแบบสถานะนั้น**
- ฝั่งเซิร์ฟเวอร์ตรวจซ้ำอีกชั้นทุกครั้งที่กดปุ่มใดๆ (ทุก action ยกเว้นการผูกบัญชีตัวเอง ต้องเป็นแอดมิน) ดังนั้นถึงจะแฮกเข้ามาก็ทำอะไรไม่ได้

**อุปกรณ์และสถานการณ์จริง**
- **Mobile-first** — ทั้งระบบถูกวางไว้เป็น mobile-first อย่างชัดเจน (ระบบดีไซน์ระบุว่า toast ต้องลอยอยู่เหนือแถบแท็บล่าง และ modal ต้องเป็น bottom sheet บนมือถือ) หน้านี้ต้องเดินตามกฎเดียวกัน
- **แต่หน้านี้ไม่ใช่งานภาคสนาม** — ต่างจากหน้าไรเดอร์/แพ็คตรงที่ไม่มีการถ่ายรูป ไม่มี GPS ไม่มีสแกน ไม่ใช่งานมือเดียวกลางแดด สถานการณ์จริงคือ **แอดมิน/เจ้าของร้านนั่งอยู่ในร้าน ถือมือถือหรือแท็บเล็ต** ตอนรับพนักงานใหม่เข้าทีม หรือตอนตั้งค่ากะรอบเปิดร้าน
- **ความถี่: ต่ำมาก** — เป็นงานตั้งค่า ไม่ใช่งานประจำวัน อาจใช้เดือนละครั้งตอนมีพนักงานใหม่ หรือปีละครั้งตอนปรับกะ
- **ความเร่งรีบ: ต่ำ–ปานกลาง** — จังหวะที่เร่งที่สุดคือตอนพนักงานใหม่ยืนรออยู่ตรงหน้า แอดมินต้องสร้างบัญชี + อ่านรหัสผูก LINE ให้พนักงานพิมพ์ลงมือถือตัวเอง ณ ตรงนั้นเลย → **ช่วงที่ต้อง "อ่านออกจากระยะไกล/แชร์จอกันดู" คือกล่องรหัสผูก LINE**
- สรุป: **Mobile-first แต่ไม่ต้องบีบสุดโต่ง** ควรออกแบบให้อ่านสบายบนมือถือ และขยายได้ดีบนแท็บเล็ต/จอใหญ่ (พื้นที่คอนเทนต์กว้างสุดประมาณจอกลาง มีระยะขอบรอบด้าน)

## 2. โครงหน้า

**ไม่มีแท็บใดๆ ในหน้านี้** — เป็นสกรอลล์เดียวจากบนลงล่าง แบ่งเป็น 2 บล็อกใหญ่ที่มีสถานะแยกกันอิสระ (พังทีละบล็อกได้)

ลำดับจากบนลงล่าง:

1. **แถบเตือนของ Shell (อาจมีหรือไม่มี)** — ถ้าตัวแอดมินเองยังไม่ผูก LINE จะมีแถบสีเหลืองลอยอยู่เหนือทุกหน้ารวมถึงหน้านี้ ข้อความ: `"⚠️ ยังไม่ได้ผูก LINE — เพิ่ม PKM Shop OA เป็นเพื่อน แล้วพิมพ์รหัส {link_code} เพื่อรับแจ้งเตือนงาน"` (ไม่ใช่ของหน้านี้โดยตรง แต่ดีไซเนอร์ต้องเผื่อพื้นที่)

2. **หัวข้อหน้า "ทีมงาน"** — แถวเดียว จัดชิดซ้าย–ขวา
   - ซ้าย: หัวข้อ `"ทีมงาน"` ตัวหนา ขนาดใหญ่กว่าตัวปกติเล็กน้อย
   - ขวา: ปุ่มหลักขนาดเล็ก (สีแบรนด์ ตัวหนังสือขาว) `"+ เพิ่มพนักงาน"`

3. **รายการการ์ดพนักงาน** — การ์ดพื้นขาว ขอบมน เรียงต่อกันแนวตั้ง มีระยะห่างเล็กน้อยระหว่างใบ **1 ใบ = พนักงาน 1 คน** เรียงตามลำดับวันที่สร้างบัญชี (เก่า → ใหม่) **ไม่มีช่องค้นหา ไม่มีตัวกรอง ไม่มีการแบ่งหน้า**

   โครงในการ์ด 1 ใบ (บนลงล่าง):
   - **แถวบน** — ซ้าย/ขวา:
     - ซ้าย: **ชื่อพนักงาน** ตัวหนาขนาดเล็ก ตัดด้วย ellipsis ถ้ายาว (ถ้าไม่มีชื่อ แสดงขีดกลาง `"—"`) ตามด้วยป้ายสถานะที่อาจมีหรือไม่มี:
       - ป้ายเทา `"ปิดใช้งาน"` — โผล่เฉพาะเมื่อพนักงานถูกปิดใช้งาน
       - ป้ายเหลือง `"ยังไม่มีบัญชีเข้าระบบ"` — โผล่เฉพาะเมื่อยังไม่มีบัญชีล็อกอินผูกอยู่
       - ป้ายทั้งหมดขึ้นบรรทัดใหม่ได้ (wrap)
     - ขวา: **ปุ่มโปร่ง (ghost) ขนาดเล็ก** ข้อความ `"ปิด"` ถ้ากำลังใช้งานอยู่ / `"เปิด"` ถ้าถูกปิดใช้งานอยู่ — ปุ่มนี้ไม่ยุบตัว (shrink-0)
   - **แถวป้ายหน้าที่** — ป้ายสีแบรนด์อ่อน (พื้นแบรนด์โปร่ง ตัวหนังสือสีแบรนด์) **1 ป้ายต่อ 1 หน้าที่** ขึ้นบรรทัดใหม่ได้ ข้อความไทยจริง 5 แบบ: `"แอดมิน"` / `"สต็อก"` / `"แพ็ค"` / `"ไรเดอร์"` / `"ทั่วไป"` (คนหนึ่งมีได้หลายหน้าที่พร้อมกัน)
   - **บล็อกสถานะการผูก LINE** — มี 2 หน้าตาที่เกิดพร้อมกันไม่ได้:
     - (ก) **ผูกแล้ว** → บรรทัดข้อความสีเขียวเข้มขนาดเล็ก `"LINE: ผูกแล้ว ✅"` — ไม่มีปุ่ม ไม่มีกล่อง
     - (ข) **ยังไม่ผูก** → กล่องพื้นเหลืองอ่อน ขอบมน มีระยะขอบใน ประกอบด้วย ข้อความสีเหลืองเข้มขนาดเล็ก `"ยังไม่ผูก LINE"` + ถ้ามีรหัสผูกจะแสดง **รหัสในกล่องขาว ฟอนต์ monospace ตัวหนา มีเส้นขอบเหลืองบาง** + ปุ่มโปร่งขนาดเล็ก `"คัดลอก"`
     - **เคสที่ต้องรองรับ:** ถ้ายังไม่ผูกและ **ไม่มีรหัส** → เหลือแค่กล่องเหลืองที่มีข้อความ `"ยังไม่ผูก LINE"` ลอยอยู่ตัวเดียว ไม่มีรหัส ไม่มีปุ่ม
   - **ล่างสุด** — ปุ่มข้อความล้วนสีแบรนด์ `"แก้หน้าที่"` (ไม่มีพื้นหลัง ไม่มีขอบ)

   > **สำคัญ: การ์ดพนักงานทั้งใบ "ไม่ใช่" ปุ่ม** — ต้องกดปุ่มย่อยเฉพาะจุดเท่านั้น

4. **หัวข้อรอง "กะการทำงาน"** — ใช้คอมโพเนนต์หัวข้อ **ตัวเดียวกับหัวข้อหน้า** จึงออกมาขนาดและน้ำหนักเท่ากับ `"ทีมงาน"` เป๊ะ (นี่คือข้อบกพร่อง ดูข้อ 8)
   - ซ้าย: `"กะการทำงาน"`
   - ขวา: ปุ่มโปร่งขนาดเล็ก (ไม่ใช่ปุ่มหลัก) `"+ เพิ่มกะ"`

5. **รายการการ์ดกะ** — การ์ดพื้นขาว ขอบมน เรียงต่อกันแนวตั้ง เรียงตาม **เวลาเริ่มกะจากน้อยไปมาก**
   - **การ์ดทั้งใบเป็นปุ่มกดได้** — แตะตรงไหนก็เปิดหน้าต่างแก้ไข
   - ซ้าย: **ชื่อกะ** ตัวหนาขนาดเล็ก ตัดด้วย ellipsis + ป้ายเทา `"ปิดใช้งาน"` (เฉพาะเมื่อกะถูกปิด); บรรทัดล่าง = **ช่วงเวลา** ตัวเล็กสีเทา
   - ขวา: **ข้อความ** สีแบรนด์ `"แก้ไข"` — เป็นแค่ข้อความประกอบ ไม่ใช่ปุ่มจริง (แต่กดได้เพราะการ์ดทั้งใบเป็นปุ่ม)

## 3. ทุก flow ทีละสเต็ป

### 3.1 เพิ่มพนักงาน + สร้างบัญชีเข้าระบบ

**ขั้นตอน**
1. กดปุ่ม `"+ เพิ่มพนักงาน"` มุมขวาบนของหัวข้อ `"ทีมงาน"`
2. หน้าต่างเปิดขึ้น หัวข้อ `"เพิ่มพนักงาน + บัญชีเข้าระบบ"` — บนมือถือเลื่อนขึ้นมาจากขอบล่าง (bottom sheet มุมบนมน), บนจอใหญ่เป็นกล่องกลางจอ; มีปุ่ม `✕` มุมขวาบน (ป้ายสำหรับ screen reader = `"ปิด"`); แตะพื้นหลังดำโปร่ง = ปิด
3. กรอกฟอร์ม ทุกช่องมีป้ายชื่ออยู่ **ด้านบน** ช่อง (ตัวหนาขนาดเล็กสีเทาเข้ม) เรียงจากบนลงล่าง มีระยะห่างเท่ากัน:
   1. ป้าย `"ชื่อพนักงาน"` — ช่องข้อความธรรมดา **จำเป็น**
   2. ป้าย `"อีเมล (สำหรับเข้าสู่ระบบ)"` — ช่องอีเมล (เบราว์เซอร์ช่วยตรวจรูปแบบ) **จำเป็น**
   3. ป้าย `"รหัสผ่าน (อย่างน้อย 6 ตัว)"` — **ช่องข้อความธรรมดาที่มองเห็นตัวอักษร ไม่ใช่จุดๆ (ตั้งใจ)** placeholder `"ตั้งรหัสให้พนักงาน"` **จำเป็น**
   4. ป้าย `"เบอร์โทร (ไม่บังคับ)"` — ช่องข้อความที่เรียกแป้นตัวเลขโทรศัพท์ **ไม่จำเป็น**
   5. ป้าย `"หน้าที่"` — แถวชิปกลม (ขอบมนเต็ม มีเส้นขอบ) เรียงต่อกันขึ้นบรรทัดใหม่ได้ **5 ชิป**: `"แอดมิน"` / `"สต็อก"` / `"แพ็ค"` / `"ไรเดอร์"` / `"ทั่วไป"`
      - **เลือกได้หลายอัน** กดซ้ำเพื่อเอาออก
      - **ค่าเริ่มต้น: เลือก `"แพ็ค"` ไว้แล้ว 1 อัน**
      - ชิปที่เลือก = พื้นสีแบรนด์เต็ม ตัวหนังสือขาว ขอบสีแบรนด์ / ชิปที่ไม่เลือก = พื้นขาว ขอบเทาอ่อน
4. กดปุ่มหลักเต็มความกว้าง `"สร้างบัญชี"` ที่ล่างสุด (**หน้าต่างนี้ไม่มีปุ่ม "ยกเลิก"**)
5. ระหว่างส่ง ปุ่มเปลี่ยนข้อความเป็น `"กำลังสร้าง…"` และกดไม่ได้ (**ไม่มี spinner ในปุ่ม ไม่มีฉากทึบทับหน้าต่าง — ช่องกรอกยังพิมพ์แก้ได้ระหว่างรอ**)
6. สำเร็จ → toast เขียว → หน้าต่างนี้ปิด → รายชื่อรีเฟรชเอง → **หน้าต่างผลลัพธ์เปิดต่อทันที**
7. หน้าต่างผลลัพธ์ หัวข้อ `"สร้างบัญชีสำเร็จ"` มี **2 หน้าตา**:
   - (ก) **ได้รหัสผูกกลับมา** → ข้อความจัดกลาง `"ส่งรหัสนี้ให้พนักงานพิมพ์ใน LINE OA เพื่อผูกบัญชี"` + **กล่องรหัสตัวใหญ่มาก** (พื้นเทาอ่อน ขอบมน ฟอนต์ monospace ขนาดใหญ่ ตัวหนา ระยะห่างตัวอักษรกว้าง สีเทาเข้ม) + ปุ่มหลักเต็มความกว้าง `"คัดลอกรหัส"`
   - (ข) **ไม่ได้รหัสกลับมา** → ข้อความอย่างเดียว `"สร้างบัญชีแล้ว — ดูรหัสผูก LINE ได้ที่การ์ดพนักงานในหน้านี้"` (ไม่มีกล่องรหัส ไม่มีปุ่มคัดลอก)
8. ทั้งสองหน้าตามีปุ่มโปร่งเต็มความกว้าง `"ปิด"` ล่างสุด

**กฎการตรวจสอบ (validation)**
| ช่อง | กฎฝั่งหน้าจอ | กฎฝั่งเซิร์ฟเวอร์ (หน้าจอไม่กัน) |
|---|---|---|
| ชื่อพนักงาน | จำเป็น ห้ามเว้นว่าง/มีแต่ช่องว่าง | ยาวไม่เกิน 80 ตัวอักษร (หน้าจอไม่จำกัดความยาว → พิมพ์เกินได้แล้วค่อยโดนปฏิเสธ) |
| อีเมล | จำเป็น + รูปแบบอีเมลตามเบราว์เซอร์ | ต้องเป็นอีเมลที่ถูกต้อง; **อีเมลซ้ำจะรู้ตอนกดส่งเท่านั้น** ไม่มีการเช็คสดขณะพิมพ์ |
| รหัสผ่าน | จำเป็น **อย่างน้อย 6 ตัวอักษร** | ไม่เกิน 72 ตัวอักษร |
| เบอร์โทร | ไม่บังคับ **ไม่มีการตรวจรูปแบบใดๆ** | ยาวไม่เกิน 30 ตัวอักษร |
| หน้าที่ | **ต้องเลือกอย่างน้อย 1** | ต้องเลือกอย่างน้อย 1 |

- ไม่มีการอัปโหลดรูป
- **การตรวจทั้งหมดเกิดตอน "กดปุ่มส่ง" เท่านั้น** ไม่มีข้อความ error ใต้ช่องใดๆ — ทุกอย่างเด้งเป็น toast ก้อนเดียว

**ข้อความสำเร็จ (verbatim)**
- toast เขียว: `"สร้างบัญชีพนักงานแล้ว"`
- แล้วตามด้วยหน้าต่าง `"สร้างบัญชีสำเร็จ"`
- การ์ดพนักงานใบใหม่โผล่ **ท้ายรายการ**

**ข้อความล้มเหลว (verbatim)**
- ตรวจฝั่งหน้าจอไม่ผ่าน → toast แดง: `"กรอกชื่อ อีเมล รหัสผ่าน (≥6 ตัว) และหน้าที่อย่างน้อย 1 อย่าง"` (ไม่ยิงไปเซิร์ฟเวอร์เลย ปุ่มไม่เข้าสถานะกำลังส่ง)
- เซิร์ฟเวอร์ปฏิเสธ → toast แดง = ข้อความจริงจากเซิร์ฟเวอร์ เช่น `"สร้างบัญชีเข้าสู่ระบบไม่สำเร็จ"` — **หรืออาจหลุดเป็นข้อความภาษาอังกฤษจากระบบล็อกอิน เช่นกรณีอีเมลซ้ำ**
- fallback สุดท้าย → toast แดง: `"สร้างไม่สำเร็จ"`
- ทุกกรณีที่ล้มเหลว: **หน้าต่างไม่ปิด ข้อมูลที่กรอกยังอยู่ครบ ปุ่มกลับมากดได้**

---

### 3.2 แก้หน้าที่ของพนักงาน

**ขั้นตอน**
1. ที่การ์ดพนักงาน กดข้อความสีแบรนด์ `"แก้หน้าที่"` ที่ล่างสุดของการ์ด
2. หน้าต่างเปิดขึ้น **หัวข้อเป็นข้อความไดนามิก** รูปแบบ: `หน้าที่ของ {ชื่อพนักงาน}` — ตัวอย่างจริง: `"หน้าที่ของ สมชาย"`; **ถ้าพนักงานคนนั้นไม่มีชื่อ หัวข้อจะเป็น `"หน้าที่ของ พนักงาน"` เป๊ะๆ**
3. แสดง **รายการช่องติ๊ก (checkbox) 5 บรรทัดแนวตั้ง** — **ไม่ใช่ชิปแบบตอนเพิ่มพนักงาน!** แต่ละบรรทัดเป็นแถบเต็มความกว้างกดได้ทั้งแถว สูงอย่างน้อย 44px ขอบมน มีสถานะ hover พื้นเทาอ่อนมาก; ช่องติ๊กเป็นสี่เหลี่ยมขอบมนขนาดกลางอยู่ซ้าย ตามด้วยข้อความตัวหนาขนาดเล็ก: `"แอดมิน"` / `"สต็อก"` / `"แพ็ค"` / `"ไรเดอร์"` / `"ทั่วไป"`
4. **ค่าเริ่มต้น = หน้าที่ปัจจุบันของคนนั้นถูกติ๊กไว้แล้ว** ติ๊ก/เอาติ๊กออกได้อิสระทุกช่อง
5. แถวปุ่มล่าง 2 ปุ่มกว้างเท่ากัน: ปุ่มโปร่ง `"ยกเลิก"` (ปิดหน้าต่าง ทิ้งการแก้ไข ไม่มีการเตือน) | ปุ่มหลัก `"บันทึก"`
6. ระหว่างบันทึก ปุ่มขวาเปลี่ยนเป็น `"กำลังบันทึก…"` และกดไม่ได้
7. สำเร็จ → toast เขียว → หน้าต่างปิด → ป้ายหน้าที่บนการ์ดอัปเดต

**กฎการตรวจสอบ**
- **ต้องติ๊กอย่างน้อย 1 หน้าที่** — ถ้าไม่ติ๊กเลยแล้วกด `"บันทึก"` → toast แดงทันที ไม่ยิงไปเซิร์ฟเวอร์
- **กฎเซิร์ฟเวอร์ที่หน้าจอไม่กันล่วงหน้า (สำคัญมาก):** **แอดมินถอดสิทธิ์ "แอดมิน" ของตัวเองไม่ได้** — แต่หน้าจอปัจจุบัน **ยอมให้ติ๊กออกและกดบันทึกได้จนสุด** แล้วค่อยเด้ง toast แดงจากเซิร์ฟเวอร์ (ไม่มีการปิดใช้งานช่องติ๊กล่วงหน้า)
- ถ้าหาพนักงานคนนั้นไม่เจอในร้านนี้ → เซิร์ฟเวอร์ตอบ error

**ข้อความสำเร็จ (verbatim)**
- toast เขียว: `"บันทึกหน้าที่แล้ว"`

**ข้อความล้มเหลว (verbatim)**
- ไม่ติ๊กเลย → toast แดง: `"ต้องมีอย่างน้อย 1 หน้าที่"`
- ถอดสิทธิ์แอดมินตัวเอง → toast แดง: `"ถอดสิทธิ์แอดมินของตัวเองไม่ได้ — ให้แอดมินคนอื่นเป็นคนแก้"`
- หาพนักงานไม่เจอ → toast แดง: `"ไม่พบพนักงาน"`
- fallback → toast แดง: `"ไม่สำเร็จ"`
- ทุกกรณี: **หน้าต่างยังเปิดค้าง ปุ่มกลับมากดได้**

---

### 3.3 เปิด / ปิดใช้งานพนักงาน

**ขั้นตอน**
1. ที่การ์ดพนักงาน กดปุ่มโปร่งขนาดเล็กมุมขวาบน — ข้อความ `"ปิด"` (ถ้าคนนั้นกำลังใช้งานอยู่) หรือ `"เปิด"` (ถ้าถูกปิดใช้งานอยู่)
2. หน้าต่างยืนยันเปิดขึ้น:
   - หัวข้อ: `"ปิดใช้งานพนักงาน?"` (ตอนกำลังจะปิด) หรือ `"เปิดใช้งานพนักงาน?"` (ตอนกำลังจะเปิด)
   - เนื้อความ: **เป็นชื่อพนักงานล้วนๆ ไม่มีคำอธิบายอื่นเลย** ตัวอักษรขนาดเล็กสีเทา — ตัวอย่างจริง: `"สมชาย"`
3. ปุ่มใน dialog 2 ปุ่มกว้างเท่ากัน: ปุ่มโปร่ง `"ยกเลิก"` | ปุ่มยืนยัน `"ยืนยัน"`
   - **ตอน "ปิด" ใช้งาน → ปุ่มยืนยันเป็นสีแดงเต็ม ตัวหนังสือขาว** (โหมดอันตราย)
   - **ตอน "เปิด" ใช้งาน → ปุ่มยืนยันเป็นสีแบรนด์ปกติ**
4. กด `"ยกเลิก"` (หรือแตะพื้นหลัง หรือปุ่ม `✕`) → ไม่ทำอะไรต่อ
5. กด `"ยืนยัน"` → **dialog ปิดทันที** แล้วยิงคำสั่งไปเซิร์ฟเวอร์
   - **⚠️ ไม่มีสถานะกำลังส่งเลย** ไม่มี spinner ปุ่มไม่ถูกปิดใช้งาน → ระหว่างรอเซิร์ฟเวอร์ **หน้าจอนิ่งสนิท ไม่มีอะไรบอกว่ากำลังทำงาน** และกดปุ่มรัวซ้ำได้
6. สำเร็จ → toast เขียว → ป้าย `"ปิดใช้งาน"` โผล่/หาย และข้อความปุ่มสลับ `"ปิด"` ↔ `"เปิด"`

**กฎการตรวจสอบ**
- **ไม่มีการตรวจฝั่งหน้าจอเลย**
- **กฎเซิร์ฟเวอร์ที่หน้าจอไม่กันล่วงหน้า:** **แอดมินปิดใช้งานบัญชีของตัวเองไม่ได้** — กดได้ ยืนยันได้ แล้วเด้ง toast แดง
- **ผลข้างเคียงที่ต้องเข้าใจ:** พนักงานที่ถูกปิดใช้งาน เมื่อล็อกอินจะเจอหน้ากั้น (ไอคอน 🚫 หัวข้อ `"บัญชีถูกปิดใช้งาน"`) เข้าแอปไม่ได้เลย — **นี่คือการกระทำที่กระทบชีวิตคนจริง ควรออกแบบให้ dialog สื่อสารผลลัพธ์นี้ ไม่ใช่แค่โชว์ชื่อ**

**ข้อความสำเร็จ (verbatim)**
- toast เขียว: `"บันทึกแล้ว"`

**ข้อความล้มเหลว (verbatim)**
- ปิดตัวเอง → toast แดง: `"ปิดใช้งานบัญชีของตัวเองไม่ได้ — ให้แอดมินคนอื่นเป็นคนปิด"`
- หาไม่เจอ → toast แดง: `"ไม่พบพนักงาน"`
- fallback → toast แดง: `"ไม่สำเร็จ"`

### 3.4 คัดลอกรหัสผูก LINE

**ขั้นตอน**
1. มี **2 ทางเข้า**:
   - ทางที่ 1: ที่การ์ดพนักงานที่ยังไม่ผูก LINE (กล่องเหลือง) กดปุ่มโปร่งขนาดเล็ก `"คัดลอก"` ข้างรหัส
   - ทางที่ 2: ในหน้าต่าง `"สร้างบัญชีสำเร็จ"` กดปุ่มหลักเต็มความกว้าง `"คัดลอกรหัส"`
2. ระบบคัดลอกลงคลิปบอร์ด — **ไม่มีสถานะกำลังส่ง ปุ่มไม่เปลี่ยนหน้าตาใดๆ** (ไม่มี "คัดลอกแล้ว ✓" ค้างบนปุ่ม)
3. สำเร็จ → toast เขียว
4. ล้มเหลว (เช่น เบราว์เซอร์บล็อกคลิปบอร์ด หรือไม่ได้เปิดผ่าน HTTPS) → toast แดง — **รหัสยังแสดงบนจอให้จดเองได้เสมอ**

**กฎการตรวจสอบ**
- ปุ่ม `"คัดลอก"` บนการ์ดจะโผล่ **ก็ต่อเมื่อมีรหัสจริงเท่านั้น** ถ้าไม่มีรหัส กล่องเหลืองจะเหลือแค่ข้อความ `"ยังไม่ผูก LINE"`
- ปุ่ม `"คัดลอกรหัส"` ในหน้าต่างผลลัพธ์จะโผล่ก็ต่อเมื่อได้รหัสกลับมาจริง

**ข้อความสำเร็จ (verbatim)**
- toast เขียว: `"คัดลอกรหัสแล้ว"`

**ข้อความล้มเหลว (verbatim)**
- toast แดง: `"คัดลอกไม่สำเร็จ — กรุณาจดรหัสด้วยตนเอง"`

### 3.5 เพิ่มกะการทำงานใหม่

**ขั้นตอน**
1. กดปุ่มโปร่งขนาดเล็ก `"+ เพิ่มกะ"` ที่หัวข้อ `"กะการทำงาน"`
2. หน้าต่างเปิดขึ้น หัวข้อ `"เพิ่มกะ"` — **ฟอร์มว่างเปล่า** (ชื่อว่าง เวลาว่างทั้งคู่)
3. กรอกฟอร์ม:
   1. ป้าย `"ชื่อกะ"` — ช่องข้อความ **จำเป็น** + **คำใบ้ใต้ช่อง** (ตัวเล็กมากสีเทาอ่อน): `"เช่น กะเช้า / กะบ่าย / กะดึก"` — **นี่คือช่องเดียวในหน้านี้ที่มีคำใบ้ใต้ช่อง**
   2. **แถวเวลา 2 คอลัมน์กว้างเท่ากัน**: ป้าย `"เวลาเริ่ม"` และ ป้าย `"เวลาสิ้นสุด"` — ทั้งคู่เป็น **ช่องเลือกเวลาของระบบปฏิบัติการ** **จำเป็นทั้งคู่**
   3. **ช่องติ๊กเต็มแถว** สูงอย่างน้อย 44px ข้อความ `"เปิดใช้งานกะนี้"` — **ค่าเริ่มต้น: ติ๊กไว้แล้ว**
4. กดปุ่มหลักเต็มความกว้าง `"บันทึก"` ล่างสุด (**ไม่มีปุ่มยกเลิก ไม่มีปุ่มลบ**)
5. ระหว่างส่ง ปุ่มเปลี่ยนเป็น `"กำลังบันทึก…"` และกดไม่ได้
6. สำเร็จ → toast เขียว → หน้าต่างปิด → กะใหม่โผล่ในรายการตามลำดับเวลาเริ่ม (อาจโผล่กลางรายการ ไม่ใช่ท้ายรายการ)

**กฎการตรวจสอบ**
- ชื่อกะ: **จำเป็น** ห้ามเว้นว่าง — **ไม่มีการจำกัดความยาว**
- เวลาเริ่ม: **จำเป็น**
- เวลาสิ้นสุด: **จำเป็น**
- **ไม่มีการตรวจว่าเวลาสิ้นสุดต้องมากกว่าเวลาเริ่ม** → **กะข้ามเที่ยงคืน (เช่น 22:00 – 06:00) บันทึกได้ปกติ โดยตั้งใจ**
- **ไม่มีการตรวจชื่อกะซ้ำ ไม่มีการตรวจเวลาทับซ้อนกับกะอื่น** → สร้างกะชื่อเดียวกัน/เวลาชนกันได้
- ต้องมีข้อมูลร้านในเซสชัน ถ้าเซสชันเพี้ยน → toast แดง `"ไม่พบข้อมูลร้าน — กรุณาเข้าสู่ระบบใหม่"` และไม่ยิงคำสั่ง
- การตรวจทั้งหมดเกิดตอน **กดปุ่มส่ง** เท่านั้น เด้งเป็น toast ก้อนเดียว ไม่มี error ใต้ช่อง

**ข้อความสำเร็จ (verbatim)**
- toast เขียว: `"บันทึกกะแล้ว"`

**ข้อความล้มเหลว (verbatim)**
- ตรวจไม่ผ่าน → toast แดง: `"กรอกชื่อกะ เวลาเริ่ม และเวลาสิ้นสุด"`
- เซสชันเพี้ยน → toast แดง: `"ไม่พบข้อมูลร้าน — กรุณาเข้าสู่ระบบใหม่"`
- ฐานข้อมูลปฏิเสธ → toast แดง = **ข้อความดิบ ซึ่งมักเป็นภาษาอังกฤษยาวๆ** (ส่วนกะเขียนตรงเข้าฐานข้อมูล ไม่ผ่านชั้นแปลข้อความไทย)
- fallback → toast แดง: `"ไม่สำเร็จ"`
- ทุกกรณี: หน้าต่างไม่ปิด ข้อมูลยังอยู่

---

### 3.6 แก้ไขกะการทำงานเดิม

**ขั้นตอน**
1. กดที่ **การ์ดกะใบใดก็ได้** — แตะตรงชื่อ ตรงเวลา หรือตรงคำว่า `"แก้ไข"` สีแบรนด์ ก็ได้ผลเหมือนกันหมด (ทั้งใบเป็นปุ่ม)
2. หน้าต่างเปิดขึ้น หัวข้อ `"แก้ไขกะ"` — **ฟอร์มถูกเติมค่าเดิมไว้ครบ**: ชื่อกะเดิม, เวลาเริ่ม/สิ้นสุดเดิม, ช่องติ๊ก `"เปิดใช้งานกะนี้"` ตามสถานะเดิม
3. แก้ค่าที่ต้องการ
   - **การ "ปิดกะ" ทำได้โดยเอาติ๊ก `"เปิดใช้งานกะนี้"` ออกเท่านั้น — ไม่มีปุ่มลบกะในหน้านี้เลย นี่คือวิธีเดียวที่มี**
4. กดปุ่มหลักเต็มความกว้าง `"บันทึก"` → เปลี่ยนเป็น `"กำลังบันทึก…"` และกดไม่ได้
5. สำเร็จ → toast เขียว → หน้าต่างปิด → การ์ดกะอัปเดต (ชื่อ/เวลา/ป้าย `"ปิดใช้งาน"`)

**กฎการตรวจสอบ**
- เหมือนโหมดสร้างทุกข้อ (ชื่อกะ/เวลาเริ่ม/เวลาสิ้นสุด จำเป็น, ไม่ตรวจลำดับเวลา, ไม่ตรวจซ้ำ)
- **เพิ่มเติม:** ระบบตรวจว่าฐานข้อมูลยืนยันการบันทึกจริง ถ้าถูกปฏิเสธแบบเงียบๆ (ไม่มีแถวไหนถูกแก้) จะถือว่าล้มเหลวและแจ้งเตือน
- **ไม่มีฟังก์ชันลบกะโดยเจตนา** — ปิดใช้งานเท่านั้น

**ข้อความสำเร็จ (verbatim)**
- toast เขียว: `"บันทึกกะแล้ว"`

**ข้อความล้มเหลว (verbatim)**
- toast แดง: `"กรอกชื่อกะ เวลาเริ่ม และเวลาสิ้นสุด"`
- toast แดง: `"ไม่พบข้อมูลร้าน — กรุณาเข้าสู่ระบบใหม่"`
- toast แดง: `"บันทึกไม่สำเร็จ — เซิร์ฟเวอร์ไม่ยืนยันข้อมูล"`
- toast แดง = ข้อความดิบจากฐานข้อมูล (มักภาษาอังกฤษ)
- fallback → toast แดง: `"ไม่สำเร็จ"`

---

### 3.7 โหลดข้อมูลล้มเหลว → ลองใหม่

**ขั้นตอน**
1. **สองบล็อกพังแยกกันได้อิสระ** — ถ้ารายชื่อพนักงานโหลดไม่ได้ บล็อกนั้นถูกแทนที่ด้วยสถานะผิดพลาด แต่ **ส่วน "กะการทำงาน" ด้านล่างยังแสดงปกติ** และกลับกันก็ได้
2. สถานะผิดพลาด (จัดกลาง มีระยะห่างบน–ล่าง) แสดง:
   - ไอคอน `⚠️` ขนาดใหญ่
   - หัวข้อ: `"โหลดข้อมูลไม่สำเร็จ"`
   - คำอธิบายตัวเล็กสีเทาอ่อน: `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"`
   - ปุ่มโปร่งขนาดเล็ก: `"ลองใหม่"`
3. กด `"ลองใหม่"` → กลับไปแสดงตัวโหลดหมุน แล้วโหลดใหม่เฉพาะบล็อกนั้น

**ลำดับการแสดงผลในแต่ละบล็อก (สำคัญ — เกิดพร้อมกันไม่ได้)**
> กำลังโหลด → **ตัวโหลดหมุน**
> ผิดพลาด → **สถานะผิดพลาด**
> ไม่มีข้อมูล → **สถานะว่าง**
> มีข้อมูล → **รายการการ์ด**

**ข้อความสำเร็จ**
- รายการข้อมูลกลับมาแสดงตามปกติ

**ข้อความล้มเหลว (verbatim)**
- ยังคงแสดงสถานะผิดพลาดเดิม (`"โหลดข้อมูลไม่สำเร็จ"`) พร้อมปุ่ม `"ลองใหม่"` ให้กดซ้ำได้ไม่จำกัด

## 4. สถานะของหน้า

| สถานะ | ผู้ใช้เห็นอะไร | ข้อความจริง |
|---|---|---|
| **ปกติ — ทั้งสองบล็อกมีข้อมูล** | หัวข้อ + ปุ่ม + รายการการ์ดพนักงาน แล้วต่อด้วยหัวข้อรอง + ปุ่ม + รายการการ์ดกะ | `"ทีมงาน"`, `"+ เพิ่มพนักงาน"`, `"กะการทำงาน"`, `"+ เพิ่มกะ"` |
| **กำลังโหลด (บล็อกพนักงาน)** | วงกลมหมุนเล็ก (ขอบเทาอ่อน ส่วนบนสีแบรนด์) + ข้อความสีเทาอ่อน จัดกลาง มีระยะห่างบน–ล่างพอสมควร | `"กำลังโหลด…"` |
| **กำลังโหลด (บล็อกกะ)** | เหมือนกันเป๊ะ แต่แสดงใต้หัวข้อ `"กะการทำงาน"` — **สองบล็อกโหลดแยกกัน อาจเสร็จไม่พร้อมกัน** | `"กำลังโหลด…"` |
| **กำลังโหลดระดับแอป (ตรวจสิทธิ์)** | เต็มจอ ข้อความจัดกึ่งกลางจอ สีเทา | `"กำลังโหลด…"` |
| **ว่าง — ไม่มีพนักงาน** | จัดกลาง: ไอคอน `👥` ขนาดใหญ่ / หัวข้อตัวหนาขนาดเล็กสีเทาเข้ม / คำใบ้ตัวเล็กมากสีเทาอ่อน — **ไม่มีปุ่ม CTA ในสถานะว่าง** ผู้ใช้ต้องเงยขึ้นไปกดปุ่มที่หัวข้อ | หัวข้อ: `"ยังไม่มีพนักงาน"` / คำใบ้: `"กด “เพิ่มพนักงาน” เพื่อสร้างบัญชีเข้าระบบ"` |
| **ว่าง — ไม่มีกะ** | เหมือนกัน ไอคอน `🕐` — **ไม่มีปุ่ม CTA เช่นกัน** | หัวข้อ: `"ยังไม่มีกะ"` / คำใบ้: `"กด “เพิ่มกะ” เพื่อกำหนดช่วงเวลาเข้างาน"` |
| **ผิดพลาด (แยก 2 บล็อก)** | จัดกลาง: ไอคอน `⚠️` ขนาดใหญ่ / หัวข้อตัวหนาขนาดเล็ก / คำอธิบายตัวเล็กสีเทาอ่อน / ปุ่มโปร่งขนาดเล็ก | `"โหลดข้อมูลไม่สำเร็จ"` / `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"` / `"ลองใหม่"` |
| **กำลังส่ง — สร้างบัญชี** | ปุ่มเปลี่ยนข้อความ + กดไม่ได้ **ไม่มี spinner ไม่มีฉากทึบ ช่องกรอกยังพิมพ์ได้** | `"สร้างบัญชี"` → `"กำลังสร้าง…"` |
| **กำลังส่ง — บันทึกหน้าที่ / บันทึกกะ** | ปุ่มเปลี่ยนข้อความ + กดไม่ได้ | `"บันทึก"` → `"กำลังบันทึก…"` |
| **กำลังส่ง — เปิด/ปิดใช้งานพนักงาน** | **⚠️ ไม่มีสถานะกำลังส่งเลย** dialog ปิดไปแล้ว หน้าจอนิ่งสนิทจนกว่า toast จะเด้ง กดปุ่มซ้ำได้ | (ไม่มีข้อความ) |
| **กำลังส่ง — คัดลอกรหัส** | **ไม่มีสถานะกำลังส่งเลย** ปุ่มไม่เปลี่ยนหน้าตา | (ไม่มีข้อความ) |
| **permission-denied** | **ไม่มีสถานะนี้ในหน้านี้** — ผู้ใช้ที่ไม่ใช่แอดมินถูกดีดออกก่อนหน้าจะเรนเดอร์ ไม่มีหน้าจอ "คุณไม่มีสิทธิ์" ให้ออกแบบ | — |
| **บัญชีถูกปิดใช้งาน (ผลจากหน้านี้ ไปเกิดที่หน้าอื่น)** | พนักงานที่ถูกปิดใช้งาน เมื่อล็อกอินจะเจอหน้ากั้น ไอคอน `🚫` | `"บัญชีถูกปิดใช้งาน"` |
| **toast (ทุกชนิด)** | ลอยด้านล่างกึ่งกลาง **เหนือแถบแท็บล่าง** กล่องขอบมน ตัวหนังสือขาวตัวหนาขนาดเล็ก มีเงา **หายเองใน 3.6 วินาที ไม่มีปุ่มปิด ซ้อนกันได้หลายก้อน** สี: สำเร็จ = เขียว, ผิดพลาด = แดง, ข้อมูล = เทาเข้ม | (ดูข้อ 5) |

| ฟิลด์ | รูปแบบการแสดงผล | ตัวอย่างจริง |
|---|---|---|
| ชื่อพนักงาน | ข้อความตัวหนาขนาดเล็ก **ตัดท้ายด้วย ellipsis ถ้ายาวเกิน** — ถ้าว่าง แสดงขีดกลาง | `สมชาย ใจดี` / `—` |
| สถานะใช้งาน | ป้ายเทาโผล่ **เฉพาะเมื่อถูกปิด** (ตอนใช้งานปกติ **ไม่มีป้ายอะไรเลย** — ไม่มีป้ายเขียว "ใช้งานอยู่") | `ปิดใช้งาน` |
| มีบัญชีล็อกอินหรือไม่ | ป้ายเหลืองโผล่ **เฉพาะเมื่อไม่มี** | `ยังไม่มีบัญชีเข้าระบบ` |
| หน้าที่ | **หลายป้าย** สีแบรนด์อ่อน 1 ป้ายต่อ 1 หน้าที่ **แปลเป็นไทยเสมอ** ถ้าเจอค่าที่ไม่รู้จักจะแสดงค่าดิบภาษาอังกฤษ | `แพ็ค` `ไรเดอร์` |
| สถานะ LINE | 2 หน้าตาเท่านั้น (ผูกแล้ว = บรรทัดเขียว / ยังไม่ผูก = กล่องเหลือง) | `LINE: ผูกแล้ว ✅` |
| รหัสผูก LINE | **ฟอนต์ monospace** ในกล่องขาวเล็ก บนการ์ด — และ **ฟอนต์ monospace ขนาดใหญ่มาก ตัวหนา ระยะห่างตัวอักษรกว้าง** ในหน้าต่างผลลัพธ์ (ออกแบบให้อ่านออกจากระยะห่าง) | รูปแบบสร้างจากฐานข้อมูล เช่น `A7K2M9` |

### 6.2 การ์ดกะ

| ฟิลด์ | รูปแบบการแสดงผล | ตัวอย่างจริง |
|---|---|---|
| ชื่อกะ | ข้อความตัวหนาขนาดเล็ก ตัดท้ายด้วย ellipsis | `กะเช้า` |
| ช่วงเวลา | **`HH:MM – HH:MM น.`** — ตัวเลข 24 ชั่วโมง **ตัดวินาทีทิ้งเสมอ**, คั่นด้วย **en-dash ` – ` (ขีดกลางยาว มีเว้นวรรคทั้งสองข้าง ไม่ใช่ hyphen)**, ลงท้ายด้วยคำว่า **` น.`** | `08:00 – 17:00 น.` / กะข้ามคืน: `22:00 – 06:00 น.` |
| สถานะกะ | ป้ายเทาโผล่เฉพาะเมื่อถูกปิด | `ปิดใช้งาน` |

### 6.3 หมายเหตุรูปแบบข้อมูล

- **หน้านี้ไม่มีการแสดงเงินเลย** — ไม่มีราคา ไม่มียอด ไม่ต้องใช้รูปแบบ `฿1,250`
- **หน้านี้ไม่มีการแสดงวันที่/เวลาแบบเต็ม** — ไม่มี "สร้างเมื่อ" ไม่มี "อัปเดตล่าสุด" **ทั้งที่รายชื่อเรียงตามวันที่สร้าง แต่ไม่แสดงวันที่นั้นให้เห็น**
- เวลาในกะเป็น **เวลาท้องถิ่นของร้าน (เวลาไทย)** เก็บเป็นเวลาเปล่าๆ ไม่มีโซนเวลาผูกอยู่ — ไม่มีการแปลงโซนเวลาใดๆ
- **ไม่มีการแสดงอีเมลหรือเบอร์โทรของพนักงานบนการ์ด** — กรอกตอนสร้างแล้วมองไม่เห็นอีกเลยในหน้านี้ (และแก้ไม่ได้ด้วย)
- **ไอคอนทั้งหมดเป็น emoji ล้วน** (`👥` `🕐` `⚠️` `✅` `✕` `🚫`) ไม่ใช่ icon font หรือ SVG

## 7. ข้อมูลอัปเดตยังไง

**สรุปสั้น: โหลดครั้งเดียวตอนเปิดหน้า — ไม่มี realtime ไม่มี polling**

- **ไม่มี polling** — ไม่มีการดึงข้อมูลซ้ำตามรอบเวลา (ต่างจากหน้าคิวแพ็คที่ดึงใหม่ทุก 15 วินาที)
- **ไม่มี realtime subscription** — ไม่มีการฟังการเปลี่ยนแปลงสดจากฐานข้อมูล (ต่างจากหน้าแรกของพนักงานที่ใช้ realtime)
- **ไม่รีเฟรชตอนกลับมาโฟกัสหน้าต่าง** — ค่าตั้งต้นทั้งระบบปิดพฤติกรรมนี้ไว้ สลับแท็บไปกลับมาก็ไม่โหลดใหม่
- **ข้อมูลรีเฟรชเฉพาะ 3 กรณี:**
  1. เปิดหน้าครั้งแรก / รีโหลดหน้า
  2. หลังทำรายการสำเร็จ (สร้างบัญชี / แก้หน้าที่ / เปิด-ปิดใช้งาน → รีเฟรชรายชื่อพนักงาน; บันทึกกะ → รีเฟรชรายการกะ)
  3. กดปุ่ม `"ลองใหม่"` ในสถานะผิดพลาด
- **แหล่งข้อมูล 2 ทางที่ต่างกันโดยสิ้นเชิง** (นี่คือเหตุผลที่ 2 บล็อกพังแยกกันและข้อความ error คนละภาษา):
  - รายชื่อพนักงาน → ผ่านชั้นเซิร์ฟเวอร์เฉพาะทาง **มีข้อความ error ภาษาไทยเตรียมไว้**
  - รายการกะ → เขียน/อ่านตรงเข้าฐานข้อมูล **ข้อความ error เป็นภาษาอังกฤษดิบ**

**ผลต่อดีไซน์**
- **ข้อมูลอาจเก่าโดยที่ผู้ใช้ไม่รู้ตัว** — ถ้าแอดมินอีกคนเพิ่งเพิ่มพนักงาน หน้านี้จะไม่รู้เรื่องเลยจนกว่าจะรีโหลด **ยิ่งร้ายตรงที่สถานะ "ผูก LINE แล้ว" ของพนักงานเปลี่ยนได้จากฝั่ง LINE ตลอดเวลา — แอดมินอาจนั่งดูการ์ดเหลือง "ยังไม่ผูก LINE" ทั้งที่พนักงานผูกไปแล้วเมื่อกี้**
- **จึงควรมี:** (ก) **ปุ่มรีเฟรชที่มองเห็นได้** อย่างน้อยที่หัวข้อหน้า และ (ข) **ตัวบอกเวลาที่โหลดล่าสุด** เช่นบรรทัดเล็กๆ ใต้หัวข้อ — ทั้งสองอย่างนี้ **ยังไม่มีในดีไซน์ปัจจุบันเลย** ถือเป็นช่องว่างที่ควรเติม
- แต่เพราะความถี่การใช้งานต่ำมาก **ไม่จำเป็นต้องใส่ realtime หรือ auto-refresh ที่กระพริบตลอดเวลา** — ปุ่มรีเฟรชแบบตั้งใจกดเพียงพอ

## 8. สิ่งที่ designer ต้องรู้

### 🔴 ห้ามแตะ — ผูกกับกติกาธุรกิจ / เจตนาที่ตั้งใจไว้

- **ไม่มีปุ่มลบทั้งพนักงานและกะ — โดยเจตนา** วิธีเดียวที่มีคือ "ปิดใช้งาน" (soft-disable) **ห้ามออกแบบปุ่มลบเพิ่มโดยไม่ถามเจ้าของระบบ** ข้อมูลต้องคงอยู่เพื่อการตรวจสอบย้อนหลัง
- **ช่องรหัสผ่านแสดงตัวอักษรเปิดเผย ไม่ใช่จุดๆ — โดยเจตนา** เพราะแอดมินเป็นคน **ตั้งรหัสให้พนักงานแล้วต้องอ่านบอกต่อ** ณ ตรงนั้น **ห้ามเปลี่ยนเป็นช่องรหัสผ่านซ่อนโดยไม่มีปุ่มดู/ซ่อน**
- **รหัสผูก LINE คือหัวใจของการรับพนักงานใหม่** — พนักงานต้องเอารหัสนี้ไปพิมพ์ใน LINE OA ถึงจะรับแจ้งเตือนงานได้ รหัสปรากฏ **3 ที่**: หน้าต่างหลังสร้างบัญชี / กล่องเหลืองบนการ์ดพนักงาน / แถบเตือนใน Shell (สำหรับตัวแอดมินเอง) **ต้องอ่านง่าย คัดลอกง่าย และจดตามได้เสมอ** — ห้ามลดขนาดกล่องรหัสในหน้าต่างผลลัพธ์
- **ข้อความไทยทุกบรรทัดในข้อ 5 ต้องใช้ verbatim** รวมถึงอัญประกาศไทย “ ” และ en-dash ` – ` **ห้ามเขียนใหม่ให้สวยขึ้น** ถ้าคิดว่าข้อความไหนควรเปลี่ยน ให้เสนอเป็น proposal แยก
- **ป้ายหน้าที่ 5 แบบตายตัว** (`แอดมิน` / `สต็อก` / `แพ็ค` / `ไรเดอร์` / `ทั่วไป`) **เพิ่ม/ลด/เปลี่ยนชื่อไม่ได้** เป็นค่าที่ผูกกับสิทธิ์ทั้งระบบ; และ **คนหนึ่งมีได้หลายหน้าที่พร้อมกัน** — ห้ามออกแบบเป็นตัวเลือกเดียว (radio / dropdown)
- **ค่าเริ่มต้นตอนเพิ่มพนักงาน = เลือก `"แพ็ค"` ไว้แล้ว** — เป็นหน้าที่ที่พบบ่อยสุด อย่าเปลี่ยนเป็นไม่เลือกอะไรเลย
- **กะข้ามเที่ยงคืนบันทึกได้โดยเจตนา** (เช่น `22:00 – 06:00 น.`) **ห้ามใส่ validation ห้าม** — แต่ควรออกแบบวิธี **สื่อสารว่า "ข้ามวัน"** ให้ชัด เพราะตอนนี้ไม่มีเลย และรายการเรียงตามเวลาเริ่มเท่านั้น กะดึกจึงไปโผล่ท้ายรายการเสมอ
- **การ์ดพนักงานไม่แสดงป้ายอะไรเลยตอนสถานะปกติ** — ป้ายมีไว้บอก "ผิดปกติ" เท่านั้น อย่าเพิ่มป้ายเขียว "ใช้งานอยู่" ให้ทุกใบ จะกลบสัญญาณ
- **ป้าย `"ยังไม่มีบัญชีเข้าระบบ"` แทบไม่มีทางเกิดจากหน้านี้** (การสร้างผ่านหน้านี้จะสร้างบัญชีล็อกอินให้เสมอ) เกิดได้เฉพาะกับข้อมูลที่ถูกใส่เข้าระบบทางอื่น **แต่ยังต้องรองรับ อย่าตัดทิ้ง**

### 🟡 ปรับได้ — จุดที่ดีไซน์ปัจจุบันอ่อนและควรถูกปรับปรุง

- **ความไม่สม่ำเสมอ #1 (ร้ายแรงที่สุด) — ตัวเลือก "หน้าที่" มี 2 หน้าตาในหน้าเดียวกัน**: ตอน **เพิ่มพนักงาน** เป็น **ชิปกลม** แต่ตอน **แก้หน้าที่** เป็น **รายการช่องติ๊กแนวตั้ง** ทั้งที่เป็นข้อมูลชุดเดียวกันเป๊ะ (5 ตัวเลือก เลือกได้หลายอัน) **ควรรวมเป็นแบบเดียว**
- **ความไม่สม่ำเสมอ #2 — ปุ่มปิดหน้าต่างไม่เท่ากัน**: หน้าต่างแก้หน้าที่มีปุ่มคู่ `"ยกเลิก"` | `"บันทึก"` แต่หน้าต่างเพิ่มพนักงานและหน้าต่างกะมีแค่ปุ่มส่งเต็มความกว้าง ต้องปิดด้วย `✕` หรือแตะพื้นหลัง **ควรทำให้เหมือนกัน**
- **ความไม่สม่ำเสมอ #3 — ลำดับชั้นหัวข้อผิด**: `"กะการทำงาน"` ใช้คอมโพเนนต์หัวข้อตัวเดียวกับ `"ทีมงาน"` จึงออกมาเป็นหัวข้อระดับ 1 ขนาดเท่ากันทั้งคู่ → **หน้าเดียวมีหัวข้อระดับ 1 สองอัน** ผิดทั้งเชิงสายตาและเชิงความหมาย (screen reader) **ควรออกแบบหัวข้อระดับรองเพิ่มเข้าระบบ**
- **ความไม่สม่ำเสมอ #4 — affordance ของการ์ดขัดกัน**: **การ์ดกะทั้งใบกดได้** (คำว่า `"แก้ไข"` เป็นแค่ข้อความประกอบ) แต่ **การ์ดพนักงานทั้งใบกดไม่ได้** ต้องกดปุ่มย่อยเฉพาะจุด — **2 รายการในหน้าเดียวกันทำงานคนละแบบ ผู้ใช้จะสับสน**
- **กฎที่เซิร์ฟเวอร์บังคับแต่หน้าจอไม่กันล่วงหน้า — 2 ข้อ**: แอดมิน **ถอดสิทธิ์แอดมินของตัวเองไม่ได้** และ **ปิดใช้งานบัญชีตัวเองไม่ได้** ตอนนี้หน้าจอปล่อยให้กดจนสุดแล้วค่อยเด้ง toast แดง **ควรปิดใช้งานตัวเลือกล่วงหน้าพร้อมข้อความอธิบาย** — ทำได้เพราะหน้านี้รู้อยู่แล้วว่าใครคือผู้ใช้ปัจจุบัน
- **ไม่มีสถานะกำลังส่งตอนเปิด/ปิดใช้งานพนักงาน** — หลังกด `"ยืนยัน"` dialog ปิดทันที **หน้าจอนิ่งสนิท ไม่มีอะไรบอกว่ากำลังทำงาน** จนกว่า toast จะเด้ง และกดปุ่มรัวซ้ำได้ **ควรออกแบบ optimistic update หรือ inline spinner บนการ์ดใบนั้น**
- **ไม่มีการยืนยันตอนปิดหน้าต่างที่กรอกข้อมูลไปแล้ว** — แตะพื้นหลังพลาดครั้งเดียว ข้อมูลทั้ง 5 ช่องในหน้าต่างเพิ่มพนักงานหายหมด **ไม่มีเตือน**
- **การตรวจสอบทั้งหมดเป็น toast ก้อนเดียว ไม่มี error ใต้ช่อง** — toast บอกแค่ว่า "อะไรสักอย่างผิด" ไม่ได้ชี้ว่าช่องไหน และ **ลอยอยู่ที่ขอบล่างจอ ไกลจากฟอร์มที่ผิด** ควรออกแบบ error ระดับช่อง
- **toast อยู่ 3.6 วินาทีแล้วหายเอง ไม่มีปุ่มปิด** — ข้อความยาวๆ อย่าง `"ถอดสิทธิ์แอดมินของตัวเองไม่ได้ — ให้แอดมินคนอื่นเป็นคนแก้"` อาจอ่านไม่ทัน ยิ่งถ้าเป็นข้อความ error ภาษาอังกฤษดิบจากฝั่งกะยิ่งยาวกว่านั้น **กล่อง toast ต้องรองรับข้อความยาวหลายบรรทัด และควรมีปุ่มปิด/หยุดนับถอยหลังสำหรับ error**
- **หน้าต่างยืนยันปิดใช้งานพนักงานมีแค่ชื่อเป็นเนื้อความ** — ไม่บอกเลยว่าผลคือ **คนนั้นจะล็อกอินเข้าแอปไม่ได้อีก** ควรเขียนข้อความอธิบายผลลัพธ์ (ต้องเสนอข้อความใหม่ให้เจ้าของอนุมัติ)
- **ไม่มีช่องค้นหา ไม่มีตัวกรอง ไม่มีการแบ่งหน้า ทั้ง 2 รายการ** — ร้านที่มีพนักงาน 50+ คนจะเป็นสกรอลล์ยาวมาก **ควรเสนอ:** ค้นหาชื่อ, กรองตามหน้าที่, กรอง "เฉพาะที่ยังไม่ผูก LINE", กรอง "ปิดใช้งาน"
- **จังหวะรายการไม่สม่ำเสมอ** — การ์ดพนักงานที่ยังไม่ผูก LINE **สูงกว่า** คนที่ผูกแล้ว (กล่องเหลืองเพิ่มบรรทัด) ทำให้รายการกระตุก ไม่เป็นจังหวะ
- **ไม่มีปุ่มรีเฟรช ไม่มีตัวบอกเวลาโหลดล่าสุด** ทั้งที่ข้อมูลไม่อัปเดตเอง (ดูข้อ 7) **ควรเติม**
- **เป้าแตะบางจุดเล็กเกิน** — แถวช่องติ๊กใส่ความสูงขั้นต่ำ 44px ไว้ถูกต้องแล้ว **แต่ปุ่มข้อความล้วน `"แก้หน้าที่"` ไม่มีความสูงขั้นต่ำเลย** และป้ายต่างๆ ก็เล็กกว่ามาตรฐาน **ควรตรวจทั้งหน้า**
- **ช่องเลือกเวลาเป็นของระบบปฏิบัติการ** — หน้าตาต่างกันมากตามอุปกรณ์ (iOS = ล้อหมุน, Android = หน้าปัดนาฬิกา) **ควบคุมสไตล์ไม่ได้** ถ้าดีไซน์ต้องการตัวเลือกเวลาแบบกำหนดเอง **ต้องบอกล่วงหน้าเพราะเป็นงานเพิ่มที่ไม่เล็ก**
- **แถบแท็บล่างมี 11 รายการสำหรับแอดมิน** → บนมือถือต้องเลื่อนแนวนอน **เป็นจุดที่ควรพิจารณาใหม่ทั้งระบบ** (ไม่ใช่แค่หน้านี้ แต่กระทบหน้านี้)
- **ไอคอนเป็น emoji ล้วนทั้งระบบ** — ถ้าจะเปลี่ยนเป็นชุดไอคอนจริง **ต้องเปลี่ยนทั้งระบบพร้อมกัน** อย่าเปลี่ยนเฉพาะหน้านี้
- **สถานะว่างไม่มีปุ่ม CTA ในตัว** — ผู้ใช้ต้องเงยกลับขึ้นไปกดปุ่มที่หัวข้อ ทั้งที่คำใบ้บอกให้ "กดเพิ่มพนักงาน" **ควรเสนอปุ่มในสถานะว่างเลย**
- **ข้อความ error ฝั่ง "กะ" อาจหลุดเป็นภาษาอังกฤษ** เพราะเขียนตรงเข้าฐานข้อมูล ไม่ผ่านชั้นแปลไทย **กล่อง toast ต้องรองรับข้อความอังกฤษยาวๆ ด้วย** (ระยะยาวควรเสนอให้แปลเป็นไทย แต่เป็นงานฝั่งระบบ)

### 🎨 โทนสีที่ใช้จริงในหน้านี้ (อ้างอิงสำหรับการรีดีไซน์)

| ความหมาย | สีที่ใช้ตอนนี้ |
|---|---|
| หน้าที่ / ปุ่มหลัก / ลิงก์ข้อความ | สีแบรนด์ (ป้าย = พื้นแบรนด์โปร่ง ตัวหนังสือแบรนด์) |
| ยังไม่ผูก LINE / ยังไม่มีบัญชี | เหลือง (พื้นเหลืองอ่อนมาก ตัวหนังสือเหลืองเข้ม) |
| LINE ผูกแล้ว | เขียวเข้ม (ข้อความอย่างเดียว ไม่มีพื้น) |
| ปิดใช้งาน | เทา (พื้นเทาอ่อน ตัวหนังสือเทากลาง) |
| การกระทำอันตราย (ปุ่มยืนยันตอนปิดใช้งาน) + toast ผิดพลาด | แดง |
| toast สำเร็จ | เขียว |
| toast ข้อมูลทั่วไป | เทาเข้มเกือบดำ |
