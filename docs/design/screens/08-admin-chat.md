# แชทลูกค้า (`/admin/chats`)

> แอดมินร้าน (role `admin` เท่านั้น) ใช้ตอบแชทลูกค้าที่ทักเข้ามาทาง LINE แทน AI — ดูว่าห้องไหนรอแอดมิน อ่านประวัติ พิมพ์ตอบ (ระบบ push เข้า LINE ให้เอง) แล้ว "ปิดเคส" คืนห้องให้ AI ตอบต่อ

---

## 1. บริบทการใช้งาน

**ใครใช้**
- แอดมินร้านเท่านั้น (role `admin`). ระบบกันไว้ 2 ชั้น: ชั้นหน้าเว็บ (Guard) และชั้นเซิร์ฟเวอร์ (edge function `admin-action` ตรวจ role ซ้ำ)
- พนักงานตำแหน่ง `stock` / `packer` / `rider` / `staff` **เข้าไม่ได้** — ถ้าพยายามเข้า URL นี้ตรงๆ จะถูกเด้งไปหน้าแรกตามตำแหน่งของตัวเอง (stock→หน้าแคตตาล็อก, packer→หน้าแพ็ก, rider→หน้าไรเดอร์, อื่นๆ→หน้าสตาฟ) **โดยไม่มีหน้า "คุณไม่มีสิทธิ์" ให้เห็นเลย** — เป็นการ redirect เงียบๆ
- เข้าถึงจากเมนูหลักของแอป: รายการ "แชทลูกค้า" ไอคอน 💬

**อุปกรณ์และสถานการณ์จริง**
- **ใช้จริงทั้งสองแบบ** และดีไซน์ปัจจุบันรองรับทั้งคู่:
  - **มือถือ** — แอดมินเดินอยู่ในร้าน/หน้าร้าน ถือมือถือมือเดียว ตอบลูกค้าระหว่างทำอย่างอื่น หลักฐานว่าออกแบบมาเพื่อมือถือจริง: มีปุ่มย้อนกลับ ← เฉพาะบนมือถือ, มีลิงก์กดโทรออกได้ทันที (`tel:`), กล่องพิมพ์สูงอย่างน้อย 44px (touch target มาตรฐาน), toast ลอยอยู่เหนือแถบเมนูล่าง
  - **เดสก์ท็อป** — แอดมินนั่งโต๊ะในร้าน เปิดจอใหญ่ ตอบหลายห้องสลับไปมา เห็นรายการห้อง + ห้องสนทนาพร้อมกัน
- **ตัดสิน: mobile-first** — เพราะบนมือถือหน้านี้ทำงานเหมือนแอปแชท native เต็มรูปแบบ (สลับหน้าเต็มจอ) และมีฟีเจอร์เฉพาะมือถือ (ปุ่มย้อนกลับ, กดโทร) ส่วนเดสก์ท็อปเป็นการ "ขยาย" ให้เห็น 2 คอลัมน์พร้อมกัน

**ความถี่และความเร่งรีบ**
- เปิดค้างไว้เป็นชั่วโมงระหว่างเวลาทำการ แต่ **ไม่มีเสียงเตือน ไม่มี browser notification ไม่มี badge บน favicon** — แอดมินต้องกลับมามองจอเอง
- ความเร่งรีบสูงมากในเคสธง **"ร้องเรียน"** และ **"ฉุกเฉิน"** (ลูกค้าโกรธ / เหตุด่วน) — ป้ายแดงต้องสะดุดตาที่สุดบนหน้า
- **ทุกข้อความที่กดส่ง ไปถึงลูกค้าจริงทาง LINE ทันที — ไม่มี undo ไม่มีแก้ไข ไม่มีลบ**

## 2. โครงหน้า

**ไม่มีแท็บในหน้านี้**

จากบนลงล่าง:

**A. หัวหน้าจอ (PageHeader)**
- ซ้าย: หัวข้อ **"แชทลูกค้า"** ตัวหนาขนาดกลาง
- ขวา: ป้ายสรุปจำนวนห้องที่รอ — สีเหลืองอำพัน (พื้นเหลืองอ่อน ตัวอักษรเหลืองเข้ม) ข้อความ **"รอแอดมิน {จำนวน} ห้อง"** เช่น "รอแอดมิน 3 ห้อง"
- **ป้ายนี้แสดงเฉพาะเมื่อมีห้องรออย่างน้อย 1 ห้อง** — ถ้าเป็น 0 ห้อง มุมขวาจะว่างเปล่า ไม่มี "รอแอดมิน 0 ห้อง"

**B. พื้นที่หลัก — 2 คอลัมน์บนเดสก์ท็อป / สลับหน้าบนมือถือ**

| | เดสก์ท็อป (จอกว้าง) | มือถือ (จอแคบ) |
|---|---|---|
| ยังไม่เลือกห้อง | เห็นทั้ง 2 คอลัมน์: รายการซ้าย + การ์ดว่าง "เลือกห้องแชทจากรายการ" ขวา | เห็นเฉพาะรายการห้องเต็มจอ |
| เลือกห้องแล้ว | เห็นทั้ง 2 คอลัมน์: รายการยังอยู่ (การ์ดที่เลือกมีขอบสีแบรนด์) + ห้องสนทนาขวา | รายการหายไป เห็นเฉพาะห้องสนทนาเต็มจอ + ปุ่ม ← |

- คอลัมน์ซ้ายกว้างต่ำสุด 260px สูงสุด 320px; คอลัมน์ขวายืดกินที่เหลือทั้งหมด; ระยะห่างระหว่างคอลัมน์เท่าระยะการ์ดมาตรฐาน

**C. รายการห้องแชท (คอลัมน์ซ้าย)**
- การ์ดพื้นขาว ขอบมน เรียงลงเป็นแถวๆ มีระยะห่างเล็กระหว่างการ์ด **ทั้งใบเป็นปุ่มกดได้**
- ในการ์ดแต่ละใบ:
  - ฝั่งซ้าย 2 บรรทัด: **ชื่อเล่นลูกค้า** (ตัวหนาเล็ก, ตัดด้วย … ถ้ายาว) และใต้ลงมา **เบอร์โทร** (ตัวเล็กสีเทา, ตัดด้วย … ถ้ายาว)
  - ฝั่งขวาเรียงลง 2 ชั้น: **เวลาข้อความล่าสุด** (ตัวเล็กมาก 11px สีเทาจาง) และใต้ลงมาคือ **ป้ายสถานะ**
- **การ์ดที่กำลังเปิดอยู่** ได้ขอบสีแบรนด์ (นี่คือตัวบอกสถานะ selected ตัวเดียวที่มี)
- **ลำดับการเรียง (ผูกกับกติกาธุรกิจ — ห้ามแตะ)**: ห้องที่ "รอแอดมิน" ลอยขึ้นบนสุดเสมอ แล้วจึงเรียงตามเวลาข้อความล่าสุด ใหม่→เก่า; ห้องที่ยังไม่เคยมีข้อความไปอยู่ล่างสุด
- **แสดงสูงสุด 50 ห้อง — ไม่มีปุ่มโหลดเพิ่ม ไม่มีช่องค้นหา ไม่มีตัวกรอง**
- **ไม่มีตัวอย่างข้อความล่าสุด ไม่มีตัวนับข้อความยังไม่อ่าน**

**D. ป้ายสถานะห้อง (ใช้ซ้ำ 2 ที่: บนการ์ดรายการ + บนหัวห้องสนทนา)**
- เป็นแถวของป้ายเล็ก ชิดขวา **ตกบรรทัดใหม่ได้** และ **แสดงพร้อมกันหลายป้ายได้**
- เงื่อนไข:
  1. ห้องอยู่ในโหมดคนตอบ → ป้ายเหลืองอำพัน **"รอแอดมิน"**
  2. ห้องถูกตั้งธงร้องเรียน → ป้ายแดง **"ร้องเรียน"**
  3. ห้องถูกตั้งธงฉุกเฉิน → ป้ายแดง **"ฉุกเฉิน"**
- ธงเป็นได้ครั้งละค่าเดียว → **ป้ายแดงเห็นได้ทีละ 1 ป้าย** แต่ **ป้ายเหลือง + ป้ายแดงเห็นพร้อมกันได้** (เช่น "รอแอดมิน" + "ฉุกเฉิน" 2 ป้ายในมุมขวาแคบๆ ของการ์ด)

**E. ห้องสนทนา (คอลัมน์ขวา)** — การ์ดใบเดียว ขอบมน เนื้อในไม่มี padding แบ่งเป็น 4 แถบซ้อนกันแนวตั้ง:

1. **หัวห้อง** — เส้นคั่นด้านล่าง
   - ปุ่ม **"←"** ชิดซ้าย (ปุ่มโปร่ง ขนาดเล็ก) — **มีเฉพาะบนมือถือ** ป้ายช่วยการเข้าถึง: "กลับไปรายชื่อแชท"
   - กลาง 2 บรรทัด: **ชื่อเล่นลูกค้า** (ตัวหนาเล็ก ตัดด้วย …) และใต้ลงมาคือ **ลิงก์โทร** ตัวเล็กสีแบรนด์ รูปแบบ "📞 {เบอร์}" เช่น "📞 0812345678" — กดแล้วมือถือเปิดแอปโทรออก; ถ้าไม่มีเบอร์แสดงข้อความเทา "ไม่มีเบอร์โทร" (กดไม่ได้)
   - ขวา: ป้ายสถานะห้องชุดเดียวกับข้อ D
2. **แถบเตือน handoff** — *มีเงื่อนไข* แสดงเฉพาะห้องโหมดคนตอบ (ดูข้อ 3.3)
3. **พื้นที่ข้อความ** — พื้นหลังเทาอ่อนโปร่ง แยกตัวจากการ์ดขาว **ความสูงคงที่: 55% ของความสูงจอบนมือถือ / 60% บนเดสก์ท็อป** เลื่อนขึ้นลงได้ภายในตัวเอง เลื่อนไปข้อความใหม่สุดให้อัตโนมัติ
   - แสดง **100 ข้อความล่าสุด** เรียงเก่า→ใหม่ **ไม่มีการโหลดประวัติย้อนหลังเพิ่ม**
4. **กล่องพิมพ์ (Composer)** — เส้นคั่นด้านบน อยู่ล่างสุด**ในการ์ด** (ไม่ใช่แถบติดขอบจอ)
   - ช่องพิมพ์หลายบรรทัด สูง 2 บรรทัด สูงต่ำสุด 44px **ลากขยายเองไม่ได้** ยืดกินความกว้างที่เหลือ
   - ปุ่มสีแบรนด์ **"ส่ง"** ชิดขวา ไม่หด
   - **ไม่มีปุ่มแนบรูป ไม่มีอีโมจิพิกเกอร์ ไม่มีเทมเพลต/quick reply และกด Enter ส่งไม่ได้** (Enter = ขึ้นบรรทัดใหม่)

**F. ฟองข้อความ — 3 แบบ + 1 แบบพิเศษ**

การจัดฝั่ง: **ลูกค้าอยู่ซ้าย / AI และแอดมินอยู่ขวา (ทั้งคู่คือ "ฝั่งร้าน")**

ทุกฟอง: กว้างสุด 80% ของพื้นที่, ขอบมนมาก, ข้อความขึ้นบรรทัดตามที่พิมพ์จริง + ตัดคำยาวได้, เวลากำกับชิดขวาล่างตัวเล็กมาก (10px)

| แบบ | ฝั่ง | หน้าตา | ป้ายกำกับบนสุดในฟอง |
|---|---|---|---|
| แอดมินพูด | ขวา | พื้นสีแบรนด์ ตัวอักษรขาว เวลาเป็นขาวจาง | **"แอดมิน"** (10px หนา ขาวจาง) |
| AI พูด | ขวา | พื้นขาว ขอบบางสีแบรนด์จางครึ่งหนึ่ง ตัวอักษรเทาเข้ม | **"AI"** (10px หนา สีแบรนด์) |
| ลูกค้าพูด | ซ้าย | พื้นเทาอ่อน ตัวอักษรเทาเข้ม | ไม่มีป้าย |
| ข้อความระบบ | กลาง | **ไม่ใช่ฟอง** — ข้อความเปล่าจัดกึ่งกลาง ตัวเล็กสีเทาจาง | ไม่มี |

## 3. ทุก flow ทีละสเต็ป

### 3.1 เปิดหน้า → เลือกห้องแชท

**ขั้นตอน**
1. แอดมินกดเมนู "💬 แชทลูกค้า" → เข้าหน้า `/admin/chats`
2. หน้าเริ่มโหลดรายการห้อง → คอลัมน์รายการแสดงวงกลมหมุนพร้อมข้อความ **"กำลังโหลด…"**
3. โหลดเสร็จ → ห้องที่ "รอแอดมิน" ลอยขึ้นบนสุด ตามด้วยห้องที่มีข้อความใหม่สุด
4. ถ้ามีห้องรอ ≥ 1 ห้อง → ป้าย **"รอแอดมิน {n} ห้อง"** ปรากฏที่หัวหน้าจอมุมขวา
5. แอดมินกดการ์ดห้องแชท (กดได้ทั้งใบ)
6. การ์ดที่เลือกได้ขอบสีแบรนด์ • **บนมือถือ**: รายการหายไป ห้องสนทนาขึ้นเต็มจอ • **บนเดสก์ท็อป**: ห้องสนทนาโผล่คอลัมน์ขวา รายการยังอยู่
7. ห้องสนทนาแสดงวงกลมหมุน **"กำลังโหลด…"** ในพื้นที่ข้อความสั้นๆ → แล้วข้อความ 100 รายการล่าสุดขึ้น เรียงเก่า→ใหม่ และเลื่อนลงล่างสุดอัตโนมัติ

> หมายเหตุดีไซน์: **ทุกครั้งที่สลับห้อง จะเห็นวงกลมหมุนแวบหนึ่งในพื้นที่ข้อความ** เพราะห้องสนทนาถูกสร้างใหม่ทั้งบล็อก ไม่ใช่แค่เปลี่ยนข้อมูล

**กฎการตรวจสอบ**
- ไม่มี validation — เป็นการอ่านอย่างเดียว
- แสดงสูงสุด 50 ห้อง / 100 ข้อความต่อห้อง (จำกัดแข็ง)

**ข้อความสำเร็จ:** ไม่มี toast — ผลลัพธ์คือรายการแสดงผลและห้องเปิดขึ้น

**ข้อความล้มเหลว (โหลดรายการห้องพลาด):** แทนที่ทั้งคอลัมน์ด้วยบล็อกผิดพลาด
- ไอคอน ⚠️
- หัวข้อ: **"โหลดข้อมูลไม่สำเร็จ"**
- คำอธิบาย: **"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"**
- ปุ่มโปร่งขนาดเล็ก: **"ลองใหม่"**

### 3.2 ส่งข้อความหาลูกค้า — flow หลักของหน้านี้

**ขั้นตอน**
1. แอดมินพิมพ์ในช่องพิมพ์ (ข้อความชี้นำในช่องว่าง: **"พิมพ์ข้อความถึงลูกค้า…"**)
2. ปุ่ม **"ส่ง"** จะกดได้ก็ต่อเมื่อมีข้อความ (หลังตัดช่องว่างหัวท้ายแล้วไม่ว่าง) และไม่ได้กำลังส่งอยู่ — **ตอนช่องว่างเปล่า ปุ่มอยู่ในสถานะกดไม่ได้ตั้งแต่แรก**
3. กด "ส่ง" → ปุ่มเปลี่ยนข้อความเป็น **"กำลังส่ง…"** และกดไม่ได้
4. ระบบส่งไปเซิร์ฟเวอร์: เซิร์ฟเวอร์ตรวจสิทธิ์ admin → หาห้องในร้านเดียวกัน → **ถ้าห้องยังเป็นโหมด AI จะถูกสลับเป็นโหมดคนตอบทันที (= การพิมพ์ตอบ คือการเทคโอเวอร์ AI โดยปริยาย)** → บันทึกข้อความ → อัปเดตเวลาข้อความล่าสุด → push เข้า LINE ให้ลูกค้า
5. สำเร็จ → **ช่องพิมพ์ถูกเคลียร์ทันที**
6. **ถ้าลูกค้ายังไม่ได้ผูก LINE** → toast สีเทาเข้ม: **"บันทึกข้อความแล้ว แต่ลูกค้ายังไม่ได้เชื่อม LINE"**
7. **ถ้า push LINE สำเร็จตามปกติ → ไม่มี toast ใดๆ เลย**
8. ระบบดึงข้อความใหม่ + รีเฟรชรายการห้อง → ฟองสีแบรนด์ป้าย "แอดมิน" โผล่ชิดขวา, แถบเหลืองโผล่ (ถ้าเพิ่งเทคโอเวอร์), ห้องนี้เด้งขึ้นบนสุดของรายการพร้อมป้าย "รอแอดมิน"
9. เลื่อนลงหาฟองใหม่อัตโนมัติ → ปุ่มกลับเป็น **"ส่ง"**

**กฎการตรวจสอบ**
- **ฝั่งหน้าเว็บ:** ต้องมีข้อความหลังตัดช่องว่าง; กันกดซ้ำระหว่างกำลังส่ง
- **ฝั่งเซิร์ฟเวอร์:** ข้อความยาว **1–2000 ตัวอักษร**
  - ⚠️ **ช่องพิมพ์ปัจจุบันไม่มีการจำกัดความยาว และไม่มีตัวนับตัวอักษร** — พิมพ์เกิน 2000 ได้ แล้วค่อยเด้ง toast แดงหลังกดส่ง (ปรับได้ — ควรออกแบบตัวนับ)
- **ฝั่งเซิร์ฟเวอร์:** ต้องเป็น role `admin`; ห้องต้องอยู่ในร้านเดียวกัน มิฉะนั้นตอบกลับเป็น error
- ไม่มีการบังคับแนบรูป ไม่มี GPS ไม่มีไฟล์แนบใดๆ ใน flow นี้

**ข้อความสำเร็จ (verbatim)**
- กรณีปกติ: **ไม่มีข้อความ** — ฟีดแบ็กเดียวคือช่องพิมพ์ถูกเคลียร์ + ฟองข้อความใหม่โผล่ + แถบเหลือง **"AI หยุดตอบห้องนี้ — คุณกำลังดูแลลูกค้าอยู่"** โผล่ขึ้น
- กรณีลูกค้าไม่มี LINE: **"บันทึกข้อความแล้ว แต่ลูกค้ายังไม่ได้เชื่อม LINE"**

**ข้อความล้มเหลว (verbatim)**
- toast แดง แสดง **ข้อความ error จริงจากเซิร์ฟเวอร์ ซึ่งอาจเป็นภาษาอังกฤษ** เช่น "Session not found."
- ถ้าเซิร์ฟเวอร์ไม่ส่งข้อความมา → ใช้ข้อความสำรอง: **"ส่งข้อความไม่สำเร็จ"** หรือ **"ทำรายการไม่สำเร็จ"**
- **ข้อความที่พิมพ์ไม่ถูกลบเมื่อพลาด** — แอดมินกดส่งซ้ำได้ทันที

### 3.3 ปิดเคส คืนห้องให้ AI

**ขั้นตอน**
1. ห้องที่อยู่ในโหมดคนตอบจะมี **แถบสีเหลืองอ่อน** อยู่ใต้หัวห้อง (ขอบล่างเหลืองเข้มขึ้น) — ซ้ายเป็นข้อความ **"AI หยุดตอบห้องนี้ — คุณกำลังดูแลลูกค้าอยู่"** (ตัวเล็กสีเหลืองเข้ม) ขวาเป็นปุ่มโปร่งขนาดเล็ก **"ปิดเคส ให้ AI ตอบต่อ"**
   - บนจอแคบ ข้อความกับปุ่มจะตกบรรทัดกัน
   - **ห้องโหมด AI จะไม่มีแถบนี้เลย และปิดเคสไม่ได้**
2. กดปุ่ม "ปิดเคส ให้ AI ตอบต่อ" → **เปิดหน้าต่างยืนยัน** (บนมือถือเลื่อนขึ้นจากขอบล่างแบบ bottom sheet มุมบนมน; บนจอกว้างลอยกลางจอ; ฉากหลังดำโปร่ง 40%)
3. หน้าต่างยืนยันแสดง:
   - หัวข้อตัวหนา: **"ปิดเคสห้องนี้?"**
   - ข้อความ: **"AI จะกลับมาตอบลูกค้าห้องนี้อัตโนมัติ"**
   - ปุ่ม ✕ มุมขวาบน (ป้ายช่วยการเข้าถึง: "ปิด")
   - ปุ่ม 2 ปุ่มเรียงแถวเดียว กว้างเท่ากันคนละครึ่ง: **"ยกเลิก"** (ปุ่มโปร่ง) และ **"ปิดเคส"** (ปุ่มสีแบรนด์ — **ไม่ใช่ปุ่มแดง** เพราะไม่ใช่การทำลายข้อมูล)
4. กด "ยกเลิก" หรือ ✕ หรือแตะฉากหลัง → ปิดหน้าต่าง ไม่เกิดอะไรขึ้น (แตะในตัวหน้าต่างไม่ปิด)
5. กด "ปิดเคส" → ปุ่มบนแถบเหลืองเปลี่ยนเป็น **"กำลังปิดเคส…"** และกดไม่ได้
6. เซิร์ฟเวอร์: ตรวจสิทธิ์ admin → หาห้องในร้าน → สลับห้องกลับเป็นโหมด AI **และล้างธง "ร้องเรียน"/"ฉุกเฉิน" ทิ้งไปด้วย** → เขียนข้อความระบบลงบทสนทนา
7. toast เขียว: **"ปิดเคสแล้ว — AI กลับมาตอบต่อ"**
8. หน้าจอรีเฟรช → **แถบเหลืองหายไป**, **ป้ายทั้งหมดของห้องนี้หายไป** (ทั้งเหลืองและแดง), ห้องหล่นลงจากบนสุด, ตัวเลขที่หัวหน้าจอลดลง 1 (หรือป้ายหายไปทั้งอันถ้าเหลือ 0)
9. ข้อความระบบสีเทากลางจอปรากฏท้ายบทสนทนา: **"แอดมินปิดเคส — AI กลับมาตอบอัตโนมัติ"**

**กฎการตรวจสอบ**
- ต้องผ่านหน้าต่างยืนยันเสมอ — ไม่มีทางลัด
- กันกดซ้ำ: ปุ่มกดไม่ได้ระหว่างกำลังทำงาน
- ฝั่งเซิร์ฟเวอร์: ต้องเป็น role `admin` และห้องต้องอยู่ในร้านเดียวกัน
- เซิร์ฟเวอร์ **ไม่ได้เช็ค**ว่าห้องเป็นโหมดคนตอบอยู่ก่อนหรือไม่ — หน้าเว็บอาศัยการซ่อนปุ่มแทน

**ข้อความสำเร็จ (verbatim):** toast เขียว **"ปิดเคสแล้ว — AI กลับมาตอบต่อ"** + ข้อความระบบในบทสนทนา **"แอดมินปิดเคส — AI กลับมาตอบอัตโนมัติ"**

**ข้อความล้มเหลว (verbatim):** toast แดงแสดง error จริงจากเซิร์ฟเวอร์ (อาจเป็นอังกฤษ เช่น "Session not found.") — ข้อความสำรอง: **"ปิดเคสไม่สำเร็จ"** • แถบเหลืองยังอยู่ ปุ่มกลับมากดได้อีก

---

### 3.4 โทรหาลูกค้า

**ขั้นตอน**
1. ในหัวห้องสนทนา ถ้าลูกค้ามีเบอร์ → แสดงลิงก์สีแบรนด์ตัวเล็ก รูปแบบ **"📞 {เบอร์}"**
2. แอดมินกดลิงก์ → มือถือเปิดแอปโทรออกพร้อมเบอร์
3. ถ้าไม่มีเบอร์ → แสดงข้อความเทา **"ไม่มีเบอร์โทร"** (ไม่ใช่ลิงก์ กดไม่ได้)

**กฎการตรวจสอบ:** ไม่มี — เบอร์ถูกส่งไปดิบๆ ไม่มีการจัดรูปแบบหรือตรวจความถูกต้อง

**ข้อความสำเร็จ:** ไม่มีในตัวเว็บ — ระบบปฏิบัติการเปิดแอปโทรออกเอง

**ข้อความล้มเหลว:** ไม่มี — บนเดสก์ท็อปที่ไม่มีแอปรองรับ กดแล้วอาจไม่เกิดอะไรขึ้นเลย (จุดอ่อน: ไม่มีทางรู้ว่ากดติดหรือไม่)

---

### 3.5 กลับไปหน้ารายการ (เฉพาะมือถือ)

**ขั้นตอน**
1. ขณะเปิดห้องสนทนาบนมือถือ กดปุ่ม **"←"** มุมซ้ายบนของหัวห้อง (ป้ายช่วยการเข้าถึง: "กลับไปรายชื่อแชท")
2. ห้องสนทนาถูกซ่อน รายการห้องกลับมาเต็มจอทันที

**กฎการตรวจสอบ:** ไม่มี

**ข้อความสำเร็จ / ล้มเหลว:** ไม่มีทั้งคู่ (เปลี่ยนหน้าเงียบๆ)

> ⚠️ **บนเดสก์ท็อปไม่มีปุ่มนี้ และไม่มีวิธียกเลิกการเลือกห้องเลย** — เปลี่ยนได้แค่คลิกห้องอื่นเท่านั้น (ปรับได้)

---

### 3.6 รับข้อความใหม่จากลูกค้าแบบเรียลไทม์ (ไม่ต้องทำอะไร)

**ขั้นตอน**
1. เมื่อเปิดห้องใดห้องหนึ่ง ระบบจะเฝ้าฟังข้อความใหม่ของ **ห้องนั้นห้องเดียว**
2. ลูกค้าส่งข้อความเข้ามาทาง LINE → ระบบดึงข้อความใหม่ทันที
3. ฟองสีเทาชิดซ้ายโผล่ขึ้น + หน้าจอเลื่อนลงล่างสุดอัตโนมัติ
4. เมื่อเปลี่ยนห้องหรือออกจากหน้า ระบบเลิกเฝ้าฟังห้องเดิม

**กฎการตรวจสอบ:** ไม่มี

**ข้อความสำเร็จ:** ไม่มีข้อความ — ฟองใหม่ปรากฏเองโดยไม่ต้องรีเฟรช

**ข้อความล้มเหลว:** **ไม่มีการแจ้งเตือนใดๆ เมื่อการเชื่อมต่อเรียลไทม์หลุด** — ระบบเงียบๆ ตกไปใช้การดึงข้อมูลใหม่ทุก 15 วินาทีแทน แอดมินไม่มีทางรู้ว่าตอนนี้เร็วหรือช้า

> ⚠️ **เรียลไทม์ครอบคลุมเฉพาะห้องที่เปิดอยู่** — รายการห้องฝั่งซ้ายไม่มีเรียลไทม์เลย

---

## 4. สถานะของหน้า

| สถานะ | ผู้ใช้เห็นอะไร | ข้อความจริง |
|---|---|---|
| **กำลังโหลด — รายการห้อง** | วงกลมหมุนเล็ก (ขอบเทา ส่วนบนสีแบรนด์) + ข้อความเทาจาง จัดกึ่งกลาง มีระยะห่างบนล่างมาก แทนทั้งคอลัมน์ • **ไม่มี skeleton ไม่มี shimmer** | "กำลังโหลด…" |
| **กำลังโหลด — ข้อความในห้อง** | วงกลมหมุนตัวเดียวกัน อยู่ในกล่องสูง 55vh/60vh พื้นเทาอ่อน • เห็นทุกครั้งที่สลับห้อง | "กำลังโหลด…" |
| **ว่าง — ไม่มีห้องแชทเลย** | ไอคอน 💬 ขนาดใหญ่ + หัวข้อเทาเข้ม + คำใบ้เทาจาง จัดกึ่งกลาง | หัวข้อ: "ยังไม่มีห้องแชท" / คำใบ้: "เมื่อลูกค้าทักผ่าน LINE จะแสดงที่นี่" |
| **ว่าง — ยังไม่เลือกห้อง (เดสก์ท็อปเท่านั้น)** | การ์ดขาวในคอลัมน์ขวา บรรจุไอคอน 👈 + หัวข้อ + คำใบ้ • บนมือถือไม่เห็นสถานะนี้ (คอลัมน์ถูกซ่อน) | หัวข้อ: "เลือกห้องแชทจากรายการ" / คำใบ้: "ห้องที่รอแอดมินจะอยู่บนสุด" |
| **ว่าง — ห้องที่ยังไม่มีข้อความ** | ไอคอน 💬 + หัวข้อ **ไม่มีคำใบ้** อยู่ในกล่องสูง 55vh/60vh พื้นเทาอ่อน | หัวข้อ: "ยังไม่มีข้อความ" (ไม่มีคำใบ้) |
| **ผิดพลาด — โหลดข้อมูลไม่ได้** (ใช้ทั้งรายการห้องและพื้นที่ข้อความ เนื้อหาเหมือนกันเป๊ะ) | ไอคอน ⚠️ ขนาดใหญ่ + หัวข้อเทาเข้ม + คำอธิบายเทาจาง + ปุ่มโปร่งเล็ก จัดกึ่งกลาง | หัวข้อ: "โหลดข้อมูลไม่สำเร็จ" / คำอธิบาย: "อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ" / ปุ่ม: "ลองใหม่" |
| **กำลังส่ง (busy)** | ปุ่มสีแบรนด์เปลี่ยนข้อความ + เข้าสถานะกดไม่ได้ • **ช่องพิมพ์ยังพิมพ์ต่อได้** • **ไม่มีฟองข้อความล่วงหน้า (no optimistic bubble)** — หน้าจอนิ่งสนิทระหว่างรอ | ปุ่ม: "กำลังส่ง…" |
| **ปุ่มส่งกดไม่ได้ (ช่องว่าง)** | ปุ่มสีแบรนด์ในสถานะจาง กดไม่ได้ ตั้งแต่ยังไม่พิมพ์อะไร | ปุ่ม: "ส่ง" |
| **กำลังปิดเคส (busy)** | ปุ่มบนแถบเหลืองเปลี่ยนข้อความ + กดไม่ได้ | ปุ่ม: "กำลังปิดเคส…" |
| **ห้องอยู่ในมือแอดมิน (handoff)** | แถบเหลืองอ่อนใต้หัวห้อง + ป้ายเหลือง "รอแอดมิน" ในหัวห้องและในการ์ดรายการ | "AI หยุดตอบห้องนี้ — คุณกำลังดูแลลูกค้าอยู่" + ปุ่ม "ปิดเคส ให้ AI ตอบต่อ" |
| **ไม่มีสิทธิ์ (role อื่น)** | **ไม่มีหน้าจอ permission-denied ใดๆ** — ถูกเด้งไปหน้าแรกของ role ตัวเองเงียบๆ ก่อนที่หน้านี้จะแสดง | (ไม่มีข้อความ) |
| **สิทธิ์ข้อมูลถูกตัด (RLS)** | ⚠️ กรณีขอบ: ถ้าฐานข้อมูลไม่คืนห้องใดเลย จะเห็น **สถานะว่าง ไม่ใช่สถานะผิดพลาด** — แอดมินอาจเข้าใจผิดว่า "ไม่มีลูกค้าทักมา" | "ยังไม่มีห้องแชท" |

## 5. ข้อความไทยทั้งหมดบนหน้านี้

| ที่ใช้ | ข้อความจริง (verbatim) |
|---|---|
| หัวหน้าจอ | `แชทลูกค้า` |
| ป้ายนับที่หัวหน้าจอ (แบบไดนามิก) | `รอแอดมิน {จำนวน} ห้อง` — เช่น `รอแอดมิน 3 ห้อง` |
| ป้ายบนการ์ด/หัวห้อง (เหลือง) | `รอแอดมิน` |
| ป้ายบนการ์ด/หัวห้อง (แดง) | `ร้องเรียน` |
| ป้ายบนการ์ด/หัวห้อง (แดง) | `ฉุกเฉิน` |
| ชื่อลูกค้าเมื่อไม่มีชื่อเล่น (fallback) | `ลูกค้า` |
| เบอร์โทรเมื่อไม่มีเบอร์ (fallback, ทั้งการ์ดและหัวห้อง) | `ไม่มีเบอร์โทร` |
| สถานะว่าง — รายการห้อง (หัวข้อ) | `ยังไม่มีห้องแชท` |
| สถานะว่าง — รายการห้อง (คำใบ้) | `เมื่อลูกค้าทักผ่าน LINE จะแสดงที่นี่` |
| สถานะว่าง — ยังไม่เลือกห้อง (หัวข้อ) | `เลือกห้องแชทจากรายการ` |
| สถานะว่าง — ยังไม่เลือกห้อง (คำใบ้) | `ห้องที่รอแอดมินจะอยู่บนสุด` |
| สถานะว่าง — ห้องไม่มีข้อความ (หัวข้อ, ไม่มีคำใบ้) | `ยังไม่มีข้อความ` |
| ปุ่มย้อนกลับ ← (ป้ายช่วยการเข้าถึง) | `กลับไปรายชื่อแชท` |
| แถบเตือน handoff | `AI หยุดตอบห้องนี้ — คุณกำลังดูแลลูกค้าอยู่` |
| ปุ่มปิดเคส (ปกติ) | `ปิดเคส ให้ AI ตอบต่อ` |
| ปุ่มปิดเคส (กำลังทำงาน) | `กำลังปิดเคส…` |
| ช่องพิมพ์ (ข้อความชี้นำ) | `พิมพ์ข้อความถึงลูกค้า…` |
| ปุ่มส่ง (ปกติ) | `ส่ง` |

## 7. ข้อมูลอัปเดตยังไง

**ผสม 2 กลไก และเร็วไม่เท่ากัน — นี่คือประเด็นดีไซน์สำคัญ**

| ส่วน | กลไก | ความหน่วง |
|---|---|---|
| **ข้อความในห้องที่เปิดอยู่** | **Realtime** — เฝ้าฟังข้อความใหม่ของห้องนั้นโดยตรง + ดึงซ้ำทุก 15 วินาทีเป็นตัวสำรองเงียบๆ | แทบทันที |
| **รายการห้องแชทฝั่งซ้าย** | **Polling อย่างเดียว ทุก 15 วินาที** — ไม่มี realtime เลย | **ช้าได้ถึง 15 วินาที** |
| **หลังกดส่ง / กดปิดเคส** | ดึงข้อมูลใหม่ทันทีทั้งข้อความและรายการห้อง | ทันที |

**ผลต่อดีไซน์**
- **ห้องใหม่ที่รอแอดมิน อาจใช้เวลาถึง 15 วินาทีกว่าจะโผล่ในรายการ** และป้าย "รอแอดมิน {n} ห้อง" กับป้ายแดง "ร้องเรียน"/"ฉุกเฉิน" ก็ช้าตามกันหมด — สำหรับเคสฉุกเฉิน นี่คือช่องว่างที่ควรถูกแก้
- **ไม่มีตัวบอกว่า "อัปเดตล่าสุดเมื่อไหร่" เลย** และไม่มีปุ่มรีเฟรชด้วยตัวเอง — แอดมินไม่มีทางรู้ว่าที่เห็นอยู่สดหรือค้าง **แนะนำให้ดีไซน์ใหม่พิจารณาเพิ่ม** (ปรับได้)
- **ไม่มีสัญญาณบอกว่าการเชื่อมต่อ realtime ยังดีอยู่หรือหลุดไปแล้ว** — เวลาหลุด ประสบการณ์เปลี่ยนจาก "ทันที" เป็น "ช้า 15 วิ" แบบเงียบสนิท
- **ไม่มี typing indicator ไม่มี read receipt ไม่มี presence ไม่มีเสียงเตือน ไม่มี browser notification ไม่มี badge บน favicon**

## 8. สิ่งที่ designer ต้องรู้

### 🔒 ห้ามแตะ — ผูกกับกติกาธุรกิจ (Ready.md §4, §7)

- **ลำดับสายตา "ห้องไหนรอแอดมิน" คือหัวใจของหน้านี้** — มีสัญญาณ 4 ตัวที่สื่อเรื่องเดียวกันและต้องคงไว้ทั้งหมด: (1) ป้ายนับที่หัวจอ, (2) ห้องรอลอยขึ้นบนสุดของรายการ, (3) ป้ายเหลือง "รอแอดมิน" บนการ์ด, (4) แถบเหลืองในห้องสนทนา
- **ความหมายของสี — ห้ามสลับ:**
  - **เหลืองอำพัน = ต้องการแอดมิน** (ห้องอยู่โหมดคนตอบ, AI หยุดแล้ว)
  - **แดง = ธงร้องเรียน / ฉุกเฉิน** — ต้องสะดุดตาสูงสุดบนหน้า เพราะเป็นเคสลูกค้าโกรธหรือเหตุด่วนจริง
  - **สีแบรนด์ = แอดมินพูด** / **ขาวขอบแบรนด์ = AI พูด** / **เทา = ลูกค้าพูด**
- **การพิมพ์ตอบ = การเทคโอเวอร์ AI โดยปริยาย** — ไม่มีปุ่ม "รับเคส" และไม่ควรมี: แค่ส่งข้อความ AI ก็หยุดตอบห้องนั้นทันทีจนกว่าจะปิดเคส นี่คือกติกาฝั่งเซิร์ฟเวอร์ ดีไซน์เปลี่ยนไม่ได้
- **ปิดเคส = คืนห้องให้ AI** และเป็นทางเดียวที่ AI จะกลับมาตอบ — ต้องผ่านหน้าต่างยืนยันเสมอ
- **ปุ่มยืนยัน "ปิดเคส" ไม่ใช่ปุ่มแดง** โดยตั้งใจ — ไม่ใช่การทำลายข้อมูล อย่าเปลี่ยนเป็นสีอันตราย
- **ข้อความไทยทุกคำในตารางข้อ 5 คือของจริง — ห้ามคิดคำใหม่เอง** ถ้าเห็นว่าคำไหนควรแก้ ให้เสนอเป็นข้อเสนอแยก อย่าเปลี่ยนเงียบๆ
- **ทุกข้อความที่กดส่งถึงลูกค้าจริงทาง LINE ทันที — ไม่มี undo ไม่มีแก้ไข ไม่มีลบ** ดีไซน์ต้องช่วยไม่ให้กดส่งพลาด (นี่เป็นเหตุผลหนึ่งที่ Enter ไม่ส่ง)
- **ข้อความระบบ (system_notice) มาจากฐานข้อมูลดิบๆ** — ดีไซน์คุมเนื้อหาไม่ได้ คุมได้แค่รูปลักษณ์

### ⚠️ กับดักและข้อจำกัดที่ต้องรู้

- **ป้ายซ้อนกันได้ในพื้นที่แคบมาก** — ห้องหนึ่งอาจมี "รอแอดมิน" + "ฉุกเฉิน" พร้อมกัน 2 ป้าย ตกบรรทัดในมุมขวาของการ์ดที่กว้างแค่ 260–320px รวมกับเวลาข้างบน **ต้องทดสอบเคสนี้จริง**
- **ชื่อเล่นและเบอร์ตัดด้วย … ได้ทั้งคู่** ในคอลัมน์แคบ
- **ผลข้างเคียงที่หน้าต่างยืนยันบอกไม่ครบ:** การปิดเคส **ล้างธง "ร้องเรียน"/"ฉุกเฉิน" ทิ้งไปด้วย** แต่ข้อความปัจจุบันพูดถึงแค่ "AI จะกลับมาตอบ" — เป็นการสูญเสียข้อมูลที่แอดมินอาจไม่คาดคิด (ควรเสนอแก้ข้อความ)
- **สลับห้องทีไรก็เห็นวงกลมหมุนแวบหนึ่งทุกครั้ง** — ไม่ใช่การเปลี่ยนข้อมูลแบบลื่นๆ
- **ช่องพิมพ์ไม่ถูกล็อกระหว่างส่ง** — ถ้าแอดมินพิมพ์เพิ่มระหว่างรอ สิ่งที่พิมพ์เพิ่มจะ**ถูกล้างทิ้ง**ตอนส่งสำเร็จ (เป็นบั๊กเชิงประสบการณ์ที่ดีไซน์ควรเสนอทางแก้)
- **บนเดสก์ท็อปยกเลิกการเลือกห้องไม่ได้** — ไม่มีปุ่มปิด/ย้อนกลับ

### 🔧 จุดที่ดีไซน์ปัจจุบันอ่อน — ปรับได้ (เสนอปรับปรุงได้เต็มที่)

- **ไม่มีตัวอย่างข้อความล่าสุดในรายการห้อง** — การ์ดมีแค่ชื่อ/เบอร์/เวลา/ป้าย แอดมินต้องเปิดห้องถึงจะรู้ว่าลูกค้าพูดอะไร **นี่คือช่องว่างที่ใหญ่ที่สุดของหน้านี้**
- **ไม่มีตัวบอกข้อความที่ยังไม่ได้อ่าน (unread) เลย** — ไม่มีจุด ไม่มีตัวเลข ไม่มีตัวหนา
- **ป้าย "AI" กับ "แอดมิน" เล็กแค่ 10px** และทั้งคู่อยู่ฝั่งขวาเหมือนกัน — ในสถานการณ์เร่งรีบ แอดมินอาจแยกไม่ออกว่าประโยคนี้ AI พูดหรือตัวเองพูด **ควรแยกให้ชัดกว่านี้**
- **ไม่มีฟองข้อความล่วงหน้า (optimistic bubble)** — หลังกดส่ง หน้าจอนิ่งสนิทยกเว้นข้อความบนปุ่ม ระหว่างรอ round-trip (เซิร์ฟเวอร์ → push LINE → ดึงข้อมูลใหม่) ซึ่งไม่ใช่เวลาสั้นๆ **ควรพิจารณาเพิ่มฟองรอส่ง**
- **ส่งสำเร็จปกติไม่มี toast เลย** — ฟีดแบ็กเดียวคือฟองที่โผล่ทีหลัง
- **ไม่มีตัวนับตัวอักษร ทั้งที่เซิร์ฟเวอร์จำกัด 2000 ตัว** — เกินแล้วเพิ่งรู้ตอนได้ toast แดง
- **พื้นที่ข้อความสูงคงที่ 55vh/60vh ไม่ยืดตามจอ** — จอสูงมากจะเหลือที่ว่างเปล่าใต้การ์ด จอเตี้ยจะเห็นข้อความน้อยมาก และ **กล่องพิมพ์อยู่ในการ์ด ไม่ใช่แถบติดขอบจอ** — ต่างจากแอปแชททั่วไปทุกตัว **นี่คือจุดที่ควรออกแบบใหม่ให้เป็น layout เต็มความสูงจริง**
- **ลิงก์โทรเป็นแค่ตัวอักษรขนาด 12px** — เล็กเกินไปสำหรับ touch target ทั้งที่มันคือ action สำคัญบนมือถือ **ควรทำเป็นปุ่มจริง**
- **ไม่มีบริบทลูกค้าเลยนอกจากชื่อเล่นกับเบอร์** — ไม่มีออเดอร์ล่าสุด ไม่มีที่อยู่ ไม่มีโซนจัดส่ง ไม่มีรูปโปรไฟล์ LINE ไม่มีลิงก์ไปหน้าออเดอร์ของลูกค้าคนนี้ แอดมินต้องเปิดหน้า "ออเดอร์" อีกแท็บมาดูเทียบเอง **ช่องว่างใหญ่อีกจุด**
- **ไม่มีช่องค้นหา ไม่มีตัวกรอง ไม่มีการโหลดเพิ่ม** — ห้องที่ 51 เป็นต้นไปเข้าไม่ถึงเลย และประวัติที่เก่ากว่า 100 ข้อความดูไม่ได้ (ถ้าดีไซน์ใหม่ต้องการ ต้องระบุเป็น requirement ให้ฝั่งพัฒนา)
- **ไม่มีเทมเพลตข้อความ / quick reply** — ทั้งที่แอดมินน่าจะพิมพ์ประโยคซ้ำๆ ทุกวัน
- **ส่งได้เฉพาะข้อความล้วน** — ไม่มีแนบรูป/สติกเกอร์/อีโมจิพิกเกอร์ (ฝั่งเซิร์ฟเวอร์ก็รองรับแค่ข้อความ ถ้าจะเพิ่มต้องแก้ทั้งระบบ)
- **ไม่มีเสียง ไม่มี notification** — ถ้าดีไซน์ใหม่ต้องการความไว ต้องระบุเป็น requirement ชัดเจน
- **สถานะไม่มีสิทธิ์ไม่มีหน้าจอรองรับ** — role อื่นถูกเด้งเงียบๆ ไม่มีอะไรให้ดีไซน์ (แต่ถ้าเห็นว่าควรมีหน้าอธิบาย ก็เสนอได้)

### 📐 รายละเอียดที่มีอยู่แล้วและควรคงไว้

- toast เด้งจากล่าง หายเองใน **3.6 วินาที ปิดเองไม่ได้** วางที่เหนือแถบเมนูล่างบนมือถือ / ใกล้ขอบล่างบนเดสก์ท็อป กว้างสุดประมาณครึ่งจอเล็ก จัดกึ่งกลาง
- หน้าต่างยืนยันเป็น **bottom sheet บนมือถือ / dialog กลางจอบนเดสก์ท็อป** สูงสุด 90% ของจอ เลื่อนได้ แตะฉากหลังเพื่อยกเลิกได้ กว้างสุดขนาดกลาง
- flow มือถือถูกออกแบบมาให้ **ใช้มือเดียว**: ปุ่มย้อนกลับซ้ายบน, กล่องพิมพ์ล่างสุด, toast อยู่เหนือแถบเมนู — คงหลักนี้ไว้
- บล็อกสถานะว่างกับบล็อกผิดพลาด **จงใจแยกกันคนละหน้าตา** เพื่อไม่ให้แอดมินเข้าใจผิดว่า "โหลดพลาด" คือ "ไม่มีข้อมูล" — อย่ารวมเป็นคอมโพเนนต์เดียวกัน
