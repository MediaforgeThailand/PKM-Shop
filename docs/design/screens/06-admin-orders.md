# ออเดอร์ — ศูนย์ควบคุมงานออเดอร์ของแอดมิน (`/admin/orders`)

> แอดมินร้าน (role `admin` เท่านั้น) ใช้ตลอดทั้งวันทำการ เพื่อดูออเดอร์วันนี้ คุมรอบไรเดอร์ จัดการรอบพัสดุ Kerry ค้นหาออเดอร์ย้อนหลัง และสร้างออเดอร์ให้ลูกค้าที่โทรเข้ามา

---

## 1. บริบทการใช้งาน

**ใครใช้**
- แอดมินร้านเท่านั้น (role `admin`) ตรวจสิทธิ์ 2 ชั้น: ชั้นแรกที่ route (`Guard need={['admin']}`) ชั้นที่สองที่ server ทุกครั้งที่กดปุ่ม
- คนที่ไม่ใช่แอดมินจะไม่มีทางเห็นหน้านี้เลย — ระบบเด้งไปหน้าแรกของ role ตัวเอง (stock → `/catalog`, packer → `/packer`, rider → `/rider`, อื่นๆ → `/staff`) **ไม่มีหน้าจอ "คุณไม่มีสิทธิ์" ให้ออกแบบ**
- หน้านี้คือ **หน้าแรกที่แอดมินเห็นทันทีหลัง login** (`homeFor('admin')` → `/admin/orders`) → เป็น first impression ของระบบหลังบ้านทั้งหมด

**อุปกรณ์และสถานการณ์จริง**
- **Mobile-first** — และนี่คือของจริง ไม่ใช่คำพูดลอยๆ: ปุ่มหลักของหน้าชื่อ `"+ โทรสั่ง"` แปลว่าสถานการณ์การใช้งานคือ **แอดมินถือหูโทรศัพท์ไว้ข้างหนึ่ง อีกมือกรอกฟอร์มตามที่ลูกค้าพูด** ฟอร์มสร้างออเดอร์ต้องกรอกมือเดียวได้ และลำดับช่องต้องตรงกับลำดับที่ลูกค้าพูด (สินค้า → วิธีส่ง → ชื่อ → เบอร์ → ที่อยู่) — ลำดับปัจจุบันถูกต้องแล้ว **ห้ามสลับ**
- แต่ต้องใช้ได้ดีบนจอใหญ่ด้วย: แอดมินนั่งโต๊ะคุมบอร์ดทั้งวัน เนื้อหากว้างสุด `max-w-5xl`
- ตัวหน้าถูกครอบด้วยโครง Shell: header ติดบนตลอด (`PKM Shop / หลังบ้าน` + ปุ่ม `"ออก"`), แถบเมนูด้านบนบนจอ sm ขึ้นไป, แถบแท็บล่างจอแบบ fixed สูง 56px บนมือถือ (เนื้อหาจึงเว้นล่างไว้กันทับ) โดยเมนู `"ออเดอร์"` ไอคอน 🧾 เป็นรายการแรก

**ความถี่และความเร่งรีบ**
- เปิดค้างไว้ทั้งวัน กลับมามองบ่อยมาก (แทบทุกครั้งที่มีออเดอร์เข้า)
- เร่งรีบสูงเฉพาะตอนรับสาย (ลูกค้ารออยู่ปลายสาย) และตอน Kerry มารับพัสดุ (มีหน้าต่างเวลาจำกัด)
- **ไม่มีการถ่ายรูปและไม่มี GPS ในหน้านี้เลย** — งานภาพถ่าย (สลิป/หลักฐานส่ง) อยู่หน้า packer/rider หน้านี้เป็นงาน "อ่านสถานะ + กรอกเลข + กดปุ่ม" ล้วนๆ
- **ไม่มี dark mode** ทั้งระบบ — ทุกอย่างเป็นโทนสว่าง ถ้าจะใช้กลางแจ้งหน้าร้านต้องคิดเรื่อง contrast

## 2. โครงหน้า

เรียงจากบนลงล่าง:

1. **หัวข้อหน้า** — แถวเดียว ซ้าย-ขวา
   - ซ้าย: ข้อความ `"ออเดอร์"` (ตัวหนา ขนาดใหญ่กว่าเนื้อความ)
   - ขวา: ปุ่มหลักขนาดเล็ก `"+ โทรสั่ง"` → เปิดฟอร์มสร้างออเดอร์
   - **ปุ่มนี้อยู่ติดหน้าตลอดทุกแท็บ** ไม่ผูกกับแท็บใดแท็บหนึ่ง

2. **แถบแท็บ 4 อัน** — กล่องพื้นเทาอ่อน ขอบมน เลื่อนแนวนอนได้ถ้าไม่พอ; แต่ละปุ่มสูงอย่างน้อย 40px และแบ่งพื้นที่เท่ากัน; แท็บที่เลือกเป็นพื้นขาว ตัวหนังสือเข้ม มีเงาบางๆ ส่วนที่ไม่ได้เลือกเป็นตัวหนังสือสีเทา
   - `"บอร์ดวันนี้"` (ค่าเริ่มต้นเมื่อเปิดหน้า)
   - `"รอบไรเดอร์"`
   - `"Kerry"`
   - `"ค้นหา"`
   - **หมายเหตุสำคัญ: ทั้ง 4 แท็บไม่มีตัวเลขแจ้งเตือน** ทั้งที่คอมโพเนนต์แท็บรองรับ badge ตัวเลขอยู่แล้ว (วงกลมสี brand ตัวหนังสือขาว)

3. **พื้นที่เนื้อหาของแท็บที่เลือก** (รายละเอียดแต่ละแท็บด้านล่าง)

### แท็บ 1 · `"บอร์ดวันนี้"`
ไม่ใช่ kanban แนวนอน แต่เป็น **5 กลุ่มเรียงลงมาแนวตั้ง** มีระยะห่างระหว่างกลุ่ม แต่ละกลุ่มประกอบด้วย:
- หัวข้อกลุ่ม (ตัวเล็ก กึ่งหนา สีเทา) + **ตัวนับจำนวนในวงกลมพื้นเทา** (ตัวเลขหนาขนาด 11px)
- รายการการ์ดออเดอร์ใต้หัวข้อ
- ถ้ากลุ่มว่าง → กล่องเส้นประ ตัวหนังสือเทาจาง `"ไม่มีออเดอร์"` — **กลุ่มยังแสดงเสมอ ไม่ถูกซ่อน**

5 กลุ่ม (ครอบคลุมครบทั้ง 11 สถานะ):

| ลำดับ | หัวข้อกลุ่ม (verbatim) | รวมสถานะอะไรบ้าง |
|---|---|---|
| 1 | `"รอชำระ"` | pending |
| 2 | `"ชำระแล้ว / เข้ารอบ"` | paid, confirmed |
| 3 | `"กำลังแพ็ค"` | packing, packed |
| 4 | `"กำลังส่ง"` | out_for_delivery, delivering |
| 5 | `"เสร็จ / ตีกลับ / ยกเลิก"` | delivered, returned, awaiting_redelivery_fee, cancelled |

ดึงเฉพาะออเดอร์ที่สั่งใน **"วันนี้" ตามเวลากรุงเทพ** สูงสุด 300 รายการ เรียงใหม่→เก่า

### การ์ดออเดอร์ (ใช้ร่วมกันระหว่างแท็บ "บอร์ดวันนี้" และ "ค้นหา")
การ์ดพื้นขาว ขอบมน มีเงาบาง — **กดได้ทั้งใบ** (ทั้งการ์ดเป็นปุ่มเดียว) กดที่ไหนก็เปิดหน้าต่างรายละเอียด
- ซ้ายบน: เลขออเดอร์ (ฟอนต์ monospace กึ่งหนา) + วันเวลาสั่ง (ตัวเล็ก 11px สีเทาจาง)
- ซ้ายล่าง: บรรทัดเดียวตัดท้ายด้วย … รูปแบบ `ชื่อผู้รับ · วิธีส่ง · ยอดเงิน`
- ขวา: ป้ายสถานะ (ไม่ยุบขนาด)

### แท็บ 2 · `"รอบไรเดอร์"`
การ์ดรอบเรียงตามเวลารอบ (เก่า→ใหม่ ไม่จำกัดจำนวน) — เฉพาะรอบของวันนี้ แต่ละการ์ด:
- หัวการ์ด: ซ้าย `"รอบ {ชช:นน} น."` (กึ่งหนา) / ขวา ป้ายสถานะรอบพื้นเทา
- บรรทัดรอง (ตัวเล็กสีเทา): `🛵 {ชื่อไรเดอร์} · {n} จุด` — ถ้ายังไม่มีไรเดอร์แสดง `"ยังไม่มีไรเดอร์รับรอบ"` แทนชื่อ
- รายการออเดอร์ในรอบ: แต่ละแถวเป็นปุ่มเต็มความกว้าง ขอบบางสีอ่อน กดแล้วพื้นเปลี่ยนสี แสดง `จุด {n} · {order_no}` + บรรทัดรอง `ชื่อผู้รับ · ยอดเงิน` + ป้ายสถานะทางขวา
- เรียงตามลำดับจุด โดย**จุดที่ยังไม่ถูกจัดลำดับจะขึ้นก่อน** และแสดงเป็น `จุด - · {order_no}`
- **แต่ละการ์ดโหลดออเดอร์ของตัวเองแยกกัน** → สถานะกำลังโหลด/ว่าง/ผิดพลาด เกิดขึ้น **ในการ์ดใบนั้นใบเดียว** ไม่กระทบการ์ดอื่น

สถานะรอบ 5 แบบ (verbatim): `"เปิดรับออเดอร์"` / `"ปิดรอบ รอไรเดอร์รับ"` / `"ไรเดอร์รับแล้ว"` / `"กำลังส่ง"` / `"จบรอบ"`

### แท็บ 3 · `"Kerry"`
เรียงจากบนลงล่าง:
1. **แถบเตือนพื้นเหลืองอ่อน ขอบมน ตัวหนังสือสีน้ำตาลอำพัน** — เป็นข้อความคงที่ ไม่ใช่ข้อมูลจริงจากระบบ:
   `"📦 Kerry เข้ารับพัสดุช่วง 11:00–14:00 น. (แล้วแต่วัน) — เปิดรอบแล้วเตรียมของให้พร้อมก่อนช่วงเวลานี้"`
2. **การ์ดรอบ Kerry วันนี้**
   - มีรอบแล้ว → ซ้าย `"รอบ Kerry วันนี้"` / ขวา ป้ายสถานะรอบ
   - ยังไม่มีรอบ → ข้อความสีเทา `"ยังไม่ได้เปิดรอบ Kerry ของวันนี้"`
   - ใต้ลงมา: ปุ่มหลักเต็มความกว้าง **ที่เปลี่ยนข้อความตามสถานะ** (ดู flow ที่ 3)
3. **รายการพัสดุ Kerry ค้างส่งมอบ** — สูงสุด 100 รายการ เรียงใหม่→เก่า **ไม่ได้กรองเฉพาะวันนี้** จึงมีพัสดุค้างข้ามวันโผล่ได้ แต่ละใบเป็นการ์ดที่ **แบ่งเป็น 2 ส่วนกดได้คนละที่**:
   - ส่วนบน (กดเปิดรายละเอียด): เลขออเดอร์ + เวลา + บรรทัดรอง `ชื่อผู้รับ · ยอดเงิน` และถ้ามีเลขพัสดุแล้วจะต่อท้ายว่า ` · เลขพัสดุ {external_ref}` + ป้ายสถานะทางขวา
   - ส่วนล่าง: ถ้าสถานะเป็น `"แพ็คแล้ว"` → ปุ่มหลักขนาดเล็กเต็มความกว้าง `"ส่งมอบแล้ว + เลขพัสดุ"`; ถ้ายังไม่แพ็ค → **ไม่มีปุ่ม** แต่เป็นข้อความกลางสีเทาจาง `"ต้องแพ็คเสร็จก่อนจึงบันทึกส่งมอบได้"`

### แท็บ 4 · `"ค้นหา"`
- ฟอร์มแถวเดียว: ช่องกรอกยืดเต็ม (คีย์บอร์ดมือถือขึ้นปุ่มค้นหา) + ปุ่มหลัก `"ค้นหา"` ที่ไม่ยุบขนาด
- **ค้นเมื่อกด submit เท่านั้น ไม่ใช่พิมพ์แล้วค้นทันที**
- ผลลัพธ์แสดงเป็นการ์ดออเดอร์แบบเดียวกับบอร์ด สูงสุด 30 รายการ
- **แท็บนี้ไม่มีการอัปเดตอัตโนมัติ**

## 3. ทุก flow ทีละสเต็ป

### 3.1 ดูออเดอร์วันนี้แล้วเปิดรายละเอียด (flow หลัก)

**ขั้นตอน**
1. แอดมิน login → ระบบพามาที่หน้านี้อัตโนมัติ → แท็บ `"บอร์ดวันนี้"` ถูกเลือกไว้แล้ว
2. ระบบดึงออเดอร์ของวันนี้ (เวลากรุงเทพ) สูงสุด 300 รายการ → ระหว่างรอแสดงตัวหมุน + `"กำลังโหลด…"` เต็มพื้นที่แท็บ
3. เห็น 5 กลุ่มเรียงลงมา แต่ละกลุ่มมีตัวเลขนับจำนวนข้างหัวข้อ; กลุ่มที่ว่างแสดงกล่องเส้นประ `"ไม่มีออเดอร์"`
4. หน้าจะดึงข้อมูลใหม่เองทุก 20 วินาที ออเดอร์ใหม่เด้งขึ้นกลุ่ม `"รอชำระ"` เอง (ไม่ต้องกดรีเฟรช)
5. กดที่การ์ดออเดอร์ใบใดก็ได้ (กดได้ทั้งใบ) → เปิดหน้าต่างรายละเอียด: **เป็นแผ่นเลื่อนขึ้นจากขอบล่างบนมือถือ / กล่องกลางจอบนเดสก์ท็อป** สูงสุด 90% ของความสูงจอ เลื่อนดูข้างในได้
6. ปิดด้วยปุ่ม ✕ มุมขวาบน (ป้ายกำกับสำหรับ screen reader คือ `"ปิด"`) หรือกดพื้นหลังดำโปร่ง → กลับมาที่บอร์ด

**กฎการตรวจสอบ**
- ไม่มีช่องกรอก — เป็น flow อ่านอย่างเดียว
- ช่วง "วันนี้" ล็อกตายที่เวลากรุงเทพ (UTC+7 คงที่) **ไม่มีตัวเลือกเปลี่ยนวัน ไม่มี date picker ไม่มีปุ่ม "เมื่อวาน"** — ดูย้อนหลังในบอร์ดไม่ได้ ต้องไปแท็บค้นหา
- จำกัด 300 รายการ/วัน — เกินกว่านี้ถูกตัดทิ้ง **โดยไม่มีข้อความเตือนใดๆ**

**ข้อความสำเร็จ** — ไม่มี toast (เป็นการอ่านข้อมูล) ข้อมูลขึ้นบนบอร์ดพร้อมตัวนับต่อกลุ่ม

**ข้อความล้มเหลว** — กล่องผิดพลาดเต็มพื้นที่แท็บ: ⚠️ + `"โหลดข้อมูลไม่สำเร็จ"` + `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"` + ปุ่มโปร่ง `"ลองใหม่"`

### 3.2 ดูรอบไรเดอร์วันนี้ + เปิดออเดอร์ในรอบ

**ขั้นตอน**
1. กดแท็บ `"รอบไรเดอร์"`
2. ระบบดึงรอบไรเดอร์ของวันนี้ เรียงตามเวลารอบ → ระหว่างรอแสดงตัวหมุน + `"กำลังโหลด…"`
3. เห็นการ์ดรอบเรียงตามเวลา แต่ละใบมี `"รอบ {ชช:นน} น."` + ป้ายสถานะรอบ + บรรทัด `🛵 {ชื่อไรเดอร์} · {n} จุด`
4. แต่ละการ์ดโหลดออเดอร์ในรอบของตัวเองแยกกัน → ระหว่างรอแสดงตัวหมุน + `"กำลังโหลดออเดอร์…"` **เฉพาะในการ์ดนั้น**
5. ออเดอร์ในรอบเรียงตามลำดับจุด (จุดที่ยังไม่มีลำดับขึ้นก่อน แสดงเป็น `จุด -`)
6. กดแถวออเดอร์ → เปิดหน้าต่างรายละเอียด (ตัวเดียวกับ flow 3.1)
7. รอบและออเดอร์ในรอบดึงใหม่เองทุก 20 วินาที — สถานะรอบไล่ตามไรเดอร์เอง (`"เปิดรับออเดอร์"` → `"ปิดรอบ รอไรเดอร์รับ"` → `"ไรเดอร์รับแล้ว"` → `"กำลังส่ง"` → `"จบรอบ"`)

**กฎการตรวจสอบ**
- ไม่มีช่องกรอก — อ่านอย่างเดียว
- **แอดมินสร้าง / แก้ / ปิดรอบไรเดอร์จากหน้านี้ไม่ได้** — รอบไรเดอร์ถูกสร้างอัตโนมัติ (ต่างจาก Kerry ที่มีปุ่มเปิดรอบ) ห้ามออกแบบปุ่มเปิดรอบไรเดอร์ขึ้นมาเอง
- ไม่จำกัดจำนวนรอบและจำนวนออเดอร์ในรอบ

**ข้อความสำเร็จ** — ไม่มี toast; การ์ดรอบแสดงชื่อไรเดอร์และจำนวนจุดครบ

**ข้อความล้มเหลว** — 2 ระดับ ใช้ข้อความเดียวกัน (`"โหลดข้อมูลไม่สำเร็จ"` + `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"` + `"ลองใหม่"`) ต่างกันที่ขอบเขต: ระดับแท็บ = เต็มพื้นที่ / ระดับการ์ด = อยู่ในการ์ดใบนั้นใบเดียว

---

### 3.3 เปิดรอบ Kerry ประจำวัน / กวาดออเดอร์เข้ารอบอีกครั้ง

**ขั้นตอน**
1. กดแท็บ `"Kerry"` → เห็นแถบเหลืองเตือน `"📦 Kerry เข้ารับพัสดุช่วง 11:00–14:00 น. (แล้วแต่วัน) — เปิดรอบแล้วเตรียมของให้พร้อมก่อนช่วงเวลานี้"`
2. ถ้ายังไม่มีรอบวันนี้ → การ์ดแสดง `"ยังไม่ได้เปิดรอบ Kerry ของวันนี้"` และปุ่มเขียนว่า `"เปิดรอบ Kerry วันนี้"`
3. กดปุ่ม → ปุ่มถูก disable และข้อความเปลี่ยนเป็น `"กำลังทำรายการ…"` → ยิงคำสั่งทันที **ไม่มีหน้าต่างยืนยัน**
4. สำเร็จ → toast สีเขียว `"เปิดรอบ Kerry วันนี้แล้ว — กวาดออเดอร์ที่ชำระแล้วเข้ารอบ"` → การ์ดเปลี่ยนเป็น `"รอบ Kerry วันนี้"` + ป้ายสถานะรอบ และรายการพัสดุด้านล่างถูกดึงใหม่
5. เมื่อมีรอบแล้ว **ปุ่มเดิมเปลี่ยนข้อความเป็น** `"กวาดออเดอร์ที่ชำระแล้วเข้ารอบอีกครั้ง"` — กดซ้ำได้เรื่อยๆ เพื่อดูดออเดอร์ที่เพิ่งชำระเข้ารอบ (ฝั่ง server เป็นแบบ "มีแล้วใช้ของเดิม" จึงกดซ้ำได้ปลอดภัย ไม่สร้างรอบซ้ำ)

**กฎการตรวจสอบ**
- ไม่มีฟิลด์ให้กรอก — ปุ่มเดียวจบ
- ไม่มีหน้าต่างยืนยัน
- กันกดซ้ำด้วยการ disable ปุ่มระหว่างทำรายการ
- **หนึ่งรอบต่อหนึ่งวัน** (ตามวันเวลากรุงเทพ) — บังคับที่ฝั่งฐานข้อมูล ไม่ใช่ฝั่งหน้าจอ

**ข้อความสำเร็จ** — toast สีเขียว `"เปิดรอบ Kerry วันนี้แล้ว — กวาดออเดอร์ที่ชำระแล้วเข้ารอบ"`
⚠️ ข้อความนี้ใช้ทั้งกรณีเปิดรอบใหม่ **และ** กรณีกวาดซ้ำ — ทั้งที่ผู้ใช้กดปุ่มคนละข้อความกัน (จุดที่ควรปรับ)

**ข้อความล้มเหลว** — toast สีแดง แสดงข้อความจริงจาก server ถ้ามี มิฉะนั้นใช้ `"เปิดรอบไม่สำเร็จ"`

---

### 3.4 บันทึกส่งมอบพัสดุ Kerry + เลขพัสดุ

> ⚠️ นี่คือ action ที่ **เปลี่ยนออเดอร์เป็น "ส่งสำเร็จ" และแจ้งลูกค้าทาง LINE ทันที** แต่ปัจจุบัน **ไม่มีหน้าต่างยืนยัน**

**ขั้นตอน**
1. ในแท็บ `"Kerry"` ดูรายการพัสดุค้างส่งมอบ (สถานะ pending / paid / confirmed / packing / packed)
2. ถ้าออเดอร์ยัง **ไม่** เป็น `"แพ็คแล้ว"` → ใต้การ์ดเป็นข้อความสีจาง `"ต้องแพ็คเสร็จก่อนจึงบันทึกส่งมอบได้"` — ไม่มีปุ่มให้กด
3. เมื่อสถานะเป็น `"แพ็คแล้ว"` → ปรากฏปุ่มหลักขนาดเล็กเต็มความกว้าง `"ส่งมอบแล้ว + เลขพัสดุ"`
4. กดปุ่ม → เปิดหน้าต่างหัวข้อ `"ส่งมอบ Kerry · {order_no}"` (ตัวอย่าง: `"ส่งมอบ Kerry · 2607-0012"`)
5. ในฟอร์มมีช่องเดียว:
   - ป้ายกำกับ `"เลขพัสดุ (Tracking No.)"`
   - คำอธิบายใต้ช่อง `"บังคับกรอก — ลูกค้าจะได้รับแจ้งเลขนี้ทาง LINE"`
   - ตัวอย่างในช่องว่าง `"เช่น KERDO1234567890"`
   - **ค่าเริ่มต้น = เลขอ้างอิงเดิมถ้าเคยบันทึกไว้แล้ว**
6. ปุ่ม `"ยืนยันส่งมอบแล้ว"` **จะถูก disable จนกว่าจะพิมพ์เลขพัสดุ** (ช่องว่างล้วนก็ยังนับว่าว่าง)
7. กดยืนยัน → ปุ่มเปลี่ยนเป็น `"กำลังบันทึก…"` และถูก disable
8. สำเร็จ → toast `"บันทึกส่งมอบ {order_no} แล้ว"` → หน้าต่างปิดเอง → ออเดอร์หายจากรายการค้างส่งมอบ (เพราะกลายเป็น "ส่งสำเร็จ") และไปโผล่กลุ่ม `"เสร็จ / ตีกลับ / ยกเลิก"` ในบอร์ด

**กฎการตรวจสอบ**
- เลขพัสดุ: **บังคับกรอก** — กันไว้ 3 ชั้น (ช่อง required + กัน submit เมื่อว่าง + disable ปุ่ม)
- ฝั่ง server: เลขพัสดุยาว **1–120 ตัวอักษร** (ยังไม่มีการจำกัดความยาวที่ช่องกรอก — พิมพ์เกินได้แล้วค่อย error ตอน submit)
- ฝั่ง server: ออเดอร์ต้องเป็นพัสดุ Kerry เท่านั้น มิฉะนั้นตอบกลับ `"ออเดอร์นี้ไม่ใช่พัสดุ Kerry"`
- ฝั่ง server: ออเดอร์ต้องอยู่สถานะ `"แพ็คแล้ว"` ก่อน (หน้าจอกันไว้ชั้นแรกด้วยการซ่อนปุ่ม)
- ค่าที่ส่งจะถูกตัดช่องว่างหัว-ท้ายให้อัตโนมัติ
- **ไม่มีหน้าต่างยืนยัน** — กดปุ๊บทำปั๊บ

**ข้อความสำเร็จ** — toast สีเขียว `"บันทึกส่งมอบ {order_no} แล้ว"` (ตัวอย่าง: `"บันทึกส่งมอบ 2607-0012 แล้ว"`)

**ข้อความล้มเหลว** — toast สีแดง แสดงข้อความจริงจาก server (เช่น `"ออเดอร์นี้ไม่ใช่พัสดุ Kerry"` หรือ error เรื่องสถานะ) มิฉะนั้นใช้ `"บันทึกไม่สำเร็จ"` — **หน้าต่างไม่ปิด** ผู้ใช้แก้เลขแล้วลองใหม่ได้ทันที

### 3.5 ค้นหาออเดอร์ย้อนหลังด้วยเลขออเดอร์

**ขั้นตอน**
1. กดแท็บ `"ค้นหา"` → เห็นสถานะเริ่มต้น: 🔎 + `"ค้นหาออเดอร์ย้อนหลัง"` + `"ใส่เลขออเดอร์บางส่วนได้ เช่น 0012"`
2. พิมพ์เลขออเดอร์ในช่อง (คีย์บอร์ดมือถือขึ้นปุ่มค้นหา) ตัวอย่างในช่องว่าง: `"พิมพ์เลขออเดอร์ เช่น 2607-0012"`
3. กดปุ่ม `"ค้นหา"` หรือกด Enter → ค่าถูกตัดช่องว่างหัว-ท้ายแล้วเริ่มค้น
4. ระหว่างค้นแสดงตัวหมุน + `"กำลังค้นหา…"`
5. เจอ → แสดงผลลัพธ์เป็นการ์ดออเดอร์ เรียงใหม่→เก่า สูงสุด 30 รายการ
6. กดการ์ด → เปิดหน้าต่างรายละเอียดเหมือนแท็บอื่น
7. ไม่เจอ → 🤔 + `"ไม่พบออเดอร์ "{คำค้น}""` + `"ลองพิมพ์เลขให้สั้นลงหรือตรวจตัวสะกด"` (ตัวอย่าง: `ไม่พบออเดอร์ "9999"`)

**กฎการตรวจสอบ**
- **ไม่ค้นแบบทันที** — ต้อง submit ก่อน (พิมพ์เฉยๆ ไม่ค้น) ไม่มี debounce
- พิมพ์แต่ช่องว่างล้วน → กลับไปแสดงสถานะเริ่มต้น 🔎 ไม่ยิงคำค้น
- **ค้นได้เฉพาะเลขออเดอร์เท่านั้น** — ค้นชื่อลูกค้า / เบอร์โทรไม่ได้
- จำกัด 30 รายการ — เกินถูกตัดเงียบๆ ไม่มีข้อความบอก ไม่มี pagination
- **ไม่จำกัดช่วงวัน** — ค้นได้ทุกอายุออเดอร์ (ต่างจากบอร์ดที่ล็อกที่วันนี้)

**ข้อความสำเร็จ** — ไม่มี toast; รายการผลลัพธ์ขึ้นแทนที่สถานะเริ่มต้น

**ข้อความล้มเหลว** — ⚠️ + `"โหลดข้อมูลไม่สำเร็จ"` + `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"` + `"ลองใหม่"`
🔴 **ดีไซน์ต้องแยก 2 เคสนี้ให้ชัด**: "ไม่พบออเดอร์" (ค้นสำเร็จแต่ไม่มีผล → 🤔 ไม่มีปุ่ม) กับ "โหลดไม่สำเร็จ" (เน็ตหลุด → ⚠️ + ปุ่มลองใหม่) ห้ามยุบรวมกัน

---

### 3.6 ดูรายละเอียดออเดอร์ (หน้าต่างที่เปิดได้จากทุกแท็บ)

หัวข้อหน้าต่าง = **เลขออเดอร์** (เช่น `2607-0012`) เนื้อหาเรียงลงมา:

1. **แถวป้าย**: ป้ายสถานะ (สีตามสถานะ) + ป้ายวิธีส่งแบบเต็ม (พื้นสี brand โปร่ง ตัวหนังสือสี brand — ใช้ชื่อเต็ม เช่น `"ส่งปกติ (ไรเดอร์)"` ไม่ใช่ตัวย่อแบบในการ์ด) + วันเวลาสั่ง (ตัวเล็กสีเทาจาง)
2. **กล่องผู้รับ** (ขอบบางสีเทา ขอบมน):
   - ชื่อผู้รับ (กึ่งหนา) — ถ้าไม่มีชื่อแสดง `"ลูกค้า"`
   - เบอร์ผู้รับ = **ลิงก์โทรออกได้จริง** สี brand นำหน้าด้วย 📞 (แสดงเมื่อมีเบอร์เท่านั้น)
   - ที่อยู่ — ถ้าไม่มีแสดง `"ยังไม่มีที่อยู่"`
   - บรรทัดล่างสุด (ตัวเล็กสีเทาจาง): `บัญชีลูกค้า: {ชื่อเล่น}` — ถ้าไม่มีชื่อเล่นแสดง `บัญชีลูกค้า: -` และถ้ามีเบอร์จะต่อท้ายว่า ` · {เบอร์}`
3. **กล่องรายการสินค้า + ยอด** (พื้นเทาอ่อนมาก ขอบมน):
   - แต่ละรายการ: `{ชื่อสินค้า} × {จำนวน}` คู่กับราคารวมของรายการนั้น (ตัวเลขเรียงหลักตรงกัน) — ถ้าหาชื่อสินค้าไม่เจอแสดงคำว่า `"สินค้า"`
   - เส้นคั่น แล้วตามด้วย 3 บรรทัด: `"ค่าสินค้า"` / `"ค่าส่ง"` / `"รวม"` (บรรทัด `"รวม"` เป็นตัวหนาสีเข้มกว่า)
4. **กล่องเลขอ้างอิง** — **แสดงเฉพาะออเดอร์ Grab / Lalamove / Kerry เท่านั้น** (ออเดอร์ไรเดอร์ร้านจะไม่เห็นกล่องนี้เลย) ดู flow 3.7
5. **ช่องเลือกโซนลูกค้า** ดู flow 3.8
6. **`"ประวัติสถานะ"`** — ไทม์ไลน์เรียงเก่า→ใหม่: คอลัมน์ซ้ายกว้างคงที่เป็นวันเวลา (ตัวเลขเรียงหลักตรงกัน สีเทาจาง) ขวาเป็น `{สถานะเดิม} → {สถานะใหม่}` (ถ้าไม่มีสถานะเดิมจะแสดงแค่สถานะใหม่) และถ้ามีหมายเหตุจะแสดงใต้ลงมาเป็นตัวเทาจาง
   - ไม่มีประวัติ → `"ยังไม่มีประวัติ"`
7. **ปุ่มยกเลิกออเดอร์** ดู flow 3.9

**⚠️ กับดักสำคัญ: ข้อมูลในหน้าต่างนี้ไม่อัปเดตอัตโนมัติ** ต่างจาก 4 แท็บหลัก — ถ้าแอดมินเปิดค้างไว้ระหว่างที่แพ็คเกอร์เปลี่ยนสถานะ ป้ายสถานะและประวัติจะไม่ขยับ ต้องปิดแล้วเปิดใหม่

---

### 3.7 บันทึก / แก้ไขเลขอ้างอิง Grab/Lalamove/Kerry

**ขั้นตอน**
1. เปิดรายละเอียดของออเดอร์ที่วิธีส่งเป็น Grab / Lalamove / Kerry
2. เห็นกล่องเลขอ้างอิง: ป้ายกำกับตัวเล็กสีเทา `"เลขอ้างอิง Grab/Lalamove/Kerry"` และใต้ลงมาคือค่าปัจจุบัน (ฟอนต์ monospace) หรือ `"— ยังไม่บันทึก —"` ถ้ายังว่าง
3. ทางขวาเป็นปุ่มโปร่งขนาดเล็ก: `"บันทึกเลข"` (ถ้ายังว่าง) หรือ `"แก้ไข"` (ถ้ามีค่าแล้ว)
4. กดปุ่ม → เปิดหน้าต่าง **ซ้อนบน** หน้าต่างรายละเอียด หัวข้อ `"เลขอ้างอิง · {order_no}"`
5. ในฟอร์ม: ป้ายกำกับ `"เลขอ้างอิง Grab/Lalamove/Kerry"` + คำอธิบาย `"เลขงานจากแอปที่เรียกรถ หรือเลขพัสดุ"` + ตัวอย่างในช่องว่าง `"เช่น GR-123456"` (เติมค่าเดิมไว้ให้ถ้ามี)
6. ปุ่ม `"บันทึก"` **disable จนกว่าจะพิมพ์ค่า**
7. กดบันทึก → `"กำลังบันทึก…"`
8. สำเร็จ → toast `"บันทึกเลขอ้างอิงแล้ว"` → หน้าต่างเลขอ้างอิงปิด (**หน้าต่างรายละเอียดยังเปิดอยู่**) → ค่าที่แสดงอัปเดตทันที และปุ่มเปลี่ยนจาก `"บันทึกเลข"` เป็น `"แก้ไข"`

**กฎการตรวจสอบ**
- บังคับกรอก (กัน 3 ชั้นเหมือน flow 3.4)
- ฝั่ง server: **1–120 ตัวอักษร**
- ฝั่ง server: อนุญาตเฉพาะออเดอร์ Grab / Lalamove / Kerry มิฉะนั้นตอบ `"เลขอ้างอิงใช้กับออเดอร์ Grab/Lalamove/Kerry เท่านั้น"`
- ค่าถูกตัดช่องว่างหัว-ท้ายอัตโนมัติ
- ไม่มีหน้าต่างยืนยัน
- **action นี้ไม่เปลี่ยนสถานะออเดอร์ และไม่แจ้งลูกค้า** (ต่างจากการส่งมอบ Kerry ที่ยิง LINE)

**ข้อความสำเร็จ** — toast สีเขียว `"บันทึกเลขอ้างอิงแล้ว"`

**ข้อความล้มเหลว** — toast สีแดง ข้อความจริงจาก server มิฉะนั้น `"บันทึกไม่สำเร็จ"`; หน้าต่างไม่ปิด

---

### 3.8 เปลี่ยนโซนลูกค้า (zone override)

**ขั้นตอน**
1. เปิดรายละเอียดออเดอร์ → เลื่อนลงถึงช่อง ป้ายกำกับ `"โซนลูกค้า (override)"` คำอธิบาย `"มีผลกับการคิดค่าส่งของออเดอร์ถัดไปของลูกค้าคนนี้"`
2. ระหว่างโหลดข้อมูลลูกค้า แสดงตัวหมุน + `"กำลังโหลดข้อมูลลูกค้า…"` **แทนที่ช่องเลือก**
3. เลือกค่าจากช่องเลือก 3 ตัวเลือก: `"ตามระยะจริง"` (ค่าว่าง = ให้ระบบคิดเอง) / `"ในเขต"` / `"นอกเขต"`
4. **เปลี่ยนค่าปุ๊บยิงคำสั่งปั๊บ — ไม่มีปุ่มยืนยัน** ระหว่างทำรายการช่องเลือกถูก disable
5. สำเร็จ → toast `"อัปเดตโซนลูกค้าแล้ว"` → ช่องแสดงค่าใหม่

**กฎการตรวจสอบ**
- 3 ค่าเท่านั้น; `"ตามระยะจริง"` ส่งเป็นค่าว่าง (null)
- ช่องเลือกถูก disable เมื่อ: กำลังทำรายการ / โหลดข้อมูลลูกค้าไม่สำเร็จ / ไม่มีข้อมูลลูกค้า
- ฝั่ง server: ถ้าหาลูกค้าไม่เจอ → ตอบกลับ `"Customer not found."` (**ข้อความนี้เป็นภาษาอังกฤษจาก server จริง** ไม่ใช่ข้อความที่ designer เขียนขึ้น — ควรเสนอให้แปลเป็นไทย)
- **🔴 สำคัญมาก**: แก้ที่ระดับ **"ลูกค้า"** ไม่ใช่ระดับ "ออเดอร์" — **ไม่มีผลกับค่าส่งของออเดอร์ใบที่กำลังเปิดอยู่** มีผลกับออเดอร์ **ถัดไป** ของลูกค้าคนนั้นเท่านั้น
- ไม่มีหน้าต่างยืนยัน แม้จะกระทบการคิดเงินรอบถัดไป
- **เป็น action เดียวในหน้านี้ที่ไม่มี feedback ทางข้อความระหว่างทำงาน** (แค่ช่องเลือกถูก disable เฉยๆ)

**ข้อความสำเร็จ** — toast สีเขียว `"อัปเดตโซนลูกค้าแล้ว"`

**ข้อความล้มเหลว** — toast สีแดง ข้อความจริงจาก server มิฉะนั้น `"อัปเดตโซนไม่สำเร็จ"`
หมายเหตุ: ค่าในช่องเลือก **ไม่ถูกย้อนกลับทันทีเมื่อ error** (จะกลับเป็นค่าเดิมหลังระบบดึงข้อมูลใหม่) → เกิดช่วงเวลาสั้นๆ ที่หน้าจอโกหก จุดนี้ควรแก้ในดีไซน์ใหม่

### 3.9 ยกเลิกออเดอร์ (3 ชั้น: ปุ่ม → หน้าต่าง → ยืนยัน)

**ขั้นตอน**
1. เปิดรายละเอียดของออเดอร์ที่สถานะเป็น `"รอชำระ"` / `"ชำระแล้ว"` / `"เข้ารอบ"` เท่านั้น → เห็นปุ่มแดงเต็มความกว้าง `"ยกเลิกออเดอร์"` ล่างสุด
2. ออเดอร์ที่แพ็คแล้ว / กำลังส่ง / ส่งสำเร็จ → **ปุ่มไม่ปรากฏเลย** (ไม่ใช่ disable) และ **ไม่มีข้อความอธิบายว่าทำไมยกเลิกไม่ได้**
3. กดปุ่ม → เปิดหน้าต่าง **ซ้อนบน** หน้าต่างรายละเอียด หัวข้อ `"ยกเลิก · {order_no}"`
4. ในฟอร์ม: ป้ายกำกับ `"เหตุผล (ไม่บังคับ)"` + ตัวอย่างในช่องว่าง `"เช่น ลูกค้าขอยกเลิก"`
5. กดปุ่มแดง `"ยกเลิกออเดอร์"` ในหน้าต่าง → **เปิดกล่องยืนยันอีกชั้น (= modal ชั้นที่ 3)**:
   - หัวข้อ: `"ยืนยันยกเลิกออเดอร์?"`
   - ข้อความ: `"{order_no} จะถูกยกเลิกและคืนสต็อก — เหตุผล: {เหตุผล}"`
     - ท่อน `" — เหตุผล: ..."` จะต่อท้าย **ก็ต่อเมื่อกรอกเหตุผล** ถ้าไม่กรอกจะเหลือแค่ `"{order_no} จะถูกยกเลิกและคืนสต็อก"`
     - ตัวอย่างกรอกเหตุผล: `"2607-0012 จะถูกยกเลิกและคืนสต็อก — เหตุผล: ลูกค้าขอยกเลิก"`
     - ตัวอย่างไม่กรอก: `"2607-0012 จะถูกยกเลิกและคืนสต็อก"`
   - ปุ่มซ้าย `"ยกเลิก"` = **ไม่ทำอะไร** / ปุ่มขวา `"ยกเลิกออเดอร์"` = **ทำจริง** (สีแดง)
6. กด `"ยกเลิกออเดอร์"` ในกล่องยืนยัน → ปุ่มในหน้าต่างเปลี่ยนเป็น `"กำลังยกเลิก…"`
7. สำเร็จ → toast `"ยกเลิก {order_no} แล้ว"` → ปิดหน้าต่างยกเลิก **และปิดหน้าต่างรายละเอียดด้วย** → กลับสู่บอร์ด → ออเดอร์ย้ายไปกลุ่ม `"เสร็จ / ตีกลับ / ยกเลิก"`

**กฎการตรวจสอบ**
- ปุ่มแสดงเฉพาะสถานะ pending / paid / confirmed — **ยกเลิกได้ก่อนแพ็คเท่านั้น** (กฎธุรกิจ — **ห้ามแตะ**)
- เหตุผล: **ไม่บังคับ**; ฝั่ง server ยาวไม่เกิน **300 ตัวอักษร**
- เหตุผลว่างหลังตัดช่องว่าง → ไม่ส่งฟิลด์นี้ไปเลย
- **เป็น action เดียวในหน้านี้ที่มีกล่องยืนยัน**
- 🔴 **คำว่า "ยกเลิก" ถูกใช้ 2 ความหมายในกล่องยืนยันเดียวกัน** — ปุ่มซ้าย `"ยกเลิก"` = ถอย, ปุ่มขวา `"ยกเลิกออเดอร์"` = ลุย อยู่ติดกัน **เสี่ยงกดผิดสูงมาก ดีไซน์ควรแก้คำ** (เช่น `"ปิด"` / `"ไม่ยกเลิก"`)

**ข้อความสำเร็จ** — toast สีเขียว `"ยกเลิก {order_no} แล้ว"` (ตัวอย่าง: `"ยกเลิก 2607-0012 แล้ว"`)

**ข้อความล้มเหลว** — toast สีแดง ข้อความจริงจาก server มิฉะนั้น `"ยกเลิกไม่สำเร็จ"`; หน้าต่างยังเปิดอยู่

### 3.10 สร้างออเดอร์โทรสั่ง / หน้าร้าน

> **ไม่ใช่ wizard — เป็นฟอร์มยาวหน้าเดียวที่เลื่อนลงเรื่อยๆ** ในหน้าต่างสูงสุด 90% ของจอ
> ⚠️ action นี้ **ตัดสต็อก + แจ้งลูกค้าทาง LINE ทันที** แต่ **ไม่มีกล่องยืนยัน**

**ขั้นตอน**
1. กดปุ่ม `"+ โทรสั่ง"` มุมขวาบนของหน้า (กดได้จากทุกแท็บ) → เปิดหน้าต่างหัวข้อ `"สร้างออเดอร์ (โทรสั่ง/หน้าร้าน)"`

2. **ส่วนที่ 1 — เลือกสินค้า** (ป้ายกำกับ `"เลือกสินค้า"`)
   1. ช่องค้นหาสินค้า ตัวอย่างในช่องว่าง `"ค้นหาชื่อสินค้า…"` (คีย์บอร์ดมือถือขึ้นปุ่มค้นหา) — **กรองทันทีขณะพิมพ์ ไม่ต้องกด submit** (ต่างจากแท็บค้นหาออเดอร์!)
   2. ระหว่างโหลดสินค้า: ตัวหมุน + `"กำลังโหลดสินค้า…"`
   3. รายการสินค้าอยู่ใน **กล่องเลื่อนได้สูงสุดประมาณ 192px** (ขอบบางสีเทา ขอบมน) — โหลดเฉพาะสินค้าที่เปิดขาย เรียงตามชื่อ ไม่จำกัดจำนวน
   4. แต่ละแถว: ชื่อสินค้า (ตัดท้ายด้วย …) + บรรทัดรองตัวเล็กสีเทาจาง `{ราคา} · คงเหลือ {n}` (ตัวอย่าง: `฿120 · คงเหลือ 8`)
   5. ขวาสุดของแถว: ป้ายพื้นเทา `"+ เพิ่ม"` ถ้ายังไม่เลือก หรือป้ายพื้นสี brand ตัวหนังสือขาว `× {จำนวน}` ถ้าเลือกแล้ว; แถวที่เลือกแล้วพื้นหลังเป็นสี brand แบบจางมาก
   6. **กดที่แถวสินค้า = +1 ชิ้น** (กดซ้ำเพิ่มทีละชิ้น) — **ไม่มีปุ่มลบในรายการนี้**
   7. ค้นแล้วไม่เจอ → ข้อความสีเทาจาง `"ไม่พบสินค้า"` (ไม่มีไอคอน ไม่ใช่ empty state เต็มรูปแบบ)

3. **ส่วนที่ 2 — ปรับจำนวน** (โผล่เมื่อเลือกอย่างน้อย 1 อย่าง)
   1. กล่องขอบบาง หัวข้อตัวเล็กกึ่งหนาสีเทา `"รายการที่เลือก"`
   2. แต่ละรายการ: ชื่อสินค้า (กึ่งหนา) + ราคารวมของรายการนั้นทางขวา
   3. ใต้ลงมาเป็นตัวปรับจำนวน: ปุ่ม `−` (สูง 48px กว้าง 56px พื้นเทา ตัวหนังสือใหญ่หนา, ป้าย screen reader `"ลด"`) + ช่องพิมพ์ตัวเลขกลาง (สูง 48px ตัวใหญ่หนา จัดกลาง คีย์บอร์ดตัวเลข) + ปุ่ม `+` (ป้าย screen reader `"เพิ่ม"`)
   4. **ลดจำนวนลงถึง 0 → รายการถูกลบออกจากตะกร้าอัตโนมัติ** และหายจากกล่อง `"รายการที่เลือก"`

4. **ส่วนที่ 3 — วิธีจัดส่ง**
   - ป้ายกำกับ `"วิธีจัดส่ง"` คำอธิบาย `"Lalamove คิดตามระยะทาง — สร้างผ่านแชทลูกค้าเท่านั้น"`
   - เป็นแถบแท็บ 3 ตัวเลือก: `"ไรเดอร์ร้าน"` (ค่าเริ่มต้น) / `"ด่วน (Grab)"` / `"Kerry"`
   - **ไม่มี Lalamove โดยเจตนา** (server ปฏิเสธ) — **ห้ามเพิ่มเข้าไป**

5. **ส่วนที่ 4 — ข้อมูลผู้รับ** (4 ช่อง เรียงลงมา ทุกช่อง**ไม่บังคับ**)
   1. ป้ายกำกับ `"ชื่อผู้รับ"` — ตัวอย่างในช่องว่าง `"ชื่อลูกค้า"` (server: ไม่เกิน 120 ตัวอักษร)
   2. ป้ายกำกับ `"เบอร์ผู้รับ"` — ตัวอย่างในช่องว่าง `"08xxxxxxxx"` คีย์บอร์ดโทรศัพท์ (server: ไม่เกิน 30 ตัวอักษร)
   3. ป้ายกำกับ `"ที่อยู่จัดส่ง"` — ตัวอย่างในช่องว่าง `"บ้านเลขที่ / จุดสังเกต"` (server: ไม่เกิน 500 ตัวอักษร)
   4. ป้ายกำกับ `"เบอร์ลูกค้า (ผูกประวัติ)"` คำอธิบาย `"เว้นว่างได้ — ระบบจะใช้เบอร์ผู้รับแทน"` — ตัวอย่างในช่องว่าง `"08xxxxxxxx"` คีย์บอร์ดโทรศัพท์ (server: ไม่เกิน 30 ตัวอักษร)

6. **ส่วนที่ 5 — สถานะเงิน**
   - แถวสูงอย่างน้อย 44px: ช่องติ๊กขนาด 20×20px + ข้อความ `"รับเงินแล้ว (เก็บเงินสด/โอนตรง)"`
   - **ติ๊กไว้เป็นค่าเริ่มต้น**
   - 🔴 ตัวนี้เป็นตัวตัดสินว่าลูกค้าจะได้รับ LINE แจ้ง "ชำระแล้ว" หรือ "ขอให้ชำระเงิน" — **สำคัญมากแต่หน้าตาเหมือนของรอง**

7. **แถบสรุปล่างสุด** (มีเส้นคั่นด้านบน)
   - ซ้าย (สีเทา): `{n} ชิ้น · ค่าสินค้า` (ตัวอย่าง: `3 ชิ้น · ค่าสินค้า`)
   - ขวา (ตัวหนา): ยอดรวมค่าสินค้า
   - **ไม่แสดงค่าส่งและยอดสุทธิ** เพราะค่าส่งคำนวณฝั่ง server หลังกดสร้าง

8. **ปุ่มสร้าง** — ปุ่มหลักเต็มความกว้าง `"สร้างออเดอร์"` → ระหว่างทำรายการเปลี่ยนเป็น `"กำลังสร้าง…"` และถูก disable
   - ปุ่มนี้ **อยู่ล่างสุดของ scroll ไม่ติดขอบจอ** ต้องเลื่อนผ่านทุกอย่างกว่าจะถึง

9. สำเร็จ → toast `"สร้างออเดอร์ {order_no} แล้ว"` → หน้าต่างปิดเอง → ออเดอร์ใหม่ขึ้นบอร์ด (กลุ่ม `"ชำระแล้ว / เข้ารอบ"` ถ้าติ๊กรับเงินแล้ว หรือกลุ่ม `"รอชำระ"` ถ้าไม่ติ๊ก)

**กฎการตรวจสอบ**
- **สินค้า: บังคับอย่างน้อย 1 รายการ** — ถ้ายังไม่เลือกแล้วกดสร้าง → toast สีแดง `"เลือกสินค้าอย่างน้อย 1 อย่าง"` และ **ไม่ยิงคำสั่ง** (สังเกต: ปุ่มไม่ได้ disable — กดได้แล้วค่อยเตือน ต่างจากหน้าต่างอื่นในหน้านี้ที่ disable ปุ่มไว้)
- จำนวนต่อรายการ: **1–99 ชิ้น** (ตัวปรับจำนวนตั้งขั้นต่ำ 0 เพื่อใช้ลบรายการ)
- วิธีส่ง: rider / express_grab / parcel_kerry เท่านั้น — **lalamove ถูกปฏิเสธโดยเจตนา**
- ทุกช่องข้อความถูกตัดช่องว่างหัว-ท้าย และถ้าว่างจะไม่ส่งฟิลด์นั้นไปเลย
- เบอร์ลูกค้าเว้นว่าง → ระบบใช้เบอร์ผู้รับแทนอัตโนมัติ
- **ไม่มีการตรวจรูปแบบเบอร์โทรเลย** ทั้งฝั่งหน้าจอและ server (จำกัดแค่ความยาว 30 ตัวอักษร) — พิมพ์อะไรก็ผ่าน
- **ไม่มีการเช็คสต็อกฝั่งหน้าจอ** — กดเพิ่มเกิน "คงเหลือ" ได้ (แม้จนติดลบ) server เป็นคนปฏิเสธทีหลัง
- **ไม่มีกล่องยืนยัน** แม้จะตัดสต็อก + ยิง LINE หาลูกค้า
- **ฝั่งหน้าจอยังไม่มีการจำกัดความยาวที่ช่องกรอกเลยสักช่อง** — พิมพ์เกินได้แล้วค่อย error ตอน submit

**ข้อความสำเร็จ** — toast สีเขียว `"สร้างออเดอร์ {order_no} แล้ว"` (ตัวอย่าง: `"สร้างออเดอร์ 2607-0013 แล้ว"`); ถ้า server ไม่ส่งเลขกลับมาจะใช้ `"สร้างออเดอร์แล้ว"`

**ข้อความล้มเหลว** — toast สีแดง ข้อความจริงจาก server มิฉะนั้น `"สร้างไม่สำเร็จ"` — **หน้าต่างไม่ปิดและข้อมูลที่กรอกไว้ยังอยู่ครบ** กรอกใหม่ได้ทันที (พฤติกรรมนี้ดีมากสำหรับคนที่กำลังคุยโทรศัพท์อยู่ — **ห้ามทำหาย**)

## 7. ข้อมูลอัปเดตยังไง

**Polling ล้วน — ไม่มี realtime subscription เลยแม้แต่จุดเดียว** ดึงข้อมูลใหม่ทุก **20 วินาที**

| ส่วน | อัปเดตอัตโนมัติ? |
|---|---|
| บอร์ดวันนี้ | ✅ ทุก 20 วิ |
| รายการรอบไรเดอร์ | ✅ ทุก 20 วิ |
| ออเดอร์ในแต่ละการ์ดรอบ | ✅ ทุก 20 วิ — **แยกต่อการ์ด** (มี 6 รอบ = ยิง 7 คำขอทุก 20 วิ) |
| รอบ Kerry วันนี้ | ✅ ทุก 20 วิ |
| พัสดุ Kerry ค้างส่งมอบ | ✅ ทุก 20 วิ |
| แท็บค้นหา | ❌ ไม่อัปเดตเลย |
| **รายการสินค้า / ประวัติสถานะ / ข้อมูลลูกค้า ในหน้าต่างรายละเอียด** | ❌ **ไม่อัปเดตเลย** |

**หลังกดปุ่มทำรายการ** ระบบดึงข้อมูลที่เกี่ยวข้องใหม่ทันทีโดยไม่รอรอบ 20 วิถัดไป

**ผลต่อดีไซน์ — ต้องแก้:**
- 🔴 **ไม่มีตัวบอกว่าอัปเดตล่าสุดเมื่อไหร่ และไม่มีสัญญาณว่าเพิ่งรีเฟรช** — ตัวหมุนขึ้นเฉพาะการโหลดครั้งแรกเท่านั้น ทุกรอบหลังจากนั้นข้อมูลเปลี่ยนเงียบสนิท ไม่มี animation ไม่มี toast ไม่มี badge "ใหม่" **ควรเพิ่ม timestamp "อัปเดตเมื่อ …" หรือ indicator เบาๆ**
- 🔴 **การ์ดกระโดดข้ามกลุ่มกลางคัน** — ผู้ใช้กำลังเล็งจะกดการ์ดหนึ่ง แล้ว polling ย้ายมันไปกลุ่มอื่น/หายไป **ควรออกแบบ transition ให้ตามทัน**
- 🔴 **ไม่มี pull-to-refresh และไม่มีปุ่มรีเฟรชเอง** — ผู้ใช้ทำได้แค่รอ 20 วิ (ยกเว้นปุ่ม `"ลองใหม่"` ตอน error) บนมือถือคนปัดลงโดยสัญชาตญาณ **ควรพิจารณาเพิ่ม**
- 🔴 **หน้าต่างรายละเอียดค้างข้อมูลเก่า** — เปิดค้างไว้ตอนแพ็คเกอร์แพ็คเสร็จ ป้ายสถานะจะยังโชว์ของเก่าอยู่ ต้องปิดเปิดใหม่ **ป้ายสถานะที่โกหกคือปัญหาที่ดีไซน์ต้องเผชิญ**
- **ตัวเลข "จำนวนจุด" ในการ์ดรอบมาจากการนับออเดอร์ของการ์ดนั้นเอง** → ระหว่างยังโหลดไม่เสร็จจะแสดง `0 จุด` ชั่วครู่ก่อนเด้งเป็นเลขจริง — สะดุดตา ควรออกแบบ skeleton

## 8. สิ่งที่ designer ต้องรู้

### 🚫 ห้ามแตะ (ผูกกับกฎธุรกิจ — ต้องถามเจ้าของระบบก่อนถ้าจะเปลี่ยน)

- **กฎธุรกิจทั้งหมดอยู่ใน `Ready.md`** (หน้านี้อ้างอิง §3.1–3.5 และ §3.3 สำหรับโซนลูกค้า) — **ห้ามคิดกฎธุรกิจใหม่เอง** ถ้าดีไซน์จะเปลี่ยนพฤติกรรม (เพิ่มสถานะ / เปลี่ยนเงื่อนไขยกเลิก / เปลี่ยนกลุ่มบนบอร์ด) ต้องถามก่อนเสมอ
- **ยกเลิกออเดอร์ได้ก่อนแพ็คเท่านั้น** (pending / paid / confirmed) — จะปรับได้แค่ *วิธีสื่อสาร* ว่าทำไมทำไม่ได้ ไม่ใช่ *เงื่อนไข*
- **ส่งมอบ Kerry ต้องแพ็คเสร็จก่อน** — server บังคับอีกชั้น ซ่อนปุ่มไว้เฉยๆ ไม่พอ
- **Lalamove ถูกตัดออกจากตัวเลือกวิธีส่งโดยเจตนา** (คิดราคาตามระยะทาง ต้องสร้างผ่านแชทลูกค้า) — **ห้ามเพิ่มกลับเข้าไป** server ปฏิเสธอยู่ดี แต่ออเดอร์ Lalamove ที่สร้างจากทางอื่นยังมาโผล่บนบอร์ดและมีกล่องเลขอ้างอิงได้ปกติ
- **แอดมินสร้าง/แก้/ปิดรอบไรเดอร์เองไม่ได้** — รอบถูกสร้างอัตโนมัติ ห้ามออกแบบปุ่มขึ้นมา
- **บอร์ดล็อกที่ "วันนี้" ตามเวลากรุงเทพเสมอ** — ไม่มี date picker การดูย้อนหลังทำได้ทางเดียวคือแท็บค้นหา
- **โซนลูกค้าแก้ที่ระดับ "ลูกค้า" มีผลกับออเดอร์ถัดไป** ไม่ใช่ใบที่เปิดอยู่ — ข้อความ hint บอกไว้แล้ว ห้ามตัดทิ้ง
- **ขีดจำกัดความยาวจาก server ที่ดีไซน์ต้องเคารพ**: เลขพัสดุ 1–120 / เลขอ้างอิง 1–120 / เหตุผล ≤ 300 / ที่อยู่ ≤ 500 / ชื่อผู้รับ ≤ 120 / เบอร์ ≤ 30 / จำนวนต่อรายการ 1–99 / อย่างน้อย 1 รายการสินค้า
- **ต้องแยก "ไม่มีข้อมูล" ออกจาก "โหลดไม่สำเร็จ" เสมอ** — เป็นเจตนาที่เขียนไว้ในโค้ด เพราะแอดมินต้องไม่เข้าใจผิดว่า "ไม่มีออเดอร์" ทั้งที่จริงเน็ตหลุด สถานะผิดพลาดต้องมี ⚠️ + ปุ่ม `"ลองใหม่"` เสมอ / สถานะว่างต้องไม่มีปุ่ม
- **เลขออเดอร์และเลขอ้างอิงใช้ฟอนต์ monospace, ตัวเลขเงินเรียงหลักตรงกัน** — ตั้งใจให้อ่านทางโทรศัพท์และเทียบเลขง่าย **ควรคงไว้**
- **ลำดับช่องในฟอร์มสร้างออเดอร์** (สินค้า → วิธีส่ง → ชื่อ → เบอร์ → ที่อยู่) ตรงกับลำดับที่ลูกค้าพูดทางโทรศัพท์ — **ห้ามสลับ**
- **เมื่อสร้างออเดอร์ล้มเหลว ข้อมูลที่กรอกไว้ต้องไม่หาย** — คนกำลังคุยโทรศัพท์อยู่

### ⚠️ ข้อจำกัดเชิงระบบ (ดีไซน์แก้เองไม่ได้ แต่ต้องรับรู้)

- **ฟอร์มสร้างออเดอร์ไม่โชว์ค่าส่งและยอดสุทธิ** — แสดงแค่ `{n} ชิ้น · ค่าสินค้า` เพราะค่าส่งคำนวณฝั่ง server หลังกดสร้าง **แอดมินจึงบอกลูกค้าทางโทรศัพท์ไม่ได้ว่ารวมเท่าไหร่จนกว่าจะกดสร้างเสร็จ** — นี่คือช่องว่างเชิง UX ที่ใหญ่มาก แต่แก้ด้วยดีไซน์อย่างเดียวไม่ได้
- **ค้นหาได้เฉพาะเลขออเดอร์** — ค้นชื่อลูกค้า/เบอร์โทรไม่ได้ ทั้งที่แอดมินรับสายแล้วลูกค้ามักบอกชื่อ/เบอร์ ไม่ใช่เลขออเดอร์ (ต้องแก้ที่ query ไม่ใช่แค่ดีไซน์ — ควรยกให้เจ้าของระบบพิจารณา)
- **limit เงียบทุกจุด** — บอร์ด 300 / Kerry 100 / ค้นหา 30: เกินแล้วตัดทิ้งโดยไม่บอกผู้ใช้ ไม่มี pagination ไม่มี "โหลดเพิ่ม" ไม่มี "แสดง 30 จาก N"
- **ไม่มีการตรวจรูปแบบเบอร์โทรเลย** ทั้งฝั่งหน้าจอและ server — พิมพ์อะไรก็ผ่าน
- **ทุกช่องในฟอร์มสร้างออเดอร์ไม่บังคับ ยกเว้นสินค้า** — สร้างออเดอร์ไรเดอร์โดยไม่มีชื่อ ไม่มีเบอร์ ไม่มีที่อยู่ ได้จริง (จะไปโชว์ `"ลูกค้า"` และ `"ยังไม่มีที่อยู่"` ในรายละเอียด) → ปัญหาตกไปที่ไรเดอร์ปลายทาง
- **ไม่มีการเช็คสต็อกฝั่งหน้าจอ** — แถวสินค้าโชว์ `คงเหลือ {n}` แต่กดเพิ่มเกินได้ (แม้ติดลบ) server ปฏิเสธทีหลัง
- **ทุกการเปลี่ยนสถานะวิ่งผ่านช่องทางเดียว** (edge function `admin-action`, 6 คำสั่ง: create_kerry_round / kerry_handover / set_external_ref / set_customer_zone / cancel_order / create_manual_order) และ **ข้อความ error ภาษาไทยจริงจาก server จะถูกเอามาโชว์ใน toast** → **toast error ต้องรองรับข้อความยาวหลายบรรทัด** ไม่ใช่แค่คำสั้นๆ

### 🔧 จุดที่ดีไซน์ปัจจุบันอ่อน — ปรับได้ ควรปรับ

**โครงสร้าง / เลย์เอาต์**
- 🔴 **บนเดสก์ท็อปหน้านี้แทบไม่ได้ออกแบบเลย** — การ์ดออเดอร์ยืดเต็ม `max-w-5xl` ทำให้บรรทัดยาวโล่งมาก **ไม่มี grid / หลายคอลัมน์แม้แต่จุดเดียวทั้งหน้า** บอร์ดจริงๆ ควรเป็น kanban หลายคอลัมน์บนจอใหญ่ — **นี่คือโอกาสใหญ่ที่สุดของการ redesign**
- 🔴 **ฟอร์มสร้างออเดอร์ = nested scroll บนมือถือ** — กล่องเลื่อนสูงสุด 90% ของจอ ข้างในมีกล่องรายการสินค้าที่เลื่อนได้อีกชั้น (สูงสุด ~192px) ใช้งานยากมาก และปุ่ม `"สร้างออเดอร์"` อยู่ล่างสุดของ scroll ไม่ติดขอบจอ — ต้องเลื่อนผ่านทุกอย่างกว่าจะถึง **ควรพิจารณาทำเป็น multi-step หรือปุ่มติดขอบล่าง**
- 🔴 **หน้าต่างซ้อนได้ถึง 3 ชั้น** (รายละเอียด → ยกเลิก → กล่องยืนยัน) ทั้ง 3 ชั้นใช้พื้นหลังดำโปร่งทับกันทำให้จอมืดขึ้นเรื่อยๆ และเป็นแผ่นเลื่อนซ้อนแผ่นเลื่อนบนมือถือ **แต่ละชั้นกดพื้นหลังปิดได้ → เสี่ยงปิดผิดชั้น**
- **ความสูงของหน้าต่างรายละเอียดต่างกันสุดขั้วตามออเดอร์** (กล่องเลขอ้างอิงและปุ่มยกเลิกซ่อน/แสดงตามเงื่อนไข): ออเดอร์ไรเดอร์ที่ส่งสำเร็จ = สั้นมาก ไม่มีทั้งสองอย่าง / ออเดอร์ Grab ที่รอชำระ = ยาวสุด — **ต้องออกแบบเผื่อทุก combination**
- **toast ลอยเหนือแถบแท็บล่างบนมือถือ กว้างสุดขนาดเล็ก หายเองใน 3.6 วิ** — **จะทับปุ่ม submit ที่อยู่ล่างสุดของหน้าต่าง** ต้องเช็คจุดนี้

**บอร์ด**
- 🔴 **ป้ายสถานะกลางน้ำ 4 ตัวเป็นสีเทาเหมือนกันหมด แยกไม่ออกด้วยสี** ต้องอ่านตัวหนังสือเท่านั้น — รหัสสีปัจจุบัน: เขียว = `ส่งสำเร็จ` / แดง = `ยกเลิก`, `ตีกลับ` / เหลืองอำพัน = `รอชำระ`, `รอค่าส่งใหม่` / ฟ้า = `ออกส่ง`, `กำลังส่ง` / **เทา = `ชำระแล้ว`, `เข้ารอบ`, `กำลังแพ็ค`, `แพ็คแล้ว` ทั้งหมด**
- **11 สถานะยุบเหลือ 5 กลุ่ม** และกลุ่มสุดท้าย `"เสร็จ / ตีกลับ / ยกเลิก"` รวม 4 สถานะที่ความหมายต่างกันสุดขั้ว (ส่งสำเร็จ vs ยกเลิก vs ตีกลับ vs รอค่าส่งใหม่) ไว้ด้วยกัน
- **ลำดับความสำคัญกลับหัว**: กลุ่มที่ต้องลงมือทำ (`"รอชำระ"`) อยู่บนสุด — ถูกแล้ว แต่กลุ่ม "เสร็จแล้ว" ที่ไม่ต้องทำอะไร **บวมขึ้นเรื่อยๆ ตลอดวันและกินพื้นที่มากที่สุด** ผู้ใช้ต้องเลื่อนผ่านทุกกลุ่มเสมอ **ไม่มีการพับเก็บ**
- **แท็บไม่มีตัวเลขแจ้งเตือน** ทั้งที่คอมโพเนนต์รองรับอยู่แล้ว — แอดมินไม่รู้ว่ามีพัสดุ Kerry ค้างส่งมอบกี่ชิ้นจนกว่าจะกดเข้าแท็บ **เป็น quick win ที่ชัดเจนที่สุดในหน้านี้**

**คำและปุ่ม**
- 🔴 **คำว่า "ยกเลิก" กำกวมในกล่องยืนยันเดียวกัน** — ปุ่มซ้าย `"ยกเลิก"` = ถอย / ปุ่มขวา `"ยกเลิกออเดอร์"` = ลุย อยู่ติดกัน **เสี่ยงกดผิดสูงมาก ควรเปลี่ยนคำ** (เช่น `"ปิด"` หรือ `"ไม่ยกเลิก"`)
- 🔴 **ปุ่มเดียวเปลี่ยนความหมายตามสถานะ (Kerry)** — ปุ่มเดียวกันเป็นทั้ง `"เปิดรอบ Kerry วันนี้"` และ `"กวาดออเดอร์ที่ชำระแล้วเข้ารอบอีกครั้ง"` สองความหมายนี้ต่างกันมากในหัวผู้ใช้ **ควรแยกเป็นคนละปุ่ม/คนละลำดับความสำคัญ** (และ toast สำเร็จก็ใช้ข้อความเดียวกันทั้งสองกรณี)
- 🔴 **ปุ่มยกเลิกหายไปเฉยๆ เมื่อยกเลิกไม่ได้** (ไม่ใช่ disable + ไม่มีคำอธิบาย) — แอดมินที่เคยเห็นปุ่มจะงงว่าปุ่มหายไปไหน **ควรมีข้อความบอกเหตุผล เหมือนที่แท็บ Kerry ทำไว้ดีแล้ว** (`"ต้องแพ็คเสร็จก่อนจึงบันทึกส่งมอบได้"` คือแบบอย่างที่ดีที่ควรลอกไปใช้)
- **ช่องติ๊ก `"รับเงินแล้ว (เก็บเงินสด/โอนตรง)"` ติ๊กไว้เป็นค่าเริ่มต้น** และหน้าตาเป็นช่องติ๊กธรรมดา ทั้งที่เป็น**ตัวตัดสินว่าจะยิง LINE แจ้ง "ชำระแล้ว" หรือ "ขอให้ชำระเงิน" ไปหาลูกค้า** — สำคัญมากแต่ดูเหมือนของรอง **ควรยกระดับเป็นตัวเลือกที่ชัดเจน เช่น 2 ปุ่ม**
- **`"Customer not found."` เป็นภาษาอังกฤษ** — เป็นข้อความ error เดียวในหน้านี้ที่ยังไม่แปล ควรเสนอให้แปล

**ความปลอดภัยของ action**
- 🔴 **หลาย action ที่มีผลจริงยิงทันทีโดยไม่มีการยืนยัน**: เปิดรอบ Kerry / **บันทึกส่งมอบ Kerry (→ เปลี่ยนเป็น "ส่งสำเร็จ" + ยิง LINE หาลูกค้าทันที)** / เปลี่ยนโซนลูกค้า (→ กระทบค่าส่งรอบหน้า) / **สร้างออเดอร์ (→ ตัดสต็อก + ยิง LINE)** — **มีแค่ "ยกเลิกออเดอร์" เท่านั้นที่มีการยืนยัน** ทั้งที่บาง action ข้างต้นย้อนกลับยากกว่า
- **เปลี่ยนโซนลูกค้าไม่มี feedback ระหว่างทำงานเลย** (แค่ช่องเลือกถูก disable เฉยๆ) และ **ค่าในช่องไม่ย้อนกลับทันทีเมื่อ error** → เกิดช่วงที่หน้าจอโกหก
- **หน้าต่างปิดได้ระหว่างกำลังส่งข้อมูล** — ปุ่ม ✕ และพื้นหลังไม่ถูกล็อก ผู้ใช้อาจปิดกลางคันแล้วไม่ทันเห็นผลลัพธ์
- **ปุ่ม `"สร้างออเดอร์"` ไม่ disable ตอนตะกร้าว่าง** (กดได้แล้วค่อยเตือนด้วย toast) — ไม่สอดคล้องกับหน้าต่างอื่นในหน้าเดียวกันที่ disable ปุ่มไว้เมื่อกรอกไม่ครบ **ควรเลือกแนวทางเดียวให้ทั้งหน้า**

**รายละเอียดเล็กที่ควรเก็บ**
- **ปุ่มโทรออกใช้ได้จริง** — เบอร์ผู้รับกดแล้วโทรออกจากมือถือได้ทันที เป็น touch target สำคัญของงาน "รับสาย" **แต่ตอนนี้เป็นแค่ลิงก์ข้อความเล็กๆ ควรทำให้เด่นขึ้นมาก**
- **ที่อยู่เป็นข้อความล้วน ไม่มีปุ่มคัดลอก ไม่มีลิงก์แผนที่** ทั้งที่แอดมินอาจต้องส่งต่อให้ Grab/Lalamove
- **รูปแบบเวลาไม่สม่ำเสมอ** — การ์ดในบอร์ดมี ` น.` ต่อท้าย แต่การ์ด Kerry และหน้าต่างรายละเอียดไม่มี **ควรทำให้เหมือนกัน**
- **สถานะว่างเห็นบ่อยกว่าที่คิด ไม่ใช่ edge case** — บอร์ดว่างทุกเช้าก่อนออเดอร์แรก (🌤️ `"ยังไม่มีออเดอร์วันนี้"`) และแท็บค้นหาขึ้นสถานะว่างเสมอตอนเปิด (🔎) **ทั้งสองควรออกแบบให้สวยและมี CTA**
- **ไม่มี skeleton screen เลยทั้งหน้า** — มีแต่ตัวหมุน และตัวหมุนในแท็บหลักกินพื้นที่ทั้งแท็บ (เนื้อหาหายหมด)
- **การกรองสินค้าในฟอร์มสร้างออเดอร์ทำงานทันทีขณะพิมพ์ แต่การค้นหาออเดอร์ต้องกด submit** — พฤติกรรมช่องค้นหา 2 แบบในหน้าเดียวกัน สร้างความสับสน
