# สินค้า & สต็อก (`/catalog`)

> พนักงานสต็อก (role `stock`) และแอดมิน (role `admin`) ใช้ระหว่างวัน — ตอนยืนอยู่หน้าชั้นของ/ในโกดัง — เพื่อดูของคงเหลือ เพิ่มสินค้า รับของเข้า (ต้องถ่ายรูป) ตัดสต็อก (ต้องระบุเหตุผล) และย้อนดูประวัติความเคลื่อนไหว

---

## 1. บริบทการใช้งาน

**ใครใช้**
- **พนักงานสต็อก (role `stock`)** — คนหลังร้าน/คนคุมของ. หลัง login ระบบพาเขามาหน้านี้เป็นหน้าแรกโดยอัตโนมัติ และเมนูของเขามีแค่ **2 อัน** (📦 สินค้า, 👤 เช็คอิน). **หน้านี้คือหน้าจอหลักหน้าเดียวของงานทั้งวันของเขา** — ถ้าหน้านี้ใช้ยาก งานสต็อกทั้งวันพัง
- **แอดมิน (role `admin`)** — เข้าได้ด้วยสิทธิ์ admin bypass. แอดมินเห็นเมนู 11 อัน ทำให้แถบเมนูล่างเลื่อนแนวนอนได้
- **คนอื่นเข้าไม่ได้เลย** — route ป้องกันด้วย guard ที่ต้องมี role `admin` หรือ `stock`; ถ้าไม่มีสิทธิ์จะถูก redirect ไปหน้าแรกของ role ตัวเองทันที ผู้ใช้ **ไม่เคยเห็นหน้านี้ในสภาพ permission-denied** (ไม่มีข้อความ "คุณไม่มีสิทธิ์" บนหน้านี้). ชั้นที่สองอยู่ฝั่งเซิร์ฟเวอร์: ถ้าโปรไฟล์ถูกปิดใช้งานระหว่างทาง การกดบันทึกจะเด้ง error กลับมาเป็น toast

**อุปกรณ์และสถานการณ์จริง**
- **มือถือ ใช้มือเดียว ยืนอยู่ ถือของอยู่อีกมือ** — อาจอยู่ในโกดังแสงน้อย หรือหน้าร้านกลางแดด เน็ตร้านช้า
- ต้องถ่ายรูปของจริงตรงหน้าเป็นหลักฐานทุกครั้งที่รับของเข้า → มือถือคือ device หลัก ไม่ใช่ตัวเลือกรอง
- แอดมินอาจเปิดบนจอใหญ่เพื่อตรวจ/แก้ราคา แต่ไม่ใช่กรณีหลัก

**ความถี่และความเร่งรีบ**
- รับของเข้า = เป็นล็อต วันละไม่กี่ครั้ง แต่ครั้งละหลายรายการติดกัน
- ตัดสต็อก = ประปราย แต่**ทุกครั้งคือของหาย/ของเสียจริง มีผลต่อยอดพร้อมขายทันที** — ต้องช้าลงและยืนยันให้แน่
- เพิ่มสินค้า = นาน ๆ ครั้ง

**คำตัดสิน: Mobile-first เต็มรูปแบบ ไม่ใช่ responsive ตามหลัง**
เหตุผล: ผู้ใช้หลัก (role `stock`) ทำงานบนมือถือ 100%, การถ่ายรูปเป็นแกนกลางของทุก flow, และดีไซน์ปัจจุบันวางโครงแบบมือถือมาแล้ว (modal เป็น bottom sheet บนมือถือ / กล่องกลางจอบนจอใหญ่, toast ลอยเหนือแถบเมนูล่าง, เนื้อหาเว้นที่ด้านล่างกันแถบเมนูทับ). จอใหญ่คือ **โบนัส** — จำกัดความกว้างเนื้อหาไว้ ไม่ต้องออกแบบ layout ใหม่ทั้งชุด

## 2. โครงหน้า

หน้านี้ **ไม่มีแท็บ** — เป็นคอลัมน์เดียวเรียงจากบนลงล่าง มีระยะห่างระหว่างบล็อกสม่ำเสมอ

### บล็อกที่ 1 — หัวหน้าจอ
แถวเดียว ซ้าย-ขวา:
- **ซ้าย:** หัวข้อตัวหนา ขนาดใหญ่กว่าเนื้อหา — `"สินค้า & สต็อก"`
- **ขวา:** ปุ่มหลักขนาดเล็ก — `"+ เพิ่มสินค้า"` → เปิดฟอร์มเพิ่มสินค้า

### บล็อกที่ 2 — การ์ดเตือน "หมวดสินค้าโหลดไม่สำเร็จ" (แสดงเฉพาะเมื่อเกิดปัญหา)
การ์ดพื้นขาว ขอบมน ภายในมีหัวข้อเล็กตัวหนา `"หมวดสินค้า"` ตามด้วยกล่อง error มาตรฐาน (⚠️ + `"โหลดข้อมูลไม่สำเร็จ"` + คำอธิบาย + ปุ่ม `"ลองใหม่"`)

> **สำคัญสำหรับดีไซเนอร์:** การ์ดนี้แสดง **พร้อมกัน** กับรายการสินค้าได้ — สินค้ายังโหลดสำเร็จแม้หมวดจะพัง. ผลที่ตามมาคือ ชื่อหมวดจะหายไปจากบรรทัดราคาในทุกการ์ด และ dropdown หมวดในฟอร์มจะเหลือแค่ตัวเลือกเดียว. ต้องออกแบบให้เห็นการ์ดนี้แต่ไม่บังลิสต์

### บล็อกที่ 3 — รายการสินค้า (เนื้อหาหลัก)
รายการการ์ดเรียงต่อกันแนวตั้ง มีช่องไฟบาง ๆ ระหว่างการ์ด **เรียงตามชื่อสินค้า** (ไม่ใช่ตามยอดคงเหลือ ไม่ใช่ตามหมวด)

แต่ละการ์ด = แถวแนวนอน 3 โซน ซ้าย → ขวา:

**โซนที่ 1 — รูปย่อ (สี่เหลี่ยมจัตุรัสประมาณ 56px มุมมน)**
- ถ้ามีรูปสินค้า → แสดงรูปจริง ครอปเต็มกรอบ
- ถ้าไม่มีรูป → กล่องพื้นเทาอ่อน + emoji 🛍️ สีเทาจาง

**โซนที่ 2 — ข้อมูล (ยืดเต็มพื้นที่เหลือ) 3 บรรทัด**
1. **ชื่อสินค้า** ตัวหนาขนาดเล็ก ตัดด้วย … ถ้ายาวเกิน + ถ้าสินค้าถูกปิดขายจะมี **ป้ายเทา** ข้อความ `"ปิดขาย"` ต่อท้าย
2. **ราคา + หมวด** ตัวเล็กสีเทา — รูปแบบ: `฿1,250 · เครื่องดื่ม` (ส่วน ` · ชื่อหมวด` จะหายไปถ้าสินค้าไม่มีหมวด หรือหมวดโหลดไม่สำเร็จ)
3. **ยอดคงเหลือ** ตัวเล็ก — ข้อความ `"คงเหลือ 12"` และถ้ามีของถูกจองจะต่อท้ายเป็น `"คงเหลือ 12 (จอง 3)"`
   - **สีเป็นสัญญาณ:** ปกติสีเทา / **สีแดงเมื่อยอดพร้อมขายเหลือ 0 หรือติดลบ** — นี่คือสัญญาณ "ของหมด" ที่ต้องคงไว้

**โซนที่ 3 — ปุ่มแอ็กชัน (กริด 2×2 = 4 ปุ่ม)**

| ปุ่ม | ลักษณะ | ปลายทาง |
|---|---|---|
| `"รับเข้า"` | ปุ่มโปร่ง เล็ก | เปิดฟอร์มรับของเข้า |
| `"ตัดสต็อก"` | ปุ่มโปร่ง เล็ก **ตัวอักษรสีแดง** — **จางลงและกดไม่ได้เมื่อสินค้าไม่มีของในสต็อกเลย** | เปิดฟอร์มตัดสต็อก |
| `"ประวัติ"` | ปุ่มโปร่ง เล็ก | เปิดประวัติความเคลื่อนไหว |
| `"แก้ไข"` | ปุ่มโปร่ง เล็ก | เปิดฟอร์มแก้ไขสินค้า |

> **การ์ดทั้งใบกดไม่ได้** — ต้องกด 1 ใน 4 ปุ่มเท่านั้น ไม่มีหน้ารายละเอียดสินค้าแยก

### บล็อกที่ 4–7 — Modal ทั้ง 5 ตัว (ไม่ใช่หน้าใหม่)
ทุก modal ใช้โครงเดียวกัน: **บนมือถือเป็น bottom sheet** (เต็มความกว้าง มุมบนมน เด้งขึ้นจากขอบล่าง เว้นระยะ safe area ด้านล่าง) / **บนจอใหญ่เป็นกล่องกลางจอ** (มุมมนทุกด้าน กว้างสุดประมาณ 448px). สูงสุด 90% ของจอ แล้วเลื่อนภายใน. ปิดได้ด้วยปุ่ม ✕ มุมขวาบน (ป้ายช่วยการเข้าถึงว่า `"ปิด"`) หรือแตะฉากหลังสีดำโปร่ง 40%

1. **ฟอร์มสินค้า** — หัวข้อ `"เพิ่มสินค้า"` หรือ `"แก้ไขสินค้า"`
2. **ฟอร์มรับของเข้า** — หัวข้อ `"รับของเข้า · {ชื่อสินค้า}"`
3. **ฟอร์มตัดสต็อก** — หัวข้อ `"ตัดสต็อก / แก้ยอด · {ชื่อสินค้า}"`
4. **กล่องยืนยันตัดสต็อก** — ซ้อนทับฟอร์มตัดสต็อกอีกชั้น หัวข้อ `"ยืนยันตัดสต็อก"`
5. **ประวัติสต็อก** — หัวข้อ `"ประวัติสต็อก · {ชื่อสินค้า}"`

> **ข้อจำกัดที่ควรรู้:** modal ปัจจุบัน **ไม่ปิดด้วยปุ่ม Esc, ไม่มี focus trap, ไม่ประกาศตัวเป็น dialog กับ screen reader** — เป็นจุดที่ควรแก้ตอนรีดีไซน์

## 3. ทุก flow ทีละสเต็ป

### 3.1 เพิ่มสินค้าใหม่ (พร้อมสต็อกตั้งต้น)

**ขั้นตอน**
1. กดปุ่ม `"+ เพิ่มสินค้า"` ที่มุมขวาบนของหัวหน้าจอ
2. เปิด modal หัวข้อ `"เพิ่มสินค้า"` (bottom sheet บนมือถือ)
3. *(ไม่บังคับ)* แตะกล่องรูปขนาด 80×80 ที่มีเส้นประและ emoji 📷 ข้าง ๆ มีคำใบ้ `"แตะเพื่อใส่/เปลี่ยนรูปสินค้า"` → **เปิดตัวเลือกไฟล์แบบทั่วไป (เลือกจากคลังภาพได้ — จงใจ เพราะรูปสินค้ามักเป็นรูปโปรโมทที่มีอยู่แล้ว)** → เลือกรูป → ตัวอย่างรูปแทนที่กล่องเส้นประทันที
4. กรอกช่อง `"ชื่อสินค้า"` — placeholder `"เช่น ข้าวหอมมะลิ 5 กก."`
5. กรอกช่อง `"ราคา (บาท)"` — placeholder `"0"`, คีย์บอร์ดตัวเลข
6. เลือก `"หมวดสินค้า"` จาก dropdown (ค่าเริ่มต้นคือ `"— ไม่ระบุ —"`) หรือกดปุ่ม `"+ หมวดใหม่"` ข้าง ๆ เพื่อเข้า flow 3.2
7. *(ไม่บังคับ)* กรอก `"รายละเอียด (ไม่บังคับ)"` — placeholder `"รายละเอียดสั้น ๆ"`
8. ตั้ง `"สต็อกตั้งต้น"` ด้วยตัวนับ (ปุ่ม − / ช่องพิมพ์ตัวเลขตัวใหญ่ / ปุ่ม +) พร้อมปุ่มลัด `"+10"` `"+50"` `"+100"`. ใต้ label มีคำใบ้ `"จำนวนของที่มีพร้อมขายตอนเพิ่มสินค้า — ถ้ามากกว่า 0 ต้องถ่ายรูปของ"`. ค่าเริ่มต้น = 0
   - **ปุ่มลัดคือการ *บวกเพิ่ม* จากค่าปัจจุบัน ไม่ใช่ตั้งค่าใหม่** — กด `"+10"` สองครั้งได้ 20
9. **ถ้าสต็อกตั้งต้น > 0** → ช่อง `"รูปของที่รับเข้า"` โผล่ขึ้นมาอัตโนมัติ → แตะกล่องถ่ายรูป (เส้นประ สูงอย่างน้อย 96px, emoji 📷, ข้อความ `"ถ่ายรูปสต็อกตั้งต้น (บังคับ)"`) → **เปิดกล้องหลังโดยตรง เลือกจากคลังภาพไม่ได้ (จงใจ: ต้องถ่ายของจริงเป็นหลักฐาน)** → ถ่าย → ตัวอย่างรูปเต็มความกว้าง (สูงสุด ~224px) พร้อมปุ่ม `"ถ่ายใหม่"` ทับอยู่มุมขวาล่างบนไล่เฉดดำ
10. กดปุ่ม `"บันทึก"` (ปุ่มหลัก เต็มความกว้าง ล่างสุดของฟอร์ม)
11. ปุ่มเปลี่ยนเป็น `"กำลังบันทึก…"` และกดไม่ได้ (ระหว่างนี้รูปสินค้าถูกย่อในเครื่องก่อนส่ง; ถ้ามีสต็อกตั้งต้น ระบบจะบันทึกรายการเคลื่อนไหวให้อัตโนมัติด้วยเหตุผลว่า `"สต็อกตั้งต้น"`)
12. สำเร็จ → modal ปิดเอง → รายการรีเฟรช → toast เขียว

**กฎการตรวจสอบ**
- **ชื่อสินค้า:** บังคับ, ห้ามเป็นช่องว่างล้วน, **ยาวไม่เกิน 120 ตัวอักษร (เซิร์ฟเวอร์บังคับ — แต่ช่องกรอกปัจจุบันไม่จำกัด ผู้ใช้พิมพ์เกินได้แล้วค่อยเด้ง error)**
- **ราคา:** บังคับ, ต้องเป็นตัวเลข ≥ 0, **ต้องเป็นจำนวนเต็มบาท ไม่รับสตางค์** (0–10,000,000). ถ้าพิมพ์ `12.5` จะผ่านฝั่งหน้าจอแต่เซิร์ฟเวอร์ปฏิเสธ → ควรสื่อในดีไซน์ว่าไม่รับทศนิยม
- **รายละเอียด:** ไม่บังคับ, เซิร์ฟเวอร์จำกัด 500 ตัวอักษร (ช่องกรอกไม่จำกัด)
- **สต็อกตั้งต้น:** จำนวนเต็ม 0–1,000,000, กดต่ำกว่า 0 ไม่ได้
- **รูปสต็อกตั้งต้น: บังคับเมื่อสต็อกตั้งต้น > 0** — ตรวจทั้งฝั่งหน้าจอและเซิร์ฟเวอร์
- **ขนาดรูป:** ทั้งสองรูปต้องไม่เกิน 5MB และต้องเป็น JPEG / PNG / WebP
- **ไม่มีช่อง "รหัสสินค้า"** — ระบบสร้างเองจากชื่อ (จงใจ ให้เจ้าของร้านที่ไม่เทคนิคใช้ได้)

**ข้อความสำเร็จ (verbatim)**
- toast เขียว: `"บันทึกสินค้าแล้ว"`

**ข้อความล้มเหลว (verbatim)** — modal **ไม่ปิด** ค่าที่กรอกอยู่ครบ ปุ่มกลับมากดได้ + toast แดง
- `"กรอกชื่อและราคาให้ถูกต้อง"`
- `"สต็อกตั้งต้นมากกว่า 0 ต้องถ่ายรูปของที่รับเข้า"`
- `"รูปใหญ่เกินไป (เกิน 5MB)"` *(จากเซิร์ฟเวอร์)*
- `"สต็อกตั้งต้นต้องแนบรูปของที่รับเข้า"` *(จากเซิร์ฟเวอร์)*
- `"ไม่สามารถประมวลผลรูปได้"`
- `"ไม่พบหมวดสินค้า"` *(จากเซิร์ฟเวอร์)*
- fallback สุดท้าย: `"บันทึกไม่สำเร็จ"`

### 3.2 สร้างหมวดสินค้าใหม่แบบ inline (flow ย่อยในฟอร์มสินค้า)

**ขั้นตอน**
1. ในช่อง `"หมวดสินค้า"` กดปุ่ม `"+ หมวดใหม่"` (ปุ่มโปร่งเล็กด้านขวาของ dropdown)
2. แถว dropdown **ถูกแทนที่ทันที** ด้วย: ช่องพิมพ์ placeholder `"ชื่อหมวดใหม่"` (โฟกัสอัตโนมัติ — คีย์บอร์ดเด้งขึ้นเลย) + ปุ่มหลักเล็ก `"สร้าง"` + ปุ่มโปร่งเล็ก `"ยกเลิก"`
3. พิมพ์ชื่อหมวด → กด `"สร้าง"`
4. สำเร็จ → **หมวดใหม่ถูกเลือกให้อัตโนมัติ** → กลับสู่ dropdown ปกติ → ช่องชื่อถูกล้าง → toast เขียว
5. กด `"ยกเลิก"` → กลับเป็น dropdown ทันที **แต่ชื่อที่พิมพ์ค้างไว้ยังไม่ถูกล้าง** (กด `"+ หมวดใหม่"` อีกครั้งจะเห็นชื่อเก่าค้างอยู่ — ข้อบกพร่องที่ควรแก้)

**กฎการตรวจสอบ**
- **ชื่อหมวดว่าง → กดปุ่ม `"สร้าง"` แล้วไม่เกิดอะไรเลย ไม่มี toast ไม่มีเส้นแดง ไม่มี feedback ใด ๆ** — **จุดที่ดีไซเนอร์ต้องแก้** (เสนอ: ทำปุ่มเป็น disabled เมื่อช่องว่าง หรือแสดงข้อความเตือน)
- ชื่อหมวดยาวไม่เกิน 60 ตัวอักษร (เซิร์ฟเวอร์บังคับ, ช่องกรอกไม่จำกัด)
- ไม่มีช่องจัดลำดับหมวดใน UI
- ปุ่ม `"สร้าง"` / `"ยกเลิก"` ไม่ทำให้ฟอร์มสินค้าถูกส่ง

**ข้อความสำเร็จ (verbatim)**
- toast เขียว: `"สร้างหมวดแล้ว"`

**ข้อความล้มเหลว (verbatim)**
- toast แดงด้วยข้อความจากเซิร์ฟเวอร์ หรือ fallback: `"สร้างหมวดไม่สำเร็จ"`
- ยังค้างอยู่ในโหมดสร้างหมวด ช่องยังเปิด ชื่อที่พิมพ์ไม่หาย

### 3.3 แก้ไขสินค้า / เปิด-ปิดขาย

**ขั้นตอน**
1. ในการ์ดสินค้ากดปุ่ม `"แก้ไข"`
2. เปิด modal หัวข้อ `"แก้ไขสินค้า"` โดยกรอกค่าเดิมไว้ให้ครบแล้ว: ชื่อ, ราคา, หมวด, รายละเอียด, สถานะเปิดขาย และรูปสินค้าเดิม
3. แก้ค่าที่ต้องการ; แตะรูปเพื่อเปลี่ยนรูปสินค้า (ถ้าเปิดตัวเลือกไฟล์แล้วยกเลิก จะกลับไปแสดงรูปเดิม)
4. ติ๊ก/ปลดติ๊ก checkbox ขนาดใหญ่ (20×20) ข้อความ `"เปิดขายสินค้านี้"`
5. กด `"บันทึก"` → `"กำลังบันทึก…"` (ส่งรูปใหม่เฉพาะเมื่อเลือกไฟล์ใหม่จริงเท่านั้น)
6. สำเร็จ → modal ปิด → การ์ดในลิสต์อัปเดต → ถ้าปิดขาย การ์ดจะขึ้นป้ายเทา `"ปิดขาย"`

**กฎการตรวจสอบ**
- เหมือน flow 3.1 ทุกข้อ (ชื่อบังคับ, ราคาจำนวนเต็ม ≥ 0, รายละเอียดไม่เกิน 500)
- **ในโหมดแก้ไข ไม่มีช่อง "สต็อกตั้งต้น" และไม่มีช่องรูปสต็อกเลย — ถูกซ่อนทั้งหมด**
- **ไม่มีช่องแก้จำนวนสต็อก** — สต็อกเปลี่ยนได้ผ่านปุ่ม `"รับเข้า"` / `"ตัดสต็อก"` เท่านั้น
- **ไม่มีปุ่มลบสินค้า** — ปิดขายได้อย่างเดียว
- ถ้าสินค้าถูกลบไปแล้วระหว่างนั้น → เซิร์ฟเวอร์ตอบ error

**ข้อความสำเร็จ (verbatim)**
- toast เขียว: `"บันทึกสินค้าแล้ว"`

**ข้อความล้มเหลว (verbatim)**
- `"กรอกชื่อและราคาให้ถูกต้อง"`
- `"ไม่พบสินค้า"` *(จากเซิร์ฟเวอร์)*
- `"รูปใหญ่เกินไป (เกิน 5MB)"` *(จากเซิร์ฟเวอร์)*
- fallback: `"บันทึกไม่สำเร็จ"`

### 3.4 รับของเข้าสต็อก (บังคับถ่ายรูป — ไม่มีกล่องยืนยัน)

**ขั้นตอน**
1. ในการ์ดสินค้ากดปุ่ม `"รับเข้า"`
2. เปิด modal หัวข้อ `"รับของเข้า · {ชื่อสินค้า}"` — ตัวอย่าง: `"รับของเข้า · ข้าวหอมมะลิ 5 กก."`
3. ตั้งจำนวนในช่อง `"จำนวนที่รับเข้า"` ด้วยตัวนับ: ปุ่ม − (กว้าง ~56px สูง ≥48px พื้นเทาอ่อน) / ช่องพิมพ์ตัวเลขกลาง (ตัวใหญ่ ตัวหนา คีย์บอร์ดตัวเลข พิมพ์ได้เฉพาะตัวเลข) / ปุ่ม + (พื้นสีแบรนด์ ตัวอักษรขาว) + แถวปุ่มลัด `"+10"` `"+50"` `"+100"`. **ค่าเริ่มต้น = 1**
4. อ่าน **แถบสรุปสด** (กล่องพื้นเทาอ่อนอ่อน มุมมน) — ซ้าย `"คงเหลือปัจจุบัน 12"` (ตัวเลขหนาเข้ม) / กลาง `"→"` สีเทาจาง / ขวา `"หลังรับเข้า 22"` (**ตัวเลขสีเขียว**). อัปเดตทันทีทุกครั้งที่กดตัวนับ
5. ถ้าสินค้ามีของถูกจอง จะเห็นบรรทัดตัวเล็กสีเหลืองอำพัน: `"มีของถูกจองอยู่ 3 ชิ้น (รวมอยู่ในคงเหลือ)"`
6. แตะกล่องถ่ายรูปในช่อง `"รูปของที่รับเข้า"` (ข้อความในกล่อง: `"ถ่ายรูปของที่รับเข้า (บังคับ)"`) → **เปิดกล้องหลังของมือถือทันที** → ถ่ายรูป → ตัวอย่างรูปเต็มความกว้าง พร้อมปุ่ม `"ถ่ายใหม่"` ทับมุมขวาล่าง
7. *(ไม่บังคับ)* กรอก `"หมายเหตุ (ไม่บังคับ)"` — placeholder `"เช่น ล็อตใหม่จากซัพพลายเออร์"`
8. กดปุ่มหลักเต็มความกว้าง `"ยืนยันรับเข้า"`
9. ปุ่มเปลี่ยนเป็น `"กำลังบันทึก…"` + กดไม่ได้ — **ระหว่างนี้รูปดิบจากกล้อง (อาจหลาย MB) กำลังอัปโหลด และ ไม่มีแถบแสดงความคืบหน้าเลย มีแค่ข้อความ**
10. สำเร็จ → modal ปิด → ยอด `"คงเหลือ"` ในการ์ดเพิ่มขึ้น → toast เขียว
    - ถ้าไม่ได้กรอกหมายเหตุ ระบบจะบันทึกเหตุผลว่า `"รับของเข้า"` ให้เอง

**กฎการตรวจสอบ**
- **รูปบังคับ 100%** — ไม่มีรูป กดยืนยันไม่ผ่าน (ตรวจก่อนเข้าสถานะ busy ด้วยซ้ำ)
- ต้องมีข้อมูลผู้ใช้ในเซสชัน มิฉะนั้นหยุดทันที
- จำนวน: จำนวนเต็ม ต่ำสุด 1 (กด − ต่ำกว่า 1 ไม่ได้), **ห้ามเป็น 0**, สูงสุด 999,999. ถ้าลบตัวเลขจนช่องว่างจะเด้งกลับเป็น 1
- หมายเหตุ: เซิร์ฟเวอร์จำกัด 300 ตัวอักษร (ช่องกรอกไม่จำกัด)
- **ไม่มีกล่องยืนยันในโหมดรับเข้า — กดปุ่มแล้วบันทึกเลย** (ต่างจากโหมดตัดสต็อกโดยเจตนา: รับเข้า = ปลอดภัย)

**ข้อความสำเร็จ (verbatim)**
- toast เขียว: `"รับของเข้าสต็อกแล้ว"`
- ผลข้างเคียงที่ผู้ใช้เห็น: รายการนี้จะโผล่ในหน้า `"ประวัติ"` เป็น `+N` สีเขียว พร้อมรูปย่อที่เพิ่งถ่าย

**ข้อความล้มเหลว (verbatim)** — modal ไม่ปิด ค่าและรูปที่ถ่ายไว้ยังอยู่ครบ
- `"ต้องถ่ายรูปของที่รับเข้า"`
- `"ไม่พบข้อมูลผู้ใช้ กรุณาเข้าสู่ระบบใหม่"`
- ข้อความจากเซิร์ฟเวอร์ (บางข้อความยังเป็นอังกฤษ — ดูข้อ 8)
- fallback: `"ทำรายการไม่สำเร็จ"`

### 3.5 ตัดสต็อก / แก้ยอด (บังคับเหตุผล + ต้องยืนยันซ้ำ)

**ขั้นตอน**
1. ในการ์ดสินค้ากดปุ่ม `"ตัดสต็อก"` (ตัวอักษรสีแดง) — **ปุ่มนี้จางลงและกดไม่ได้เมื่อสินค้าไม่มีของในสต็อกเลย**
2. เปิด modal หัวข้อ `"ตัดสต็อก / แก้ยอด · {ชื่อสินค้า}"`
   - **สังเกต:** ปุ่มในลิสต์เขียน `"ตัดสต็อก"` แต่หัว modal เขียน `"ตัดสต็อก / แก้ยอด"` — คำไม่ตรงกัน เป็นจุดที่ปรับได้
3. ตั้งจำนวนในช่อง `"จำนวนที่ตัดออก"` ด้วยตัวนับ: ต่ำสุด 1, **สูงสุด = จำนวนที่มีในสต็อกจริง**, และ **ไม่มีปุ่มลัด `"+10"` `"+50"` `"+100"` (ตัดออกโดยเจตนา เพื่อกันการกดพลาดเกินยอด)**
4. อ่านแถบสรุปสด: `"คงเหลือปัจจุบัน 12"` `"→"` `"หลังตัด 9"` — **ตัวเลขเป็นสีแดงถ้าจะเหลือ 0 หรือติดลบ มิฉะนั้นเป็นสีดำเข้ม**
5. *(ถ้ามีของจอง)* เห็นบรรทัดเหลืองอำพัน `"มีของถูกจองอยู่ 3 ชิ้น (รวมอยู่ในคงเหลือ)"`
6. เลือก `"เหตุผล (บังคับ)"` จาก dropdown 4 ตัวเลือก:
   - `"ของเสีย"` **(ค่าเริ่มต้น)**
   - `"สูญหาย"`
   - `"นับสต็อกใหม่"`
   - `"อื่นๆ"`
7. ถ้าเลือก `"อื่นๆ"` → ช่อง `"ระบุเหตุผล"` โผล่ขึ้นมา (โฟกัสอัตโนมัติ, placeholder `"เช่น แพ็คเกจฉีกขาดระหว่างขนย้าย"`)
8. กดปุ่ม **พื้นแดงเต็ม ตัวอักษรขาว เต็มความกว้าง** ข้อความ `"ตัดสต็อก"`
9. **กล่องยืนยันซ้อนขึ้นมาอีกชั้น:**
   - หัวข้อ: `"ยืนยันตัดสต็อก"`
   - ข้อความ (รูปแบบ): `"{ชื่อสินค้า} จะถูกตัดออก {qty} ชิ้น (คงเหลือ {current} → {after}) และยอดพร้อมขายจะลดลงทันที"`
   - ตัวอย่างจริง: `"ข้าวหอมมะลิ 5 กก. จะถูกตัดออก 3 ชิ้น (คงเหลือ 12 → 9) และยอดพร้อมขายจะลดลงทันที"`
   - ปุ่มซ้าย: `"ยกเลิก"` (ปุ่มโปร่ง) / ปุ่มขวา: `"ตัดสต็อก"` (พื้นแดง)
10. กด `"ยกเลิก"` (หรือแตะฉากหลัง หรือกด ✕) → กล่องยืนยันปิด กลับมาที่ฟอร์มเดิม **ค่ายังอยู่ครบ ไม่มีอะไรถูกบันทึก**
11. กด `"ตัดสต็อก"` ในกล่องยืนยัน → ปุ่มหลักเปลี่ยนเป็น `"กำลังบันทึก…"`
12. สำเร็จ → ทั้งกล่องยืนยันและฟอร์มปิด → ยอด `"คงเหลือ"` ในการ์ดลดลง → toast เขียว
    - เหตุผลที่บันทึก: ถ้าเลือก `"อื่นๆ"` จะบันทึกเป็น `"อื่นๆ: {ข้อความที่พิมพ์}"` มิฉะนั้นบันทึกชื่อเหตุผลตรง ๆ

**กฎการตรวจสอบ**
- **เหตุผลบังคับเสมอ (กฎธุรกิจ Ready.md §3.5)** — dropdown มีค่าเริ่มต้นอยู่แล้วจึงว่างไม่ได้ แต่ถ้าเลือก `"อื่นๆ"` แล้วไม่พิมพ์อะไร → หยุดพร้อม toast
- **โหมดตัดสต็อกไม่มีการถ่ายรูป และไม่มี UI ให้แนบรูปเลย** — ตรงตามกฎธุรกิจ: "ตัดสต็อก (qty ลบ) ต้องมีเหตุผลเสมอ — รูปไม่บังคับ"
- จำนวนเกินสต็อก → หยุดพร้อม toast (เป็นด่านที่สอง เพราะตัวนับจำกัดสูงสุดไว้แล้ว)
- ต้องมีข้อมูลผู้ใช้ในเซสชัน
- **ต้องผ่านกล่องยืนยันเสมอ** — ไม่มีทางลัด
- ตัวเลขที่ใช้เทียบเป็นค่าที่จำไว้ตอนเปิด modal — ตัวเลขจริงบังคับที่ฐานข้อมูล

**ข้อความสำเร็จ (verbatim)**
- toast เขียว: `"ตัดสต็อกแล้ว"`
- ผลข้างเคียงที่ผู้ใช้เห็น: ยอดคงเหลือลดลง (เปลี่ยนเป็นสีแดงถ้าเหลือ ≤ 0), ปุ่ม `"ตัดสต็อก"` ในการ์ดกลายเป็นกดไม่ได้ถ้าสต็อกถึง 0, รายการโผล่ในประวัติเป็นเลขติดลบสีแดง

**ข้อความล้มเหลว (verbatim)** — modal ไม่ปิด ค่าที่กรอกยังอยู่
- `"กรุณาระบุเหตุผล"`
- `"จำนวนที่ตัดมากกว่าสต็อกคงเหลือ"`
- `"ไม่พบข้อมูลผู้ใช้ กรุณาเข้าสู่ระบบใหม่"`
- fallback: `"ทำรายการไม่สำเร็จ"`

### 3.6 ดูประวัติความเคลื่อนไหวสต็อก (อ่านอย่างเดียว)

**ขั้นตอน**
1. ในการ์ดสินค้ากดปุ่ม `"ประวัติ"`
2. เปิด modal หัวข้อ `"ประวัติสต็อก · {ชื่อสินค้า}"` → แสดงวงกลมหมุน + `"กำลังโหลด…"`
   - **การรอนี้ยาวกว่าที่คิด:** ระบบต้องดึงรายการก่อน แล้ว **ขอลิงก์รูปแยกทีละใบ** — modal ค้างที่ตัวหมุนจนกว่าลิงก์รูปทุกใบจะเสร็จ. ยิ่งมีรูปเยอะยิ่งรอนาน
3. แสดงรายการสูงสุด **20 แถว เรียงใหม่ → เก่า**. แต่ละแถวเป็นกล่องพื้นเทาอ่อน ขอบบาง มุมมน:
   - **ซ้าย:** รูปย่อ 48×48 มุมมน — **แตะแล้วเปิดรูปเต็มในแท็บใหม่**; ถ้ารายการนั้นไม่มีรูป (เช่น การตัดสต็อกทุกครั้ง) → กล่องพื้นเทา + emoji 🗒️
   - **ขวา 3 บรรทัด:** (1) จำนวนตัวหนา — `+10` **สีเขียว** เมื่อเป็นการรับเข้า / `-3` **สีแดง** เมื่อเป็นการตัด; (2) เหตุผล ตัวเล็กสีเทา ตัดด้วย … ถ้ายาว หรือ `"—"` ถ้าไม่มี; (3) วันเวลา ตัวเล็กสีเทาจาง
4. ถ้ามีครบ 20 แถว จะเห็นบรรทัดกลางล่างสุด ตัวเล็กสีเทาจาง: `"แสดง 20 รายการล่าสุด"` — **ไม่มีปุ่มดูเพิ่ม ไม่มีลิงก์ไปหน้าประวัติเต็ม**
5. ปิดด้วยปุ่ม ✕ มุมขวาบน หรือแตะฉากหลัง

**กฎการตรวจสอบ**
- อ่านอย่างเดียว ไม่มีช่องกรอก ไม่มี validation
- จำกัด 20 แถวแบบตายตัว
- ลิงก์รูปมีอายุ 1 ชั่วโมง (ถ้าเปิด modal ค้างไว้นานกว่านั้นแล้วแตะรูป อาจเปิดไม่ได้)
- รีเฟรชอัตโนมัติหลังบันทึกสินค้า / รับเข้า / ตัดสต็อก

**ข้อความสำเร็จ**
- ไม่มี toast (เป็นหน้าอ่านอย่างเดียว) — แค่แสดงรายการ

**ข้อความล้มเหลว (verbatim)** — เนื้อใน modal ถูกแทนที่ทั้งหมดด้วยกล่อง error:
- ⚠️ + `"โหลดข้อมูลไม่สำเร็จ"` + `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"` + ปุ่ม `"ลองใหม่"`

## 4. สถานะของหน้า

| สถานะ | ผู้ใช้เห็นอะไร | ข้อความจริง |
|---|---|---|
| **ปกติ** | หัวหน้าจอ + ปุ่มเพิ่มสินค้า + รายการการ์ดสินค้าเรียงตามชื่อ | `"สินค้า & สต็อก"`, `"+ เพิ่มสินค้า"` |
| **กำลังโหลด (รายการสินค้า)** | วงกลมหมุนเล็ก (16px, ขอบเทา ส่วนบนสีแบรนด์) + ข้อความสีเทาจาง จัดกลาง เว้นบน-ล่างประมาณ 40px. **ไม่มี skeleton** | `"กำลังโหลด…"` |
| **กำลังโหลด (หมวดสินค้า)** | **ไม่มีสถานะโหลดของตัวเอง** — ระหว่างรอ ชื่อหมวดจะยังไม่ขึ้นในบรรทัดราคา และ dropdown จะมีแค่ตัวเลือกเดียว (ผู้ใช้อาจเข้าใจผิดว่าไม่มีหมวดเลย) | `"— ไม่ระบุ —"` |
| **กำลังโหลด (ประวัติ)** | วงกลมหมุน + ข้อความ กลาง modal (ค้างนานถ้ามีรูปเยอะ) | `"กำลังโหลด…"` |
| **ว่าง — ไม่มีสินค้าเลย** | emoji 📦 ขนาดใหญ่มาก + หัวข้อตัวกลางสีเทาเข้ม + คำใบ้ตัวเล็กสีเทาจาง จัดกลาง เว้นบน-ล่างประมาณ 48px | `"ยังไม่มีสินค้า"` / คำใบ้: `"กด “เพิ่มสินค้า” เพื่อเริ่มขาย"` — **ใช้อัญประกาศไทยโค้ง “ ” ครอบคำ ต้องคงไว้ตามนี้เป๊ะ** |
| **ว่าง — ไม่มีประวัติสต็อก** | emoji 🗒️ ขนาดใหญ่ + หัวข้อ + คำใบ้ กลาง modal | `"ยังไม่มีความเคลื่อนไหว"` / คำใบ้: `"รายการรับเข้า/ตัดสต็อกจะแสดงที่นี่"` |
| **ว่าง — ไม่มีหมวดสินค้า** | **ไม่มี empty state เฉพาะ** — dropdown มีแค่ตัวเลือกเดียว ผู้ใช้ต้องกด `"+ หมวดใหม่"` เอง | `"— ไม่ระบุ —"` |
| **ผิดพลาด — สินค้าโหลดไม่สำเร็จ** | กล่อง error แทนที่ลิสต์ทั้งหมด: ⚠️ ใหญ่ + หัวข้อ + คำอธิบาย + ปุ่มโปร่งเล็ก. **จงใจแยกหน้าตาจาก empty state เพื่อไม่ให้ผู้ใช้เข้าใจผิดว่า "ไม่มีข้อมูล"** | `"โหลดข้อมูลไม่สำเร็จ"` / `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"` / ปุ่ม `"ลองใหม่"` |
| **ผิดพลาด — หมวดโหลดไม่สำเร็จ** | การ์ดเล็กเหนือลิสต์ (แสดง**พร้อม**ลิสต์ได้) หัวข้อ + กล่อง error เดียวกัน | `"หมวดสินค้า"` + ชุดข้อความ error ข้างบน |
| **ผิดพลาด — ประวัติโหลดไม่สำเร็จ** | กล่อง error แทนเนื้อใน modal | ชุดข้อความ error ข้างบน |
| **ผิดพลาด — บันทึกไม่สำเร็จ** | **ไม่ใช้กล่อง error** — modal ไม่ปิด ค่าที่กรอกอยู่ครบ + toast แดงลอยขึ้นมา | ดูข้อ 5 |
| **กำลังส่ง (busy)** | ปุ่ม submit กดไม่ได้ + ข้อความเปลี่ยน. **ข้อควรระวัง: ฟิลด์อื่นในฟอร์มยังแก้ได้, modal ยังปิดได้ด้วยการแตะฉากหลัง/ปุ่ม ✕ — ไม่มี overlay กันคลิก. ตอนอัปโหลดรูปรับเข้า ผู้ใช้อาจรอนานโดยไม่มีแถบความคืบหน้า** | `"กำลังบันทึก…"` |
| **ปุ่มตัดสต็อกกดไม่ได้** | ปุ่ม `"ตัดสต็อก"` ในการ์ดจางลง ~40% เมื่อสินค้าไม่มีของในสต็อก. **ไม่มีคำอธิบายว่าทำไม** | — |
| **Permission denied** | **ผู้ใช้ไม่เคยเห็นสถานะนี้บนหน้านี้** — ถูก redirect ออกก่อนหน้าจะ render. ชั้นเซิร์ฟเวอร์เด้งเป็น toast แดงตอนกดบันทึก (บางข้อความยังเป็นภาษาอังกฤษ) | `"Requires role: stock."`, `"No active staff profile for this tenant."` *(ยังไม่แปลไทย — ควรแก้)* |

## 7. ข้อมูลอัปเดตยังไง

**สรุปสั้น: โหลดครั้งเดียว — ไม่มี realtime, ไม่มี polling, ไม่มีปุ่มรีเฟรช**

- **ไม่มีการดึงข้อมูลซ้ำตามรอบเวลาเลย** — ต่างจากหน้าอื่นในแอปที่ดึงซ้ำทุก 15–30 วินาที (ออเดอร์แอดมิน, คิวแพ็ก, รอบไรเดอร์, คิวสลิป, แชท, หน้าหลักพนักงาน) และต่างจากแชท/หน้าหลักพนักงานที่ใช้ realtime จริง
- **ไม่ดึงซ้ำแม้ผู้ใช้สลับกลับมาที่แท็บ** — ตั้งค่าปิดไว้ทั้งแอป
- ข้อมูลรีเฟรชเฉพาะ **หลังผู้ใช้บันทึกสำเร็จด้วยตัวเอง** (บันทึกสินค้า / รับเข้า / ตัดสต็อก) หรือ **โหลดหน้าใหม่**
- **ไม่มีปุ่มรีเฟรชด้วยตนเอง** — ปุ่ม `"ลองใหม่"` โผล่เฉพาะตอน error เท่านั้น

**ผลต่อดีไซน์ (สำคัญ)**
- **ถ้าพนักงานสองคนแก้สต็อกสินค้าเดียวกันพร้อมกัน อีกคนจะเห็นตัวเลขค้างจนกว่าจะทำรายการเอง** — ในโกดังที่มีคนหลายคนนี่คือความเสี่ยงจริง
- **ตัวเลข "คงเหลือปัจจุบัน" ในแถบสรุปคือค่าที่จำไว้ตอนเปิด modal** ไม่ใช่ค่าสด. ตัวเลขจริงถูกบังคับที่ฐานข้อมูล — ผู้ใช้อาจเห็น "หลังตัด 9" แต่ผลจริงต่างออกไปถ้ามีคนอื่นแก้ระหว่างนั้น
- **ดีไซเนอร์ควรพิจารณาเพิ่ม:** (1) ตัวบอกเวลาอัปเดตล่าสุด เช่น "ข้อมูล ณ 14:30" ที่หัวหน้าจอ, (2) ปุ่มรีเฟรชด้วยตนเอง หรือ pull-to-refresh, (3) วิธีสื่อว่าตัวเลขในแถบสรุปอาจไม่สด
- ถ้าเสนอ realtime/auto-refresh ต้องคุยกับ dev ก่อน — เป็นการเปลี่ยนพฤติกรรมระบบ ไม่ใช่แค่ดีไซน์

## 8. สิ่งที่ designer ต้องรู้

### 🔒 ห้ามแตะ — ผูกกับกฎธุรกิจ (Ready.md §3.5)

- **รับของเข้า = บังคับถ่ายรูปเสมอ และต้องเป็นการถ่ายสด** — กล่องถ่ายรูปสต็อกเปิดกล้องหลังโดยตรง เลือกจากคลังภาพไม่ได้ **โดยเจตนา** (ต้องเป็นหลักฐานของจริงตรงหน้า). ห้ามเปลี่ยนเป็น "เลือกรูปจากเครื่องก็ได้"
- **ตัดสต็อก = บังคับระบุเหตุผลเสมอ และรูปไม่บังคับ** — ห้ามเอา dropdown เหตุผลออก ห้ามทำให้เหตุผลว่างได้ และห้ามเพิ่มการบังคับถ่ายรูปในโหมดตัด
- **ห้ามเพิ่มช่อง "จำนวนสต็อก" เข้าไปในฟอร์มแก้ไขสินค้า** — สต็อกต้องเปลี่ยนผ่านรายการเคลื่อนไหวที่มีหลักฐานครบ (รูป/เหตุผล/เวลา/ผู้ทำ) เท่านั้น. การเพิ่มช่องนี้ = ทำลาย audit trail ทั้งระบบ
- **ลำดับชั้นความอันตรายต้องคงไว้ครบทุกชั้น:**
  - รับเข้า = ปลอดภัย → ปุ่มหลัก, **ไม่มีกล่องยืนยัน**, แต่บังคับรูป
  - ตัดสต็อก = อันตราย → ตัวอักษรแดงในลิสต์ → ปุ่มพื้นแดงเต็มใน modal → **กล่องยืนยันซ้อนอีกชั้นที่บอกผลกระทบเป็นตัวเลขจริง** → ปุ่มยืนยันสีแดง
  - โหมดตัด**ตัดปุ่มลัด `+10/+50/+100` ออกโดยเจตนา** และจำกัดสูงสุดที่สต็อกจริง — **ห้ามใส่ปุ่มลัดกลับเข้าไปในโหมดตัด**
- **ห้ามเพิ่มช่อง "รหัสสินค้า" / น้ำหนัก / ค่าคอมต่อชิ้น กลับเข้ามาในฟอร์ม** — เจ้าของตัดสินใจไว้ว่า "การเพิ่มสินค้าต้องใช้ได้เลยสำหรับเจ้าของร้านที่ไม่เทคนิค". รหัสสินค้าระบบสร้างเองจากชื่อ, ค่าคอมตั้งที่ระดับร้าน
- **ไม่มีปุ่มลบสินค้า มีแค่ปิดขาย** — เป็นการตัดสินใจเชิงธุรกิจ ห้ามเพิ่มปุ่มลบ
- **ราคาเป็นจำนวนเต็มบาทเท่านั้น ไม่รับสตางค์** (0–10,000,000)
- **ข้อความไทยทุกบรรทัดในข้อ 5 คือข้อความจริง** — ห้ามคิดเอง ห้ามเกลาโดยพลการ. ถ้าเสนอเปลี่ยนถ้อยคำ ให้ทำเป็นข้อเสนอแยกให้เจ้าของอนุมัติ. สังเกตอัญประกาศไทยโค้ง “ ” ใน `กด “เพิ่มสินค้า” เพื่อเริ่มขาย` ต้องคงไว้เป๊ะ

### ⚠️ กับดักที่ต้องระวัง

- **คำว่า "คงเหลือ" ถูกใช้กับตัวเลขสองตัวที่ไม่เหมือนกัน!**
  - `"คงเหลือ 12"` ในการ์ดลิสต์ = **ของพร้อมขาย** (ของในสต็อก − ของที่จองแล้ว)
  - `"คงเหลือปัจจุบัน 12"` ในแถบสรุปของ modal = **ของในสต็อกดิบ รวมของที่จองแล้ว**
  - ตอนนี้โค้ดอุดด้วยบรรทัดเหลือง `"มีของถูกจองอยู่ N ชิ้น (รวมอยู่ในคงเหลือ)"` แต่ **นี่คือจุดสับสนที่ควรแก้ให้ขาด** — เสนอใช้คำต่างกันไปเลย เช่น "พร้อมขาย" vs "ในสต็อกทั้งหมด" (ต้องขออนุมัติเจ้าของเพราะเป็นการเปลี่ยนถ้อยคำ)
- **ปุ่ม `"ตัดสต็อก"` (อันตราย) อยู่ติดกับ `"รับเข้า"` (ปกติ) ในกริด 2×2 ห่างกันแค่ ~6px และเป็นปุ่มขนาดเล็ก** — เป็นเป้ากดที่เล็กที่สุดบนหน้าจอนี้ ในขณะที่ผู้ใช้ยืนถือของอยู่. **นี่คือความเสี่ยงกดพลาดที่ชัดที่สุด**
- **หัว modal เขียน `"ตัดสต็อก / แก้ยอด"` แต่ปุ่มในลิสต์เขียน `"ตัดสต็อก"`** — คำไม่ตรงกัน
- **ปุ่มลัด `+10/+50/+100` คือการ *บวกเพิ่ม* ไม่ใช่ตั้งค่า** — กด `+10` สองครั้งได้ 20 ไม่ใช่ 10. ถ้าผู้ใช้เข้าใจผิดจะรับของเข้าเกิน. ดีไซน์ปัจจุบันไม่สื่อเรื่องนี้เลย
- **ปุ่ม `"ตัดสต็อก"` ที่จางลง (กดไม่ได้) ไม่มีคำอธิบายว่าทำไม** — ผู้ใช้ต้องเดาเองว่าเพราะไม่มีของ
- **ระหว่าง `"กำลังบันทึก…"` ผู้ใช้ยังปิด modal ได้ และยังแก้ฟิลด์อื่นได้** — ไม่มีอะไรกันคลิก. ถ้าปิด modal ระหว่างอัปโหลด งานยังวิ่งต่อแต่ผู้ใช้จะไม่เห็น toast สำเร็จในบริบทเดิม
- **รูปรับเข้าอัปโหลดไฟล์ดิบจากกล้อง ไม่ย่อขนาด** (ต่างจากรูปสินค้าที่ย่อในเครื่องก่อน) — ไฟล์อาจหลาย MB บนเน็ตร้านที่ช้า และ **ไม่มีแถบความคืบหน้าเลย มีแค่ข้อความ `"กำลังบันทึก…"`**. เกิน 5MB จะเด้ง error หลังรอนาน
- **ประวัติสต็อกต้องขอลิงก์รูปทีละใบก่อนแสดงผล** — modal ค้างที่ตัวหมุนนานตามจำนวนรูป ไม่มีการทยอยแสดง
- **หมวดสินค้าไม่มีสถานะโหลดของตัวเอง** — ระหว่างรอ ชื่อหมวดหายไปเงียบ ๆ ผู้ใช้อาจเข้าใจผิดว่าสินค้าไม่มีหมวด
- **ปุ่ม `"สร้าง"` หมวดใหม่เมื่อช่องว่าง = กดแล้วเงียบสนิท ไม่มี feedback ใด ๆ**
- **ปุ่ม `"ยกเลิก"` ในโหมดสร้างหมวดไม่ล้างชื่อที่พิมพ์ไว้** — เปิดใหม่แล้วชื่อเก่าค้าง
- **ช่องกรอกไม่มีการจำกัดความยาวเลย แต่เซิร์ฟเวอร์จำกัด:** ชื่อสินค้า 120, รายละเอียด 500, ชื่อหมวด 60, เหตุผล 300 ตัวอักษร → ผู้ใช้พิมพ์ยาวเกินได้แล้วค่อยโดน error หลังกดบันทึก
- **error บางข้อความจากเซิร์ฟเวอร์ยังเป็นภาษาอังกฤษ** และถูกแสดงตรง ๆ ใน toast ให้ผู้ใช้ไทยเห็น: `"Requires role: stock."`, `"No active staff profile for this tenant."`, `"Invalid photo path."`
- **toast หายเองใน 3.6 วินาที ไม่มีปุ่มปิด ซ้อนกันเป็นชั้นแนวตั้งได้** ลอยอยู่เหนือแถบเมนูล่างบนมือถือ. ถ้าข้อความ error ยาว ผู้ใช้อาจอ่านไม่ทัน
- **ดีไซน์ต้องรองรับแถบเมนูล่างสองความหนาแน่น:** role `stock` เห็นเมนู 2 อัน / admin เห็น 11 อันและแถบต้องเลื่อนแนวนอนได้

### ✏️ ปรับได้ — จุดอ่อนที่ควรถูกปรับปรุง

1. **[ใหญ่ที่สุด] ลิสต์ยาวได้ไม่จำกัดแต่ไม่มีเครื่องมือจัดการเลย** — ไม่มีช่องค้นหา, ไม่มีตัวกรองตามหมวด, ไม่มีตัวกรอง "เฉพาะของหมด" หรือ "เฉพาะที่ปิดขาย", ไม่มีการแบ่งหน้า, ไม่มี virtualization. ดึงสินค้าทั้งหมดทีเดียวเรียงตามชื่อ. **ถ้าร้านมีสินค้าหลักร้อยรายการ พนักงานต้องเลื่อนหาเอง — นี่คือช่องว่างที่ใหญ่ที่สุดของหน้านี้**
2. **สินค้าที่ปิดขายปนอยู่ในลิสต์เดียวกัน** ไม่ได้แยกกลุ่ม ไม่ได้ลดความเด่น
3. **ไม่มีตัวบอกว่าข้อมูลอัปเดตล่าสุดเมื่อไหร่ และไม่มีปุ่มรีเฟรช** (ดูข้อ 7)
4. **ปุ่ม 4 ปุ่มเล็กเกินไปสำหรับคนยืนถือของ** และปุ่มอันตรายอยู่ติดปุ่มปกติ — ควรพิจารณาแยกกลุ่ม เพิ่มระยะห่าง หรือย้ายแอ็กชันรองไปอยู่หลัง overflow
5. **ไม่มี skeleton ระหว่างโหลด** มีแค่วงกลมหมุนเล็ก ๆ กลางที่ว่าง
6. **ไม่มีแถบความคืบหน้าตอนอัปโหลดรูป** — จุดที่ผู้ใช้รอนานที่สุดกลับได้ feedback น้อยที่สุด
7. **ประวัติจำกัด 20 รายการแบบตายตัว** ไม่มีปุ่ม "ดูเพิ่ม" ไม่มีลิงก์ไปหน้าประวัติเต็ม — มีแค่ข้อความ `"แสดง 20 รายการล่าสุด"`. ถ้าธุรกิจต้องตรวจย้อนหลังต้องออกแบบทางออกใหม่
8. **แถบสรุป `"คงเหลือปัจจุบัน N → หลังตัด M"` คือหัวใจ UX ของ modal นี้** — คนหน้างานตัดสินใจจากตัวเลขนี้ ไม่ใช่จากตัวเลขในช่องกรอก. **ควรทำให้เด่นขึ้น ไม่ควรลดเป็นข้อความเล็ก ๆ**
9. **Accessibility ที่ควรแก้:** modal ไม่มี focus trap / ไม่ปิดด้วย Esc / ไม่ประกาศตัวเป็น dialog; หลายสถานะสื่อด้วยสีอย่างเดียว (ยอดคงเหลือแดง, `+N` เขียว / `-N` แดง — แม้จะมีเครื่องหมาย +/− ช่วยอยู่บ้าง); ป้าย `"ปิดขาย"` คอนทราสต์ต่ำ (เทาเข้มบนเทาอ่อน); ไอคอนทุกตัวเป็น emoji (📦 🛍️ 🗒️ 📷 ⚠️) ไม่ใช่ SVG จึงหน้าตาไม่เหมือนกันข้าม OS
10. **การสร้างหมวดแบบ inline คือ pattern ที่ดีและตั้งใจ** — ผู้ใช้ไม่ต้องออกจากฟอร์มไปหน้าตั้งค่า และหมวดใหม่ถูกเลือกให้อัตโนมัติ. **รักษาแนวคิดนี้ไว้** แค่ซ่อม feedback ที่ขาด (ปุ่มเงียบเมื่อว่าง, ยกเลิกแล้วชื่อค้าง)
