# งานไรเดอร์ (`/rider`)

> ไรเดอร์เปิดหน้านี้บนมือถือตลอดกะงาน เพื่อกดรับรอบส่ง ไล่ส่งของทีละจุดพร้อมถ่ายรูปยืนยัน และเช็ครายได้สัปดาห์นี้

---

## 1. บริบทการใช้งาน

**ใครใช้**
- ผู้ใช้หลักคือ **ไรเดอร์** (role `rider`) — เมื่อล็อกอิน ระบบพาเข้าหน้านี้เป็นหน้าแรกอัตโนมัติ ไม่ต้องกดเมนู
- **แอดมิน** (role `admin`) เข้าดูได้ด้วย และเซิร์ฟเวอร์ยกเว้นกฎ "ออเดอร์ต้องเป็นของไรเดอร์คนที่กด" ให้แอดมิน (ใช้ช่วยเคลียร์งานแทนได้)
- ผู้ใช้ role อื่นเข้าไม่ได้เลย — ถูกกันตั้งแต่ระดับ route ไม่ได้เห็นหน้านี้ในสภาพ permission-denied จึงไม่ต้องออกแบบหน้าจอ "ไม่มีสิทธิ์" สำหรับหน้านี้

**อุปกรณ์และสถานการณ์จริง**
- **มือถือเท่านั้นในทางปฏิบัติ** ไรเดอร์ยืนกลางแดด ถือกล่องของอยู่มืออีกข้าง ใช้นิ้วโป้งมือเดียว บางครั้งนั่งคร่อมมอเตอร์ไซค์ สวมถุงมือ เน็ตมือถืออาจช้าหรือหลุดเป็นช่วง
- หน้าจอนี้เป็น **mobile-first อย่างเด็ดขาด** — ไม่ใช่แค่ "รองรับมือถือด้วย" เดสก์ท็อปเป็นแค่ผลพลอยได้สำหรับแอดมินที่นั่งดู ทุกการตัดสินใจด้านดีไซน์ต้องเริ่มจากจอ ~375–430px กว้าง และนิ้วโป้งข้างเดียว
- ปุ่มหลักในหน้านี้จึงถูกทำให้ใหญ่กว่าปกติมาก (สูง 52–56px เต็มความกว้าง) — เป็นการตัดสินใจที่ตั้งใจ ไม่ใช่ความเผลอ

**ความถี่และความเร่งรีบ**
- เปิดค้างไว้ทั้งกะ ดูซ้ำ ๆ ทุกจุดส่ง — รอบหนึ่งอาจมี 5–20 จุด แต่ละจุดต้องแตะอย่างน้อย 2–3 ครั้ง (เริ่มส่ง → ถ่ายรูป → ยืนยัน)
- **แท็บ "รอบพร้อมรับ" มีการแข่งขัน** — รอบเป็นแบบใครกดก่อนได้ก่อน ไรเดอร์หลายคนเห็นรอบเดียวกันพร้อมกัน จังหวะนี้เร่งรีบสูงสุดในหน้าจอทั้งหมด
- ช่วงถ่ายรูป POD คือช่วงที่มือไม่ว่าง มีเวลาไม่กี่วินาที ลูกค้ายืนรออยู่ตรงหน้า

## 2. โครงหน้า

ลำดับจากบนลงล่าง:

1. **แถบ Shell (กรอบนอก ไม่ใช่ของหน้านี้)** — แถบหัวติดขอบบน มีชื่อระบบ `"PKM Shop"` และเมนู บนมือถือมีแถบแท็บล่างจอติดขอบล่าง ไอคอน 🛵 ป้าย `"ไรเดอร์"` เป็นแท็บของหน้านี้ พื้นที่เนื้อหาถูกเว้นด้านล่างไว้แล้วให้พ้นแถบนี้
2. **แบนเนอร์เตือนผูก LINE (มีเงื่อนไข)** — ถ้าไรเดอร์ยังไม่ผูกบัญชี LINE จะมีแถบเหลืองคาดหัวทุกหน้ารวมถึงหน้านี้ ข้อความจริง: `"⚠️ ยังไม่ได้ผูก LINE — เพิ่ม PKM Shop OA เป็นเพื่อน แล้วพิมพ์รหัส {link_code} เพื่อรับแจ้งเตือนงาน"` (เช่น รหัส `A7K2M9`) — **กินความสูงเพิ่ม ต้องเผื่อในดีไซน์**
3. **หัวข้อหน้า** — ข้อความ `"งานไรเดอร์"` ตัวหนาขนาดกลาง ชิดซ้าย ไม่มีปุ่มใด ๆ ทางขวา (ช่องว่างขวานี้ว่างอยู่จริง)
4. **แถบแท็บแบบ segmented** — พื้นเทาอ่อน ขอบมน 3 ช่องแบ่งเท่ากัน แท็บที่เลือกเป็นการ์ดขาวมีเงาจาง แท็บที่ไม่ได้เลือกเป็นตัวอักษรเทา ความสูงขั้นต่ำ 40px แต่ละแท็บมีตัวเลข badge วงกลมสีแบรนด์ตัวขาวเล็ก ๆ ต่อท้ายป้าย
5. **เนื้อหาตามแท็บที่เลือก** (ดูรายละเอียดแต่ละแท็บด้านล่าง)
6. **Toast แจ้งผล** — ลอยเหนือแถบแท็บล่าง (บนจอใหญ่ลอยมุมล่าง) กล่องเดียวกว้างไม่เกิน ~384px หายเองใน 3.6 วินาที **ไม่มีปุ่มปิด**

### แท็บที่ 1 — `"รอบพร้อมรับ"` (แท็บเริ่มต้นเมื่อเปิดหน้า)

- **Badge** = จำนวนรอบว่างที่รับได้ตอนนี้ **ซ่อนเมื่อเป็น 0** (ไม่แสดงเลข "0")
- เนื้อหาเป็นรายการ **การ์ดรอบว่าง** เรียงจากรอบเวลาเก่าสุดไปใหม่สุด แต่ละการ์ด (พื้นขาว ขอบมน มีระยะห่างระหว่างบล็อกภายใน) ประกอบด้วย:
  - แถวบนแบ่งซ้าย-ขวา:
    - **ซ้าย**: เวลารอบ ตัวใหญ่มากตัวหนา (ขนาดพาดหัว) เช่น `"รอบ 14:00 น."` และใต้ลงมาเป็นตัวเล็กสีเทา จำนวนจุด เช่น `"8 จุดส่ง"`
    - **ขวา**: ป้ายสถานะพื้นเหลืองอ่อนตัวอักษรเหลืองเข้ม ข้อความ `"พร้อมรับ"`
  - **ปุ่มหลักเต็มความกว้าง** สีแบรนด์ทึบ สูงขั้นต่ำ 56px ตัวอักษรใหญ่ ข้อความ `"รับรอบนี้"`

### แท็บที่ 2 — `"งานของฉัน"`

- **Badge** = จำนวนรอบที่รับไว้และยังไม่จบ ซ่อนเมื่อเป็น 0
- เนื้อหาเป็นรายการ **การ์ดรอบของฉัน** (ถ้ารับหลายรอบจะเรียงต่อกันลงมา แต่ละรอบเป็นการ์ดใหญ่หนึ่งใบ) แต่ละใบมี:
  - **หัวการ์ด** แบ่งซ้าย-ขวา:
    - **ซ้าย**: เวลารอบ ตัวหนาขนาดกลาง เช่น `"รอบ 14:00 น."`
    - **ขวา**: ป้ายสถานะ 2 แบบ — ถ้ารอบกำลังวิ่ง = พื้นฟ้าอ่อนตัวอักษรฟ้าเข้ม `"กำลังวิ่ง · เหลือ 3 จุด"` / ถ้ารับแล้วแต่ยังไม่กดเริ่มจุดแรก = พื้นเหลืองอ่อนตัวอักษรเหลืองเข้ม `"รับแล้ว รอเริ่ม"`
  - **รายการจุดส่ง** ซ้อนอยู่ในการ์ดรอบอีกชั้น (การ์ดในการ์ด) เรียงตามลำดับจุดจากน้อยไปมาก จุดที่ยังไม่มีเลขลำดับจะไปอยู่ท้ายสุด แต่ละจุดเป็นกล่องขอบมนมีกรอบ **3 หน้าตาตามสถานะ**:
    - **จบแล้ว** (ส่งสำเร็จ / ตีกลับ / รอค่าส่งใหม่ / ยกเลิก) → กรอบและพื้นเทาจาง ทั้งกล่องจางลงประมาณ 70%
    - **กำลังส่งอยู่** → กรอบสีแบรนด์โปร่ง พื้นสีแบรนด์จาง ๆ (ไฮไลต์ให้เด่นสุดในรายการ)
    - **รอคิว** → กรอบเทาอ่อนธรรมดา พื้นขาว
  - ภายในกล่องจุดส่ง เรียงจากบนลงล่าง:
    1. แถวบนแบ่งซ้าย-ขวา — **ซ้าย**: วงกลมทึบสีเข้มขนาด 32px ตัวเลขขาวตัวหนาแสดงลำดับจุด (ถ้ายังไม่มีเลขแสดง `"-"`) ถัดมาเป็นเลขออเดอร์ตัวอักษรแบบ monospace ตัวเล็กหนา และชื่อผู้รับตัวหนาขนาดปกติ / **ขวา**: ป้ายสถานะภาษาไทย (ดูตารางสถานะในหมวด 6)
    2. **ที่อยู่** ตัวเล็กสีเทาเข้ม เต็มความกว้าง
    3. **ปุ่มโทร** (ถ้ามีเบอร์) — ปุ่มโปร่งเต็มความกว้าง เช่น `"📞 โทร 0812345678"`
    4. **กลุ่มปุ่มตามเงื่อนไข** เรียงลงมาเป็นคอลัมน์ (ดูหมวด 3)

### แท็บที่ 3 — `"รายได้"`

- **ไม่มี badge**
- เนื้อหาเรียงลงมา:
  1. **การ์ดตัวเลข 2 ใบวางคู่กัน (2 คอลัมน์เท่ากัน)**
     - **ใบซ้าย** โทนสีแบรนด์ (กรอบและพื้นสีแบรนด์จาง ตัวเลขสีแบรนด์): ป้ายเล็กสีเทา `"รอบนี้ (สัปดาห์นี้)"` / ตัวเลขใหญ่ตัวหนา เช่น `"฿1,250"` / คำอธิบายเล็กสีเทาอ่อน `"ตัดรอบคืนวันอาทิตย์ เที่ยงคืน"`
     - **ใบขวา** โทนปกติ (พื้นขาว): ป้าย `"จำนวนรอบที่วิ่ง"` / ตัวเลข เช่น `"5 รอบ"` / คำอธิบาย เช่น `"สะสมทั้งหมด 47 รอบ"`
  2. **หัวข้อรอง** ตัวหนาขนาดปกติ `"ประวัติการจ่ายเงิน"`
  3. **รายการการ์ดการจ่ายเงิน** (แต่ละใบพื้นขาว ขอบมน แบ่งซ้าย-ขวา ชิดกึ่งกลางแนวตั้ง เรียงจากใหม่ไปเก่า):
     - **ซ้าย**: ยอดเงินตัวหนาขนาดปกติ เช่น `"฿4,800"` และบรรทัดรองตัวเล็กสีเทา — ถ้าโอนแล้วแสดง `"โอนเมื่อ 16 ก.ค. 68 14:30"` / ถ้ายังไม่โอนแสดง `"รอผู้บริหารโอน"`
     - **ขวา**: ป้ายสถานะ — พื้นเขียวอ่อนตัวอักษรเขียวเข้ม `"โอนแล้ว"` หรือพื้นเหลืองอ่อนตัวอักษรเหลืองเข้ม `"รอโอน"` และถ้ามีไฟล์สลิปจะมีปุ่มโปร่งขนาดเล็กต่อท้าย ข้อความ `"ดูสลิป"`

## 3. ทุก flow ทีละสเต็ป

### 3.1 รับรอบ (แข่งกันกดกับไรเดอร์คนอื่น)

**ขั้นตอน**
1. เปิดหน้า `/rider` → แท็บ `"รอบพร้อมรับ"` ถูกเลือกให้อัตโนมัติ
2. ระบบแสดงเฉพาะรอบที่ **ยังไม่มีไรเดอร์คนไหนรับ** และถูกปลดล็อกให้รับได้แล้ว เรียงจากเวลารอบเก่าสุดไปใหม่สุด พร้อมจำนวนจุดของแต่ละรอบ
3. ไรเดอร์แตะปุ่ม `"รับรอบนี้"` บนการ์ดรอบที่ต้องการ
4. เปิด **กล่องยืนยัน** (บนมือถือเลื่อนขึ้นจากขอบล่างจอ บนจอใหญ่เป็นกล่องกลางจอ):
   - หัวข้อ: `"รับรอบ 14:00 น. ?"` (รูปแบบ: `รับรอบ {HH}:00 น. ?`)
   - ข้อความ: `"รอบนี้มี 8 จุดส่ง — เมื่อกดรับ ลูกค้าทุกคนในรอบจะได้รับแจ้งเตือนทันที"` (รูปแบบ: `รอบนี้มี {N} จุดส่ง — เมื่อกดรับ ลูกค้าทุกคนในรอบจะได้รับแจ้งเตือนทันที`)
   - ปุ่มสองปุ่มเรียงข้างกันกว้างเท่ากัน — ซ้าย ปุ่มโปร่ง `"ยกเลิก"` / ขวา ปุ่มสีแบรนด์ทึบ `"รับรอบนี้"`
   - **ปุ่มยืนยันไม่ใช่สีแดง** — นี่ไม่ใช่การกระทำอันตราย เป็นการรับงาน
5. ถ้ากด `"ยกเลิก"` / แตะพื้นหลังมืด / กดปุ่ม `✕` (ป้ายช่วยการเข้าถึง: `"ปิด"`) → กล่องปิด ไม่มีอะไรเกิดขึ้น
6. ถ้ากด `"รับรอบนี้"` → **กล่องปิดทันที** แล้วค่อยยิงคำขอไปเซิร์ฟเวอร์ (ไม่มีสถานะรอในกล่อง)
7. เบื้องหลังที่ผู้ใช้มองไม่เห็น: รอบถูกจองเป็นของไรเดอร์คนนี้ → ระบบไล่ให้เลขลำดับจุด 1..n กับออเดอร์ที่แพ็คเสร็จแล้ว (เรียงตามเวลาที่ลูกค้าสั่ง เก่าสุดก่อน — **ไม่ได้เรียงตามระยะทาง**) → **ออเดอร์ที่ยังแพ็คไม่เสร็จถูกเลื่อนไปรอบถัดไปเงียบ ๆ** → ลูกค้าทุกคนในรอบได้รับแจ้งเตือน LINE ทันที
8. สำเร็จ → toast เขียว → **สลับไปแท็บ `"งานของฉัน"` ให้อัตโนมัติ** → ข้อมูลทุกส่วนถูกดึงใหม่

**กฎการตรวจสอบ**
- ไม่มีช่องกรอกใด ๆ — การยืนยันผ่านกล่องยืนยันคือด่านเดียว
- รอบต้องยังไม่มีคนรับ ณ วินาทีที่กด — **มีโอกาสแพ้จริงและเกิดบ่อย**
- ผู้กดต้องมี role `rider`

**ข้อความสำเร็จ**
- toast สีเขียวพื้นทึบตัวอักษรขาว: `"รับรอบแล้ว เริ่มวิ่งได้เลย"` (แสดง 3.6 วินาที)

**ข้อความล้มเหลว** — toast สีแดงพื้นทึบตัวอักษรขาว ข้อความใดข้อความหนึ่ง:
- `"รอบนี้มีไรเดอร์รับแล้ว"` ← กรณีแพ้การแข่งขัน (พบบ่อยที่สุด)
- `"รอบนี้ยังไม่พร้อมให้รับ"` ← รอบถูกปลดล็อกกลับ/ยังไม่ถึงเวลา
- `"รับรอบไม่สำเร็จ"` ← ข้อความสำรองกรณีอื่น ๆ
- หลังขึ้น toast แดง รายการรอบว่างถูกดึงใหม่ทันที **การ์ดที่ถูกคนอื่นแย่งไปจะหายออกจากจอต่อหน้าผู้ใช้**

### 3.2 เริ่มส่งจุดถัดไป

**ขั้นตอน**
1. ไปแท็บ `"งานของฉัน"` → เห็นการ์ดรอบพร้อมรายการจุดส่งเรียงตามลำดับ
2. **ปุ่มเริ่มส่งโผล่เพียงจุดเดียวเท่านั้นในทั้งรอบ** — คือจุดแรกสุดที่ยังไม่เริ่ม และจะโผล่ก็ต่อเมื่อ **ไม่มีจุดใดกำลังส่งอยู่** (ทำทีละจุดเท่านั้น เป็นกฎธุรกิจ)
3. แตะปุ่มสีแบรนด์ทึบเต็มความกว้าง สูงขั้นต่ำ 52px ข้อความ `"เริ่มส่งจุดที่ 3"` (รูปแบบ: `เริ่มส่งจุดที่ {N}`)
4. **ไม่มีกล่องยืนยัน** — ยิงคำขอทันทีที่แตะ
5. เบื้องหลัง: รอบเปลี่ยนเป็นสถานะกำลังวิ่ง → จุดนั้นเปลี่ยนเป็นกำลังส่ง → **ลูกค้ารายนั้นได้รับแจ้งเตือน LINE ว่าไรเดอร์กำลังไปหา**
6. สำเร็จ → toast สีเทาเข้ม
7. **หน้าจอเปลี่ยนทันที**: กล่องจุดนั้นไฮไลต์สีแบรนด์ / ป้ายเปลี่ยนเป็น `"กำลังส่ง"` (พื้นฟ้าอ่อน) / ปุ่มเริ่มส่งหายไป แทนที่ด้วยปุ่ม `"📷 ส่งสำเร็จ (ถ่ายรูป POD)"` และ `"ตีกลับ"` / ป้ายหัวการ์ดเปลี่ยนเป็น `"กำลังวิ่ง · เหลือ {N} จุด"`

**กฎการตรวจสอบ**
- ไม่มีฟอร์ม — การควบคุมทำผ่านการซ่อน/แสดงปุ่มล้วน ๆ ไรเดอร์ **ข้ามจุด สลับลำดับ หรือทำหลายจุดพร้อมกันไม่ได้**
- เซิร์ฟเวอร์ตรวจว่าออเดอร์นี้อยู่ในรอบของไรเดอร์คนที่กดจริง (ยกเว้นแอดมิน)

**ข้อความสำเร็จ**
- toast สีเทาเข้มพื้นทึบตัวอักษรขาว: `"เริ่มส่งจุดที่ 3 แล้ว — ลูกค้าได้รับแจ้งเตือน"` (รูปแบบ: `เริ่มส่งจุดที่ {N} แล้ว — ลูกค้าได้รับแจ้งเตือน`)

**ข้อความล้มเหลว** — toast สีแดง:
- `"ออเดอร์นี้อยู่ในรอบของไรเดอร์คนอื่น"`
- `"ทำรายการไม่สำเร็จ"` ← ข้อความสำรอง

---

### 3.3 ปิดจุดด้วยรูป POD (ส่งสำเร็จ) — flow หัวใจของหน้านี้

**ขั้นตอน**
1. ที่จุดที่กำลังส่งอยู่ แตะปุ่มสีแบรนด์ทึบเต็มความกว้าง สูงขั้นต่ำ 52px ข้อความ `"📷 ส่งสำเร็จ (ถ่ายรูป POD)"`
2. เปิด **โมดัล** (บนมือถือเลื่อนขึ้นจากขอบล่าง มุมบนมน ขอบล่างเว้นพื้นที่ปลอดภัยของเครื่อง / บนจอใหญ่เป็นกล่องกลางจอ กว้างสูงสุด ~448px สูงสุด 90% ของจอ เลื่อนในตัวได้ / ปิดได้ด้วยการแตะพื้นหลังมืดหรือปุ่ม `✕`)
   - หัวข้อโมดัล: `"ส่งสำเร็จ · จุดที่ 3"` (รูปแบบ: `ส่งสำเร็จ · จุดที่ {N}`)
3. บรรทัดอ้างอิงตัวเล็กสีเทา: `"{เลขออเดอร์} · {ชื่อผู้รับ}"` เช่น `"PKM-2601-0042 · สมชาย ใจดี"` (ถ้าไม่มีชื่อผู้รับใช้คำว่า `"ลูกค้า"`)
4. ป้ายกำกับตัวเล็กตัวหนา: `"รูปหลักฐานการส่ง (POD)"`
5. **ตัวเลือกรูป สถานะเริ่มต้น** — กล่องขอบประ (เส้นประ 2px สีเทา) พื้นเทาอ่อนมาก ความสูงขั้นต่ำ 96px จัดกลางทั้งแนวตั้งและแนวนอน ภายในมีไอคอน 📷 ตัวใหญ่ และข้อความตัวเล็กตัวหนา `"ถ่ายรูปสินค้า ณ จุดส่ง (บังคับ)"` — คำว่า `" (บังคับ)"` ถูกต่อท้ายอัตโนมัติเพราะฟิลด์นี้บังคับ **ทั้งกล่องกดได้ ไม่ใช่แค่ไอคอน**
6. แตะกล่อง → **เปิดกล้องหลังของเครื่องโดยตรง** ไม่ผ่านหน้าเลือกรูปจากแกลเลอรี
7. หลังถ่าย → กล่องขอบประถูกแทนที่ด้วย **รูปพรีวิว** (ขอบมน สูงสุด ~224px เต็มความกว้าง ตัดขอบรูปให้เต็มกรอบ) มีแถบไล่เฉดดำจาง ๆ ทาบขอบล่างของรูป และปุ่มโปร่งพื้นขาวโปร่งแสงลอยมุมขวาล่าง ข้อความ `"ถ่ายใหม่"` (คำบรรยายรูปสำหรับ screen reader: `"ตัวอย่างรูป"`)
8. ข้อความกำกับใต้ตัวเลือกรูป ตัวเล็กสีเทาอ่อน: `"บังคับถ่ายรูปก่อนปิดจุด — ลูกค้าจะได้รับรูปนี้พร้อมแจ้งเตือน"`
9. **ปุ่มยืนยัน** สีแบรนด์ทึบเต็มความกว้าง สูงขั้นต่ำ 52px ข้อความ `"ยืนยันส่งสำเร็จ"` — **ถูกปิดการใช้งาน (จาง กดไม่ได้) จนกว่าจะมีรูป**
10. แตะยืนยัน → ปุ่มเปลี่ยนข้อความเป็น `"กำลังบันทึก…"` และถูกปิดการใช้งาน (สถานะนี้ครอบทั้งช่วงอัปโหลดรูปและช่วงบันทึก — **อาจนานหลายวินาทีบนเน็ตช้า เพราะรูปจากกล้องไม่ถูกย่อ**)
11. เบื้องหลัง: รูปถูกอัปโหลด → ออเดอร์เปลี่ยนเป็นส่งสำเร็จ → **ลูกค้าได้รับรูปนี้ทาง LINE ทันที** → ถ้าไม่เหลือจุดค้างในรอบแล้ว ระบบ **ปิดรอบและบันทึกค่ารอบให้ไรเดอร์อัตโนมัติ**
12. สำเร็จ → โมดัลปิดเอง → toast เขียว → ข้อมูลจุดส่ง รอบ และรายได้ถูกดึงใหม่
13. กล่องจุดนั้นกลายเป็นเทาจาง ป้ายเป็น `"ส่งสำเร็จ"` (เขียว) ปุ่มทั้งหมดหายไป และปุ่ม `"เริ่มส่งจุดที่ {N+1}"` โผล่ที่จุดถัดไปแทน

**กฎการตรวจสอบ**
- **รูป POD บังคับ 100%** — ปิดจุดโดยไม่มีรูปไม่ได้ทุกกรณี ปุ่มยืนยันกดไม่ได้เมื่อยังไม่มีรูป (กฎธุรกิจ **ห้ามแตะ**)
- รับเฉพาะไฟล์รูปภาพ และเปิดกล้องหลังเสมอ
- **ไม่มีการจำกัดขนาดไฟล์หรือย่อรูป** ในหน้านี้ — รูปดิบจากกล้องถูกส่งขึ้นตรง ๆ
- กันการกดซ้ำระหว่างบันทึกเรียบร้อยแล้ว

**ข้อความสำเร็จ**
- toast สีเขียว: `"ส่งสำเร็จ จุดที่ 3 🎉"` (รูปแบบ: `ส่งสำเร็จ จุดที่ {N} 🎉`)

**ข้อความล้มเหลว**
- toast สีแดง — ข้อความจริงจากระบบ หรือข้อความสำรอง `"ยืนยันส่งสำเร็จไม่สำเร็จ"`
- **สำคัญ: โมดัลไม่ปิด** รูปที่ถ่ายไว้ยังอยู่ ปุ่มกลับมากดใหม่ได้ทันที (ออกแบบมาให้ลองซ้ำได้โดยไม่ต้องถ่ายรูปใหม่ — **ห้ามเปลี่ยนพฤติกรรมนี้**)

### 3.4 ตีกลับ

**ขั้นตอน**
1. ที่จุดที่กำลังส่งอยู่ แตะปุ่มรอง — **ปุ่มโปร่ง (ไม่ทึบ) ตัวอักษรสีแดง** เต็มความกว้าง ข้อความ `"ตีกลับ"` (จงใจให้ด้อยกว่าปุ่ม POD)
2. เปิดโมดัล หัวข้อ: `"ตีกลับ · จุดที่ 3"` (รูปแบบ: `ตีกลับ · จุดที่ {N}`) — เนื้อในเป็นฟอร์ม (กด Enter ส่งได้)
3. บรรทัดอ้างอิงตัวเล็กสีเทา: `"{เลขออเดอร์} · {ชื่อผู้รับ}"`
4. **แถวชิปเหตุผลสำเร็จรูป 4 อัน** — ปุ่มแคปซูลขอบมนเต็ม ขนาดเล็ก ตัดขึ้นบรรทัดใหม่ได้ ข้อความจริงทั้ง 4:
   - `"โทรแล้วไม่รับสาย + รอครบ 5 นาที"`
   - `"ไม่มีคนอยู่รับของ"`
   - `"ที่อยู่ผิด/หาไม่เจอ"`
   - `"ลูกค้ายกเลิก"`
5. แตะชิป → **ข้อความนั้นถูกใส่ลงช่องกรอกทันที แทนที่ข้อความเดิมทั้งหมด** และชิปนั้นเปลี่ยนเป็นสถานะถูกเลือก (พื้นสีแบรนด์ทึบ ตัวอักษรขาว ขอบสีแบรนด์) ชิปที่ไม่ถูกเลือก = ขอบเทา พื้นขาว ตัวอักษรเทา
   - **กับดัก**: การเลือกเทียบจากข้อความตรงตัวเป๊ะ ๆ — ถ้าไรเดอร์แก้ข้อความในช่องแม้ตัวอักษรเดียว **ชิปจะหลุดสถานะเลือกทันที** (ดูหมวด 8)
6. **ช่องกรอกเหตุผล** — ป้ายกำกับตัวเล็กตัวหนา `"เหตุผลการตีกลับ (บังคับ)"` / ช่องพิมพ์หลายบรรทัด สูงขั้นต่ำ 88px / ข้อความตัวอย่างในช่องว่าง (placeholder): `"ระบุเหตุผล เช่น โทร 2 ครั้งไม่รับสาย รอหน้าบ้าน 5 นาที"` / คำอธิบายใต้ช่อง ตัวเล็กสีเทาอ่อน: `"ตีกลับได้เมื่อ: โทรหาลูกค้าแล้วไม่รับสาย และรอที่จุดส่งครบ 5 นาที — ระบบจะแจ้งเหตุผลนี้ให้ลูกค้าพร้อมให้ชำระค่าส่งใหม่"`
7. **ปุ่มยืนยัน** — **พื้นแดงทึบเต็ม ตัวอักษรขาว** เต็มความกว้าง สูงขั้นต่ำ 52px ข้อความ `"ยืนยันตีกลับ"` — จางและกดไม่ได้เมื่อยังไม่มีเหตุผล
8. แตะยืนยัน → ปุ่มเปลี่ยนเป็น `"กำลังบันทึก…"` และถูกปิดการใช้งาน
9. เบื้องหลัง: ออเดอร์เปลี่ยนเป็นตีกลับพร้อมบันทึกเหตุผล → **สต็อกสินค้าถูกคืนอัตโนมัติ** → สถานะเดินต่อเป็น "รอค่าส่งใหม่" → **ลูกค้าได้รับเหตุผลนี้ทาง LINE พร้อมถูกเรียกเก็บค่าส่งใหม่ และแอดมินได้รับแจ้งเตือนด้วย** → ถ้าไม่เหลือจุดค้าง ระบบปิดรอบและจ่ายค่ารอบให้เอง
10. สำเร็จ → โมดัลปิดเอง → toast เขียว → ข้อมูลจุดส่ง รอบ และรายได้ถูกดึงใหม่
11. กล่องจุดนั้นกลายเป็นเทาจาง ป้ายเปลี่ยนเป็น `"รอค่าส่งใหม่"` (สีแดง) หลังข้อมูลอัปเดต

**กฎการตรวจสอบ**
- **เหตุผลบังคับ** — เว้นว่างหรือใส่แต่ช่องว่างไม่ได้ ปุ่มกดไม่ได้
- **ยาวสูงสุด 300 ตัวอักษร** (บังคับทั้งฝั่งหน้าจอและเซิร์ฟเวอร์) — **ปัจจุบันไม่มีตัวนับตัวอักษรให้ผู้ใช้เห็น**
- ช่องว่างหน้า-หลังถูกตัดทิ้งก่อนส่ง
- กันการกดซ้ำระหว่างบันทึกเรียบร้อยแล้ว
- **ไม่มีกล่องยืนยันซ้อน** — ปุ่มแดงในโมดัลคือขั้นยืนยันสุดท้ายเพียงขั้นเดียว (ต่างจากการรับรอบที่มีกล่องยืนยัน)
- กฎธุรกิจ "โทรแล้วไม่รับ + รอครบ 5 นาที" **เป็นเพียงข้อความให้อ่าน ระบบไม่ได้บังคับตรวจ** — ไม่มีตัวจับเวลา ไม่ต้องกดโทรก่อน

**ข้อความสำเร็จ**
- toast สีเขียว: `"บันทึกตีกลับแล้ว ระบบแจ้งลูกค้าให้ชำระค่าส่งใหม่"`

**ข้อความล้มเหลว**
- toast สีแดง — ข้อความจริงจากระบบ หรือข้อความสำรอง `"บันทึกตีกลับไม่สำเร็จ"`
- **โมดัลไม่ปิด** ข้อความเหตุผลที่พิมพ์ไว้ยังอยู่ครบ กดใหม่ได้ทันที

### 3.5 โทรหาลูกค้า

**ขั้นตอน**
1. ในกล่องจุดส่งที่มีเบอร์โทร จะมีลิงก์รูปปุ่มโปร่งเต็มความกว้าง ข้อความ `"📞 โทร 0812345678"` (รูปแบบ: `📞 โทร {เบอร์โทร}`)
2. **ปุ่มนี้แสดงในทุกจุดไม่ว่าสถานะใด** รวมถึงจุดที่ส่งจบไปแล้ว (จุดที่จบแล้วทั้งกล่องจางลง แต่ปุ่มโทรยังอยู่)
3. แตะ → เครื่องเปิดแอปโทรศัพท์พร้อมเบอร์ (ออกนอกเว็บไปแอปของระบบปฏิบัติการ)
4. ถ้าออเดอร์ไม่มีเบอร์ ปุ่มนี้ **ไม่ถูกแสดงเลย** (ไม่ใช่ปุ่มจาง ๆ — หายไปทั้งปุ่ม)

**กฎการตรวจสอบ**
- ไม่มีการจัดรูปแบบเบอร์ใด ๆ — แสดงค่าดิบจากฐานข้อมูลตรง ๆ ทั้งในข้อความและในลิงก์ (เบอร์อาจมาในรูปแบบไม่สม่ำเสมอ เช่น `081-234-5678` หรือ `+66812345678`)

**ข้อความสำเร็จ / ล้มเหลว**
- ไม่มีทั้งคู่ — ระบบปฏิบัติการรับช่วงต่อไปเลย ไม่มี feedback ในแอป

### 3.6 ดูสลิปโอนเงิน

**ขั้นตอน**
1. ในแท็บ `"รายได้"` → การ์ดประวัติการจ่ายเงินที่มีไฟล์สลิปแนบ จะมีปุ่มโปร่งขนาดเล็ก ข้อความ `"ดูสลิป"` ทางขวา
2. แตะ → ระบบขอลิงก์รูปชั่วคราว (**ไม่มีสถานะกำลังโหลดใด ๆ ปุ่มไม่ถูกปิดการใช้งาน กดรัวได้**)
3. สำเร็จ → เปิดโมดัล หัวข้อ `"สลิปโอนเงิน"` แสดงรูปสลิปเต็มใบ (ขอบมน สูงสุด 70% ของความสูงจอ **แสดงทั้งรูปโดยไม่ตัดขอบ** เพราะสลิปเป็นรูปแนวตั้งที่ต้องอ่านตัวเลขได้ครบ) คำบรรยายรูปสำหรับ screen reader: `"สลิปโอนเงิน"`
4. ปิดด้วยปุ่ม `✕` หรือแตะพื้นหลังมืด

**กฎการตรวจสอบ**
- ปุ่มแสดงเฉพาะเมื่อการจ่ายเงินรอบนั้นมีไฟล์สลิปแนบจริง
- ลิงก์รูปมีอายุ **1 ชั่วโมง** — ถ้าเปิดโมดัลค้างไว้เกินนั้นรูปจะพัง และ **ไม่มีการต่ออายุอัตโนมัติ**

**ข้อความสำเร็จ**
- โมดัล `"สลิปโอนเงิน"` เปิดขึ้นพร้อมรูป (ไม่มี toast)

**ข้อความล้มเหลว**
- toast สีแดง: `"เปิดสลิปไม่สำเร็จ"`

## 4. สถานะของหน้า

| สถานะ | ผู้ใช้เห็นอะไร | ข้อความจริง |
|---|---|---|
| **กำลังโหลด (ทั้งแท็บ)** | จัดกลาง มีระยะห่างบน-ล่างพอสมควร: วงกลมหมุนเล็ก ๆ (16px ขอบเทา ส่วนบนสีแบรนด์) + ข้อความตัวเล็กสีเทาอ่อน วางเรียงแนวนอน ใช้ในแท็บ `"รอบพร้อมรับ"`, แท็บ `"งานของฉัน"`, และแท็บ `"รายได้"` (บังทั้งแท็บ) | `"กำลังโหลด…"` |
| **กำลังโหลด (ในการ์ดรอบ)** | **หัวการ์ดยังเห็นอยู่** (เลขรอบ + ป้ายสถานะ) แต่พื้นที่รายการจุดส่งถูกแทนด้วยวงกลมหมุน — ไรเดอร์ยังรู้ว่ารอบไหนกำลังโหลด | `"กำลังโหลด…"` |
| **กำลังโหลด (เฉพาะประวัติการจ่ายเงิน)** | การ์ดตัวเลข 2 ใบด้านบนแสดงปกติ เฉพาะส่วนใต้หัวข้อ `"ประวัติการจ่ายเงิน"` เป็นวงกลมหมุน | `"กำลังโหลด…"` |
| **กำลังโหลดตอนเข้าระบบ** | เต็มจอ จัดกลาง ระหว่างระบบเช็คว่าล็อกอินอยู่ไหม (มาจากกรอบนอก ไม่ใช่หน้านี้) | `"กำลังโหลด…"` |
| **ว่าง — ไม่มีรอบให้รับ** | จัดกลาง เว้นระยะบน-ล่างมาก: ไอคอนใหญ่ / หัวข้อตัวเล็กตัวหนาสีเทาเข้ม / คำอธิบายตัวเล็กมากสีเทาอ่อน | ไอคอน `🛵` / `"ยังไม่มีรอบให้รับตอนนี้"` / `"รอบจะปลดล็อกทุกนาทีที่ :30 ก่อนถึงเวลารอบ"` |
| **ว่าง — ยังไม่ได้รับรอบไหนเลย** | โครงเดียวกับข้างบน | ไอคอน `📦` / `"ยังไม่มีรอบที่รับไว้"` / `"ไปที่แท็บ 'รอบพร้อมรับ' เพื่อกดรับรอบ"` |
| **ว่าง — รอบที่รับไว้ไม่มีออเดอร์** | อยู่ **ภายในการ์ดรอบ** ใต้หัวการ์ด **ไม่มีคำอธิบายรอง** | ไอคอน `📭` / `"ไม่มีออเดอร์ในรอบนี้"` |
| **ว่าง — ยังไม่เคยได้รับเงิน** | อยู่ใต้หัวข้อ `"ประวัติการจ่ายเงิน"` — **การ์ดตัวเลข 2 ใบยังแสดงอยู่ด้านบน ไม่ได้หายไป** | ไอคอน `💰` / `"ยังไม่มีประวัติการจ่ายเงิน"` / `"ยอดจะสรุปให้อัตโนมัติเมื่อตัดรอบสัปดาห์"` |
| **ว่าง — การ์ดตัวเลขไม่มีข้อมูล** | **ไม่มีสถานะว่างแยก** — แสดงเลขศูนย์ตามปกติ | `"฿0"` / `"0 รอบ"` / `"สะสมทั้งหมด 0 รอบ"` |
| **ผิดพลาด (โหลดข้อมูลไม่สำเร็จ)** | จัดกลาง: ไอคอนใหญ่ / ข้อความตัวเล็กตัวหนาสีเทาเข้ม / คำอธิบายตัวเล็กมากสีเทาอ่อน / ปุ่มโปร่งขนาดเล็ก ใช้ 4 จุด — แท็บรอบพร้อมรับ, แท็บงานของฉัน, รายการจุดส่งในการ์ดรอบ (แสดงใต้หัวการ์ด), แท็บรายได้ | ไอคอน `⚠️` / `"โหลดข้อมูลไม่สำเร็จ"` / `"อาจเป็นเน็ตหลุดหรือเซสชันหมดอายุ — ข้อมูลยังอยู่ในระบบ"` / ปุ่ม `"ลองใหม่"` |
| **ผิดพลาดจากการกดปุ่ม** | **ไม่ใช้หน้าจอ error** — ขึ้นเป็น toast แดงเสมอ หน้าจอเดิมยังอยู่ครบ | ดูข้อความในหมวด 3 ของแต่ละ flow |
| **กำลังส่ง — ปุ่มยืนยัน POD** | ปุ่มเปลี่ยนข้อความและกดไม่ได้ ครอบทั้งช่วงอัปโหลดรูปและบันทึก | `"กำลังบันทึก…"` |
| **กำลังส่ง — ปุ่มยืนยันตีกลับ** | ปุ่มแดงเปลี่ยนข้อความ กดไม่ได้ และจางลงประมาณครึ่งหนึ่ง | `"กำลังบันทึก…"` |
| **ปุ่มยังกดไม่ได้ — POD** | ปุ่ม `"ยืนยันส่งสำเร็จ"` จาง กดไม่ได้ ตราบใดที่ยังไม่ถ่ายรูป | `"ยืนยันส่งสำเร็จ"` |
| **ปุ่มยังกดไม่ได้ — ตีกลับ** | ปุ่ม `"ยืนยันตีกลับ"` จาง กดไม่ได้ ตราบใดที่ยังไม่มีเหตุผล | `"ยืนยันตีกลับ"` |
| **ไม่มีสถานะกำลังส่ง (ช่องว่างปัจจุบัน)** | ปุ่ม `"รับรอบนี้"`, `"เริ่มส่งจุดที่ {N}"`, `"ดูสลิป"` และกล่องยืนยันรับรอบ — **ไม่เปลี่ยนหน้าตาเลยระหว่างรอเครือข่าย กดรัวได้** | — |
| **ไม่มีสิทธิ์เข้า** | ผู้ใช้ที่ไม่ใช่ `rider` หรือ `admin` **ไม่เคยเห็นหน้านี้** ถูกกันตั้งแต่ระดับเส้นทาง — ไม่ต้องออกแบบหน้าจอนี้ | — |

## 5. ข้อความไทยทั้งหมดบนหน้านี้

> **ห้ามคิดข้อความไทยขึ้นเองแม้แต่คำเดียว** — ทุกข้อความที่ผู้ใช้เห็นอยู่ในตารางนี้ ถ้าดีไซน์ต้องการข้อความใหม่ที่ไม่มีในตาราง ต้องถามเจ้าของก่อน

### หัวหน้าและแท็บ
| ที่ใช้ | ข้อความจริง (verbatim) |
|---|---|
| หัวข้อหน้า | `งานไรเดอร์` |
| แท็บที่ 1 | `รอบพร้อมรับ` |
| แท็บที่ 2 | `งานของฉัน` |
| แท็บที่ 3 | `รายได้` |

| ที่ใช้ | ข้อความจริง (verbatim) |

| เวลารอบ (ตัวใหญ่) | `รอบ {HH}:00 น.` เช่น `รอบ 14:00 น.` |

## 7. ข้อมูลอัปเดตยังไง

**ทั้งหมดเป็นการดึงข้อมูลซ้ำตามรอบเวลา (polling) — ไม่มี realtime แม้แต่จุดเดียวในหน้านี้**

| ข้อมูล | ความถี่ | หมายเหตุ |
|---|---|---|
| รายการรอบว่าง (แท็บ 1) | **ทุก 15 วินาที** | ดึงตลอดแม้ผู้ใช้อยู่แท็บอื่น — badge แท็บจึงเป็นตัวเลขสด |
| รอบของฉัน (แท็บ 2) | **ทุก 15 วินาที** | |
| จุดส่งในแต่ละรอบ | **ทุก 15 วินาที ต่อการ์ดรอบหนึ่งใบ** | รับ 3 รอบ = ยิง 3 ชุดพร้อมกันทุก 15 วินาที |
| รายได้สัปดาห์นี้ (แท็บ 3) | **ทุก 30 วินาที** | |
| ประวัติการจ่ายเงิน (แท็บ 3) | **โหลดครั้งเดียว ไม่มีรอบซ้ำ** | อัปเดตเมื่อกลับเข้าหน้าใหม่หรือหลังทำ action เท่านั้น |

**นอกจากรอบเวลาแล้ว ข้อมูลถูกดึงใหม่ทันทีหลังทำ action:**
- รับรอบสำเร็จ → **ดึงใหม่ทุกอย่างทั้งหน้า**
- ปิดจุดด้วย POD / ตีกลับ → ดึงใหม่: จุดส่งในรอบ + รอบของฉัน + รายได้
- เริ่มส่ง → ดึงใหม่: จุดส่งในรอบ + รอบของฉัน (**ไม่ดึงรายได้ใหม่**)

**ผลต่อดีไซน์ — สำคัญมาก**
- **หน้าจอขยับเองได้ตลอดเวลาโดยผู้ใช้ไม่ได้ทำอะไร** โดยเฉพาะแท็บ `"รอบพร้อมรับ"` ที่การ์ดอาจ **หายไปกลางคัน** เพราะไรเดอร์คนอื่นแย่งรับไป — ดีไซน์ต้องรองรับ ไม่ใช่สมมติว่ารายการนิ่ง (ควรพิจารณา: ไม่วางปุ่มสำคัญไว้ตำแหน่งที่การ์ดหายแล้วเลื่อนขึ้นมาโดนนิ้วพอดี)
- **ไม่มีตัวบอกว่าอัปเดตล่าสุดเมื่อไหร่** ในดีไซน์ปัจจุบัน และ **ไม่มี pull-to-refresh ไม่มีปุ่มรีเฟรชด้วยมือ** — ปุ่ม `"ลองใหม่"` มีเฉพาะตอนโหลดพลาดเท่านั้น
  - **ควรพิจารณาเพิ่ม**: ตัวบอกเวลาอัปเดตล่าสุดหรือสัญญาณ "กำลังซิงก์" จะมีค่ามากกับไรเดอร์ที่เน็ตหลุด ๆ ติด ๆ เพราะตอนนี้ **หน้าจอที่ค้างเพราะเน็ตหลุดกับหน้าจอที่สดจริง หน้าตาเหมือนกันเป๊ะ** — นี่คือช่องว่างที่สำคัญที่สุดข้อหนึ่งของหน้านี้
- การดึงข้อมูลทุก 15 วินาทีกินแบตและเน็ตของไรเดอร์ตลอดกะ — อย่าเพิ่มภาระด้วยการโหลดรูปหนัก ๆ โดยไม่จำเป็น

## 8. สิ่งที่ designer ต้องรู้

### 🔒 ห้ามแตะ — ผูกกับกฎธุรกิจโดยตรง

- **รูป POD บังคับเสมอ ไม่มีทางลัด** — ปิดจุดโดยไม่ถ่ายรูปไม่ได้ ห้ามเพิ่มปุ่ม "ข้าม" "ไม่มีรูป" หรือทางเลี่ยงใด ๆ ทั้งสิ้น
- **เหตุผลตีกลับบังคับเสมอ** — ห้ามทำให้ข้ามได้ และห้ามให้ชิปเหตุผลส่งได้ทันทีโดยไม่ผ่านช่องกรอก (ชิปเป็นแค่ตัวช่วยเติมข้อความ)
- **ทำทีละจุดเท่านั้น** — ปุ่มเริ่มส่งโผล่จุดเดียวในทั้งรอบ ไรเดอร์ **ข้ามจุด สลับลำดับ หรือเปิดหลายจุดพร้อมกันไม่ได้** ห้ามออกแบบ UI ที่ให้เลือกจุดเองหรือทำขนานกัน
- **ลำดับจุดถูกกำหนดตอนกดรับรอบ และเรียงตามเวลาที่ลูกค้าสั่ง ไม่ใช่ตามระยะทาง** — ห้ามใส่ affordance ที่สื่อว่าจัดลำดับใหม่ได้ (ลากสลับ ฯลฯ)
- **ทุกข้อความไทยในหมวด 5 ต้องใช้ตามตัวอักษรเป๊ะ ๆ** — ห้ามคิดคำใหม่ ห้ามเกลาสำนวน
- **copy ที่บอกผลกระทบภายนอกห้ามตัดทิ้ง** — ทุก action มีผลที่ผู้ใช้มองไม่เห็นและกู้คืนไม่ได้:
  - รับรอบ → ลูกค้าทุกคนในรอบได้ LINE **ทันที**
  - เริ่มส่ง → ลูกค้ารายนั้นได้ LINE
  - POD → **ลูกค้าได้รูปที่เพิ่งถ่าย**
  - ตีกลับ → ลูกค้าได้เหตุผลนี้ตรง ๆ + ถูกเรียกเก็บค่าส่งใหม่ + สต็อกถูกคืน + แอดมินได้แจ้งเตือน
  - วลีอย่าง `"ลูกค้าทุกคนในรอบจะได้รับแจ้งเตือนทันที"` และ `"ลูกค้าจะได้รับรูปนี้พร้อมแจ้งเตือน"` คือสิ่งเดียวที่บอกไรเดอร์ว่ากำลังจะเกิดอะไร — **ห้ามตัดออกเพื่อความสวยงาม**
- **ลำดับความสำคัญของปุ่ม POD เหนือปุ่มตีกลับ ต้องคงไว้** — POD = ปุ่มทึบสีแบรนด์ / ตีกลับ = ปุ่มโปร่งตัวอักษรแดง แต่ **ปุ่มยืนยันในโมดัลตีกลับเป็นแดงทึบเต็ม** (จุดที่ตัดสินใจจริงต้องหนักแน่น)
- **ปุ่มหลักห้ามเล็กลง** — `"รับรอบนี้"` สูงอย่างน้อย 56px, `"เริ่มส่งจุดที่ {N}"` และปุ่ม POD สูงอย่างน้อย 52px ทั้งหมดเต็มความกว้าง ออกแบบมาเพื่อนิ้วโป้งกลางแดดขณะถือของ
- **ไม่มีปุ่ม "จบรอบ" และห้ามเพิ่ม** — ระบบปิดรอบและจ่ายค่ารอบให้เองเมื่อไม่เหลือจุดค้าง การ์ดรอบจะหายจากแท็บ `"งานของฉัน"` เอง และยอดในแท็บ `"รายได้"` ขยับเอง

### ⚠️ ต้องถามเจ้าของก่อนทำ

- **เพิ่มปุ่มนำทาง / แผนที่ / GPS** — ที่อยู่เป็นข้อความล้วน **ไม่มีข้อมูลพิกัดในระบบเลย** ปุ่ม "เปิด Google Maps" จึงทำไม่ได้ตรง ๆ ด้วยข้อมูลที่มี ถ้าอยากได้ต้องคุยเรื่องข้อมูลก่อน
- **เพิ่มตัวจับเวลา 5 นาที หรือบังคับกดโทรก่อนตีกลับ** — กฎ "โทรแล้วไม่รับ + รอครบ 5 นาที" มีอยู่จริงในกติกาธุรกิจ แต่ **ระบบไม่ได้บังคับตรวจ** เป็นแค่ข้อความให้อ่าน การเพิ่มการบังคับคือการเปลี่ยนกฎ ต้องได้ไฟเขียวก่อน
- **แสดงยอดเงินของออเดอร์ให้ไรเดอร์เห็น** — ข้อมูลถูกดึงมาแล้ว (ค่าสินค้า ค่าส่ง ยอดรวม สถานะการชำระ ประเภทการส่ง) แต่ **จงใจหรือบังเอิญไม่แสดง** ก็ไม่ทราบ การเอามาโชว์คือการตัดสินใจเชิงธุรกิจ (เช่น ไรเดอร์ต้องเก็บเงินปลายทางไหม) — ถามก่อน

### 🔧 ปรับได้ — จุดที่ดีไซน์ปัจจุบันอ่อนจริง

- **ปุ่มสำคัญ 3 ปุ่มไม่มีสถานะกำลังส่ง**: `"รับรอบนี้"`, `"เริ่มส่งจุดที่ {N}"`, `"ดูสลิป"` — กดแล้วหน้าจอนิ่งสนิทจนกว่าจะสำเร็จหรือพลาด บนเน็ตมือถือช้านี่คือคำเชิญให้กดรัว โดยเฉพาะ `"รับรอบนี้"` ที่ผู้ใช้กำลังแข่งกับคนอื่นและมีแรงจูงใจสูงที่จะกดซ้ำ ๆ **ควรเพิ่มสถานะกำลังส่ง/ปิดการใช้งานชั่วคราว** (โมดัล POD และตีกลับทำถูกแล้ว ใช้เป็นแบบอ้างอิงได้)
- **กล่องยืนยันรับรอบปิดทันทีที่กด แล้วค่อยยิงคำขอ** — ผู้ใช้เห็นกล่องหาย แต่ไม่รู้ว่าได้รอบหรือยัง ต้องรอ toast **เป็นช่วงเวลาว่างที่น่าอึดอัดในจังหวะที่ผู้ใช้ต้องการคำตอบมากที่สุด**
- **ไม่มีตัวบอกความสด / ไม่มี pull-to-refresh** — ดูรายละเอียดในหมวด 7 นี่คือช่องว่างใหญ่สำหรับผู้ใช้ที่เน็ตไม่เสถียร
- **รับหลายรอบ = หน้ายาวมหาศาล** — การ์ดรอบซ้อนการ์ดจุดส่ง และ **ไม่มีการพับ/ยุบเลย** ทุกจุดของทุกรอบกางออกหมด จุดที่ส่งจบแล้วก็ยังกินที่เต็ม ๆ (แค่จางลง) ไรเดอร์ต้องเลื่อนผ่านจุดที่จบแล้วเพื่อไปหาจุดถัดไป **ควรพิจารณาการพับกลุ่มจุดที่จบแล้ว หรือการยึดจุดปัจจุบันให้เห็นเสมอ**
- **จำนวนจุดที่เห็นตอนกดรับ อาจไม่ตรงกับที่ได้จริง** — ออเดอร์ที่ยังแพ็คไม่เสร็จถูกเลื่อนไปรอบถัดไปเงียบ ๆ ไรเดอร์กดรับ `"8 จุดส่ง"` แล้วอาจเจอแค่ 6 จุด **UI ปัจจุบันไม่อธิบายเลย** — เป็นช่องว่างด้าน copy ที่ควรแก้ (ข้อความใหม่ต้องขออนุมัติ)
- **ชิปเหตุผลตีกลับหลุดสถานะเลือกทันทีที่พิมพ์แก้** — ไรเดอร์เลือกชิป `"โทรแล้วไม่รับสาย + รอครบ 5 นาที"` แล้วพิมพ์เติม `" โทร 2 ครั้ง"` → ชิปกลับเป็นสีขาวเหมือนไม่เคยเลือก ดูเหมือนระบบพัง ทั้งที่ข้อความยังอยู่ครบ **ควรทบทวนรูปแบบปฏิสัมพันธ์นี้**
- **ช่องเหตุผลจำกัด 300 ตัวอักษร แต่ไม่มีตัวนับ** — ผู้ใช้พิมพ์แล้วตัวอักษรหยุดขึ้นเฉย ๆ โดยไม่มีคำอธิบาย
- **รูปจากกล้องไม่ถูกย่อก่อนอัปโหลด** — รูปจากมือถือรุ่นใหม่อาจหลายเมกะไบต์ ช่วง `"กำลังบันทึก…"` จึงอาจนานผิดคาดบนเน็ตมือถือ **ควรพิจารณาแสดงความคืบหน้าการอัปโหลด แทนที่จะเป็นปุ่มนิ่ง ๆ**
- **ลิงก์รูปสลิปหมดอายุใน 1 ชั่วโมงโดยไม่มีการต่ออายุ** — เปิดโมดัลค้างไว้นานเกินไปแล้วรูปจะพังเงียบ ๆ ไม่มีข้อความอธิบาย
- **ไม่มีโหมดมืด (dark mode) เลยในระบบปัจจุบัน** — ไม่มีร่องรอยของการรองรับใด ๆ ทั้งที่ผู้ใช้ทำงานกลางแดดจ้าและกลางคืน **เป็นข้อควรพิจารณาเชิงกลยุทธ์** (คอนทราสต์กลางแดดสำคัญกว่าโหมดมืดด้วยซ้ำ — ควรตรวจสอบว่าสีเทาอ่อนทั้งหลาย เช่น คำอธิบายใต้ empty state และ helper text ยังอ่านออกไหมกลางแดด)
- **ป้ายจางในกล่องจุดที่จบแล้ว (จาง 70%) ทับซ้อนกับสีป้ายที่จางอยู่แล้ว** — ป้าย `"ส่งสำเร็จ"` และ `"รอค่าส่งใหม่"` ในกล่องที่จางลง อาจอ่านยากกลางแดด ควรตรวจคอนทราสต์
- **ที่อยู่ยาว ๆ ไหลไม่มีขอบเขต** — ไม่มีการตัดบรรทัดหรือย่อ กล่องจุดส่งจึงสูงไม่เท่ากันมาก และที่อยู่คือข้อมูลสำคัญที่สุดที่ไรเดอร์ต้องอ่านเร็ว **ควรพิจารณาลำดับชั้นการอ่านใหม่** (ตอนนี้ที่อยู่เป็นตัวเล็กสีเทา เล็กกว่าชื่อผู้รับด้วยซ้ำ — น่าสงสัยว่าถูกต้องหรือไม่สำหรับงานส่งของ)
- **ปุ่มโทรแสดงในจุดที่จบแล้วด้วย** — เป็นปุ่มเต็มความกว้างในกล่องที่ไม่ควรต้องทำอะไรแล้ว กินพื้นที่โดยไม่จำเป็น
- **พื้นที่ขวาของหัวข้อหน้าว่างอยู่** — โครงสร้างรองรับปุ่ม action ตรงนั้นได้ ถ้าดีไซน์ต้องการ (เช่น ปุ่มรีเฟรช)

### 📐 ข้อจำกัดเชิงเทคนิคที่ต้องรู้

- **โมดัลทุกตัวใช้รูปแบบเดียวกัน**: บนมือถือเลื่อนขึ้นจากขอบล่างจอ มุมบนมน เว้นพื้นที่ปลอดภัยขอบล่าง / บนจอใหญ่กล่องกลางจอ / กว้างสูงสุด ~448px / สูงสุด 90% ของจอ เลื่อนในตัวได้ / ปิดด้วยการแตะพื้นหลังมืดหรือปุ่ม `✕` — **การเปลี่ยนรูปแบบนี้กระทบทุกหน้าในระบบ ไม่ใช่แค่หน้านี้**
- **Toast**: ลอยเหนือแถบแท็บล่างบนมือถือ / มุมล่างบนจอใหญ่ / กว้างไม่เกิน ~384px / หายเองใน 3.6 วินาที / **ไม่มีปุ่มปิด** / สีเขียว = สำเร็จ, แดง = ผิดพลาด, เทาเข้ม = ข้อมูล ทั้งหมดพื้นทึบตัวอักษรขาว — **ใช้ร่วมทั้งระบบ**
- **แถบแท็บล่างของกรอบนอกกินพื้นที่ขอบล่างจอบนมือถือ** — พื้นที่เนื้อหาถูกเว้นด้านล่างไว้แล้ว (บนจอใหญ่เว้นน้อยกว่ามาก)
- **กล้องเปิดกล้องหลังโดยตรงเสมอ** ไม่ผ่านหน้าเลือกรูปจากแกลเลอรี — เป็นพฤติกรรมที่ตั้งใจ (หลักฐานต้องถ่าย ณ จุดส่ง ไม่ใช่หยิบรูปเก่ามาใส่)
- **ไม่มีรูปสินค้าหรือรายการสินค้าในจุดส่ง** — ไรเดอร์เห็นแค่เลขออเดอร์ ชื่อ เบอร์ ที่อยู่ สถานะ เท่านั้น
