> สิ่งที่ต้องการคือ **ดีไซน์ที่ดีขึ้น ไม่ใช่ระบบใหม่**

## 2. ธุรกิจนี้คืออะไร (60 วินาที)

**PKM Shop คือร้านค้าที่ทั้งร้านอยู่ในห้องแชท LINE**

- **ลูกค้าไม่มีแอป ไม่มีเว็บ ไม่มีหน้าเช็คเอาต์** — ทักแชท LINE → AI ขายของให้ → เลือกสินค้า → แจ้งที่อยู่ → เลือกวิธีส่ง → ได้ QR พร้อมเพย์ → โอน → ส่งรูปสลิป → ระบบตรวจสลิปอัตโนมัติ → ของมาส่ง จบในห้องแชทเดียว
- **พนักงานมีเว็บหลังบ้าน** — เว็บที่เอกสารชุดนี้พูดถึง: รับของเข้า / แพ็ค / ขี่รถส่ง / ตรวจสลิป / คุมออเดอร์ / ปิดเงินเดือน
- **ของถูกส่งเป็น "รอบ" ทุกชั่วโมง 24 ชั่วโมง ไม่มีวันหยุด** ← นี่คือสิ่งที่ทำให้ธุรกิจนี้ไม่เหมือนร้านออนไลน์ทั่วไป

### 👉 กติกาข้อเดียวที่ต้องจำให้ได้

> **ทุกนาทีที่ :30 ของทุกชั่วโมง ระบบปิดรอบ แล้วงานทั้งรอบเด้งเข้าหาคนแพ็คพร้อมกัน — เขามีเวลา 30 นาทีก่อนไรเดอร์มารับ**
> เกิดขึ้น **24 ครั้งต่อวัน ทุกวัน รวมตี 3**

ทุกการตัดสินใจด้านดีไซน์ของหน้า Packer และ Rider ต้องวัดด้วยไม้บรรทัดอันนี้: *"คนที่ยืนอยู่หน้าโต๊ะแพ็คตอนตี 3 มีเวลา 30 นาที เขาอ่านหน้านี้ออกใน 2 วินาทีไหม"*

รายละเอียดเต็มอยู่ใน [`14-domain-vocabulary.md` §7](14-domain-vocabulary.md)

## 3. คนใช้งาน 5 กลุ่ม

| บทบาท | ใช้อะไร | อยู่ที่ไหนตอนใช้ | เห็นกี่เมนู |
|---|---|---|---|
| **แอดมิน / เจ้าของร้าน** | ทุกหน้า | นั่งโต๊ะ จอใหญ่ — **desktop-first** | 11 |
| **สต็อก** | สินค้า + รับของเข้า | ในคลัง ยืน/นั่ง มือถือหรือแท็บเล็ต | 2 |
| **แพ็ค** | คิวแพ็คของ | **ยืนที่โต๊ะแพ็ค มือเดียว อีกมือถือของ** — mobile-first | 2 |
| **ไรเดอร์** | รอบส่ง | **บนมอเตอร์ไซค์ กลางแดด มือเดียว** — mobile-first | 2 |
| **พนักงานทั่วไป** | เช็คอิน + แชททีม | มือถือ | 1 |

> คนหนึ่งถือได้หลายบทบาทพร้อมกัน — UI ต้องรองรับป้ายหลายอันต่อคน

## 5. กติกาที่ห้ามฝ่าฝืน

ระบบนี้จัดการ**เงินจริง**และ**ของจริง** ข้อพวกนี้ผูกกับกติกาธุรกิจและฐานข้อมูล ไม่ใช่รสนิยม:

1. **ปุ่มต้องใหญ่อย่างน้อย 44px** — Packer/Rider ใส่ถุงมือ ถือของ ยืนกลางแดด (Stepper 48px, แท็บล่าง 56px)
2. **ช่องกรอกต้องเป็น 16px** — เล็กกว่านี้ Safari ซูมเองแล้วเลย์เอาต์พัง
3. **ห้ามใช้ `alert()` / `confirm()` / `prompt()` ของเบราว์เซอร์** — นี่คือข้อร้องเรียนโดยตรงจากเจ้าของงาน ต้องใช้ Modal/Toast/Stepper ที่มีอยู่
4. **เงินเป็นจำนวนเต็มบาท ไม่มีสตางค์** — ห้ามออกแบบช่องรับทศนิยม แสดงผลเป็น `฿1,250` เสมอ
5. **ห้ามพิมพ์ตัวเลขค่าส่งตายตัวใน mockup** — 40/55/100/25 เป็นแค่ค่าเริ่มต้น แอดมินแก้ได้ ทุกที่ที่โชว์ราคาต้องเป็นช่องที่ดึงค่าจริง
6. **ข้อความไทยทุกคำในเอกสารนี้คือของจริง** — ห้ามคิดคำใหม่ใส่ mockup ถ้าคำไหนไม่ดี ให้เสนอเป็นข้อเสนอแยกที่ระบุชัดว่ากำลังเสนอเปลี่ยน
7. **รูปคือหลักฐาน ไม่ใช่ของประดับ** — ของเข้าคลัง / แพ็คเสร็จ / ส่งถึง / เช็คอิน / โอนเงินเดือน ทั้งหมด**บังคับ**แนบรูป ห้ามออกแบบให้ข้ามได้
8. **สถานะเปลี่ยนได้เฉพาะตามตารางใน `14-domain-vocabulary.md` §8** — ถ้าวางปุ่มที่ทำนอกตาราง ระบบจะปฏิเสธ
9. **ปุ่ม "ยืนยันโอนแล้ว" ไม่ได้สั่งจ่ายเงิน** — มันคือการบันทึกว่ามนุษย์โอนไปแล้ว ห้ามออกแบบให้ดูเหมือนปุ่มจ่ายเงิน

## 6. ปัญหาที่รู้อยู่แล้ว — นี่คืองานที่อยากได้จริง ๆ

ระบบทำงานถูกต้อง แต่ดีไซน์ยังอ่อนหลายจุด เรียงตามความคุ้มค่า:

| # | ปัญหา | อยู่ที่ไหน |
|---|---|---|
| 1 | **ยังไม่มี Rich menu ใน LINE เลย** ทั้งที่ข้อความในระบบอ้างถึงปุ่ม "ดูสินค้า" — พื้นที่ว่างที่ใหญ่ที่สุด | [LINE](13-line-customer-flow.md) |
| 2 | **แท็บล่าง 11 อันของแอดมินบนมือถือ** ต้องเลื่อนแนวนอน — ใช้จริงแย่มาก | [design-system §5](00-design-system.md) |
| 3 | **QR + การ์ดชำระเงินถูกส่งซ้ำทุกครั้งที่ลูกค้าพิมพ์** — คุย 5 ประโยค ได้ QR 5 ชุด | [LINE §2](13-line-customer-flow.md) |
| 4 | **ลูกค้าที่ขอคุยกับแอดมินถูกทิ้งให้เงียบ** — ไม่รู้ว่ารอนานแค่ไหน ไม่มีสัญญาณว่ามีคนเห็นแล้ว | [LINE §4](13-line-customer-flow.md) |
| 5 | **การ์ดสถานะออเดอร์ใน LINE อ่านอย่างเดียว ไม่มีปุ่มเลย** | [LINE §3](13-line-customer-flow.md) |
| 6 | **ป้ายสถานะไม่มีระบบสีกลาง** — แต่ละหน้าเขียนสีเอง | [design-system §1](00-design-system.md) |
| 7 | **`out_for_delivery` กับ `delivering` ดูเหมือนกัน** ทั้งที่คนละความหมายสำหรับลูกค้า | [domain §1](14-domain-vocabulary.md) |
| 8 | **วิธีจัดส่งมีชื่อไทย 2 ชุดขัดกัน** — การ์ดกับ modal เรียกคนละชื่อ | [domain §4](14-domain-vocabulary.md) |
| 9 | **ไม่มี dark mode** ทั้งที่ไรเดอร์ทำงานตี 3 | [design-system §1](00-design-system.md) |
| 10 | **ไม่มีโลโก้เป็นรูป ไม่มี favicon ไม่มี icon library** (ไอคอนเป็น emoji ล้วน) | [design-system §7](00-design-system.md) |
| 11 | **ไม่มี skeleton loading** — มีแค่ spinner กลางจอ | [design-system §6](00-design-system.md) |
| 12 | **การ์ดสินค้าใน LINE ไม่มีตัวเลือกจำนวน** — ซื้อ 2 ชิ้นต้องพิมพ์เอา | [LINE §2](13-line-customer-flow.md) |

## 7. คำถามที่ต้องให้เจ้าของงานเคาะก่อนเริ่ม

1. **`payment_status` ไม่มีป้ายไทยเลยทั้งระบบ** — คำที่เสนอไว้ใน [`14-domain-vocabulary.md` §2](14-domain-vocabulary.md) ต้องได้รับการยืนยัน
2. **ชื่อวิธีจัดส่ง 2 ชุด** — เลือกชุดเดียว หรือกำหนดให้ชัดว่าที่ไหนใช้ชุดไหน
3. **แอดมินต้องใช้บนมือถือจริงไหม** หรือ desktop-only ได้ (ตัดสินชะตาปัญหาข้อ 2 ในตารางบน)
4. **`drop_option` (วางไว้ / รอรับ)** มีในฐานข้อมูลและสเปค แต่ยังไม่ถูกใช้ในเว็บ — จะออกแบบเผื่อไหม

> กติกาของโปรเจกต์นี้เขียนไว้ชัด: **"ห้ามเดากติกาธุรกิจใหม่เอง ถ้าเจอกรณีที่สเปคไม่ครอบ ให้หยุดถามเจ้าของงาน"**

## 8. วิธีอ่านแต่ละบทของหน้าจอ

ทั้ง 12 บทใน [`screens/`](screens/) ใช้โครงเดียวกัน:

1. **บริบทการใช้งาน** — ใครใช้ ที่ไหน อุปกรณ์อะไร เร่งแค่ไหน
2. **โครงหน้า** — บล็อกจากบนลงล่าง
3. **ทุก flow ทีละสเต็ป** — มีหมายเลข + กฎ validation + ข้อความสำเร็จ/ล้มเหลว (คำจริง)
4. **สถานะของหน้า** — ปกติ / โหลด / ว่าง / ผิดพลาด / กำลังส่ง
5. **ข้อความไทยทั้งหมด** — คัดลอกไปใช้ได้เลย
6. **ข้อมูลที่แสดง** — รูปแบบการแสดงผล
7. **ข้อมูลอัปเดตยังไง** — เรียลไทม์ / poll กี่วินาที
8. **สิ่งที่ designer ต้องรู้** — ระบุชัดว่าอะไร **[ห้ามแตะ]** อะไร **[ปรับได้]**
